## Q7: TS interface 和 type 区别

**核心句**：两者 90% 场景可互换，差别在两点：合并 vs 联合

**展开**："它们 90% 场景能互换。两个关键差别——**第一，interface 支持声明合并**：同名 interface 自动合并字段，所以第三方库类型增强（比如给 `Window` 加属性）必须用 interface；type 重复定义会报错。**第二，type 能表达 interface 表达不了的**：联合类型（`A | B`）、元组、映射类型、条件类型、`Partial<T>`/`Pick<T,K>` 这些工具类型的底层都是 type。我的准则是：对象形状用 interface（更友好的错误提示+可合并），需要联合/工具类型用 type。"

来源：[TypeScript 官方 Declaration Merging](https://www.typescriptlang.org/docs/handbook/declaration-merging.html)、[Effective TypeScript](https://effectivetypescript.com/2021/06/03/interface/)。

## Q8: TS 泛型作用（🔴 硬伤题）

**核心句**：泛型是类型的"参数"，让一份代码复用多种类型同时不丢类型信息

**展开**：泛型本质是**类型层面的参数化**——就像函数参数化值一样，泛型参数化类型。三个典型场景：
1. **容器类型**：`Array<T>`、`Map<K,V>`、`Promise<T>`——你知道里面装的是什么，调 `.map()`/`.then()` 时类型不丢。
2. **工具函数**：`useState<T>(initial: T): [T, (v: T) => void]`——返回值的类型跟入参绑定，不写泛型就只能用 any。
3. **类型工具**：`Partial<T>`、`Pick<T,K>`、`Readonly<T>`——这些是泛型+映射类型，T 是入参，输出新的类型。

关键是**类型推导**——大多数时候不用显式写 `<T>`，TS 会从入参推导，但泛型让这种推导成为可能。

来源：[TypeScript Generics Best Practices](https://medium.com/@robinviktorsson/a-comprehensive-guide-to-typescript-generics-ed01f1f83ac6)、[Reddit r/typescript](https://www.reddit.com/r/typescript/comments/85aupt/what_are_generics_used_for_in_a_real_world/)。

**📌 失分教训**：场上挤了 30 秒只挤出 Partial，且 Partial 是**映射类型**不是泛型本身。下次面试前必须把"Array/Map/Promise/useState"四类例子刻进肌肉记忆。

## Q11: LRU 缓存（手写）

**核心句**：用 Map 的插入顺序特性，O(1) get/put

**代码**：
```js
class LRUCache {
  constructor(capacity) { this.capacity = capacity; this.cache = new Map(); }
  get(key) {
    if (!this.cache.has(key)) return -1;
    const val = this.cache.get(key);
    this.cache.delete(key);
    this.cache.set(key, val);  // 移到末尾（最新）
    return val;
  }
  put(key, val) {
    if (this.cache.has(key)) this.cache.delete(key);
    else if (this.cache.size >= this.capacity) {
      const oldestKey = this.cache.keys().next().value;  // 取最老的
      this.cache.delete(oldestKey);
    }
    this.cache.set(key, val);
  }
}
```

## Q12: get 里为什么要 delete 再 set？

**核心句**：Map 按插入顺序遍历，delete+set 等于把这条挪到末尾（最新位置）

**展开**："ES2015 规范规定 Map 迭代顺序 = 插入顺序。delete 之后 set 会把这个 key 当成新插入的，挪到末尾。所以 get 操作不只是查值，还要更新访问时间——把刚访问的标记为最新。"

---

## Q13: Map.keys().next() 为什么能取到最老的？

**核心句**：Map 按规范维护插入顺序，delete+set 把 key 挪末尾，所以 keys().next() 永远是最老的

**展开**："能这么写的根本原因是 ES2015 规范明确规定 **Map 的迭代顺序 = 插入顺序**，并且 delete 再 set 会把这个 key 当成新插入的，挪到末尾。所以 `keys().next()` 永远返回最早插入的那个——这是 LRU 用 Map 能简化到 30 行的关键。

如果不用 Map，**经典实现是 HashMap + 双向链表**：HashMap 做 O(1) 查找，双向链表维护访问顺序，每次 get/put 把节点挪到头部，淘汰时摘尾部。两者复杂度都是 O(1)，但代码量是 Map 方案的 3 倍。

Map 方案的代价是空间效率略低（Map 内部就是类似 HashMap+链表的实现），但浏览器/V8 已经做过优化，日常前端场景完全够用。如果做高频 LRU（比如 React 内部 fiber 缓存），可能要上 typed array 池化节点。"

来源：[MDN Map](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Map)、[LRU 实现 Stack Overflow](https://stackoverflow.com/questions/54730706/why-use-a-doubly-linked-list-and-hashmap-for-an-lru-cache-instead-of-a-deque)。

## Q14: SDD 体系是什么？

**核心句**：用 skill 工作流把 PRD 蒸馏成 spec，靠 SSOT + 证据链 + 多层门控保质量

**展开**：90 秒讲完，**三句话三层**：
- **是什么**：一套 skill 工作流，PRD 进，结构化 spec + plan 出，团队 20+ 人推广半年。
- **三大支柱**：SSOT——6 个 Reference 文件做唯一事实来源；证据链——每条结论必须带 evidenceId 溯源到源码；多层门控——摄入门控/章节门控/readiness 评分/确定性脚本兜底。
- **闭环**：spec 产出后会生成 drift-report 对比 reference，增量更新机制让人工确认后回流。

然后等面试官追问细节，不要一开始全抛。

## Q15: 反馈回流闭环怎么做？

**核心句**：两条回流路径：spec 产出后的 drift-report 对比 + 日常 commit 的增量更新

**展开**："两条路径——**主动回流**：每次跑完 PRD 工作流，对比当前 reference，输出 drift-report 列出'需要更新的领域文件 + diff 预览'，人工确认后写入。**被动回流**：日常 git commit 时，reference 的增量更新模式自动对比 gitlog 和当前 reference，提示是否同步。两种都需要人工确认，不让 AI 自动写入——因为 reference 是 SSOT，错了全链路塌。"

## Q16: SDD 跟 OpenSpec 区别？

**核心句**：OpenSpec 解决"怎么执行 spec"，我解决"PRD 怎么蒸馏成可执行的 spec"

**展开**："OpenSpec 和 GitHub Spec Kit 是**执行层**——你给它一份详细 spec，它帮 Copilot/Claude Code 跑 vibe coding 流水线。但它们有个**共同缺口：从 PRD 到 spec 的蒸馏没人做**。

我的工作流就填这个缺口——核心是 SSOT + 证据链 + 多层门控。SSOT 让 AI 有前置业务知识（OpenSpec 没有这层），证据链强制每条结论带源码出处防幻觉（OpenSpec 是黑盒产出），门控让产出可验证（OpenSpec 靠人工 review）。

一句话：**OpenSpec 做代码生成，我做 PRD 到 spec 的可信蒸馏。** 我们的 spec 产出完全可以喂给 OpenSpec 跑后续步骤——是上下游关系不是替代关系。"

来源：[I Tested Three Spec-Driven AI Tools](https://ranthebuilder.cloud/blog/i-tested-three-spec-driven-ai-tools-here-s-my-honest-take/)、[GitHub Discussion #1536](https://github.com/github/spec-kit/discussions/1536)、[Microsoft Developer Blog - Spec Kit](https://developer.microsoft.com/blog/spec-driven-development-spec-kit)。

**📌 二次增强（融入面试官洞察）**：网易大数据团队**已经在用 spec-driven 开发方式**（OpenSpec/Spec Kit），且明确把"工程化"和"产品"分开看。如果二面碰到这个团队的人，**不要单方面推销自己的方案**，而是同方向交流："你们用 OpenSpec 跑下来感觉最大的痛点是什么？"——他们大概率会说"PRD 到 spec 那段没人做"，正好是你的差异化。

## Q17: Spec 蒸馏具体是什么？

**核心句**：Spec 三段 + Plan 三段 + 三大支柱（SSOT/证据链/门控）

**展开**：分两层讲——
- **架构层**：三大支柱——SSOT（6 个 Reference 文件做唯一事实来源）、证据链（每条结论带 evidenceId 溯源到源码 file:line）、多层门控（摄入/章节/readiness/确定性脚本）。
- **产物层**：Spec 三段（需求理解 + 契约对齐 + 风险检查），Plan 三段（架构概览 + 开发顺序 + QA 矩阵）。Spec 是"AI 能直接理解的需求文档"，Plan 是"开发能直接执行的任务清单"。

**关键差异化**：风险检查段——AI 不确定的点强制生成表格标红需人工确认，不让 AI 编造。这点是团队推广时大家最认可的——AI 不再"自信地胡说"。

## Q18: Monorepo 为什么 Lerna + Nx Workspace？

**核心句**：历史项目选 Lerna+Nx，新项目我会选 pnpm workspace + Turborepo

**展开**："这个项目是 2022 年搭的，当时 Lerna 是主流选型——它的独立发版能力强，我们的 benefit 组件库就是要独立发包给 8 个业务用，所以选了。Nx 主要是要它的增量构建和任务缓存，因为 monorepo 全量构建太慢。

但**如果今天搭新项目，我会选 pnpm workspace + Turborepo**——pnpm 的 hard link 装包快、磁盘省，Turborepo 配置简单、缓存机制更现代。Lerna 进入维护模式后被 Nx 接管了，新项目没必要再上 Lerna。"
