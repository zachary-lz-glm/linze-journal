## Q1: 你对微前端架构底层代码、底层架构了解多少？乾坤是直接用还是包了一层？怎么做灰度发布？

**分析**：
- 表面问的：乾坤底层架构 + 灰度发布机制
- 实际考察的：是否真懂底层架构，还是只会用 SDK；对工程化关键场景（灰度）有没有思考
- 答到的要点：乾坤路由分发机制、基座监听 URL、动态加载子应用
- 漏掉的关键点：
  - 直接用乾坤 SDK 意味着业务方接触原生 API，缺少团队抽象层（标准答案应该在乾坤之上包一层）
  - 灰度发布根本答不上来——这是高频生产场景
  - 没主动展示对乾坤底层（JS 沙箱、CSS 隔离、prefetch）的理解

| 维度 | 分数 | 问题 |
|------|------|------|
| 知识完整度 | 5/10 | 基础架构答对，灰度 0 分 |
| 表达清晰度 | 6/10 | "具体不在我这边"重复 2 次，回避感强 |
| 追问防御力 | 3/10 | 灰度追问直接放弃 |

**改进版答案**：
- **核心句**：乾坤是基座路由 + JS 沙箱 + CSS 隔离的微前端框架，但直接裸用 SDK 缺少团队抽象层
- **展开**：乾坤底层四件事——基座监听 URL 变化、动态加载子应用资源、JS 沙箱隔离全局变量、CSS 隔离防样式污染。灰度我们目前主要靠阿波罗配置控制——基座读 feature flag 决定加载新老版本子应用的入口。**比较成熟的做法是在乾坤之上再包一层团队 SDK**，把灰度、监控、错误边界、子应用注册统一收口，业务方不直接接触乾坤 API。我们中台这块确实没做完整，是后续要补的。

## Q2: 400+ 组件库是一个个手动封装的吗？动态组装是不是服务端的事，前端只提供渲染引擎？

**分析**：
- 表面问的：组件库是手动封装吗 + 前端职责边界
- 实际考察的：能否区分"基础组件"和"业务规格"；对"动态渲染引擎"职责的理解
- 答到的要点：3-4十个基础组件 → 400+ 规格；JSON Schema 协议；阿波罗下发；前端渲染引擎 + 联动解析
- 漏掉的关键点：可以主动讲"规格参数空间爆炸 → 用基础组件 + 参数化解决"的本质

| 维度 | 分数 | 问题 |
|------|------|------|
| 知识完整度 | 8/10 | 本质讲清了 |
| 表达清晰度 | 7/10 | 略散 |
| 追问防御力 | 8/10 | 主动引出联动 |

**改进版答案**：
- **核心句**：400+ 不是手写——是 3-4十个基础组件 × N 维参数（国家/业务线/城市）组合出来的规格空间
- **展开**：基础组件是技术维度（按 UI 类型分），规格是业务维度（按权益类型分）。前端职责就两块：①**渲染引擎**接收 Schema 配置渲染 UI ②**联动解析器**处理组件间依赖。配置数据由后端阿波罗下发，前后端通过 JSON Schema 协议对齐——后端控制下发什么，前端控制怎么画。本质是用"少量基础组件 + 参数化"解决"规格爆炸"问题。

## Q3: 两个独立业务组件之间的联动怎么做？A 发事件，B 怎么监听？特殊属性如何处理？

**分析**：
- 表面问的：组件间联动机制
- 实际考察的：能否把联动抽象成数据结构（依赖图），而不是硬编码
- 答到的要点：有向图依赖、拓扑排序、循环检测、Schema 声明、发布订阅、模板字符串/三目、visible/disable/数据
- 漏掉的关键点：可以主动把联动分 3 类讲清楚（属性/数据/接口）

| 维度 | 分数 | 问题 |
|------|------|------|
| 知识完整度 | 8/10 | 底层抽象讲清了 |
| 表达清晰度 | 7/10 | 偏长 |
| 追问防御力 | 8/10 | 主动分类 |

**改进版答案**：
- **核心句**：联动本质是字段级有向图依赖——A 影响 B 是 Schema 里声明的依赖关系，引擎层做拓扑排序 + 循环检测
- **展开**：联动在 Schema 里用自定义 directive 语法声明。运行时三类联动：①**属性联动**（visible/disable，模板字符串求值）②**数据联动**（依赖项变化触发重新请求接口）③**组件间状态联动**（被依赖项组件里写死转换逻辑）。引擎层在页面初始化时注册所有订阅，依赖项值变化触发重新求值，结果覆盖到对应组件的 props。这种设计的好处是组件本身不需要知道依赖谁，只暴露标准 props——解耦。

## Q4 ⚠️: 你们里面应该多内置引擎规范，靠人为勾兑很难保障——你怎么看？（失分点）

**分析**：
- 表面问的：组件解耦的设计哲学
- 实际考察的：是否理解"通用属性/事件抽象层"——Event Bus / Pub-Sub 解耦模式
- 答到的要点：反问场景（被动）；监听机制在组件外部
- 漏掉的关键点：
  - 面试官核心点：**组件 A 不应该让组件 B 去监听某个具体值**——应该是事件总线 + 通用属性抽象
  - 应该立刻接住"通用属性抽象"——visible/disabled/loading/value 是组件契约的通用属性
  - 应该提到 Event Bus / Pub-Sub 解耦模式（Thoughtworks、AWS 微前端架构都推荐）

| 维度 | 分数 | 问题 |
|------|------|------|
| 知识完整度 | 4/10 | 没接住核心点 |
| 表达清晰度 | 5/10 | 反问回去显得被动 |
| 追问防御力 | 3/10 | 没主动接设计哲学 |

**改进版答案**（搜索增强）：
- **核心句**：通用属性/事件抽象——组件之间只通过标准契约（visible/disabled/value）联动，业务属性由组件内部消化
- **展开**：您说的对，引擎层应该做的是**通用属性抽象**——所有基础组件都遵守统一契约（visible/disable/loading/value 等标准属性），引擎只调度这些通用属性，组件之间不应该让对方监听具体业务字段。具体业务字段由组件内部处理并暴露成标准事件输出。这是微前端 Event Bus / Pub-Sub 解耦模式的标准做法（Thoughtworks、AWS 微前端架构都推荐这套）。**我们目前引擎层做了通用属性的发布订阅，但业务字段的契约规范还不够强**——靠文档约束，容易被新组件破坏。这块我们正在补一份组件契约规范，把 visible/disabled/value 等通用属性和业务事件输出格式固化下来，新组件必须通过 lint 校验。

## Q6: PRD 只有一句话——"后台用户列表页需要支持筛选分页批量导出"——怎么蒸馏成 AI 可识别的 spec？

**分析**：
- 表面问的：模糊需求→精准执行的核心矛盾
- 实际考察的：是否真正解决了模糊 PRD 的工程化方案
- 答到的要点：reference 提取 → 6 份领域知识 → 需求点匹配 → 源码验证 → 证据链 → 风险表 → 人工 check → plan
- 漏掉的关键点：没主动强调"风险点表"的核心价值——把模糊变确定的最后一公里

| 维度 | 分数 | 问题 |
|------|------|------|
| 知识完整度 | 8/10 | 流程讲清了 |
| 表达清晰度 | 8/10 | 顺畅 |
| 追问防御力 | 8/10 | 主动举例 |

**改进版答案**：
- **核心句**：模糊 PRD 靠"项目记忆 + 证据链 + 风险表"三层兜底——AI 不胡编，不确定就标风险
- **展开**：一句话 PRD 进来，先在前置 reference 里找类似场景（列表页筛选是常见需求，领域知识里有标准模板），然后源码验证每个推断。每条结论带 verified_by ID（源码 file:line / git log / 技术问答出处）。最后产出 spec + 风险表：spec 告诉你"我推断的是什么"，风险表告诉你"我不确定什么需要你确认"。比如筛选字段具体内容 AI 给推测样式，但明确标"未在代码中找到对应字段，建议与产品/后端对接"。人工 check 风险点后，再做一轮蒸馏产生 plan。**核心是不胡编——不确定的就标记，让人决策**。

## Q8: 8 种证据类型怎么用？前端 AI 幻觉接口，结合负向搜索，不依赖 AI 怎么快速验证接口在项目中没定义过？

- 漏掉的关键点：
  - 应该举具体工具（ast-grep / ripgrep / OpenAPI spec 比对）

| 维度 | 分数 | 问题 |

| 表达清晰度 | 7/10 | 顺畅 |

## Q9 ⚠️: AI 自作聪明引入非标准库（lodash 大版而非 lodash-es），CR 时你怎么处理？（被直接挑战）

**分析**：
- 表面问的：幻觉出现后的处理方案
- 实际考察的：是否承认问题存在 + 是否有完整的自校验/自修复方案
- 答到的要点：自校验 4 层（语法/字段对齐/规则/集成）+ 自修复循环（差异性修正/全量重演/震荡检测）
- 漏掉的关键点：
  - 应该先承认"会幻觉"——这是行业共识
  - 应该先讲"如何避免"（包白名单 + import lint 规则），再讲"出现后怎么修"

| 维度 | 分数 | 问题 |
|------|------|------|
| 知识完整度 | 6/10 | 修复机制有但防止层薄 |
| 表达清晰度 | 6/10 | "不会编造"被挑战 |
| 追问防御力 | 7/10 | 自校验机制兜住了 |

**改进版答案**（搜索增强）：
- **核心句**：先防后修——包白名单 + import lint 防止引入非标准库，4 层自校验 + 震荡检测兜底
- **展开**：先承认问题——**AI CR 准确率低、第三方库理解不到位是 2025 行业共识**（研究显示即使最好的工具也只能把幻觉降低 96% 不能完全消除）。所以**防止优先于修复**：①import 白名单 lint 规则（禁止 lodash，强制 lodash-es）②package.json 依赖审查 ③第三方库版本锁。**出现后自校验 4 层**：①语法（AST/TS/ESLint 编译）②字段双向对齐（spec vs 代码）③代码规则层（i18n/命名反模式）④集成层（导入导出路径/历史 diff）。**自修复循环**：差异性修正（不重写）→ 全量重演（A 改可能影响 B，跑整个依赖链）→ 震荡检测（A→B→A 来回横跳找局部最优）。**承认我一个人搞不完——这是工作流原型，规模化需要团队投入**。

- **展开**：①**文件匹配率**（构造数据：plan 涉及文件 vs 实际开发涉及文件，目前 80%）②**plan 稳定性**（同一需求跑多次，文件清单一致率）③**真实需求回测**（您的建议）——拿线上已完成的需求，跑 plan 看人工有没有改、改了多少行，git 都有记录。三个维度互补：构造数据可控但不真，稳定性反映确定性但不是准确性，真实回测最准但需要等需求跑完。**我现在想做的是把这三个维度做成一个 dashboard，让准确度可视化**。

- 实际考察的：能否跳出"PRD→开发计划"的窄定位，看到完整研发工作流的价值

- 漏掉的关键点：

  - 没有主动接住"开发计划 + 测试计划 + 上线计划三全"的扩展
  - 没讲清楚 PRD→Plan 只是入口，完整价值在后续闭环

**改进版答案**（搜索增强）：

## Q13: 最近出的 Loop 你怎么用？是不是噱头？

**分析**：
- 表面问的：Loop 工程的判断 + 不盲从
- 实际考察的：对 AI 前沿趋势的判断
- 答到的要点：用 Loop 整理个人知识库；特定场景有点帮助
- 漏掉的关键点：
  - **Loop Engineering 是 Anthropic 官方在推的范式**，不是噱头——Boris Cherny 原话 "my job is to write loops, not prompts"
  - 应该区分"概念是不是噱头" vs "落地是不是过度营销"
  - 跟上面试官的"噱头"判断有点摇摆，没有坚持自己的判断

| 维度 | 分数 | 问题 |
|------|------|------|
| 知识完整度 | 5/10 | 没掌握概念本质 |
| 表达清晰度 | 6/10 | 摇摆 |
| 追问防御力 | 5/10 | 没坚持判断 |

**改进版答案**（搜索增强）：
- **核心句**：Loop Engineering 不是噱头——是 Anthropic 官方在推的范式，但落地被过度营销了
- **展开**：概念本身不噱头——Boris Cherny（Claude Code 创始人）的原话是 **"my job is to write loops, not prompts"**，意思是工程师的工作从逐个 prompt 变成设计自治循环系统。我自己用 Loop 整理几百个知识卡片就体会到——让 AI 自己搜集整理校验，加式循环，比手动 prompt 高效很多。**噱头的是落地过度营销**——很多团队把 Loop 包装成银弹，但实际场景有限：①**批量同构任务**（卡片校验、文档归一化）适合 ②**需要深度业务判断的任务**（架构决策）不适合 ③**需要跨上下文的任务**（长链路分析）容易爆上下文。所以"对特定场景有用"是对的，但**概念本身是有价值的范式转变**，不能简单说噱头。

## 面试官洞察（快手电商 B 端商家团队）

> 面试官团队 2024 年开始做 AI 巡检，自己写微前端框架，对 AI 工程化有真实生产经验。以下是高价值洞察。

### 洞察 1：B 端 AI 落地的核心是「AI for 业务」而非「AI for 研发效率」

**面试官原话**：第一是业务支撑，第二是 AI for 业务。AI 怎么帮 B 端商家/运营提升人效。**AI for 研发效率基本不涉及。** 除了 AI for 研发效率不做，其他 AI 都会做。

**价值**：这个定位非常清晰——B 端业务团队（商家、运营、经营）的 AI 投入优先级是**帮业务方提效**（to 商家、to 运营），而不是帮研发提效。to AIS 是整个大专项，包含端智能、AI to UI、批量创活动、端模型、AI 开发上下文。**求职启示**：面试 B 端业务团队时，AI 工程化的故事要往"帮业务提效"靠，而不是只讲研发效能。

### 洞察 2：双 CR 机制——智能 CR 在前，人工 CR 在后

**面试官原话**：代码交付两条路（端到端 AI 流水线 + 纯编码），两条路出来的代码都走 智能 CR + 人工 CR（智能在前，人工在后）。**AI CR 准确率较低。**

**价值**：这是快手电商团队的真实生产实践。**行业共识**：2024-2025 研究 AI CR 工具降低幻觉只能到 96%，不能完全消除。快手电商的解法是"**AI 先粗筛 + 人工精审**"——AI 处理机械性问题（语法、规范、格式），人工处理业务性问题（功能正确性、交互完整性）。

### 洞察 3：B 端场景纯 CR 看不出问题——必须配合功能性巡检

**面试官原话**：B 端场景偏功能交互，**纯看代码看不出问题**。我们 2024 年开始做 AI 巡检，专门一套环境（物理机）跑代码稳定性/功能正确性校验。给足权限后，AI 自己调用页面功能。

**价值**：这是面试官团队的核心差异化实践——**AI 巡检物理机 + AI 自动调用页面**。B 端逻辑复杂、UI 交互密集，单测和 CR 都覆盖不到功能层。他们的解法是把 AI 当 QA 用——给 AI 浏览器权限，让它按 PRD 自动操作页面验证功能。**这是端到端工作流后半段（测试 plan）的工业化实践**。

### 洞察 4：金句——"我连 case 都不应该写"

**面试官原话**：PRD 分析完除了开发计划，应该给我蒸馏测试功能文档。**我连 case 都不应该写**。开发计划/测试计划/上线计划三全给我上来。

**价值**：这是面试官对端到端工作流的明确价值定位——**PRD 输入应该产出三套 plan**（开发 plan + 测试 plan + 上线 plan），代码提交跑测试 plan 的自动化 case，测完给准出报告，再跑上线 plan。**这个判断直接重塑了我对 prd-tools 价值定位的理解**——单纯 PRD→Plan 不够，必须扩成三 plan 才能把整个研发工作流串起来。

### 洞察 5：Loop Engineering 是噱头？——这个判断不准确

**面试官原话**：我先说我的感受，五破（音译词，疑似指某个具体工具）就是个噱头。

**价值**：**面试官这个判断不准确**——Loop Engineering 是 Boris Cherny（Claude Code 创始人）2025 年在推的核心范式，原话 "my job is to write loops, not prompts"。被过度营销的是落地（很多团队包装成银弹），但概念本身是有价值的范式转变。**面试时遇到面试官下"噱头"判断，不要盲从**——可以礼貌坚持自己的判断："概念本身不噱头，噱头的是过度营销的落地"。

### 洞察 6：知识提取第一步出错，后续全错

**面试官原话**：第一步如果一旦出错，你后续的所有流程都是错的。我最近就遇到了这样的问题，最近两天刚把它解决完。

**价值**：**面试官自己也在做类似的管理业务平台，对知识提取准确性有同款痛点**。这意味着面试官对候选人的 AI 工程化能力判断标准很高——不是"会不会用 AI"，是"能不能解决 AI 工程化的核心难题"。**这是同款痛点的同行**，未来如果有合作机会，知识提取这块是可以深度交流的方向。

### 洞察 7：用 git 记录做真实场景准确率回测

**面试官原话**：可以根据开发计划出来后他有没有改过你这个开发计划。git 都有记录。**实验室数据准确，但肯定没有用真实线上数据跑更准确**。

**价值**：这是非常实用的准确率度量方法论——**用 git diff 看人工对 AI 输出的修正量**，比构造数据准。**这个建议我下次面试可以直接用**——讲准确率时不只讲构造数据，还讲 git 真实回测，可信度立刻提升。
