> 日期：2026-07-22 | 通过概率：约 45%（理由见文末整体评估）

> 岗位：AI 研发（面试官自曝"我们也在做类似的事，知道得比较细"——同行内行审内行，bar 极高）
> 候选人主打：Spec-Driven PRD 工作流（团队推广版 Claude Code 插件）+ 2 个个人 agent（总管多 agent 编排 / code agent）

## 核心教训（面试官当面定调）

面试官原话："**我其实主要问的是框架这块的设计，而不是说问你具体的功能是怎么实现的。**……常规来说，基本上都是从背景，从问题到方案，到整体的框架设计，然后才是一些流程，然后具体的功能。"

病根不是没料——证据链 / 多层门控 / readiness 五维评分 / 能力面适配器都是真东西，面试官也认可"这个我大概了解了"。**栽在三个层次/方向错位上**：

1. **把"代码定位"包装成"领域知识"**：你做的是 BM25→文件定位（code localization），却一直用"隐性知识/领域知识蒸馏"的话术。面试官反复在"领域知识建模 vs 全局拓扑/代码定位"两个维度上拽你，最终你承认"没做业务领域分类"。前后话术打架 = 不懂行的信号。
2. **"上下文控制"答非所问**：面试官明确考察"agent 跑整个仓库会超上下文、产生幻觉，怎么拆/调度"（**输入端** context control），你答的是"证据链+门控+readiness"（**输出端**产物审计）。两端不是一回事。
3. **全程在"功能/实现细节"层打转**（6 模板、BM25、文件类型），没从架构框架层降维讲。

> ⚠️ 雷区提醒：本场说了"大概 20 人团队推广"。按 ADR-0005 实际口径是 **1–5 人小范围试用**——"20 人"是早期美化，本场没被追穿，但下不为例，产物里统一软化成"小范围团队试用"。

## Q1+Q2：介绍一下你的工作流 / 应用在什么场景 / 对场景的帮助

**核心句**：把 PRD 自动蒸馏成结构化、有证据的开发计划，治"口口相传丢信息"。

**展开**：它面向需求开发流程。我今年观察团队需求链路有两个痛点——一是认知偏差（PM 写 PRD → 评审 → 技术评审一路口口相传，信息损耗、产生返工）；二是隐性知识（项目经验在老员工脑子里，新人上手和交接是沉没成本）。工作流核心是用工程化手段把这条链路规范化：把自然语言 PRD，蒸馏成有证据支撑的结构化开发计划（spec + plan）。落地是 SSOT 领域知识文件 + 全链路证据链（防幻觉）+ 多层门控（保质量），目标是降低开发工时。

**诊断**：答到要点了，但**开场没一句话锐利定位**，直接铺"是什么→两个痛点→三个设计点"的流水账；ownership（我发起/我主导）拖到 Q3 才讲。改进：先一句 ownership 定位（"我今年主动发起、主导的这个工作流"），再痛点→方案→效果，最后收一句边界。

## Q3：你在这个工作流里承担什么角色？一个人弄的吗？

**核心句**：我主动发起、主导架构设计到推广，现在负责前后端多方迭代的 merge。

**展开**：这项目是我主动发起、主导的，从架构设计、实现到测试推广。目前得到前端和后端两个老板认可——团队在推 AI 工程化，前端这边我继续迭代工具，后端基于它开分支做定制，端上也在尝试接入，我现在的角色更像把各方迭代 merge 到一起、推动推广。

**诊断**：这题答得相对好，立了 ownership。但"联系/merge 角色"略自我矮化——下次收尾加一句"核心架构和证据链机制是我设计的，这是工具能跨前后端复用的前提"，把 drive 感补回来。

## Q4：两个痛点里，"隐性知识"是不是 PRD 质量的子集？你的解决手段除了蒸馏仓库还有啥？

**核心句**：两个痛点是不同维度——认知偏差治流程，隐性知识治经验不可继承。

**展开**：这俩不是子集关系。**认知偏差**是流程问题：PRD 到开发一路口口相传，信息对齐有损耗，治它的手段是把流程规范化、自动化（PRD 蒸馏流水线做成团队标准工具）。**隐性知识**是经验问题：项目的前置业务背景、踩坑点、上下游契约，散在人脑和历史文档里，换人就丢——治它的手段是 reference 蒸馏，把这些前置知识沉淀成项目内的 SSOT 文件。两个手段对应两个痛点，边界清晰。

**方向校验（关键）**：⚠️ 这里是你后面失分的**总开关**。你说 reference 解决"隐性知识/业务知识缺失"，但面试官的判断是——**reference 做的"BM25→文件定位"是 code localization，不是补全业务语义**。业界定论也分两件事：Code Localization（LocAgent 类，定位"改哪里"，ACL 2025）≠ Domain Knowledge Extraction（离线建知识图谱、建模业务抽象）。**正确方向**：一开始就划界——"reference 的核心能力是帮 AI 在新需求下快速定位要改的文件，不是建模业务领域概念；业务语义是人工沉淀在 reference 文件里的，AI 不负责从代码硬抽业务抽象"。这样后面 Q6/Q13 全顺。

（*来源：LocAgent, ACL 2025, graph-guided LLM agents for code localization；Codebase-Memory, tree-sitter 知识图谱；面试官 Q6/Q7 反复区分两个维度*）

## Q5：reference 文件从哪来？业务知识信息从哪取？

**核心句**：三源喂料——git log + 历史 PRD/技术文档 + 源码，多轮分析出领域知识文件。

**展开**：reference 蒸馏的输入有三个源：①git log（每个需求 commit 里含改了哪个文件/函数，是高含金量的开发模式经验）；②历史 PRD 和技术文档（体现项目负责哪个模块、上下游契约、前后端/BFF 分工）；③源码本身。三源经多轮分析，产出几个领域知识文件：代码结构、编码规则、跨层契约、PRD 路由（常见需求场景→该改哪些文件/函数）、业务领域知识、代码实体索引（帮 AI 快速定位候选文件）。

**诊断**：内容对，但**没讲清"哪些是 AI 能 reliably 抽的、哪些得人工兜底"**。改进：明确分层——"代码结构/实体索引是静态扫描能 reliably 出的；PRD 路由是结合历史文档经验产出的、渐进式回流；纯业务语义靠人工沉淀"。把 AI 能力边界讲清，正好接住 Q6 面试官的质疑。

## Q6：从代码直接蒸馏，AI 很难理解业务抽象——AI 汇总成领域知识具体怎么做？（方向题·核心失分点）

**核心句**：我没让 AI 建模业务抽象——reference 是定位能力，业务语义靠人工沉淀。

**展开**：您这个问题点到关键了。**AI 从代码硬抽业务抽象，确实做不到可靠**——代码里没有"机柜生命周期"这种业务概念，只有函数和配置。所以我的设计是划清边界：reference 里 AI 负责 reliably 能做的事——扫代码结构、抽代码实体、建索引；而真正的业务语义（"这个活动类型对应哪段业务逻辑""什么场景前端负责什么后端负责什么"）是**结合历史 PRD + 技术文档 + git log，半自动产出、人工校准**，沉淀成 SSOT，后续渐进式回流越来越准。我没做、也刻意没让 AI 自动做"业务领域建模"——那是需要人写本体的事，AI 从代码硬抽会幻觉。

**方向校验**：✅ **方向就是"承认 AI 做不了业务抽象建模，划界给人+机"**。面试官原话"拿 AI 做汇总效果做不到很好，因为 AI 很难理解业务抽象"——他其实在**引导你承认这个边界**，你却一直在用"领域知识"话术绕，越绕越像不懂。下次主动划界，反而显得懂行。

（*来源：面试官 Q6 原话引导；LocAgent vs Domain Knowledge Extraction 业界区分*）

## Q7（面试官澄清）：你解决的是"领域知识建模"还是"代码定位"？

**核心句**：我做的是代码定位——帮 AI 找到要改的文件，不是建模业务领域。

**展开**：您区分得对，这是两个维度。**我解决的是后者**：给 AI 一个快速索引，让它在拿到新 PRD 时能定位到要改哪些文件、哪些函数。它**不是**真正的知识图谱，不建模业务领域的抽象（比如充电宝按设备/机柜/生命周期这种业务本体我没做）。这两件事能相辅相成——定位准了，AI 拿到的上下文才对——但它们边界明确，我做的这一侧是 localization。

**诊断**：录音里你绕了 4 个来回（13:11–15:52）才承认做的是后者，而且还在硬撑"边界不那么严格、相辅相成"。**诚实划界要快、要主动**——第一句就认"我做的是定位不是建模"，反而显得清醒。

## Q9+Q10：代码拓扑生成 / reference 怎么蒸馏仓库生成图谱？仓库蒸馏流程？

**核心句**：先定义产出模板（SSOT 化、每事实只存一处），再静态扫描抽实体建索引。

**展开**：reference 工作流分模式，冷启动是 0→1 全量构建——先扫项目有没有 reference 产物，没有就从 git log 分析近期变化，结合源码，并提示输入历史 PRD/技术文档，对文档做语义提取（文字/图片/表格，大部分靠大模型分析），重点是提取业务关键字和项目关键字做匹配。代码侧用 TS 静态扫描脚本抽代码实体，构建调用关系和 BM25 索引。最后按预先定义的模板，整合出几个 SSOT 文件。核心是"每条事实只存在一处，AI 按需查询"。

**诊断**：内容有，但**这是"流程流水账"，不是"架构框架"**。面试官 Q10 说"我不问格式，只问你要什么东西、目录分类规则"——他在帮你拉到框架层。改进：先讲**架构骨架**（三源 → 提取层 → 索引层 → SSOT 产出层 → 检索层），再讲每层做什么、为什么这么分，最后才落具体模板。先框架后细节。

## Q11+Q12：6 模板的产物是什么？/（你轻度抵触"问到这么细"）

**核心句**：模板是 SSOT 产出的骨架——代码结构/编码规则/跨层契约/PRD 路由/领域知识/实体索引。

**展开**：6 个模板对应 6 类 SSOT 文件：codebase（静态结构）、编码规则、跨层契约、PRD 路由、领域知识、代码实体索引。模板给 AI 的是"大致结构 + 字段示例"，工作流跑时喂给 AI 做定位参考。设计原则是每个文件职责单一、事实只存一处。

**Q12 态度减分点**：录音里你说"咱需要问到这么细吗？哈哈，这不成代码实现了吗"——**这是轻度抵触**。面试官解释了合理理由："我们也在做类似的事……主要考察架构设计、上下文控制怎么做。"改进：别抵触，直接接"明白，您想看的是上下文控制策略，我说架构层不铺字段"，把对话拉到你该讲的框架层。

## Q13+Q14：业务分层怎么分？关键词分类还是领域分类？（方向题·核心失分点）

**核心句**：我没做业务领域分类，只做 PRD 路由——常见需求场景→该改的文件。

**展开**：坦白讲，**我没做业务领域的分类**（像充电宝按设备/机柜/网络/生命周期那种业务本体）。我做的是 PRD 路由：把常见的需求场景（比如"新增活动配置、改某条规则"这种一句话需求点），映射到它通常要改的那几类文件——因为我们 diff 大多是复用度高的配置化模板，每次要改的就那么几种文件，把这些抽象成路由表。所以是关键字提取 + BM25 回搜定位，**不是按业务领域分类**。

**方向校验**：⚠️ 这题你**诚实承认了没做领域分类**——这本身没错，但和你前面用"领域知识/隐性知识"包装 reference 的话术**前后打架**，显得概念不清。根因还是 Q4 的总开关：一开始就该把 reference 定位成"代码 localization + 人工沉淀的业务语义"，别叫"领域知识蒸馏"。下次统一口径：reference = 代码定位能力 + 人工 SSOT，不声称做了业务领域建模。

**技术精度提醒**：Q14 你和面试官确认"把关键字做成向量，通过 BM25 回搜"——⚠️ **BM25 是词法检索（词频/逆文档频率），不是向量检索**。向量是 dense embedding（语义相似度）。两者都用才叫 hybrid search（RRF/加权融合）。"把关键字做成向量再用 BM25 搜"是概念混用，内行面试官会注意到。下次讲清：若只用了 BM25 就别提向量；若两者都用就说"hybrid（BM25 词法 + embedding 语义）"。

（*来源：Weaviate/Google Vector Search hybrid search 文档；BM25 = sparse/lexical，dense embedding = semantic*）

## Q15：找到文件后，依赖（组件/样式/接口）怎么处理？蒸馏吗？粒度多细？

**核心句**：上下游契约 + 能力面适配器——前端/BFF/后端各 ~10 个能力面定位关注维度。

**展开**：这块靠两个设计：①**上下游契约**——蒸馏时会前置判断这需求是前端/后端/BFF 哪侧的，结合技术文档和代码静态配置提炼"哪些归前端、哪些归后端"（比如校验多在后端、配置改多在前端）；②**能力面适配器**——不管目录怎么变，前端本质都有路由/组件/表单/API 调用，我针对前端/BFF/后端各设计了约 10 个能力面，帮 AI 根据需求描述定位关注维度，快速判断"这个功能点我们具不具备这个能力"。能力面是前置约束，蒸馏时还会借助历史技术文档和 reference 做交叉验证。

**诊断**：✅ **这题答得最好**——能力面适配器是真正的架构设计，面试官也是这里最认可。**这就是你该多讲的"框架层"**。下次把这种"抽象出稳定的能力面、屏蔽目录差异"的设计提到开场讲，比铺 6 模板有说服力得多。

## Q16：怎么保证扫描不漏？agent 跑整个仓库会超上下文、产生幻觉，怎么拆/调度？（方向题·第二核心失分点）

**核心句**：诚实说没法 100%——但输入端靠轻量符号索引按需取子集，输出端靠证据链+门控+readiness 兜底。

**展开**：先诚实讲：**任何 agent 都没法 100% 保证不漏**——把整个仓库喂进去必然超上下文、注意力发散、产生幻觉。所以我在**两端**做控制：

- **输入端（上下文控制，这才是您问的）**：不把整个仓库喂给 agent，先静态扫描建一个**轻量符号索引（类似 repo map）**，agent 拿到需求后先在索引层定位候选文件子集，再把子集的实体+上下文喂进去——只把最相关的部分进上下文，从源头控制 token 和注意力。
- **输出端（质量兜底）**：①**证据链**——AI 每条结论（拆需求点/拆功能/拆文件）都记录"从哪得出"，找不到的强制标记（防强完成倾向编造），共若干证据类型；②**多层门控**——步骤间校验顺序和中间产物完整性（有没有跳步、有没有充分利用上下文包）；③**readiness 五维自评**——产物产出前先自评是否达标，不达标回查证据链定位问题，达标才放行。

**方向校验（关键纠偏）**：⚠️ 录音里你**只答了输出端（证据链+门控+readiness），漏了输入端（上下文控制）**——而面试官明确说考察的是"agent 跑整个仓库超上下文怎么拆调度"。证据链是事后审计、防编造，**不解决"检索时漏文件"**。正确答法必须先接"输入端怎么控制上下文"（轻量索引/repo map 思路：aider 用 PageRank 符号图 + ctags 抽签名 + 动态子集选择，只塞相关子集），再说输出端兜底。**两端配合**才完整。

（*来源：aider repo map（PageRank 符号图 + ctags + 动态子集），aider.chat/docs/repomap.html；面试官 Q16 原话"考察上下文控制"*）

## Q19：你说的"静态分析"是分析文件依赖吗？（Python 脚本）

**核心句**：那个 Python 脚本是门禁用的（校验质量），不是分析文件依赖。

**展开**：那个脚本在门禁环节用，做整体质量校验，不是用来分析文件依赖的。

**诊断**：这题你占理（面试官理解错了，你纠正回来了），但沟通上绕了好几轮才对上。改进：直接一句"那个脚本在门控里用，做产物质量校验，不是静态依赖分析"，别反问"我在哪里提到"。

## Q23（直接要反馈）：对我面试有什么建议？

**面试官原话（当面定调，刻进肌肉记忆）**：
> "结构化的东西可能没准备……回答有点'点点'（碎）。常规来说，基本上都是从背景，从问题到方案，到整体的框架设计，然后才是一些流程，然后具体的功能。**我其实主要问的是框架这块的设计，而不是说问你具体的功能是怎么实现的。** 因为我一般沟通肯定都是通过框架去设计去，先了解架构，然后才能代入。"

**核心句**：项目要按"背景→问题→方案→框架→流程→功能"降维讲，先架构后细节。

**展开**：三条刻进肌肉记忆——①**层次要对**：先框架设计，后具体功能。你全程在功能实现层打转（6 模板、BM25、文件类型），面试官想听的是架构层（三源→提取→索引→SSOT→检索的骨架、上下文控制策略）。②**别碎**：回答要有开头-展开-收尾的结构，别"点点"。③**口径要统一**：别前面说"领域知识"、后面承认"没做分类"，概念前后打架。

## 面试官洞察（美团 · 共享充电宝 · 运营中心 AI 研发）

> 核心定调：面试官自己也在做同类"代码仓库蒸馏/定位"系统，是同行内行；最看重架构设计与上下文控制，不是功能实现。

### 洞察1：AI 岗核心考"架构设计/上下文控制"，不是功能实现
**原话**："我其实主要问的是框架这块的设计，而不是说问你具体的功能是怎么实现的……通过大模型去做代码仓库的分析这块，其实是比较考验上下文控制的。所以我需要了解你的技术方向、技术方案，然后判断你的上下文控制是怎么做的。"
**价值**：通用 AI 工程岗铁律。讲项目必须先框架后细节，且"上下文控制"是 codebase-agent 的核心考点——下次任何"仓库分析/RAG/检索"类项目，开场就准备好"上下文怎么控制"的答案（见洞察4）。

### 洞察2：AI 做领域知识建模不可靠——这是行业共识，要主动划界
**原话**："从代码中直接蒸馏……让 AI 去汇总，它的效果其实做不到很好，因为 AI 很难理解你具体的业务的抽象形式。"
**价值**：别再用"领域知识/隐性知识"包装"代码定位"。业界定论分两件事：Code Localization（LocAgent 类，定位改哪里）≠ Domain Knowledge Extraction（建模业务抽象）。**主动承认 AI 做不了业务抽象、划界给"人工沉淀 + AI 定位"，反而显得懂行**。这是本场最该刻进去的认知。

### 洞察3：代码定位分两层——多仓库拆分 + 代码拓扑生成
**原话**："如果做代码定位的话……第一层就是前端如果做微服务拆分的话，其实分多个仓库……第二件事就是代码的一个拓扑的生成。"
**价值**：内行的分层视角。讲 code localization 项目，先回答"单仓还是多仓"（影响是否要跨仓检索），再讲"拓扑/索引怎么生成"。这是讲该类项目的标准起手式。

### 洞察4（标杆对照）：上下文控制的业界做法 = 轻量符号索引 + 按需取子集（aider repo map）
**对照**：面试官考的"输入端上下文控制"，业界标杆是 aider 的 **repo map**——PageRank 符号图 + ctags 抽签名 + **动态子集选择**（只把最相关的符号子集塞进上下文，不喂整个仓库）。
**价值**：你录音里只答了输出端（证据链/门控/readiness），漏了输入端。下次补一句"输入端用轻量符号索引按需取子集（aider repo map 思路），输出端用证据链兜底"——两端配合才算完整。**这是本场最该补的硬知识。**

### 洞察5：美团 AI 代码 CR 质保 = 静态质检 + 评估委员真实环境模拟（双轨）
**原话**："在 SDE/SDD 流程中……有一个阶段做质检，代码质量的检测、覆盖率、安全之类的，有一种原则就是静态分析；还有是要质量，评估委员/QA 做真实环境模拟，可以理解为 UI 自动化测试。"
**价值**：头部公司 AI 代码质量保障的真实架构——**静态分析（机器可量化：质量/覆盖率/安全）+ 真实环境模拟（人工/自动化验收）双轨**。和你工作流的"多层门控 + readiness 自评"同源（都是"自动质检 + 验收"双保险），反问环节可主动对标："我们的门控+readiness 类似你们的静态质检阶段，缺的是真实环境模拟的验收——这也是我想补的。"

### 洞察6：美团 AI 工程化成熟度——token 不限 + 绝大部分需求 AI 研发
**原话**："token 现在是不限的……基本上绝大部分需求都是用 AI 去研发的。"
**价值**：美团侧开发工具 token 不限量、AI 研发已深度铺开。说明"AI 工程化成熟度"是头部公司硬指标——你做的 PRD 工作流方向正踩在他们需求上，是隐性加分项；也意味着面试官对这块要求极高（洞察1 的根源）。

## 整体评估

| 项目 | 结果 |
|------|------|
| 通过概率 | **约 45%**（见下） |
| 最大加分点 | 证据链 + 多层门控 + readiness 五维评分这套质量保证机制是真东西（面试官"这个我大概了解了"）；能力面适配器（Q15）是真正的架构设计；ownership 立得可以（Q3） |
| 最大失分点 | ①层次错位（全程功能实现层，没到框架层）②方向混淆（代码定位包装成领域知识，前后话术打架）③上下文控制答非所问（答输出端审计，漏输入端）④Q12 轻度抵触情绪 |
| 面试官风格 | 同行内行深挖型——自己也在做同类系统，问得很细但会解释理由；偏架构派（背景→问题→方案→框架→功能），重"上下文控制"；会主动引导（Q6/Q11 都在帮你拉到对的层次） |

**通过概率 45% 的理由**：
- 减分：面试官是内行、bar 高，反复发现层次错位和方向混淆，最后给了"结构化没准备/回答碎/主要问框架不是功能"的直接负面反馈；Q12 有抵触情绪。
- 加分（没到更低）：面试官多次主动引导你到对的层次（Q6 引导承认 AI 做不了业务抽象、Q11 澄清"我不问格式只问架构"），说明他认可你有料、只是没组织好；明确说"我这边没什么问题了"、还提"想面两轮也可以"；证据链/门控/能力面是真东西。
- 综合：**"有料但没组织好"** 是核心定性，不是"不懂"。这种状态下二面/复盘后翻盘概率不低，但一面当场印象是偏下的。

### 表达诊断
- **口头禅**："对""就是""就是"高频（转写里大量"对对对对对"），显得犹豫、像在等面试官确认。
- **回答长度**：偏长，习惯铺实现细节。
- **结构性**：差——经常跳进细节，没有"一句话定位→展开→收尾边界"的骨架。
- **最大问题（storytelling 镜头）**：讲自己项目掉进"功能流水账"——按"是什么→痛点→怎么做"时间线铺，没有先立最关键决策（tentpole）；被 why 追到 2–3 层就散，没到业务/战略/反思层；讲完事实就停，没有"做前以为 X、做完发现 Y、下次 Z"的拔高。

### 项目回答表达诊断（storytelling 四查）
1. **ownership**：Q3 立了（主动发起/主导/双老板认可），但**开场没立、被问才讲** → 不够主动。
2. **spreadsheet-story**：✅ 中招。全程按时间线铺项目，没先立 tentpole（"这工具的核心决策是：把代码 localization 和人工业务语义分两层"）。
3. **被问 why 答几层**：卡在 2–3 层（表层→工程）就散，没到业务/战略/反思。面试官每个 why（为什么这么蒸馏/怎么保证不漏/怎么分类）你都没答到"本质矛盾 + 取舍 + 反思"。
4. **反思**：缺。讲完事实就停。

### AI 工程化 3 硬伤扫描
- **① 数据被追穿**："20 人团队推广"（穿帮雷区，本场没被追但口径不实，必须软化成 1–5 人小范围试用）；"百分之七八十转化效率"（口径不清，下次要么备"怎么算的"话术，要么软化成"冷启动能到七八成定位效率，后续渐进回流"）。
- **② 原理卡壳**：BM25 vs 向量概念混用（Q14）——BM25 是词法不是向量；"若干证据类型"具体哪几种没展开（面试官没追，但要备齐话术）。
- **③ 闭环回避**：本场没被问端到端闭环。但"证据链+门控+readiness"本质是局部自验证闭环，可主动接——**诚实说**这是团队推广版（PRD→Plan，Claude Code 插件，1–5 人试用）聚焦的高 ROI 环节，端到端（PRD→Code）是个人原型探索，不是生产级"6 步闭环"。
