# opendesign 一面 — 面试优化答案

> 日期：2026-07-21（7/23 录音核实更正） | 通过概率：约 15%（理由见文末整体评估）
> 公司：OpenDesign（AI 设计工具，方向 = 网站/UI 模式抽离复用，代表功能"灵感按钮"）
> 候选人主打：两个个人 AI 项目 —— Manager Agent（39节点 multi-agent 编排）+ Coding Co-Assistant（LangGraph，18内置工具）

---

## 核心教训（面试官当面定调）

面试官原话："**就感觉你准备好像不太充分了，很多问题好像没准备好。**" 技术问答仅 4 题约 10 分钟即结束。病根不是知识盲区，是**两个个人项目完全没讲清楚 + multi-agent 生态认知停在产品体验层 + 从没拿自研项目对标过业界**。用个人项目打 AI 岗，却栽在"项目表达 + 行业视野"上。

> **7/23 录音核实更正**：原记录把面试官点名的竞品误记为 OpenHands/OpenManus/Genie，实为 **opencode / ohmyopencode**（ASR 把"波分扣的/欧麦欧麦口"听岔了）；Q4 候选人提的是**企业版 Copilot**（非 Cursor），被追问"源码看过吗"。已据此更正 Q4 与洞察——复盘产物本身也会犯"没核实就猜"的错，和面试同病根。

---

## Q1: 有没有对比过主流 multi-agent 产品？（Claude Code 的 dynamic workflow，是固定状态机还是 AI 自主判断调度？）

**核心句**：我的 Manager Agent 是 LangGraph 那种固定状态机——节点和边我预先定义死，好处是可观测、可重放。

**展开**：主流现在分两派。一派是 Claude Code 的 **dynamic workflow**——模型运行时自己写一段 JS 脚本来编排 subagent，最多上千个、十几个并行，脚本本身持有计划/分支逻辑，灵活但不确定性强，适合探索、迁移、审计这类一次性大任务。另一派是 **LangGraph 式固定状态机**——我把流转逻辑写死在图里，确定性强、可调试、有完整状态快照，适合需要稳定复跑的生产链路。我的项目选后者，是想练"显式控制流"这一套；**方向其实没错**（Anthropic《Building effective agents》的定论就是"能用确定性 workflow 就别上自主 agent，尤其高风险要审计的场景"），**错在我没 defend**——录音里直接认怂"可能人家自主进化更好"，等于把分白送。下次要讲清"固定 vs 动态是场景选择不是优劣"，再补一句"我知道动态调度是趋势、固定图的边界是结构外情况不灵活"。
（*来源：code.claude.com/docs/en/workflows；Anthropic《Building effective agents》；面试官 Q1 暗示"模型自主调度"是方向*）

## Q2: 为什么是 39 个节点？而不是简单任务、甚至根本不需要这么多？

**核心句**：39 个是学习目的堆出来的上限，真要上生产我会砍到 5–8 个核心节点。

**展开**：这项目本质是学习练手，我把每个学到的能力——多路由、状态机、子任务拆解——都做成独立节点试了一遍，所以膨胀到 39 个，坦白讲有过度工程的成分。如果重做，我会按"感知→规划→执行→校验"收敛成 5–8 个核心节点，剩下的能力下沉成工具调用。节点数本身不是优势，**编排的清晰度和可收敛性才是**——这点 Claude Code 的 dynamic workflow 思路反而更轻。
（*注意雷区：讲节点时别穿帮——Manager 实际 9 外 3 内、阿里云 qwen-flash、失败归因纯规则、自动晋级默认关*）

## Q3: Coding Co-Assistant 是 LangGraph 工具，18 个内置工具是不是太多了？

**核心句**：核心是 5 步主线，剩下十几个是这 5 步的细分工具。

**展开**：18 个看起来多，但主线就 5 步：读仓库摸结构 → 语义检索定位相关代码 → 静态分析（AST 解析 + 依赖追溯 + 调用图）理解上下文 → 出 diff → 写盘。剩下的是每一步的细分工具，比如静态分析下挂着 AST、依赖、调用图三个。**工具一多，模型选错工具的概率会上升**——这是我下一步想优化的：高频工具显式描述、低频的合并。这项目和 Manager Agent 一样是学习目的，所以刻意把了解到的东西都铺开试了一遍。
（*别自我矮化说"冗余复杂"——说"学习期探索全面、生产会收敛"，同样是事实但 framing 完全不同*）

## Q4: 有没有学习过成熟开源的 coding agent / multi-agent 框架？（头号失分点）

**核心句**：面试官点名的 opencode / ohmyopencode 我了解定位；企业版 Copilot 闭源看不了源码，但公开架构文档读过。

**展开**：录音里我只答了 AutoGPT（自主任务拆解，2023 的思路、已非当下主流）和企业版 Copilot（代码检索/上下文感知强），还被追问"**企业版源码看过吗**"——我承认只到产品体验，这正是 AI 岗的档次分水岭（用过 ≠ 懂原理）。但面试官真正想听的是他点名的 **opencode / ohmyopencode**，我**一个都没提**：

- **opencode**：SST 团队（已重组为 Anomaly）的开源 terminal coding agent，Go 写、75+ 模型、不锁 vendor；主 agent + subagent（General/Explore）**动态委托**，单消息里塞多个 tool call 并发。
- **ohmyopencode**：opencode 上的社区**多 agent 编排插件**（不是配置工具）——最多 7 个并行后台 agent + 内置 LSP/AST 工具，把单个 agent 编排成"协调的开发团队"。
- 这俩走 **LLM 动态调度**，和我 Code Agent 的固定状态机是不同编排路线（见 Q1）。下次要先讲清它们的定位和编排模型，再落到"我做什么它不做（企业可审计的受控改码）、不做什么它做了（通用生态）"。
- **企业版 Copilot 源码**：闭源商业产品，看不了——但别只认"没看"，要补一句"公开的架构博客/文档我读过，核心是上下文感知 + 代码知识检索"，把"没源码"转化成"有公开资料的理解"。

（*来源：opencode.ai/docs/agents；GitHub sst/opencode；ohmyopencode 社区插件文档（opensoft/oh-my-opencode）；面试官 Q4/反问点名 opencode + ohmyopencode*）

## Q5（反问）：对我有什么建议？

**核心句**：项目讲述条理性 > 项目复杂度；AI agent 方向必须持续追新。

**展开**：面试官给的三条都值得刻进肌肉记忆——①任何面试，项目介绍都是超重点考察，要有条理有组织地讲亮点；②持续追新框架（opencode/ohmyopencode/Claude Code dynamic workflow），这块迭代极快；③看得出来面试官更认可"模型自主调度"的动态编排方向，而不是固定堆节点——我固定状态机方向没错，但要主动讲"我知道它的边界、也知道动态是趋势"。

## Q6（聊产品）：对 OpenDesign 产品的看法（全场唯一答得顺的）

**核心句**：最亮点是"灵感按钮"——内置浏览器把好网站按固定模式抽离复用。

**展开**：我之前就在用 OpenDesign，最打动我的是灵感按钮：它能在内置浏览器里把你认为好的网站，按固定模式抽离成可复用的设计资产。这反衬出一个事实——我不是不会表达，是项目没准备好就上了。

## 面试官洞察（opendesign · 主面试官）

> 核心定调：项目表达 + 行业视野是 AI 岗核心考察，候选人两者都没准备好。

### 洞察1：项目讲述的条理性，比项目复杂度更重要
**原话**："要多准备怎么介绍和表达自己做过的项目……不管什么面试都会非常重点考察。深入挖掘亮点，有条理、有组织地讲出来。"
**价值**：通用面试铁律。本次正好反了——项目复杂却讲成流水账。任何项目都要先备好"痛点→方案→亮点→取舍"四段式。

### 洞察2：AI agent 方向，必须主动追新，迭代极快
**原话**："参考更新的项目，opencode、ohmyopencode 这些 multi-agent 框架……这块迭代很快，有很多新东西值得学。"
**价值**：AI 岗隐性门槛——期待你持续追新。AutoGPT 已过时，opencode / ohmyopencode / Claude Code dynamic workflow 才是当下，高频考点。

### 洞察3：编排范式正从"固定堆叠"转向"模型自主调度"
**原话（Q1）**："dynamic workflow……它动态调度多个 agent，是 A 自己判断要不要调其他 agent 去解决问题。"（对照你的 39 节点固定编排）
**价值**：面试官暗示你的方向偏旧。趋势是 dynamic workflow / 自主调度（搜索印证：Claude Code 让模型运行时写编排脚本；Anthropic 官方"workflow 优先"也承认动态是兜底趋势）。项目要主动讲"我知道固定状态机的边界，也知道动态调度是趋势"——方向没错但要会 defend。

### 洞察4：用产品 ≠ 读源码，是两个档次
**原话（Q4追问）**："企业版 Copilot 你能看到源码吗？还是单纯产品层面体验？"
**价值**：AI agent 岗硬区分"用过"和"懂原理"。答"产品体验"直接降档。备框架至少读 1 个开源实现（opencode 源码可读、是最佳切入点）。

### 洞察5：OpenDesign 方向 = 网站/UI 模式抽离复用
**共识**：OpenDesign"灵感按钮"——内置浏览器把好网站按固定模式抽离复用。
**价值**：这家方向是设计生成，和你的 coding agent（代码生成）是相邻赛道。后续聊可往"设计稿→代码"衔接靠，是差异化切入点。

## 整体评估（见复盘报告第四步）

- **通过概率**：约 15%。面试官当面定调"没准备好"+ 技术问答 10 分钟即结束 + 核心考察点（项目表达/行业视野）双失分。唯一加分：产品环节答得有细节、态度好、多面官面试。
- **最大失分点**：Q1（编排范式空白、还认怂）+ Q4（只答 AutoGPT/Copilot、停在产品体验，面试官点名的 opencode/ohmyopencode 一个没提）。
- **最大加分点**：Q6 产品环节有具体细节（灵感按钮），证明表达力没问题，纯粹是准备不足。
- **根因**：**闭门造车**——两个 agent 项目都是"学习目的"，从没拿它们对标过业界。面试官每一题（Q1/Q2/Q3/Q4 全是对比型）都精准戳在这个盲区上。
