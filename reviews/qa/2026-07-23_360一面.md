## 面试官风格 & 这场的核心信号

- **360 AI 平台团队**，做公司内部的 AI Coding + Worker 工具（类 Cursor / Claude Code / Coda），全员在用，工作模式已转「全栈模糊化」。

## Q1: 介绍一下第一个项目（PRD 工作流 / spec-driven）

**表面问的**：项目是啥。
**实际考察的**：你能不能在 60 秒内让一个完全不了解的人听懂「为什么做、做了啥、怎么运转、产出什么」。
**答到的**：问题背景（信息损耗/返工/新人成本/AI 不可控）、三个支柱、结果。
**漏掉/出问题的**：**一上来就进 SSOT / 证据链 / 质量门控这种术语，altitude 太低太早**，面试官明确说「一开始有点懵，后来多追问才懂」。这正是 spreadsheet-story 的反面——结构有，但缺一个高空的「背景 + 流程总览」钩子。

| 维度 | 分数 | 问题 |
|------|------|------|
| 知识完整度 | 8 | 内容全 |
| 表达清晰度 | 5 | 开场 altitude 太低，术语前置，面试官懵 |
| 追问防御力 | 7 | 追问能接住 |

**改进版答案（主陈述，先背景再细节）**：

- **核心句**：把 PRD 自动蒸馏成结构化开发计划，靠「单一事实源 + 证据链 + 多层门控」保证 AI 结论可信、可追溯。
- **展开**：
  1. **为什么做（背景）**：需求到代码这条链路——PM 写 PRD → 口口相传给开发 → 开发理解再写，每一步都有信息损耗，最后返工；历史经验沉淀在老员工脑子里，新人上手成本高；直接把 PRD 喂大模型，没有项目兜底，结果不可控（连该改哪个文件都可能编造）。
  2. **我的角色（ownership）**：主动发起、主导落地，从设计、开发、测试 case 到团队推广分享都是我；我负责基建和通用能力，前后端各自迭代、我 review 两边代码，双周对进度，是整体负责人。
  3. **本质矛盾（难点）**：从 PRD 到代码生成不是最难的——最难的是**结论是否可信、可追溯、有证据支撑**（改动依据、影响范围、方案能否复现）。
  4. **怎么运转（流程总览，先给骨架再讲支柱）**：输入 PRD → ① reference 插件扫已有项目、产出 6 个领域知识文件 → ② 蒸馏插件把 PRD 拆需求点、定位候选文件、产出 SPEC → ③ 人工确认风险点和待对齐信息 → ④ 执行。三个关键决策：SSOT（6 文件无冗余，压缩 AI 编造空间）、证据链（每条结论带 file:line，指不出的也标记成一条证据）、质量门控（readiness 多维自评分 + 确定性脚本检测跳步/中间产物）。
  5. **结果**：在营销团队小范围推广试用，效率有提升；后端团队也在参考迭代内部使用。
  6. **方法论 + 边界**：真正的价值不在「能生成代码」，在「把 AI 每条结果变成可溯源、可验证的工程产物」。

> ⚠️ **雷区自纠**：录音里说了两次「20 多人推广」——按 ADR-0005 实际口径是 1–5 人小范围试用，**20+ 是早期简历美化、被追问会穿帮**。改进答案已软化为「小范围推广试用」。

## Q4: 领域知识 / RAG 知识库怎么做的？有哪些文档入库？（本场最长一答）

**实际考察**：你这套知识库是真沉淀还是套个词。
**答到的**：6 个核心 YAML、按开发生命周期角色切分、逐个讲了项目画像/codebase 清单/code rules/字段契约/PRD 路由能力面/业务知识、倒排查引（字段→函数→文件）。
**问题**：**太长、太细**，跟 Q1 一样的毛病——面试官要的是「形态 + 入库逻辑」，你给了 6 个文件逐个展开。
**改进版答案（压缩到 3 句 + 结构）**：
- **形态**：6 个 YAML，按知识在开发生命周期的角色切分（项目画像 / 静态清单 / 编码规则 / 字段契约 / PRD→能力面路由 / 业务术语），单一事实源落在一个文件，其他文件用类 ID 引用，避免冗余给 AI 编造空间。
- **入库逻辑**：reference 插件扫 git log / 历史 PRD / 技术文档，多轮分析产出这 6 个文件；每个文件顶部写「放什么/不放什么」的声明。
- **检索**：正则 + BM25 做代码实体索引和倒排，**字段→函数→文件倒查**，扫到 PRD 关键词能快速定位到具体字段/方法。没上向量，因为 B 端项目体量下 BM25 够用且可解释。

## Q9: 验证代码内容是否匹配，怎么验证？（强项）

**答到的**：4 步——① 语法（TS 编译 + 类型校验 + 完整性）② 字段上下匹配（生成字段必须包含于计划字段，防幻觉编造）③ 隐性项目约束（i18n / 枚举白名单 / 规范）④ 模块引入导出 / 路径 / 接口类型检查；外加**一致性检查**（同需求连续跑 10 次结论是否一致）、**环境样本检查**（已有分支的 git log + 改动文件对比，准确率约 80%）。
**评价**：**完整、有口径、防住了「数据被追穿」硬伤**。80% 这个数字备好了「怎么算的」话术。
**改进**：可补一句**为什么是 80% 而不是更高**——剩下 20% 主要是 PRD 歧义和需求点边界 case，不是生成机制本身的问题（诚实给误差归因）。

## Q11: 第二个项目 Manager Agent

**实际考察**：多 agent 编排的设计深度。
**答到的**：统一 WebSocket 入口 + 语义路由 + 任务规划调度下游十多种 agent；解决入口分散；编排层 + 状态机；难点是「编排成可信链路」（请求派对、中间状态丢失找回）；4 点（硬编码条件边状态机非动态 LLM / agent 存活验证 / 节点审查 / 高风险 HITL 确认）；**诚实说业余研究未接正式流程**。

**改进**：补一句**和动态编排（如纯 LangGraph LLM 路由）的取舍**——动态灵活但不可控，硬编码可控但扩展性靠人工加节点，目前节点规模（39）下硬编码 ROI 更高。

## Q12: WebSocket 干啥的？为什么不用 SSE？（方向校验题 ⚠️）

**方向校验结论**：**方向对**——业界共识是「需要双向工具确认（human-in-the-loop）的 agentic 场景，WebSocket 是更强选择」（来源见下）。但**你的理由偏弱**：只说「SSE 单向没法推 confirm」，被追问「你还有这个逻辑？」。实际上 SSE + 一个独立 POST 也能做 confirm，所以「单向没法做」这个说法被高估了。

**改进版答案**：
- **核心句**：流式输出 SSE 也够，但这是多 agent 编排器、有频繁的逐节点人工确认和中断，所以选 WebSocket。
- **展开**：
  1. SSE 单向（server→client），OpenAI/Anthropic 这些厂商流式都用它，简单、HTTP/2 友好、自动重连。
  2. 但我这个场景不是单纯流 token——中间有 agent 调度、节点审查、高风险操作要 confirm 往前走、还要支持中断/取消，**是频繁的 client→server 双向交互**。
  3. SSE 理论上能配一个独立 POST 做 confirm（hybrid 方案），但每步都开一个请求、状态对齐麻烦；WebSocket 全双工、延迟低、状态在一个连接里维护，对这种 step-by-step HITL 更干净。
  4. 所以选 WebSocket 不是「SSE 做不了」，是「频繁双向 + 单连接状态」这个场景它更合适。

> 来源：[sniki.dev - SSE vs WebSockets for AI chat](https://www.sniki.dev/posts/sse-vs-websockets-for-ai-chat/)、[websocket.org - WebSockets and AI](https://websocket.org/guides/websockets-and-ai/)、[Reddit r/webdev](https://www.reddit.com/r/webdev/comments/1seqym1/i_chose_sse_over_websockets_for_streaming_ai/)

## Q18: 系统提示词怎么设计？（方向校验：方向对，印证）

**答到的**：4 维度——角色定位 / 步骤引导 / 禁令红线 / 格式输出（JSON vs 纯文本，给清晰决策树）。
**方向校验**：**方向对**，和业界共识一致（role / instructions / constraints / output format）。
**改进（补面试官追问的「哪些放系统提示词、缓存、重构」）**：
1. **什么放系统提示词**：稳定的、每次都用的——角色、硬规则、输出格式、工具使用红线。**动态的、一次性的内容放 user message**，别污染 system。
2. **缓存**：系统提示词是稳定的 → 天然命中 prompt cache，所以**把高频稳定内容沉到 system 前缀**能省成本；改 system 会击穿缓存，所以重构时把易变部分隔离到后面。
3. **重构注意**：系统提示词膨胀后要做分层（全局角色 → 任务级规则 → 工具级说明），别堆成一坨。

## Q19: 工具设计原则？（强项）

**答到的**：① 工具描述写清楚（触发场景/参数约束/必需参数/默认值）② 调度层红绿灯（优先用工具非编造、优先专业工具）③ 并行/串行决策（并行提效但上下文不全/任务依赖）④ 重试机制（限流/工具错误重试接链路）⑤ 路由分发（意图路由器、渐进式分类、先大类再最相关工具，省 token + 避免选择困难）。
**评价**：**完整、有体系**。这题是加分项。保持。

## Q20: 「北京今天天气怎么样」之后，模型收到哪些消息、怎么流转？

**实际考察**：你对 LLM 消息协议/tool calling 的底层理解。
**答到的**：message 含用户输入 + 工具 JSON（名称/功能/参数）+ 系统提示词 → 模型思考决策 → 调天气工具传参 → 结果观察 → 汇总成人能听懂的回答。
**评价**：基本对，但偏流水账。**改进（用消息角色讲更硬）**：
1. 模型收到的 messages 是一个数组，有 4 种 role：**system**（系统提示词，全局）、**user**（「北京今天天气怎么样」）、**assistant**（模型历史回复）、**tool**（工具返回结果）。
2. 同时收到 **tools 定义**（每个工具的 name / description / JSON Schema 参数）。
3. 模型推理后输出一个 **tool_call**（决定调 weather 工具 + 按 schema 填参数），不是直接回答。
4. 系统执行工具、把结果包成 **role=tool** 的 message 塞回上下文。
5. 模型拿到 tool 结果再推理，这次输出**最终自然语言回答**给用户。整个是一个 think → act → observe 的循环。

## 面试官洞察（360 AI 平台团队）

> 360 在做公司内部的 AI Coding + Worker 工具，团队已从纯前端转型「全栈模糊化」，急需 agent 经验的人。

### 洞察1：核心产品 = 内部 AI Coding + Worker（类 Cursor / Coda）

**面试官原话**：最核心项目是做一个公司内部类似 Coda 的东西，加上类 GPT 的 work 模式；UI 和功能可以理解为和 Cursor 非常像；做了一两个月，现在基本给公司全员在用。内部叫 code。

**价值**：360 的 AI 平台不是玩具——全员在用的内部 Coding 工具，意味着你进去是做**真在生产环境跑的 agent 产品**，不是 demo。这跟你的 Manager/Code Agent（业余研究）正好形成补位：你有原型探索经验，他们有规模化落地场景。

### 洞察2：工作模式 = 全栈模糊化、模块到人

**面试官原话**：原来是前端团队（前端 + 设计 + 少量运营），现在转型，技术栈划分和岗位职责模糊化，每个人都可以理解为全栈；一个模块分给你，从需求到上线整条链路交给一个人做；UI 可以参考 Cursor，服务端、前端怎么做自己思考。

**价值**：**这是 360 这类 AI-native 团队的前沿信号**——AI 把「前后端分工」打破了，因为 AI 能补齐你弱的栈。对你（国际化高级前端）意味着：进去后要**主动扩服务端能力**，用 AI 做杠杆。面试和入职心态都要从「我是前端」切到「我端到端交付模块」。

### 洞察3：技术栈 = OpenAI Agent SDK + codex SDK + 自研 + opencode + 模型侧微调

**面试官原话**：有一部分 agent 能力用了 OpenAI 的 agent sdk、codex 的 sdk，也在自研；也参考 opencode；有算法同学做微调、做意图识别（在模型侧）。

**价值**：他们**多轨并用**（官方 SDK + codex + 自研 + opencode 开源），不是 All-in 某一个。这跟你「先复用 Claude Code plugin、自研只做领域逻辑」的思路一致。聊技术选型时可以接：**官方 SDK 解决通用工具循环，自研解决领域知识和质量门控，模型侧微调解决意图识别**——三层分工。

### 洞察4：面试官直接反馈 —— 先讲背景再讲细节（本场最关键）

**面试官原话**：表达能力很 OK；但第一个项目一开始讲了很多细节（reference 等），我一开始没太听懂……可以先介绍背景——这个东西是什么？为什么要做？流程是什么（输入 PRD → 第一步什么 → 落几个文档 → 输出什么代码）？再说怎么做、技术细节。因为你太了解了，但我只看到核心技术，不清楚背景和为什么。

**价值**：**这是本场唯一硬伤，也是最值钱的反馈**。它精准命中你 stealth 重构一直在攻的 storytelling 问题——不是讲不出，是**开场 altitude 太低、术语前置**。面试官给了明确的结构模板：**是什么 → 为什么 → 流程总览 → 技术细节**。下一场开场严格按这个四段走。

### 表达诊断
- 口头禅：「对对对对对」「然后」「就是」偏多，Q4 那段几乎每句一个「对」。**适度收**。
- 回答长度：Q1/Q4 偏长且术语前置；Q8/Q9/Q19 长度合适。
- 结构性：多数题有开头-展开，但**开场缺「背景钩子」**（Q1 最明显）。

### 项目回答表达诊断（storytelling 镜头）
- ① **ownership**：PRD 项目有（主动发起/基建/review/整体负责人）✓；Manager/Code 偏研究、ownership 弱但诚实说了。
- ② **spreadsheet-story**：**PRD 项目开场犯了**——有结构但 altitude 太低太早，没先立「背景 + 流程总览」钩子就直接下到 SSOT/证据链。**这是面试官反馈的根因**。
- ③ **被问 why 答几层**：SSE 那题答一层（单向）就被追问；精确到行号答了 tradeoff 但可更深。
- ④ **反思拔高**：缺。没有「做前以为 X、做完发现 Y」的收尾。下次每道项目题补一句反思。

### AI 工程化 3 硬伤扫描
- ① **数据被追穿**：80% 准确率有口径（环境样本 git log 对比）✓；**但「20 多人」说了两次，是穿帮风险**（实际 ADR-0005 是 1–5 人），今晚必须软化公开 repo 表述。
- ② **原理卡壳**：SSE vs WS 方向对但理由弱；LLM 消息流转基本对但偏流水账。无严重原理硬伤。
- ③ **闭环回避**：没回避——讲了飞轮回流，诚实说 Manager/Code 是业余研究、PRD 是团队推广版。没吹「6 步闭环」（好）。

## 今晚必做 3 件事

1. **改开场话术（最高优先）**：按面试官给的模板重写 PRD 项目开场——是什么 → 为什么 → 流程总览（输入 PRD → 6 文件 → SPEC → 代码）→ 技术细节。**对着镜子掐表 90 秒讲一遍**，第一句必须是背景不是术语。→ 验证：能不看稿讲完且第一句是背景。
2. **软化公开 GitHub repo 的规模表述**：把「20+ 人 / 团队推广」改成「小范围团队试用」，删掉任何「6 步闭环 / 生产级」字样。→ 验证：repo README grep 不到「20」「闭环」。
3. **补 SSE vs WS 的完整答案 + LLM 消息流转的 role 讲法**：背熟「4 种 role（system/user/assistant/tool）+ tools 定义 + tool_call + tool 结果回填」这条链。→ 验证：能口述完整流转不卡。

## 下次面试注意
1. **每个项目题开场先 30 秒背景**（是什么/为什么/流程），再下技术细节——这是本场唯一明确反馈。
2. **数字一律软化**：凡提效/规模/准确率，先备「怎么算的」，不确定就说「小范围 / 约」。
3. **被追问别硬撑单一理由**（如 SSE「单向没法做」）——承认替代方案存在、再说为什么你的场景选这个，更显体系。
