## Q3: Spec 蒸馏是什么概念？跟 PRD 怎么结合？

**核心句**：把 PRD 自动转换成结构化规约文档（Spec），是 SDD 在我们场景的落地

**展开**：借用"蒸馏"是强调从非结构化自然语言里把需求骨架提炼出来，跟模型蒸馏压缩参数有相似性。本质是 PRD 输入 → 自动拆解 → Spec + Plan 输出，中间靠证据链 + 多层门控保证可信。OpenSpec 是通用脚手架，我们的痛点是 PRD→Spec 这段闭环它不做，所以自研

**搜索增强**：OpenSpec（Fission-AI）是开源 SDD 框架，提供 `/opsx:propose` artifact-guided workflow，但缺乏对企业级 PRD（多源、2-3万字、跨上下文）的深度拆解闭环——这是我们自研的差异化点（来源：[GitHub - Fission-AI/OpenSpec](https://github.com/Fission-AI/openspec)、[Augment - 6 Best SDD Tools](https://www.augmentcode.com/tools/best-spec-driven-development-tools)）

## Q9: 证据链只能说明 TRD 从 PRD 来，怎么确保真正理解了 PRD？（关键失分题）

**核心句**：证据链只解决"溯源"——结论来自哪；"对错"靠三层独立机制：人审 + 字段双向对齐 + 黄金样本回归

**展开**：完全新增、零历史上下文的需求，本来就是我工作流的边界外的场景——这种场景应该用 OpenSpec 探讨方案或直接对话。在我设计的场景内（已有项目承接新需求），AI 的"理解"靠三件事保证：①证据链保证每条结论可追溯到 PRD 原文/代码/技术文档，②字段双向对齐（Spec 里字段 ⊆/≟ PRD 字段），③Readiness Score 多维评分 + 人审硬中止。面试官的逻辑是对的——证据链不等于正确性，正确性靠的是另一套验证机制

**搜索增强**：行业共识把 LLM 输出质量保障分两层——溯源（provenance / chain-of-evidence）和验证（validation / deterministic gates）。Sonar 的 Loop Engineering 文章明确指出"deterministic code verification tier"是和溯源独立的硬中止层（来源：[Sonar](https://www.sonarsource.com/blog/loop-engineering-without-verification-is-just-automation/)）

## Q13: 完全新增功能模块，怎么判断 TRD 对错？

**核心句**：完全新增、零上下文的需求是我工作流的边界外——不强行兜底，建议改用 OpenSpec 或对话式 AI

**展开**：我的工作流解决的是"已有项目承接增量需求"——AI 复用前置 reference（业务背景 + 代码契约 + 历史方案）做拆解。完全新增场景下，reference 失效，工具的边际价值很低。强行让工具兜底反而是反模式。这种场景应该走 OpenSpec 探讨架构，或者直接 BaseMode/Claude Code 对话——工具选型本来就是按场景来

## Q15: 对错和一致是两件事，幂等怎么保证？（关键追问）

**核心句**：一致性靠"约束概率系统的轨道"——LLM 是概率的，工作流的确定性脚本 + 中间产物校验 + 锚点覆盖把它约束在固定输出空间

**展开**：面试官的排序例子很精准——结果对不等于实现稳。我的工作流把"实现稳"拆成三层约束：①plan 精确到文件维度（同一需求拆出的文件路径必须一致），②每个 step 间用 Python 脚本做上下文消费检查 + 锚点覆盖校验，③XML 标记告诉 AI 该做/不该做。这三层都是确定性脚本，不依赖模型自觉。最终 80% 文件覆盖率 = AI 输出与黄金样本（人工 diff）的重合度

**搜索增强**：行业共识——temperature=0 并不保证完全确定性，因 GPU 浮点、batching、模型版本都会引入扰动；工程上的最佳实践是把 LLM 当概率系统，外层用 deterministic gates + idempotent pure functions 约束输出（来源：[arXiv - Non-Determinism](https://arxiv.org/html/2408.04667v5)、[Thinking Machines Lab](https://thinkingmachines.ai/blog/defeating-nondeterminism-in-llm-inference/)）

## Q16: AI 生成代码怎么判断每行对错？（闭环回避·最严重失分）

**核心句**：四层校验（语法 → 字段双向对齐 → 规则 → 集成）+ 自修复循环 + 震荡检测，端到端原型 prd2code-gen 已跑通

**展开**：端到端原型 prd2code-gen 完整 6 步闭环：①蒸馏（PRD → Spec）②分析（拆需求点 + 文件定位）③Plan（技术方案）④双路径生成（Handlebars 模板优先 + LLM 兜底）⑤自验证自修复（四层校验 + 震荡检测 A→B→A）⑥eval/auto-tune 反哺。AI 写完代码后跑四层校验：①语法层（TS emit / AST 完整性），②字段双向对齐（蒸馏出的 Spec 字段 ⊇ 实际改动字段），③规则层（i18n key 格式 / 命名规范 / 反模式），④集成层（导入导出 / 破坏性变更，diff 警察）。验证失败进自修复循环——不是改完就完，要重新全量验证。还有震荡检测：如果 A→B→A 横跳，识别切换策略找局部最优

**双轨定位讲清**：团队推广版聚焦 PRD→Plan 是 ROI 选择（人工确认后才开发、风险低、提效明显），端到端原型已设计跑通作为下一代探索——代码生成涉及质量责任和 CI 对齐，先用原型验证设计可行性再决定工程化投入。被问"既然完整为啥不推广"→ "PRD→Plan 是 ROI 最高风险最低环节，先用原型验证可行性再决定工程化投入"

**搜索增强**：行业共识——AI 代码验证分"deterministic verification tier"（语法/lint/类型/字段对齐）和"self-healing loop"（验证失败 → 反馈 → 重试）两层，缺一不可。Sonar 把这两层叫做 Loop Engineering；ACL Digital 叫 Validation Loop（来源：[Sonar](https://www.sonarsource.com/blog/loop-engineering-without-verification-is-just-automation/)、[ACL Digital](https://www.acldigital.com/blogs/the-validation-loop-building-self-healing-agents-for-migration-pipelines)、[CodeRabbit - Self-healing code](https://coderabbit.ai/guides/self-healing-code)）

## Q17: 反问环节：业务方向、岗位 KPI、AI 工程化落地

**核心句**：京东国际电商是独立 APP 的 2C 业务；岗位 KPI 是秒开 + FPS + 不出线上问题；AI 工程化在推"AI Native + 核心资产沉淀 + 抹平技术栈边界"

**展开**：面试官透露的 AI 路线：①建各业务方向、各团队自己的知识库 + 核心资产沉淀 + Skill 规约；②推 AI Native 生产范式，把一个需求涉及的多个角色压缩到尽可能少；③抹平技术栈边界——后端写前端、前端写后端。这个方向和我们 Schema + AI 工作流高度一致。**特别注意**：面试官反问"简历里体验性能质量写得少"——这是直接反馈信号，2C 岗位要看性能优化能力

## 面试官洞察（京东国际电商 AI 方向）

> 面试官逻辑极强、对 AI 工程化前沿高度熟悉，是行业前沿实践者。

### 洞察1：质询术语迁移合法性（技术纠正）

**面试官原话**："我知道模型蒸馏大模型蒸馏成小模型，你这个带个蒸馏是个什么概念？"

**价值**：行业对术语迁移很敏感。借用"蒸馏"必须有合理解释——否则会被认为是花哨包装。下次用术语迁移类词汇（蒸馏/编译/路由等），第一句话必须先解释为什么借用，避免被挑刺。

### 洞察2：溯源 ≠ 验证（认知纠正·最珍贵）

**面试官原话**："你刚刚说的证据链，那只能说明我这个 TRD 是从这个 PRD 来的。那当然是从这个片儿里来的了。" "对错和一致是两件事儿。"

**价值**：面试官的逻辑拆解能力非常强。他帮我精确定位了一个我之前混淆的设计认知——证据链（provenance）解决"从哪来"，验证（validation）解决"对不对"，是两套独立机制。这是 AI 工程化设计的根本性区分，未来讲自己工作流时必须明确分层。

### 洞察3：京东国际电商的 AI Native 路线（行业实践）

**面试官原话**："我们现在也在建各个业务方向、各个团队自己的知识库，核心资产沉淀，我们自己的这个 Skill 规约等等这些东西。也在推进这种新的生产范式，比如说 AI Native 的这种开发方式，把一些开发一个需求涉及的多个角色，我们压缩到尽可能少。"

**价值**：京东国际电商的 AI 工程化路线和我们的方向高度一致——核心资产 + Skill 规约 + AI Native。**特别有价值的判断**：行业正在从"AI 工具"转向"AI Native 生产范式"——这意味着压缩角色（PM/前端/后端/QA）的边界是行业共识方向。我的 PRD→Spec→Plan 工作流本质就是在压缩 PM/tech lead 的角色边界。

### 洞察4：抹平技术栈边界（前瞻判断·最有价值）

**面试官原话**："后端同学他没有一点前端的知识，他就可以开发前端的 APP 前端页面。我们全栈的同学，如果后端他也沉淀了对应的核心资产，我们前端同学也没有后端的相应知识，也能做后端的事情。这不就把那个让的边界给抹平了吗？"

**价值**：这是行业最前沿的判断——AI Native 的终局不是"AI 帮前端写前端"，而是"AI 让前后端边界消失"。这个判断对我的职业规划有直接影响：
- 前端工程师的护城河不再是"会前端"，而是"会某一类问题的端到端解决"
- 我现在做的 Schema 中台 + AI 工作流本质就是在抹平前后端边界，方向是对的
- 未来 1-2 年应该主动补一些后端/全栈经验，否则会被"抹平"

### 洞察5：京东国际电商岗位核心 KPI（实用信息）

**面试官原话**："首先最基本的是不出线上问题。性能体验好，这是最基本的。秒开，FPS 高。"

**价值**：京东国际电商是 2C 业务（独立 APP），核心 KPI 是性能体验质量——秒开、FPS、稳定性。这和滴滴国际化 B 端中台是完全不同的能力模型。如果进这家公司，需要补：①移动端性能优化（FPS/秒开/Service Worker），②2C 业务的质量保障体系。

## 表达诊断

- 口头禅：粗略估"然后"出现 30+次、"对吧"出现 20+次、"嘛对吧"出现 15+次、"维也纳相当于相当于"出现 5+次
- 回答平均长度：偏长，被面试官打断 5 次（"我打断一下"、"我比较想了解的是"、"我现在不关心"、"我没懂你说的"、"我懂你意思但是"）
- 结构性：弱——多数回答没有"开头-展开-收尾"，是流水账

## AI 工程化 3 硬伤扫描

- ① **数据被追穿** ⚠️ 中度：40%→85% 一致性被追"幂等怎么保证"，绕了 4 个回合才说出"plan 精确到文件维度"。下次必须备 3 套话术：怎么测的（黄金样本+10次跑）/ 怎么算的（文件覆盖率+章节完整度）/ 怎么稳的（确定性脚本+锚点校验+XML约束）
- ② **原理卡壳** ⚠️ 轻度：被追"蒸馏"概念借用合理性时绕了，没正面解释为什么借用
- ③ **闭环回避** ⚠️ **重度**：Q16"代码怎么验证"开篇说"这是端到端链路"——直接被面试官打断"就算不是端到端，最后一个环节你怎么判断"。没主动讲端到端原型 prd2code-gen 完整 6 步——这是最严重的失分

## 今晚必做 3 件事

1. **30 分钟练"一句话定义"**：把"蒸馏/证据链/双路径/震荡检测"四个术语各练 1 句话定义 + 1 句话解释为什么借用这个词。下次面试第一句话必须正面答
2. **30 分钟练"provenance vs validation"分层话术**：把"溯源 ≠ 验证"的认知写成一张卡片的背诵稿，主动讲证据链时同时讲清楚它只解决 provenance，validation 靠另一套机制
3. **30 分钟练"端到端 6 步完整闭环"开口讲**：把 prd2code-gen 6 步（蒸馏 → 分析 → Plan → 双路径生成 → 自验证自修复 → eval/auto-tune）连讲 3 遍，第 3 遍控制在 90 秒内讲完。被问任何代码验证/闭环问题时第一句话就上这套

## 下次面试注意

1. **正面回答优先**：被问"X 是什么"，第一句话必须是定义；被问"怎么保证"，第一句话必须是机制名。绝不能绕背景
2. **主动 acknowledge 边界**：被问到工作流不擅长的场景，第一句话 acknowledge "这是边界外"，再讲边界外怎么 fallback。不要被动防守
3. **闭环问题不回避**：被问代码生成/验证/CI 对齐，第一句话就上"端到端原型已跑通 6 步闭环 + 团队推广版聚焦 PRD→Plan 是 ROI 选择"。不要用"老板让先做前两步"回避
