# opendesign 一面 — 面试优化答案

> 日期：2026-07-21 | 通过概率：约 15%（理由见文末整体评估）
> 公司：OpenDesign（AI 设计工具，方向 = 网站/UI 模式抽离复用，代表功能"灵感按钮"）
> 候选人主打：两个个人 AI 项目 —— Manager Agent（39节点 multi-agent 编排）+ Coding Co-Assistant（LangGraph，18内置工具）

---

## 核心教训（面试官当面定调）

面试官原话："**就感觉你准备好像不太充分了，很多问题好像没准备好。**" 技术问答仅 4 题约 10 分钟即结束。病根不是知识盲区，是**两个个人项目完全没讲清楚 + multi-agent 生态认知停在产品体验层**。用个人项目打 AI 岗，却栽在"项目表达"上。

---

## Q1: 有没有对比过主流 multi-agent 产品？（Claude Code 的 dynamic workflow，是固定状态机还是 AI 自主判断调度？）

**核心句**：我的 Manager Agent 是 LangGraph 那种固定状态机——节点和边我预先定义死，好处是可观测、可重放。

**展开**：主流现在分两派。一派是 Claude Code 的 **dynamic workflow**——模型运行时自己写一段 JS 脚本来编排 subagent，最多上千个、十几个并行，脚本本身持有计划/分支逻辑，灵活但不确定性强，适合探索、迁移、审计这类一次性大任务。另一派是 **LangGraph 式固定状态机**——我把流转逻辑写死在图里，确定性强、可调试、有完整状态快照，适合需要稳定复跑的生产链路。我的项目选后者，是想练"显式控制流"这一套；但要承认，dynamic workflow 把编排权交给模型的方向，是现在的趋势。
（*来源：code.claude.com/docs/en/workflows；Towards AI "dynamic workflows quietly killed my LangGraph stack"；面试官 Q1 暗示"模型自主调度"是方向*）

## Q2: 为什么是 39 个节点？而不是简单任务、甚至根本不需要这么多？

**核心句**：39 个是学习目的堆出来的上限，真要上生产我会砍到 5–8 个核心节点。

**展开**：这项目本质是学习练手，我把每个学到的能力——多路由、状态机、子任务拆解——都做成独立节点试了一遍，所以膨胀到 39 个，坦白讲有过度工程的成分。如果重做，我会按"感知→规划→执行→校验"收敛成 5–8 个核心节点，剩下的能力下沉成工具调用。节点数本身不是优势，**编排的清晰度和可收敛性才是**——这点 Claude Code 的 dynamic workflow 思路反而更轻。

## Q3: Coding Co-Assistant 是 LangGraph 工具，18 个内置工具是不是太多了？

**核心句**：核心是 5 步主线，剩下十几个是这 5 步的细分工具。

**展开**：18 个看起来多，但主线就 5 步：读仓库摸结构 → 语义检索定位相关代码 → 静态分析（AST 解析 + 依赖追溯 + 调用图）理解上下文 → 出 diff → 写盘。剩下的是每一步的细分工具，比如静态分析下挂着 AST、依赖、调用图三个。**工具一多，模型选错工具的概率会上升**——这是我下一步想优化的：高频工具显式描述、低频的合并。这项目和 Manager Agent 一样是学习目的，所以刻意把了解到的东西都铺开试了一遍。

## Q4: 有没有学习过成熟开源的 coding agent / multi-agent 框架？（头号失分点，必背）

**核心句**：开源 coding agent 我关注 OpenHands，编排框架关注 LangGraph / CrewAI / OpenManus。

**展开**：分两层看。**coding agent 层**，OpenHands（原 OpenDevin）是开源标杆——model-agnostic、有 stateful bash、MIT 协议、All-Hands-AI 维护，我读过它的 agent runtime 和事件流设计；闭源里 Cursor 的代码库索引/检索能力确实强，但我只能停在产品体验（面试官追问"能看到源码吗"，这是 AI 岗的档次分水岭，下次至少读 1 个开源实现）。**编排框架层**分两派：LangGraph 是显式状态机图，控制流清楚、能上生产；CrewAI 是角色分工团队，上手快、star 最多；OpenManus 和 SmolAgents 走轻量动态路线。还有该补的——Claude Code 的 **dynamic workflow**，让模型自己写编排脚本，和状态机完全相反的思路，代表新趋势。
（*来源：OpenHands GitHub/openhands.dev；dev.to "LangGraph vs CrewAI vs AutoGen 2026"；面试官反问环节点名 OpenHands/OpenManus/Genie*）

## Q5（反问）：对我有什么建议？

**核心句**：项目讲述条理性 > 项目复杂度；AI agent 方向必须持续追新。

**展开**：面试官给的三条都值得刻进肌肉记忆——①任何面试，项目介绍都是超重点考察，要有条理有组织地讲亮点；②持续追新框架（OpenHands/OpenManus/Genie），这块迭代极快；③看得出来面试官更认可"模型自主调度"的动态编排方向，而不是固定堆节点。

## Q6（聊产品）：对 OpenDesign 产品的看法（全场唯一答得顺的）

**核心句**：最亮点是"灵感按钮"——内置浏览器把好网站按固定模式抽离复用。

**展开**：我之前就在用 OpenDesign，最打动我的是灵感按钮：它能在内置浏览器里把你认为好的网站，按固定模式抽离成可复用的设计资产。这反衬出一个事实——我不是不会表达，是项目没准备好就上了。

## 面试官洞察（opendesign · 主面试官）

> 核心定调：项目表达 + 行业视野是 AI 岗核心考察，候选人两者都没准备好。

### 洞察1：项目讲述的条理性，比项目复杂度更重要
**原话**："要多准备怎么介绍和表达自己做过的项目……不管什么面试都会非常重点考察。深入挖掘亮点，有条理、有组织地讲出来。"
**价值**：通用面试铁律。本次正好反了——项目复杂却讲成流水账。任何项目都要先备好"痛点→方案→亮点→取舍"四段式。

### 洞察2：AI agent 方向，必须主动追新，迭代极快
**原话**："参考更新的项目，OpenHands、OpenManus、Genie 这些 multi-agent 框架……这块迭代很快，有很多新东西值得学。"
**价值**：AI 岗隐性门槛——期待你持续追新。AutoGPT 已过时，OpenHands/OpenManus/Claude Code dynamic workflow 才是当下，高频考点。

### 洞察3：编排范式正从"固定堆叠"转向"模型自主调度"
**原话（Q1）**："dynamic workflow……它动态调度多个 agent，是 A 自己判断要不要调其他 agent 去解决问题。"（对照你的 39 节点固定编排）
**价值**：面试官暗示你的方向偏旧。趋势是 dynamic workflow / 自主调度（搜索印证：Claude Code 让模型运行时写编排脚本）。项目要主动讲"我知道固定状态机的边界，也知道动态调度是趋势"。

### 洞察4：用产品 ≠ 读源码，是两个档次
**原话（Q4追问）**："企业版你能看到源码吗？还是单纯产品层面体验？"
**价值**：AI agent 岗硬区分"用过"和"懂原理"。答"产品体验"直接降档。备框架至少读 1 个开源实现（OpenHands 最佳）。

### 洞察5：OpenDesign 方向 = 网站/UI 模式抽离复用
**共识**：OpenDesign"灵感按钮"——内置浏览器把好网站按固定模式抽离复用。
**价值**：这家方向是设计生成，和你的 coding agent（代码生成）是相邻赛道。后续聊可往"设计稿→代码"衔接靠，是差异化切入点。

## 整体评估（见复盘报告第四步）

- **通过概率**：约 15%。面试官当面定调"没准备好"+ 技术问答 10 分钟即结束 + 核心考察点（项目表达/行业视野）双失分。唯一加分：产品环节答得有细节、态度好、多面官面试。
- **最大失分点**：Q1（编排范式空白）+ Q4（开源框架只说两个、还停在产品体验）。
- **最大加分点**：Q6 产品环节有具体细节（灵感按钮），证明表达力没问题，纯粹是准备不足。
