## Q4:Directive 联动语法具体是什么?(机制弹药)

- **表面**:讲联动语法的实现
- **实际考察**:能不能讲清单点机制的实现细节(数据结构 + 求值 + 重渲染)
- **答到的**:前端自闭环(visible 三元/模板字符串)+ 发布订阅监听、后端业务联动(拉接口)、监听后重渲染
- **漏的**:
  - ① **术语含糊**:把依赖收集字段说成 "sense"(应是 deps 类),把重渲染说成 "CVA/CVR"(应是 re-render / re-validate)。面试不说内部缩写,说人话。
  - ② **两类联动的边界没讲清**:纯前端求值(不碰后端)vs 需后端数据(拉接口),这是 Directive 设计的核心区分,该一句话点透。
  - ③ **发布订阅的 why 没说**:为什么用发布订阅——联动是"一处变,多处响应",天然解耦依赖方和被依赖方。

| 维度 | 分数 | 问题 |
|------|------|------|
| 知识完整度 | 7 | 机制清楚,术语含糊 |
| 表达清晰度 | 6 | CVA/CVR/sense 听感乱 |
| 追问防御力 | 8 | 接住了 |

**改进版**:
- **核心句**:"Directive 是描述'A 组件变化如何影响 B 组件'的联动语法,分两类——纯前端求值的、和需要拉后端接口的。"
- **展开**:
  1. **前端自闭环联动**:A 影响 B 的属性(显隐 / 内部属性),用模板表达式描述结果,比如 `visible = 依赖项 == 某值 ? true : false`。依赖监听靠**发布订阅**:被依赖项的 name 路径收集进 deps,最外层订阅中心监听变化,变化后求值→组件重新渲染、校验。全程不碰后端。
  2. **后端业务联动**:像"城市改了,后续一堆配置要变"这种强业务联动,监听后去拉后端的联动接口,拿返回值拼成新的 schema 再重渲染。
  3. **为什么发布订阅**:联动本质是"一处变、多处响应",发布订阅把依赖方和被依赖方解耦,新增联动不用改被依赖方代码。

## Q10:三个组件 A(父)/B/C 的 useEffect 挂载 & 卸载回调执行顺序

- **表面**:useEffect 父子执行顺序
- **实际考察**:fiber 调度 + passive effect 机制的硬核理解(经典坑题)
- **答到的**:最终落到"挂载 B→C→A,卸载 B→C→A,一样的"——**结论其实对了**
- **漏的**(纯交付问题):
  - ① **答得磕磕绊绊**:ABC → BCA → CCA → "一样的",面试官不得不喂("CCA 卸载 BCA,对")。这是**本场最明确的失分点**:结论对,但过程看着像在猜。
  - ② **没讲原理**:为什么子先于父——fiber 深度优先遍历,passive effect 子优先 fire。

| 维度 | 分数 | 问题 |
|------|------|------|
| 知识完整度 | 7 | 结论对 |
| 表达清晰度 | 3 | 磕绊,要面试官喂 |
| 追问防御力 | 5 | 不自信 |

**改进版**:
- **核心句**:"挂载和卸载都是子先于父——挂载 B→C→A,卸载 cleanup 也是 B→C→A。"
- **展开**:
  1. **挂载**:B 的 effect → C 的 effect → A 的 effect。子先于父。
  2. **卸载 cleanup**:同样是 B 的 cleanup → C 的 cleanup → A 的 cleanup,与挂载同向。
  3. **原理**:React 提交后刷 passive effect 时,用 `recursivelyTraversePassiveMountEffect` **深度优先遍历 fiber 树**,所以最深层的子组件 effect 先 fire,再冒泡到父。Strict Mode 的双调用(mount→cleanup→remount)日志可以直接验证这个顺序。
  4. **一个常被追的点**:依赖变更时,cleanup 跑在**下一次 effect 之前**(不是 unmount 才跑)。
- **依据**:React 官方 issue [#15281](https://github.com/facebook/react/issues/15281)(团队确认 effects fire children-first)。历史 issue #16728 曾讨论 cleanup 方向,现代 React 实测(Strict Mode)为子优先同向。
- **下次怎么答得稳**:先甩结论"都是子先于父,BCA",再说原理"深度优先遍历",别现场推导。

## Q13:HTTP 缓存机制 + 前端工程怎么用 + 301 会缓存吗

- **表面**:HTTP 缓存全貌
- **实际考察**:缓存策略 + 工程落地 + 一个反直觉点(301 缓存)
- **答到的**:强缓存 / 协商缓存 / 304 / 执行顺序、静态强缓存入口协商缓存、**301 会被缓存(对!)**
- **漏的**:
  - ① **字段名含糊**:"身份认证""备份资源"——协商缓存的校验字段是 **ETag / If-None-Match、Last-Modified / If-Modified-Since**,要能脱口而出。
  - ② 强缓存的 header 没说全:**Cache-Control: max-age** / **Expires**。
  - ③ **301 这题答对了,是加分点**,但可以更狠:301 默认**永久缓存、无过期**,改了重定向访客仍跳旧地址、服务端没法 remotely 清——所以测试期用 302、正式 301 加 `Cache-Control: no-store`。

| 维度 | 分数 | 问题 |
|------|------|------|
| 知识完整度 | 7 | 字段名含糊 |
| 表达清晰度 | 6 | "身份认证"是错的概念 |
| 追问防御力 | 8 | 301 答对加分 |

**改进版**:
- **核心句**:"分强缓存(不发请求读本地)和协商缓存(带校验字段问服务器),命中顺序是强→协商;301 会被浏览器永久缓存。"
- **展开**:
  1. **强缓存**:不发请求,直接读本地磁盘。header:`Cache-Control: max-age`、`Expires`。命中返回 200(from disk/memory cache)。
  2. **协商缓存**:强缓存过期后,请求带校验字段(**ETag / If-None-Match**、**Last-Modified / If-Modified-Since**),资源没变返回 **304**(继续用本地),变了返回 200 + 新资源。

  4. **301 会缓存(对)**:301 默认被浏览器**永久缓存、无过期**,改了重定向访客仍跳旧地址、服务端没法远程清。抑制用 `Cache-Control: no-store`,测试期用 302 更安全。

## 面试官洞察(动念涌现 · 运营中心)

> 动念涌现是 AI 创业团队,核心 AI 工作流处于**验证期**、还没变现;公司正转型 **AI Native 组织**,推团队全栈化。面试官整体评价"比较好"。

### 洞察 1:创业团队 AI 工作流普遍处于"验证期、难变现"

**面试官原话**:"如果是按传统的步骤来说,可能现在还在一个验证的阶段,就是还没有去那些验证 AI 工作这种,我们就非常非常难变现回去。就现在还处于一个验证的一个阶段。"

**价值**:这是一个**重要的行业现实校准**——连全力做 AI 的创业团队,都还没跑通"AI 工作流→变现"的闭环。说明:① AI 工程化目前普遍是"提效/降本"逻辑,不是直接变现逻辑,面试谈收益时别吹"直接创造多少营收",讲提效 + 降本 + 经验沉淀更稳;② 你 prd-tools 的"团队推广效果不好"其实和行业现状一致(验证期),可以更坦然——不是你做不好,是这事儿整个行业都在摸索。

### 洞察 2:公司在转型 AI Native 组织 + 推全栈

**面试官原话**:"我们公司这边再转一个 AI Native 的组织,再做了一些关联整个组织的、跨团队的知识库的建设。还有一些比如说团队,各个团队去转全栈,去做一些 app 呀后端的东西……我们现在会推荐 app 后端的东西……非常有侵略性的这么一个验证期。"

**价值**:
- ① **AI Native + 跨团队知识库建设** = 和你 prd-tools 的 reference 领域知识库**高度同构**。反问时如果深挖,可以接"你们跨团队知识库怎么解决多项目共享 / 上下文隔离"——这正是你团队模式(三方 reference 放公共仓库 + API 调用)的经验点。
- ② **团队转全栈(app + 后端)** = 明确的能力信号:他们要的是能跨界的前端。你简历有 BFF + 后端协作经验(reason-tag 报警分类就是和后端协作),可以主动亮。
- ③ **"有侵略性的验证期"** = 价值观信号:偏好能 drive、敢试、敢验证的人。你的"一个人主导发起插件 / 独立设计两个 agent"正好踩在这个点上,讲项目时 ownership 要立足这个。

### 洞察 3:面试官整体评价正面

**面试官原话**(你问建议时):"有不熟的地方。我觉得整体下来,评价还是比较好的……没感觉,就是整体下来还是比较好的。"

**价值**:温和型面试官 + 正面评价 = 大概率进下一轮或拿到一个中等偏上的反馈。但别乐观——"评价比较好"在这种温和风格下**信息量有限**,真正的 HC 决策看横向对比。所以你的 storytelling 问题即使没被当场指出,也可能在横向对比时吃亏。**这正是为什么要主动把项目讲漂亮**,不靠面试官逼。

### 改进版答案二次增强(融入面试官洞察)

- **Q1 prd-tools**(融入洞察 1):讲团队推广局限时,从"自我矮化"改成"**行业现实**"——"AI 工作流→变现整个行业都还在验证期(动念这种全力做 AI 的团队也没跑通变现),所以我们的定位很明确:团队版聚焦 PRD→Plan 提效,端到端原型作为探索"。这样既诚实、又不显得是"做不好才收窄"。
- **Q15 看机会**(融入洞察 2/3):收口句用反问信息反向锚定——"贵司在 AI Native 转型 + 推全栈 + 验证期,正是我想验证复杂 AI 场景、也想往 app/后端拓展的地方,匹配"。
