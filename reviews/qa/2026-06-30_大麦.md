## Q4: Schema 中台技术方案（核心题）

**核心句**：三层架构——BFF 用 Handlebars 模板引擎把活动元数据编译成 JSON Schema，前端按 Schema 声明式渲染；最核心是自研的联动 DSL，分 fetch 型和纯前端型两类。

**展开**：
① **数据层**：后端 API 给原始活动元数据；
② **编译层**：Node.js BFF 用 Handlebars 模板（或 JSON Schema 派生）把元数据编译成标准化的 JSON Schema，包含字段定义、校验规则、联动声明；
③ **渲染层**：前端 SchemaRenderer 按声明式渲染，不写 if-else。

**联动 DSL 是核心**，分两类：
- fetch 型：`{ when: 'activityType', fetch: '/api/coupons' }` —— 字段变化调接口
- 纯前端型：`{ when: 'country', set: { '#couponList.visible': '{{$eq country, "BR"}}' } }` —— 闭环在前端

**自检数字**：20+ 活动类型、新增活动改 Schema 不改代码、上线周期 2 周→3 天（口径：模板复用率 70%+，剩余 30% 是模板开发）。

**失分教训**：面试官**两次说"没听清"**——表达问题严重。"联动"这个核心概念第一次没讲清楚。**改进**：表达要从**具体场景举例**开始（"比如运营选了'订单奖'，下一个'奖励规则'字段才出现"），再讲语法和分类。

## Q5: 怎么解决依赖包大小问题？

**核心句**：从构建时、加载时、运行时三层做减法——构建时 tree shaking + 按需引入、加载时路由分割 + 动态 import、运行时 CDN 外置 + 轻量库替换。

**展开**：
① **构建时**：tree shaking（依赖 ESM 静态分析，sideEffects: false 标记纯净包）+ babel-plugin-import 对 antd/lodash 按需引入；
② **加载时**：webpack SplitChunksPlugin 把 vendor 分包、路由级 lazy import、首屏不用的代码延后；
③ **运行时**：moment→dayjs（230KB→2KB）、把 React 等基础库走 CDN externals 不打进包；
④ **监控**：bundle-analyzer + size-limit 卡红线，PR 超阈值直接拦。

## Q6: Schema 中台最大技术难点？

**核心句**：最大难点是联动 DSL 的"依赖图"+ 性能控制——一个字段变化可能触发多跳联动，要做拓扑排序 + 增量更新才能避免风暴。

**展开**：联动表面看是"A 影响 B"，实际是个**有向图**——A 影响 B，B 影响 C，甚至有循环（A 影响 B 又被 B 反向影响）。
三个具体难点：
① **依赖图建模**：用邻接表存字段依赖关系，change 时 BFS/DFS 找影响范围；
② **循环检测**：编译期静态扫描拒绝循环依赖，运行期用 visited Set 防护；
③ **性能**：批量 change 合并成一次更新（类似 React batch），增量 re-render 只跑受影响的字段。
这套做完后，运营加新联动只要写 5 行 DSL，不用发版。

**失分教训**：原答案"写着写着把自己陷进去"是感受不是难点——面试官想听的是**具体工程问题**（依赖图、循环、性能）。

## Q7: AI 工程工作流（核心题）

**核心句**：自研双 Skill 工作流——一个 Skill 构建项目领域知识库，一个 Skill 把 PRD 蒸馏成 spec+plan，靠 SSOT 六文件 + 证据链 + 三级门控保质量，单需求工时 5h→2h。

**展开**：
① **痛点**：PM 写 PRD → 研发口口相传 → 信息损耗 → 返工；隐性知识在老员工脑子里，新人上手慢。
② **方案**：两个 Claude Code Skill——
   - 知识构建 Skill：扫描历史项目，输出领域知识库（Reference 六文件 SSOT）
   - PRD 蒸馏 Skill：读 PRD + Reference，产出 spec（技术方案）和 plan（开发计划），每条都带 `verified_by` 证据链（file:line 溯源）
③ **质量保障三板斧**：SSOT（单一事实源，杜绝 AI 编文件）+ 证据链（每条结论必须挂在真实代码上）+ 三级门控（蒸馏→生成→Review 三道 review）
④ **收益口径**（必背）：工时 5h→2h 拆解——
   - 原 5h = 读 PRD 1h + 找历史代码 2h + 写技术方案 1.5h + Review 0.5h
   - 现 2h = AI 跑蒸馏 0.5h + 人工 Review 调整 1h + 落地 0.5h
⑤ **双轨定位**（必背，应对"既然完整为啥不推广"）：当前团队推广版聚焦 **PRD→Plan** 是 ROI 选择（人工确认后才开发、风险低、提效明显）；端到端原型 prd2code-gen 已设计跑通作为下一代探索，涉及代码生成质量责任和 CI 对齐是更重的话题，先用原型验证设计可行性再决定工程化投入。

**失分教训**：① 没讲双 Skill 体系的具体职责；② 核心机制（SSOT/证据链/门控）全漏；③ "vibe coding 前置"答法回避端到端 → 触发 AI 工程化硬伤③；④ 5h→2h 没准备口径 → 触发硬伤①。

## Q12: Promise.all 实现

**核心句**：返回新 Promise，遍历用 Promise.resolve 包裹每项存结果到数组，计数器=长度时 resolve；任一 reject 立即 reject 整体（fail-fast）。

**展开**（伪代码）：
```js
function promiseAll(list) {
  return new Promise((resolve, reject) => {
    const results = new Array(list.length);
    let count = 0;
    if (list.length === 0) return resolve([]);
    list.forEach((item, i) => {
      Promise.resolve(item).then(
        (val) => { results[i] = val; if (++count === list.length) resolve(results); },
        reject  // 任一失败立即 reject（fail-fast）
      );
    });
  });
}
```

**关键点**：① 用 `Promise.resolve(item)` 兼容非 Promise 值；② `results[i]` 保序（不能用 push）；③ fail-fast 是 all 和 allSettled 的本质差别（allSettled 不 reject，全部等完）；④ 空数组立即 resolve。

**失分教训**：原答案漏了 fail-fast + 空数组 + allSettled 对比——三个关键点。

## Q14: 最大挑战（核心题）

**核心句**：最大挑战是 AI 工程工作流的架构选型——两周 MCP 调优撞墙后转向 agent+skill，理解了"单次调用 vs 多轮上下文"的能力边界。

**展开**：
① **挑战背景**：项目主动发起，前后端老板都关注，每天盯进度，压力来自架构设计 + 精神双方面；
② **核心难点**：怎么让 AI 理解已有项目需求；
③ **走弯路**：一开始用 MCP（Function Calling），调优两周准确率卡在 60% 上不去；
④ **诊断**：MCP 本质是**单次 request-response**——AI 调一次工具拿一次结果，没有持续上下文；而需求分析需要"读代码→思考→再读→验证→再读"的多轮上下文，这是 MCP 的能力天花板，调优解决不了；
⑤ **转向**：换成 Claude Code Skill + Agent Loop——Skill 提供领域知识和工具，Agent 维护多轮上下文；自研六文件 SSOT（Reference）+ 多轮搜索证据链（每条结论挂在真实代码 file:line 上）；
⑥ **能力上限认知**：MCP 适合"取一个事实"（查字段、查 API），Skill+Agent 适合"做一个判断"（理解、推理、产出文档）；
⑦ **收获**：现在我对什么场景用 MCP / Function Calling / Skill / Agent 有清晰的选型判断。

**失分教训**：原答案"MCP 单次调用"是对的但深度不够——应该补"取事实 vs 做判断"的选型方法论。

## 面试官洞察（大麦国际前端团队）

> 大麦国际是去年成立的新部门，没有具体业务线划分，所有业务都做（C 端为主），团队规模不大，业务跟着公司战略走。

### 洞察1：大麦国际组织结构（反问1）

**面试官原话**：「我们没有具体的业务线。所且其实我们的团队目前没有特别大，能跟国内的团队不太一样。因为我们是一个去年成立的新部门吧。你可以认为我们是所有业务线都要做的。」

**价值**：① 大麦国际是新部门（去年成立），组织扁平，机会多但要适应"业务在哪儿就去哪儿"；② 没有"营销团队/AI 团队/C 端团队"这种细分——一个人要 cover 多个业务；③ 团队规模不大，进来后个人影响力容易放大，但也意味着没有专门方向深耕。

### 洞察2：业务定位（反问1）

**面试官原话**：「我们一个 C 端的业务，B 端的会相对会少一些」「大多数精力都是放在 C 端」「相当于是就做港澳台加海外的一个业务，就是不做国内」「APP 一般是用原生开发，或者是用跨平台的技术，前端参与的话，可能只是一些内嵌的处理页面」

**价值**：① 业务重心是 **C 端 + 港澳台/海外**——这和候选人滴滴国际化背景**高度对口**（这是这场面试最大的优势点，自我介绍要更突出这点）；② B 端少，意味着候选人之前在 B 端配置化的经验虽然能加分但不能直接复用——需要适配 C 端；③ APP 端用原生/跨平台，**前端主要做内嵌 H5**——这意味着前端要懂 WebView 交互、JSBridge、跨端通信。

### 洞察3：工作节奏（反问2）

**面试官原话**：「现在还是处于一个高速发展的一个阶段，业务的会有大批量的业务需求」「to C 比较多，所以就是改动会比较多，迭代会也会比较快」「来了之后才会确定，也跟业务发展有关系」

**价值**：① 高速发展期 = 业务需求密集 = 适合有工程化提效能力的人；② 候选人"AI 工作流 5h→2h"这种提效类项目应该重点突出——直接对上大麦的痛点；③ 不要期待有清晰的"前半年规划"——意味着候选人需要有较强的自适应能力。

### 洞察4：面试官考察风格

**观察**：面试官全程问得**简短**（"具体说一下""能说一下吗"），**几乎没有深挖追问**——SSR（Q3）、模态框（Q11）、Promise.all（Q12）三题都没追问，**大概率是已经放弃**——这三块是失分点。

**价值**：复盘时要重点补 SSR、基础组件封装、Promise 系列手写题——下一轮可能继续考。
