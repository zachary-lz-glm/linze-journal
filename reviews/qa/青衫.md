## Q2:你们是"解决了复杂度"还是"把代码复杂度转移到了配置复杂度"?

- **表面**:二选一判断
- **实际考察**:有没有想清楚重构的本质——是消除复杂度还是搬运复杂度(这是面试官最高质量的一问)
- **答到的**:选了"解决",理由是原先 if-else 看不见配置链路,现在一页 JSON 能看清
- **漏的**:没点透核心区别——**隐式复杂度 vs 显式复杂度**。这是这个问题的标准答案钩子。

| 维度 | 分数 | 问题 |
|------|------|------|
| 知识完整度 | 7 | 方向对,但没说到"隐式→显式"这个本质 |
| 表达清晰度 | 6 | 有点绕 |
| 追问防御力 | 8 | 站住了 |

**改进版**:
- **核心句**:"是解决了——原先的复杂度是隐式的(藏在 if-else 里看不见),现在是显式的(声明式、可静态分析)。"
- **展开**:不是搬家,是质变。隐式复杂度有三个坏处:看不见、不可静态校验、一改就发散;显式复杂度(声明式配置)对应三个好处:一页协议就能看清组件依赖、能在运行前静态校验、改动收敛在局部。复杂度总量可能没少,但**从"不可控"变成了"可控"**,这就是解决。

## Q5:做特定营销事件给特定人群配置,如何让配置人员确信配置正确?

- **表面**:AI 辅助预校验怎么做
- **实际考察**:能不能系统性拆解一个 agent 工程化问题(配置前/中/后三段)
- **答到的**:三段式完整——配置前(Superpowers 式反查/歧义消除、主动质疑、模糊需求翻译成精确规则让人确认)、配置中(实时冲突检测、预注入 reference、后台静默扫描报警)、配置后(沙盘推演)
- **漏的**:① "Superpower / Bimad"表述含糊,Bimad 是口误(应该是 v0/Lovable 类工具);② 没明确点出贯穿三段的核心原则——**"不是扩张 AI 能力,是让范围越来越精确"**(其实你后半句说了,但可以提到更显眼位置作为总纲)。

| 维度 | 分数 | 问题 |
|------|------|------|
| 知识完整度 | 9 | 三段式完整,有总纲 |
| 表达清晰度 | 7 | Bimad 口误 |
| 追问防御力 | 9 | 为后续 Q6-Q8 铺好了路 |

**改进版**:
- **核心句**:"配置前/中/后三段都注入 agent,但贯穿的核心是——不是扩张 AI 能力,是让配置范围越来越精确。"
- **展开**:① **配置前**:歧义消除 + 反查圈选。配置人用自然语言说"给活跃用户发券",agent 反查数据把"活跃用户"翻译成精确的人群规则,把模糊需求变成可确认的精确逻辑(青衫面试官观点:这一步本质是提高"配置正确"的概率)。② **配置中**:预注入历史规则/边界 case 作为 reference,agent 在后台静默扫描配置逻辑漏洞,预算/数量/阈值这类实时数据自动校准。③ **配置后**:沙盘推演——不直接发布,基于历史真实数据在沙箱里虚拟跑测。

## Q7:AI 和人都会遗忘 case,没注意到就不检索,怎么强制 AI 一定检查?

- **表面**:怎么让 agent 不漏检
- **实际考察**:对 LLM"面向概率、会遗忘"本质的认知 + 工程化兜底手段
- **答到的**:AI 是黑盒、记忆易遗忘;在流程中加强制步骤(需求澄清+合规检查);类比 Superpowers 比 OpenSpec 好(跑得久但质量高,因为有大量中间脚本/自验证)
- **漏的**:① "Superpowers 比 OpenSpec 好"这个对比有点模糊,要能讲清 Superpowers 具体好在哪(强制 brainstorming→plan→TDD 的流程骨架,每步有 checklist gate);② 没强调"强制步骤"的本质是**把'靠模型自觉'改成'靠流程骨架强制'**。

**改进版**:
- **核心句**:"不能靠 prompt 提醒——模型会遗忘;要把检查变成流程骨架里的强制步骤,不跑完不准往下走。"
- **展开**:Superpowers 的做法就是模板:它强制你先 brainstorming 再 plan 再 TDD,每一步都有 checklist gate,不勾完不让下一步。这就是"拿时间换质量"。核心转变是:从"指望模型记得要检查"变成"流程结构逼着必须检查"。

## Q8:怎么让 agent 一定检查?检查主体是谁?代码和 agent 谁是主动方?他不是也可能忘记调用?(连环追问)

- **表面**:强制执行的工程实现
- **实际考察**:对 agent 可靠性工程的底层理解——确定性代码 vs 概率模型谁主导
- **答到的**:检查通过确定性脚本执行能看到过程、agent 调用代码(主体是大模型调工具)、要一定执行就硬编进系统骨架+加网关拦截;通用框架(CC/Codex)的做法列了权限/hook/复查/门禁
- **漏的(关键)**:① **没强调 PreToolUse hook 是"可阻塞"的**——这是面试官追问的真正答案。你列了 hook,但没说清楚"工具执行前拦截、返回非零即阻止"这个硬阻塞能力,这是 Claude Code 里唯一能强制拦截的环节(PostToolUse 只能记录不能阻塞);② **没提 subagent 不继承主 agent 权限**——所以 hook 是 subagent 隔离的关键强制层;③ "加一层网关拦截"说得有点抽象,其实就是 hook,直接点破更好。

| 维度 | 分数 | 问题 |
|------|------|------|
| 知识完整度 | 7 | 方向对但没点到 PreToolUse 这个核心机制 |
| 表达清晰度 | 6 | 网关/hook/权限列了一堆,没分主次 |
| 追问防御力 | 7 | 接住了但有点拼凑感 |

**改进版**(搜索增强):
- **核心句**:"让 agent 一定执行,不能靠 prompt,要靠确定性代码在工具执行前拦截——Claude Code 的 PreToolUse hook 就是干这个的,返回非零直接阻塞。"
- **展开**:agent 可能遗忘调用,所以不能依赖模型自觉,要在 agent 和工具之间加确定性拦截层。Claude Code 的具体机制:① **PreToolUse hook**(工具执行前跑脚本,返回非零即阻塞,这是唯一能"硬阻塞"的环节;PostToolUse 只能记录日志不能阻塞——来源:Claude Code hooks 官方文档);② **permissions**(allow/deny/ask 白名单);③ **独立上下文的 subagent 做复查**(复查 agent 不共享主会话上下文,见 Q10)。有个关键坑:**subagent 不自动继承主 agent 的权限**,所以 PreToolUse hook 是 subagent 行为强制的最后一道闸(来源:GitHub issue #40580 讨论的正是这个 gap)。
- **一句话收口**:"代码和 agent 谁是主动方?——agent 是发起方,但确定性代码(hook)是裁判,关键操作必须 hook 拦截,不能让模型自己决定执不执行。"

## Q9:讲讲你的门禁机制怎么设置?

- **表面**:讲自己的质量门控设计
- **实际考察**:你自己的 prd-tools 工作流里质量保障的真实深度
- **答到的**:三支柱——SSOT、证据链(每条结论带 version ID 来源)、质量门控(事前 step 标记、事中中间文件校验、事后 agent 五维自评 + 确定性脚本验证证据链连通性)
- **漏的**:结构讲得不错,但"事后 agent 自评"可以明确点出这是 LLM-as-judge 模式(和 Q10 的上下文隔离呼应)。

**改进版**:
- **核心句**:"三个支柱:SSOT 单一事实源、证据链(每条结论带来源 ID)、质量门控(事前标记/事中校验/事后自评+脚本验证)。"
- **展开**:事前——给每步 step 打特定标签,告诉 AI 这步该干/不该干什么;事中——校验每步产出的中间文件,判断上一步是否真的完成;事后——两套并行:① agent 自评(LLM-as-judge 模式,reasoning score 五维度评分,**用独立上下文**,见 Q10);② 确定性脚本验证证据链能不能连上、结论和上下文的引用符不符合预期。

## Q10:校验任务 LLM 不共享输出 LLM 上下文,为什么不能共享?

- **表面**:上下文隔离的理由
- **实际考察**:对 LLM-as-judge 偏置/污染的理解
- **答到的**:三点——① 公正性/防幻觉(裁判被污染会误判)、② 防提示词注入(越狱内容污染校验)、③ 成本(token 爆炸)
- **改进版**:这道答得最好,三点齐全。补权威佐证提升说服力:这是 LLM-as-judge 的公认最佳实践,业界叫 **context contamination / evaluator bias**(来源:Evidently AI 的 LLM-as-judge 指南、arXiv 综述 2412.05579 都专门讨论裁判模型先验知识污染评估的问题)。可以收一句:"裁判不能看被裁判的人的草稿纸——输入要来自产物,但不能共享推理上下文。"

## Q13/Q14:国际化校验 + AST 能不能完全满足?(组件传参怎么确定 prop 用作 i18n key)

- **表面**:i18n 校验的具体实现 + AST 的边界
- **实际考察**:知不知道静态分析(AST)的边界,什么时候必须上 agent 做语义校验
- **答到的**:i18n 三步(代码扫描写死文案→标签内 t() 调用检查→沙箱渲染看是否显示 key);AST 不能完全满足(看不到 key 对不对、布局影响、组件传参 prop 是否当 i18n key)
- **漏的(关键)**:① **被面试官引导才答出 AST 的边界**——不够主动,应该一上来就划清"AST 管静态结构,管不了运行时语义";② "代码扫描"其实就是 AST,可以更干脆地说出来;③ 组件传参那题答得有点绕("吸引力""代码节点"是口误),核心应该是"prop 在运行时被当成 i18n key 用,AST 看不到运行时数据流,需要 agent 逐节点分析"。

| 维度 | 分数 | 问题 |
|------|------|------|
| 知识完整度 | 7 | 最终答对了,但是被引导的 |
| 表达清晰度 | 5 | 这段最绕,口误多 |
| 追问防御力 | 6 | 被追问穿了一层才到 |

**改进版**:
- **核心句**:"AST 管静态结构,管不了运行时语义——key 对不对、prop 有没有被当 i18n key 用、文案长度有没有撑坏布局,这些得交给 agent。"
- **展开**:AST 能查的:写死的中文/英文、强制包裹标签里 t() 有没有被调用、key 字面量存不存在。AST 查不了的:① 这个 key 在 locale 文件里到底有没有对应文案(运行时才知道);② 组件传参时 prop 在下游有没有被当成 i18n key 用(AST 看不到运行时数据流);③ 多语言文案长度变化对布局的影响。这些需要沙箱渲染 + agent 做语义层分析——翻译/文案的语义匹配是纯 agent 干的活。**下次主动划界,别等面试官引导。**

## 面试官洞察(青衫 · agent 工程化团队)

> 面试官一句话:"agent/LLM 是面向概率的体系,我们做的最多的事,就是尽量提高它成功完成任务的概率。"

### 洞察 1:公司实践——海外视频创作 agent

**面试官原话**:新加坡 AI 初创,投入最多的是**视频创作类型 agent**;工程体系包括创作端(外部界面、三方模型接入的工程化)+ 内容团队;主攻视频创作本体 + **评测** + **拉片**(逐帧拆解视频);目标 to C 出海,海外。

**价值**:这家是少数把"评测"和"拉片"放在和"创作本体"同等位置的团队——说明他们认可"生成 + 评测"双轮驱动,和面试官反问环节讲的 benchmark 方法论一脉相承。若后续面试,可围绕"视频生成的多模态校验、评测体系设计"展开。

### 洞察 2:agent 工程化的核心哲学——"提高概率"而非"追求确定"(本场最高价值)

**面试官原话**:agent 是面向概率的体系;除了底层模型能力,我们做工程最多的事就是"尽量提高成功完成任务的概率"——提示词引导、流程设计、上下文设计,都是为了"利用好 LLM 本身的特性,提高它成功的概率"。所有问题最终回归一点:**我们到底在利用 LLM 的什么特性?**

**价值**:这是整场最值钱的一句话,直接给出了 agent 工程化从业者的**第一性思维**。它解释了面试官整场为什么反复追问"怎么强制执行/怎么校验"——不是为了 100% 确定,而是为了把概率从"可能对"拉到"接近一定对"。**权威印证**:Anthropic《Demystifying Evals for AI Agents》明确把 agent 当概率系统对待,强调 evals 让问题在影响用户前可见、价值随生命周期复合增长、"越晚建 evals 越难"——和面试官"持续 benchmark 优化"完全同构。

### 洞察 3:复杂任务用"假设 → probe/benchmark 验证"而非纯逻辑推理

**面试官原话**:复杂任务很难靠纯逻辑推理直接得到准确答案,正确做法是**提出假设,然后做 probe(探针)/benchmark 去验证思路对不对**。能力分两块:① 思路好不好(能不能避免之前踩过的坑);② 后面怎么持续评价和优化(benchmark)。

**价值**:这是 agent 评测方法论的实操指南。落到自己项目上:prd-tools / prd2code-gen 应该有一套 benchmark——固定一批 PRD 样本,每次改动跑一遍看蒸馏质量/证据链连通率有没有回归,而不是靠感觉说"质量变好了"。这是当前工作流缺的一块(个人原型探索阶段还没系统化 benchmark),值得补。

### 洞察 4:面试官的追问链路暴露的考察维度

**分析**:整场追问围绕一条主线——"怎么让 agent 可靠执行"。从"怎么交给规则"→"怎么强制检查"→"代码和 agent 谁主导"→"通用框架怎么做"→"你的门禁"→"上下文隔离"→"硬软校验"→"AST 边界"。这条链考察的是:你对**确定性与概率的边界**有没有清晰认知。面试官的价值观很明确:**工程化是为了提高概率,但关键路径必须用确定性兜底**。

### 洞察 5:技术引导(面试官"喂答案")

- "代码和 agent 谁是主动方?" → 引导意识到 agent 调用工具,但工具是裁判
- "AST 能不能完全满足?" → 引导承认静态分析有边界
- "组件传参……光靠 AST 不好做" → 引导到 agent 流水线逐节点分析

**价值**:面试官在关键节点都给了梯子,说明他是在"交流"而非"考倒",对候选人有基本认可。但被引导才答出,说明这些边界认知还不够内化,需要刻意练成"主动划界"。
