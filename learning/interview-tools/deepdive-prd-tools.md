# PRD 工具（prd-tools）— 金牌项目深挖稿

> **用途**：面试当场可念版。被问"讲讲你的 AI 项目 / 最有挑战的项目 / 和 Cursor 区别 / 怎么度量效果"任意一题，都从这份稿切入。
> **念法节奏**：电梯演讲 30 秒 → 业务背景 30 秒 → 双轨架构 1 分钟 → 深挖锚点按追问展开 → 量化与边界收口。
> **总控制**：完整讲 3-4 分钟，被追问可延展到 8-10 分钟。
> **素材出处**：stealth.html A4 / A5 / A10 / G4 / G5 / G6 / G7 / G8 / G9 / G10 / E9 / E19 整合。

---

## 一、30 秒电梯演讲（必背，张口就来）

> **prd-tools 解决的是"AI 怎么理解已有项目"这个空白领域**——市面上的 AI 编码工具（Cursor、Copilot、Windsurf）都是解决从 0 到 1 写新代码，但真实项目是几万行代码、大量历史业务逻辑，AI 直接喂 PRD 让它分析，准确率只有 30-40%。我做的 prd-tools 把 PRD 自动蒸馏成有证据支撑的开发计划，**PRD→Plan 从 5h/需求降到 2h/需求，已推广到 20+ 人团队**。核心创新是证据链防幻觉机制和 Reference 六文件知识结构。

**钩子三件套**：空白领域（已有项目 AI 理解）+ 量化效果（5h→2h，20+ 人）+ 核心创新（证据链 + Reference）。

---

## 二、业务背景（30 秒，先讲为什么做）

**痛点定位**：团队 20+ 人天天写需求，但**新人和 AI 都有"已有项目理解空白"**——
- **新人**：来一个新需求，光理解几万行历史代码就要 2-3 天，写出来的计划经常漏掉隐性约束（"完单油站券"这种业务术语映射到代码哪个 enum）
- **AI**：直接把 PRD 喂给 Cursor / Claude Code，它会"猜"——给出看着完整实则编造的 plan，同事照错计划开工就是返工

**核心洞察**：**Cursor / Copilot 解决的是"写代码"，但真实瓶颈是"写代码之前想清楚改哪里"**。这一段没人做，就是 prd-tools 要填的空。

**金句钩子**：**"AI 工程的难点不是写代码，是想清楚改哪里。"**

---

## 三、双轨架构（1 分钟，先说清楚再展开）

**双轨定位**——一句话先甩清楚：**团队推广版要"稳"（聚焦 PRD→Plan 环节，20+ 人在用），端到端原型要"快"（验证 PRD→Code 全链路可行性，个人独立设计）**。

### 团队推广版（在用）— 三阶段

**第一阶段·Reference 构建知识库**：跑一遍项目代码，把几万行代码蒸馏成 **6 个领域知识文件**——
1. 代码结构（目录约定、模块划分）
2. 编码规则（命名规范、模式约束）
3. 跨层契约（API 协议、数据流向）
4. PRD 路由（PRD 关键词 → 代码模块映射）
5. 业务领域 Map（业务术语 → enum → 文件三步定位）
6. 代码实体索引（函数 / 类的位置索引）

一次性构建，后续新需求自动回流更新。

**第二阶段·PRD 蒸馏 11 步**：能力面适配器 → PRD 摄入 → 证据账本 → 需求 AR 标注 → Reference 路由 → 源码精读 → 契约差异 → 生成 Spec（**硬停止人工确认**）→ 生成 Plan → AI 五项自评 → 门禁脚本。每步产出是下一步输入，链条断任何一处都能定位。

**第三阶段·人工确认后开发**：Spec 确认后 AI 生成 Execution Plan，人工 Review 后各用各的开发工具（Copilot/Cursor/Windsurf），开发完新知识回流更新 Reference。

### 端到端原型 prd2code-gen（探索）— 6 步完整闭环

① PRD 蒸馏 → ② 项目分析 → ③ 变更计划（人工确认）→ ④ **双路径代码生成**（Handlebars 模板优先 0 token + 无模板走 LLM 兜底）→ ⑤ **自验证 + 自修复循环**（4 类校验：语法 / 字段双向 / 规则 / 集成 + 3 次重试 + 震荡检测 A→B→A 切整段重写）→ ⑥ 输出报告（人工评分）+ eval/auto-tune 反哺。

### 双轨协同逻辑（被问"既然端到端跑通为啥不推广"必答）

- **团队版要稳**：执行段成熟工具多（Cursor/Copilot），各人偏好不一，强行推广代码生成涉及**质量责任划分**——20+ 人天天用，AI 写错代码算谁的？
- **原型版要快**：验证可行性、积累工程化经验、等模型成本下降再上生产

**金句**：**"双轨不是没想清楚，是稳和快各取所需。"**

## 四、4 个深挖锚点（面试官追问任何一个，深入讲）

> 这 4 个是 prd-tools 的核心创新，每个都能讲 1-2 分钟。面试官问"具体怎么实现的"必答这里。

### 锚点 1：证据链（防幻觉的核心机制）

**问题**：AI 有"完成倾向"——为了交差会编造路径。Cursor 给的 plan 看着完整，但你分不出哪条是真的、哪条是编的。

**机制**：prd-tools 给**每条结论标 evidence_id**，就是 PRD 第几页 + 代码第几行。搜不到的也显式标 `negative_code_search`，告诉你是"没找到"不是"不存在"。

**价值**：**错误发现成本从"重做"降到"抽查"**——这是可靠性工程最关键的一跳。

**消融实验数据**：去掉证据链层，Readiness 评分从 82 掉到 54。

### 锚点 2：Reference 六文件 SSOT

**问题**：几万行代码不可能全塞进 LLM 上下文窗口。

**机制**：把代码蒸馏成 6 个领域知识文件（结构 / 规则 / 契约 / 路由 / 领域 Map / 实体索引），每次请求按需检索相关片段拼进上下文。

**本质**：这就是 Context Engineering 的工程化变体——信息从"全量"压缩成"结构化精选"，既省 token 又不丢关键事实。和 Claude Code 管 context、RAG 检索是同一套思路。

**业务术语桥接案例**：PRD 提到"完单油站券"，Claude Code init 不知道映射到代码哪个 enum，只能猜或 grep "完单" 零结果放弃；prd-tools 通过 05-domain 术语表，PRD 词 → enum → 文件三步精准定位，不是猜是查表。

### 锚点 3：质量门控（5 维 quality-gate）

**问题**：靠 AI 自觉的方案规模一上去必然漂移。

**机制**：每步产出过 5 维校验——
- **完整性**：Plan 是否覆盖所有 PRD 需求点
- **一致性**：PRD 与 Plan 是否对齐
- **可追溯性**：每条结论是否有 evidence_id
- **可行性**：代码锚点是否真实存在
- **风险标注**：Open Questions 是否显式标出

**关键认知**：**"靠自觉的方案规模一上去必然漂移，靠机制约束才能扛住 20+ 人天天用。"**

### 锚点 4：反馈回流（防退化）

**问题**：知识库一改可能引入退化。

**机制**：每次蒸馏的新知识更新 reference，配合已验证的 **golden case 集**做回归——改了知识库要重跑基准案例，**确保不劣化才上线**。

**消融数据**：去掉证据链层 Readiness 从 82 掉到 54——这就是为什么证据链是必须层不是可选层。

## 五、量化指标（最致命的追问，必须答得有数字+对照组）

> **阿里一面 34:14 真题**："PRD 转化效率提升 60% 怎么度量？分母是什么？实验组用了 Claude Code，对照组是不是纯手写？"——这题场上答弱了，复盘整理成三层指标。

### 口径先收敛一句

现在简历用 **5h→2h（5 个需求对照）**，60% 是当时表述。被问"提效怎么算"，主指标答 5h→2h 即可。

### 三层指标体系

**L1 产物质量**（v2.19 已能跑）——三个锚点：
- Code Anchor 准确率：plan.md 里标的 anchor 真实被改的比例（目标 ≥ 85%）
- OQ 命中率：Open Questions 真实阻塞率（目标 ≥ 60%）
- IMP 执行率：Implementation 计划条目真实执行率（目标 ≥ 75%）
- **readiness 评分就是基于这套**

**L2 研发效率**（Sprint 3 排期中）——三个指标：
- 端到端开发耗时（目标提升 ≥ 40%）
- QA 缺陷数（目标下降 ≥ 30%）
- **开发返工率**（主指标——开发过程中发现 plan 错了的比例）

**L3 业务价值**（定性）：
- reference 飞轮增速（每月新增 playbook / 术语数）
- 团队模式契约对齐度提升

### 严谨的 A/B 设计（堵分母和对照定义）

- **实验组** = `/prd-distill` 全流程 + Claude Code 执行 plan
- **对照组** = Claude Code init + 直接给 PRD（**关键：对照组也用 Claude Code，只差有没有走 prd-tools 全流程**，这样变量单一，只验证 prd-tools 这层证据链的价值）
- **样本量** ≥ 30 个真实需求 pair-wise 对照
- **主指标** = 开发返工率

### 收口必须诚实

v2.19 已经有 readiness / final QG 内部评分（L1 能跑），但 L2/L3 业务指标在 Sprint 3 排期，**现在没有硬 A/B——这是公认短板，正在补**。

**核心一句**：**工程化项目最忌讳"我觉得有用"，得拿指标名 + 数字 + 对照组定义说话**。三件套口诀。

## 六、弯路反思（最有故事性，必讲）

> 这是 A5 卡的核心，体现你"踩过坑 + 有架构判断"。

**业务背景**：市面 AI 编码工具都是 0→1，**怎么让 AI 理解已有项目**是空白领域，prd-tools 就填这个空。

**弯路（30 秒，体现踩过坑）**：一开始我用 **MCP** 做，两周发现不管怎么优化 prompt，准确率就是上不去。定位到根因——**MCP 是单次调用模式，但需求分析需要多轮上下文和复杂论证**，这是架构级瓶颈，不是调优能解决的。**关键认知：选错架构，再多优化都白费。**

**突破方案（1 分钟，亮架构）**：转向 **Agent + Skill 工作流模式**——
- 自研 Reference 六文件 SSOT（项目知识结构化）
- 多轮搜索（Agent Loop 不是单次）
- 证据链验证（每条结论可追溯）

**核心收获收口**：**AI 工程的难点不在调 prompt，在架构选型——单次调用 vs 多轮 Agent，决定了能力上限。** 我走过的弯路，让我对"什么场景该用 Agent 而不是简单工具调用"有体感。

## 八、边界诚实 + 改进路线（最显工程判断）

> **被问"团队模式能不能上生产"必答这段**。比硬撑"60% 够用"可信十倍。

### 团队模式 60% 准确率的三个根因

**根因一·Pull-based 禁 rg/glob（损耗最大头）**：出于安全考虑（团队推广版要 20+ 人用），强制 Pull-based 读文件模式，禁用 rg/glob 主动扫描。代价是失去源码验证能力。

**根因二·11/12 delta pending（契约对齐维度）**：契约对齐维度 45% vs 单仓 70%，数字主动甩——12 个仓里 11 个仓的 delta 还在 pending。

**根因三·min 各仓质量（木桶效应）**：团队模式准确率卡在最短板那个仓。

### 边界诚实（防被追穿的杀手锏）

**62 分定位是 needs_owner_confirm 不是 blocked**——62 分不是失败，是有意的分层定位：
- 团队模式做**粗粒度**（排期和契约对齐）
- 精度由各业务方再做**单仓蒸馏**补齐到细粒度
- **两层配合，不是一层包打天下**

主动说这句，比硬撑"60% 够用"可信十倍。

### 改进路线（roadmap 清晰最加分）

**Sprint 4 引入 A2A 协议**——团队 Agent 通过 A2A 向各仓 Agent 问源码，"不读源码"的限制解除，根因一直接消除，**预估准确率从 60 提到 75+**。

三层讲清：**已经做的**（团队模式粗粒度）+ **正在做的**（A2A 解锁源码验证）+ **预期效果**（60→75+）。比说"在优化"强十倍。

### 收口金句

**"团队模式 60% 不是终点，是'安全优先、精度分层'这个设计取舍当下的代价——根因能讲清、取舍能讲清、改进路线能讲清，这比一个干巴巴的准确率数字更能让团队信你。"**

## 十二、对照团队 SDD 的话术升级（5 个亮点，重新讲一遍已有能力）

> 我们团队做了 SDD 工程分享，框架和我做的 prd-tools 大方向一致。这节专门提炼 5 个 **"用团队 SDD 的精准表达，把 prd-tools 已有能力重新讲一遍"** 的话术。不是加新内容，是把已有内容讲到对手追不上。

### 亮点①「8-14 次迭代」精准定位（替代"已有项目"粗粒度表达）

- **原表达**："市面 AI 工具解决 0→1，我解决已有项目"
- **升级表达**："更精准的分类是 **问题类型**——有标准答案的问题（架构/算法/模板代码/性能优化）AI 几乎一次过，Cursor/Copilot 已解决；涉及业务约定的问题（联调反馈/字段细节/跨服务约定）**每条线都得 8-14 次迭代**。prd-tools 解决的就是后者。"
- **用法**：被问"和 Cursor 区别" / "为什么不用 Cursor"必答。**8-14 次是超硬数字**，比"已有项目"具体十倍。

### 亮点②「日志 14 次」故事 → 对照 prd-tools 的 3 个 Reference 文件

- **团队故事**：日志改 14 次才稳定，根因是 3 类难点——A 业务字段约定（stage 还是 env？）/ B 工具链选择（AI 默认 logrus 项目用别的）/ C 部署细节（滚动/软链/采集）
- **对照 prd-tools**：

| 团队的难点 | prd-tools 的对应 Reference |
|---|---|
| A 业务字段约定 | **业务领域 Map**（术语→enum→文件三步定位） |
| B 工具链选择 | **编码规则** |
| C 部署细节 | **跨层契约** |

- **用法**：被问"你的工作流解决什么具体问题"必答——"团队的日志 14 次是单点痛点，我做的 prd-tools 把这三类项目约定系统结构化成 6 个 Reference 文件，从'每次踩坑修一次'升级到'一次沉淀全局复用'"。**把团队故事据为己用**。

### 亮点③ Reference = 「把规范做进代码」（替代"知识库/RAG"表达）

- **原表达**："Reference 是项目知识结构化"（听起来像 RAG）
- **升级表达**："Reference 六文件本质是 **把规范做进代码**——不靠 AI 上下文记忆、不占 token、写代码时天然遵循。这是从'靠 prompt 提醒'到'工程化注入'的范式升级。"
- **用法**：被问"Reference 和 RAG 区别" / "为什么不直接 prompt 让 AI 读规范"必答。**对手从 RAG 变成 prompt 工程**。

### 亮点④ VAULT 飞轮叙事（替代"防退化"防守叙事）

- **原表达**："golden case 回归是防退化机制"（防守叙事）
- **升级表达**："这是 **FLYWHEEL · VAULT 模式**——失败不消失，结构化沉淀进 vault，成为下次提速的动能。Reference 越用越准，从冷启动到稳定态是非线性提升。"
- **用法**：被问"长期价值" / "和工作一年的 Cursor 比有什么优势"必答——**vault 积累是壁垒，单次蒸馏只是工具**。

### 亮点⑤ HUMAN GATE + 四角色（替代"质量责任划分"工程视角）

- **原表达**："Spec 后硬停止是质量责任划分"（工程视角）
- **升级表达**："Spec 后硬停止本质是 **HUMAN GATE——人是裁判不是搬运工**。我做'出题的人'（定方向·拍架构），AI 做'搬砖的人'（批量生成），五维 QG + 门禁脚本做'当眼睛的人'（联调·给反馈），Spec 后我做'把门的人'（每个 commit 授权）。这不是人和 AI 的分工问题，是 **AI 时代工程师角色重定义**。"
- **用法**：被 leader 面问"人和 AI 怎么协作" / "未来工程师会被替代吗"必答。**把讨论从工具拉到组织/哲学层级，绝杀**。

---

> **5 个亮点的共性**：全是"用更精准的表达，把已有能力重新讲一遍"。**不是加内容，是升级视角**——从工程视角到哲学视角、从防守到进攻、从单点到平台。这套话术让你在任何深度追问下都不慌，因为同一个机制有 3 层表达可以切换。**stealth.html A4/A5/A10 三张卡已同步嵌入这套话术**。

## 十三、最后提醒（面试前去洗手间默念）

1. **不吹过头**：60% 是当时的表述，现在用 5h→2h（5 个需求对照）；L2/L3 在 Sprint 3 排期，没有硬 A/B
2. **表达流畅**：钩子三件套（空白领域 / 5h→2h / 证据链）背到张口就来
3. **讲"为什么"**：每一层都先说为什么这么做，再说怎么做，最后说效果
4. **不会的不装**：被追到细节答不出，说"这块当时是同事 X 负责的，我了解的是 Y"——比硬撑可信
5. **反问钩子**：讲完主动埋钩子——"您对 AI 工程化的证据链机制怎么看？"把面试变成对话

---

> **结语**：这份稿的核心不是技术深度，是**"想清楚改哪里"的工程判断**——prd-tools 解决的不是 AI 能不能写代码，是 AI 怎么敢信。每一个量化数字、每一处诚实交代、每一个改进路线，都是为了让面试官信你能把这事扛到生产。**讲好这个故事，offer 就稳了一半。**
