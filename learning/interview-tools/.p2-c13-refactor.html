现状的问题由我主导重设计来解，我 drive 了拆、删、降级三个动作的全过程。39 个 addNode 全铺在一张图里（managerGraph.graph.ts:67-104），加 40 条边其中 20 条是条件边。看着唬人，但本质是单进程内的函数调用图，没有域边界、没有独立伸缩、没有故障隔离——这是原型级的拼装，不是生产架构。重设计核心三个动作：拆、删、降级。 瘦身的本质矛盾是可观测性与复杂度之间的取舍。节点多则每一步都可观测、单测粒度细，但配置爆炸、跨域状态耦合、故障传播、认知负担全上来；合并则图变简洁，但丢单测粒度、丢故障隔离、丢独立伸缩。难点不是"砍节点"，是判断哪些节点值得保留独立性、哪些该合并掉。我的判断：能独立演进和故障隔离的留，纯格式转换的合并。 一、按业务域拆 subgraph（7 个）：知识库（RAG）/NL2SQL/代码/爬虫/GUI/办公（报表可视化）/多模态。每个 subgraph 自带自己的 Probe + Critic + HITL + checkpoint，supervisor 只做路由和仲裁。好处有几个：multimodal / music / video 这些默认不启的域可以单独部署单独伸缩，主图回归轻量；故障隔离——爬虫域挂了不拖垮代码域；团队能并行开发各自域。落地用 LangGraph 的 subgraph + 共享状态 schema，跨域通信用现有 RPC 方案（M16 已有验证）。 二、哪些该删 / 合并：visualize 和 report 是 pipeline 尾端消费者、clean 是工具不是 Agent，三个内部 Agent 全砍。(1) visualize 和 report 这俩内部 Agent 是 pipeline 尾端的消费者，根本不该是独立 Agent——它们没决策、只是格式转换，合并成一个 render 服务。(2) clean Agent 同理，是工具不是 Agent。(3) 三个内部 Agent 全砍。这样 12 个 Agent 降到 9 个外部域 Agent，每个域一个 subgraph。 三、过度设计要降级：Critic 可插拔、Self-Evolution 砍到 shadow-only、Probe-before-plan 保留加强。(1) Critic——现在是每个 plan 都仲裁，但 Critic 自己也会错（M5）。重设计成可插拔：默认只对高风险域（NL2SQL / 代码）开 Critic，低风险域（报表渲染）直接走，用 Critic 否决率这个线上 metric 动态决定哪些域需要 Critic（M14 已有 metric 基础）。(2) Self-Evolution——现在每 30min 生成假设 + 金丝雀 A/B，但每臂 8 个样本没有统计显著性（M9 弱点）。重设计成 shadow-only：只记录假设和结果、不自动 promote，所有 promote 走人工审批 + offline power analysis，这个 SD5 单独展开。(3) Probe-before-plan——这个保留且加强，它是最有价值的设计（并行探活把客观快照喂给 LLM），但要从 5 个下游扩到按域探活。 四、tradeoff 诚实讲：拆 subgraph 增加了跨域通信成本和状态同步复杂度。原来一张图内状态天然共享，拆了要显式传消息。这是架构经典取舍：高内聚低耦合 vs 通信开销。我的判断是当前 9 个外部域已经多到单图 hold 不住（故障传播 + 认知负担），拆的收益大于通信成本。但如果是 3 个域以内，不拆。 五、方法论：节点的价值在能不能独立演进和故障隔离，不在数量。我宁可要 7 个能独立部署的 subgraph，也不要 39 个绑死在一张图里的 node。追问边界：subgraph 怎么共享经验回放——经验库（M10）是全局单例还是每域一份？基础路由经验全局共享（因为路由是跨域的），但每域的失败归因样本库独立（因为 failure mode 不同：NL2SQL 的 evidence_gap 和爬虫的 timeout 归因逻辑完全不同）。这正好对应现有 failureAttribution.ts 的 11 类要按域重分组。
