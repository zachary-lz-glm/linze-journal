◆ 质量
auto-tune 怎么自动优化生成质量
六个模块闭环——注册表 + 5 维确定性评分(规则非模型评判)+ 差距分析 + 模式检测 + 循环控制 + 对抗评审,跑黄金 Case 回归看进退

评分是确定性规则、不是模型当裁判。这是要先讲清的一点:auto-tune 的 scorer 是 5 维确定性评分引擎(File Coverage 20% / Field Accuracy 30% / Pattern Correctness 20% / Compilability 15% / Structural Similarity 15%),靠规则比对、不是让另一个大模型来判分。bmad-optimizer 里虽然有并行多 reviewer 做对抗评审(spec→execute→review),但那是优化器的对抗审查环节、不是评分器本身。把"确定性规则评分"说成"用模型当裁判打分"是口径错误,这两件事要分开。

六个模块怎么串。registry 加载测试用例注册表并抽取 ground truth;scorer 按上面 5 维给产物打分;gap-analyzer 把评分失败映射成 tuning target(哪一维失分、失在哪个字段);pattern-detector 跨 case 检测代码模式(如 injectDActions / runOn / FormDComponentType 这类结构有没有写对);loop-controller 管收敛和震荡检测、跑偏了就停;bmad-optimizer 做 spec→execute→review 的对抗评审推改进。注意这套 5 维是代码生成准确度评分,和蒸馏就绪度 readiness、benchmark oracle 是三套不同的评分、别混。

迭代节奏。维护一套人工验证过的黄金 Case 集覆盖典型场景,每次只改一个变量(Prompt / 门禁阈值 / 能力面配置),跑黄金 Case 对比前后 5 维评分和文件覆盖率,确认无退化再合入——相当于基准回归。

诚实边界。黄金 Case 是小样本,只能说"基准 case 上没劣化、关键 case 上有改善",不能说统计显著证明提升。auto-tune 的调参依据是评分失败的归因、不是线上真实缺陷回流,偏原型探索阶段的工程验证、非生产级闭环。
