设计 Agent 的第一步不是挑工具选框架，是先定"目标可控性"。具体讲就是把三个问题先答清楚——任务何时算完成、哪些步骤必须人批、失败兜底是谁。我在 code_assistant 和 prd-tools 两个项目里都是从零搭起来的，前者是带自验证的代码生成 Agent，后者是 PRD 蒸馏成干净上下文的 Claude Code 插件。不先把可控边界划清楚后面工具越多越乱，Agent 在生产里就是定时炸弹。

核心矛盾是自主性与可控性的权衡，不是功能堆得多就强。给模型的自主权越多，能力上限越高，但越容易跑偏、越难 debug；约束越多则越稳但抵消了模型本来的灵活性。本质是在两者之间定边界，不是堆功能——一个跑飞了无法定位的 Agent 比一个笨但可控的 Workflow 危险得多。

关键决策是三层架构加 HITL 闸门，工具保持小而可组合。三层指规划层定目标、执行层调工具、校验层把关产出，每层职责分开可单独测试。HITL 闸门放在高风险节点（生产路径上的代码变更、外部副作用），拒绝了"让 Agent 全自主跑完一次性交付"的方案——演示好看，生产扛不住。工具粒度上拒绝"一个大工具全包"，走小工具加可组合，因为大工具 schema 描述难写、模型选不准，小工具语义清晰更容易选对。

验证方式是每层都可观测、可单测、可回放。code_assistant 走自验证加自修复循环——max_iter 控制最大尝试次数，Readiness Gate 检查产出是否达标，不达标带错误反馈重试；prd-tools 走分阶段，/reference 蒸馏长期知识、/prd-distill 处理单次 PRD，每阶段产出独立可校验。指标看每层的成功率、重试次数、最终产出质量，不是凭感觉说"看起来 work 了"。

可迁移的方法论是"先定可控边界再配能力，而不是先堆工具再加约束"。局限也讲清楚：两个项目都偏研究型、在小范围验证过、没接大规模真实业务流量；HITL 路径目前是 stub 式占位，真正的审批 UI 加权限流转是已知改进项，下次做生产化第一件事就是把这块补上。
