4 类校验（按顺序跑）：① 语法——TypeScript 编译 tsc --noEmit + AST 完整性 + 残留 {{ 占位符检查；② 字段双向——蒸馏字段 ⊆ 生成字段（不遗漏）+ 生成字段 ⊆ 蒸馏字段（不幻觉）；③ 规则——i18n key 格式、enum 白名单、命名规范、反模式逐条检查；④ 集成——import 路径、类型兼容、不破坏已有代码（diff 检查）。

SSOT是什么（涂虎面试答不上来的教训）：6个Reference文件就是AI的唯一事实来源。举个例子——项目是BFF层，有个Activity Controller，SSOT里记录了它的入参结构（哪些字段必填、哪些枚举值）、出参结构、调了哪些下游服务。这些不是AI推断的，是从源码和契约中提取的事实。AI分析新PRD时，不是猜应该改哪个文件，而是查SSOT里已有活动类型怎么注册的。SSOT的作用：让AI每个结论都有证据链追溯到源码事实，而不是AI自己编的。

方法：跑一遍完整流程，看 AI 在哪类问题上犯错，追溯是什么信息缺失导致的。比如 AI 分配任务时经常搞混前后端职责，分析后发现是代码结构信息和 API 契约信息混在同一个文件里，于是加规则：代码结构和 API 契约必须分开存储。

核心原则：关注点分离——每个文件只负责一个领域（契约文件只管 API 定义、不碰业务规则），枚举等元信息只存在一处。避免 AI 生成冗余文件（最初会生成 10 份，同一枚举出现在多个文件），减少 token 浪费和人的维护成本。
