# loop 系统 — 金牌项目深挖稿（差异化武器）

> **用途**：面试当场的差异化武器。当面试官问"你还做过什么 AI 项目"、"你对 Agent 工程化的理解"、"你怎么用 Claude Code 做实际项目"——这个项目让面试官眼前一亮，**因为它是用 Claude Code 自己造的 AI 自动化流水线，没几个人做过**。
> **念法节奏**：定位 30 秒 → 痛点 30 秒 → 架构 1 分钟 → 8 个工程亮点（按追问展开）→ 量化产出 → 反思收口。
> **素材出处**：`.claude/loop-worker.md` / `loop-reviewer.md` / `loop-v2-启动.md` + git log v233→v251。

## 一、30 秒电梯演讲（必背）

> **loop 是我用 Claude Code 的 cron + subagent 机制自研的"AI 自动化流水线"**——它把我面试速查页 100+ 张题卡自动优化成"照念就能答好"的可念答案。**单 cron 串行调度 + 独立上下文 subagent + worker-reviewer 状态机**，从 v233 跑到 v251，自动产出 18 个版本、每轮改 1 张卡并自检 div 平衡后部署上线。**最有工程含量的不是用 AI，是把 AI 串成可无人值守、能自纠错、不污染主上下文的流水线**。

**钩子三件套**：用 Claude Code 自带机制造 AI 流水线（不是调 API）+ 18 版本真实在跑（不是 PPT）+ 工程创新 8 点（独立上下文 / 零并发 / 状态机 / div 自检）。

## 二、为什么做（30 秒，痛点定位）

**背景**：我的面试速查页 `stealth.html` 累积了 **100+ 张面试题卡**——大厂 AI 前端面试的高频题，每张都要写"钩子→业务背景→分层口语→项目桥接+念法"的可念段落。

**痛点**：
- **人工写一张高质量可念卡要 30-60 分钟**，100+ 张就是 50-100 小时——根本挤不出时间
- **风格会漂移**——一次性写完会越写越累、风格不统一；分批写每批风格也不一致
- **质量参差**——疲惫时写的卡质量掉档
- **复习时还要持续迭代**——面经在更新，卡也要跟着更新

**核心洞察**：**这事本质上是个"流水线问题"——每张卡是独立工作单元，有明确的样板（E16/E19/MCP 协议卡），有明确的质量门控（div 平衡、防幻觉），完美适合 Agent 自动化。**

**金句钩子**：**"用 AI 写面试答案不算牛，把 AI 串成无人值守的流水线才算牛。"**

## 三、架构（1 分钟，三层结构）

### 第一层：单 cron 串行调度器

**机制**：单个 cron 每 3 分钟触发一次，**每次只派 1 个 subagent**（worker 或 reviewer），**串行零并发**。

**为什么这样设计（最关键的工程判断）**：
- **v1 踩过的坑**：双 cron（worker/reviewer 各一个）并行跑，会撞车——同分钟 git index.lock 冲突、push non-fast-forward、并发改 stealth.html 互相覆盖
- **解法**：单 cron 串行零并发，**根本消除竞态条件**——不是加锁，是从架构上消除

**节奏**：worker:reviewer ≈ 4:1，每累计派完 4 个 worker 改派 1 个 reviewer。

### 第二层：独立上下文 subagent

**机制**：每轮派 general-purpose subagent，**用自己的上下文干活**，跑完只回 **1 行简报**给主会话，然后销毁。

**核心价值**：
- **主会话不爆上下文**：subagent 读 50 个文件做调研，主会话只收到 1 行摘要——原始 50 文件内容不进主上下文
- **风格不漂移**：每个 subagent 是 fresh 上下文，重新读样板卡，风格永远跟随样板而不是越跑越偏
- **故障隔离**：subagent 跑挂了不影响主会话，下一轮换一张卡继续

### 第三层：状态机标记

**机制**：每张卡的 hint 字段带状态标记：
```
无标记 → [loop·已优化·待复核] → [loop·已优化·已复核] / 退回无标记(带 <!-- loop-retry:N 原因:xxx -->) / 已放弃(入 review-log)
```

**worker 选卡逻辑**：grep `class="hint"` 找未带 `已优化` 标记的卡 → 优先级 AI 卡 > Schema 项目卡 > AI 工程化卡 > 综合答题型卡 → 改写后加 `[loop·已优化·待复核]`。

**reviewer 复核逻辑**：扫所有 `待复核` 卡 → 通过改 `已复核` / 退回删标记带 retry 注释 / 已放弃入 `.claude/loop-review-log.md` 等人工。

## 四、8 个工程亮点（面试官追问任何一个，深入讲）

> 这是 loop 系统的核心价值。每个亮点都解决一个具体的工程问题，不是炫技。

### 亮点 1：独立上下文 subagent（防主上下文污染）

**问题**：长跑 100+ 轮，主会话累积所有轮次的文件读取、改写内容、git 输出——上下文必爆。

**机制**：每轮 subagent 在**自己的独立 context window** 里运行，**主会话只留 1 行简报**。

**价值**：跑 100 轮和跑 1 轮对主会话的负担一样——这就是为什么 loop 能长跑不爆。

**金句**：**"Multi-Agent 通信成本最小化——子代理干完活回主线程一句话，不是把整个上下文搬过来。"** （这也是 Anthropic 官方推荐的 Agent 设计原则）

### 亮点 2：零并发串行（防双 cron 撞车）

**问题**：v1 是双 cron 并行，同分钟撞车导致 git `index.lock` 冲突、push non-fast-forward、文件互相覆盖。

**机制**：**单 cron 串行**——每 3 分钟只派 1 个 subagent，绝不并行派两个。

**工程判断**：**消除竞态的正确方式不是加锁，是从架构上消除**——加锁是治标，串行是治本。

### 亮点 3：硬规则绕过（最体现 Claude Code 工程理解）

**问题**：Claude Code 有两条安全硬规则，**强制人工审批、白名单也绕不过**，无人值守时一触发就卡死：
- 「`cd` + 输出重定向（`2>/dev/null` / `>` / `>>`）」的复合命令
- 「`cd` 到某目录再跑 `git`」的命令（防目标目录 hooks 被执行）

**机制（4 条对策）**：
1. **搜文件内容一律用 Grep 工具**，不要用 bash `grep`
2. **awk 自检用绝对路径、不带 cd**：`awk '...' /path/to/file`
3. **git 一律用 `git -C /path ...`，绝不写 `cd ... && git`**
4. **任何 bash 命令都不要加 `2>/dev/null` / `>` / `>>`**

**纯自动的关键**：必须用 `claude --dangerously-skip-permissions` 启动（或 settings `bypassPermissions`）——这是绕过两条硬规则的唯一可靠方式。

**金句**：**"AI 工程化的隐藏成本是工具链的安全约束——不是 AI 不会干，是工具链防它干。理解这些约束并知道怎么合规绕过，是 Agent 工程师的基本功。"**

### 亮点 4：结构自检 + 自动回滚（防破坏性改动）

**问题**：HTML 是嵌套结构，subagent 改一张卡时如果多写一个 `</div>` 或漏一个，**整页卡片点不展开、子 tab 栏不显示**。

**机制**：commit 前必跑 awk 自检——
```awk
awk '{o+=gsub(/<div/,"x");c+=gsub(/<\/div>/,"y")} END{print o-c}' stealth.html
```
- 输出**必须 0**（div 平衡）
- **不是 0 → 立即 `git checkout` 撤销本次改动，返回"结构自检失败已撤销"**
- **绝不提交不平衡的改动**

**防错关键**：Edit 的 new_string 结尾**只写 card-body 闭合的一个 `</div>`**，绝不写 card 容器闭合（它在 old_string 范围外）。

### 亮点 5：SW 缓存版本 bump（防访客拿旧缓存）

**问题**：Service Worker 缓存不更新，访客拿到的永远是旧版本。

**机制**：每次部署前必须改 `sw.js` 的 `CACHE_NAME`（`kb-vN` → `kb-vN+1`）。

**真实事故案例**：stealth.html 在 v252 之前出过 4 组卡号撞车（#2/#39/#40/#41）——c13f4f0 commit 专门修复这个。说明 SW 缓存机制必须严格流程化。

### 亮点 6：风格防漂移（每轮 fresh 读样板）

**问题**：长跑会让 AI 越跑风格越偏——用词越来越华丽、结构越来越复杂。

**机制**：每个 worker subagent 启动时**先 fresh 读 6 张样板卡**（E16 Agent 架构全解、E19 MCP 协议全解、E20 LangChain+LangGraph、E26 AI 工程化主线、Schema 驱动架构、400+ 权益组件库），**风格 = 钩子开头 → 业务背景 → 分层口语展开 → 项目/业务桥接收口 + hint 念法提示**。

**金句**：**"对抗 AI 风格漂移最有效的方式，不是写 prompt 约束，是每轮强制重新读样板。"**

### 亮点 7：防幻觉硬规则（数字必须有出处）

**问题**：AI 改写时容易编造数字——"准确率提升 50%"、"覆盖 80% 场景"看着像样但假的。

**机制**：改写里出现的**每个数字 / 百分比 / 源码细节，必须能在该卡"业务成果/痛点"段或 archive 里找到出处**；找不到出处就别写具体数字，改用定性描述。**绝不编造像样但虚假的数据**。

**多源参考**：只在需要细节时读——`/Users/didi/work/prd-tools` + `/Users/didi/work/prd-to-code-gen`（README/CLAUDE.md/src/docs/plugins/scripts）；`archive/retrospectives/`（Schema 看 04-27/04-28/04-29，prd-tools 看 05-06/05-19/05-26）；`reviews/`（qa/ 真实失分点 + transcripts/ 转写）——**提炼真实面试官追问，让卡能防追问**。

**保护机制**：读了 >5 个源文件还没动笔 → 果断放弃本轮，返回"本轮跳过:资料过多"——防上下文爆。

### 亮点 8：worker-reviewer 分离（产出/审核解耦）

**问题**：同一个 AI 既改又审，会"自证无误"。

**机制**：worker 改写加 `[loop·已优化·待复核]` → 累计 4 张后派 reviewer 复核 → reviewer 通过改 `已复核` / 退回带 retry 注释 / 已放弃入人工复核队列。

**价值**：**产出和审核解耦，多一层防线**——这是质量门控在 Agent 流水线的落地。

## 六、5 层追问预案

### 第 1 层：业务背景 / 为什么做？
→ 翻**二、为什么做**：100+ 张卡人工优化干不完 + 风格漂移 + 质量参差。

### 第 2 层：方案选型 / 为什么不用 X？
**最致命的一层**。准备 4 个对比：

| 对比项 | 为什么不用 | 为什么用现在方案 |
|---|---|---|
| **vs GitHub Actions** | 1）不是为 AI Agent 设计，调度死板<br>2）没有 Claude Code 的 subagent / cron / context 隔离原生支持<br>3）改起来重 | Claude Code 自带 cron + subagent + bypassPermissions，**直接复用原生机制零开发** |
| **vs 普通 shell 脚本调 API** | 1）没有 Agent Loop 能力<br>2）没有上下文管理<br>3）要自己实现 worker-reviewer 状态机 | Claude Code 是完整 Agent harness，**省了写 harness 的力气** |
| **vs LangChain 自己写** | 1）重写 cron + subagent + git 操作成本高<br>2）和 Claude Code 工具链割裂 | 直接用 Claude Code 的工具系统（Read/Edit/Grep/git -C） |
| **vs 一次性大模型批量改** | 1）100+ 张一次性塞爆上下文<br>2）风格必然漂移<br>3）出错没法定位 | 单卡单轮独立 context，**风格永远跟样板** |

**金句**：**"选 Claude Code 不是因为它强，是因为它把 cron + subagent + 工具系统 + 上下文管理都备齐了——我只需要写状态机和业务规则。"**

### 第 3 层：技术难点 / 最难的是什么？
**最有杀伤力的答法**——按问题分类：

- **最难的 1：硬规则绕过**——Claude Code 的安全硬规则（cd+重定向/cd+git）会卡死无人值守，**白名单都绕不过**。要发现并系统化规避（git -C / Grep 工具 / 不带重定向），**这是 Claude Code 工程化的隐藏知识**
- **最难的 2：风格防漂移**——长跑必然漂移，**解法是每轮强制 fresh 读样板**，不是写 prompt 约束
- **最难的 3：结构自检**——HTML 嵌套结构一个 div 错就全页崩，**解法是 commit 前 awk 校验 + 不平衡自动 checkout**

### 第 4 层：量化效果 / 数据指标？
→ 翻**五、量化产出**：
- v233→v251，**18 个版本真实在跑**
- 节省约 **90% 人工时间**
- 主会话上下文**长跑不爆**（核心架构指标）
- 双 cron 撞车事故**归零**（架构演进效果）

### 第 5 层：重做你会怎么改？
**最容易加分的层**。准备 3 个反思：

1. **金字塔能力建设**：现在 loop 只优化"卡的风格"，下一步应该按**五层金字塔**（L1 基础 / L2 概念 / L3 工程化 / L4 系统设计 / L5 业务判断）分配选卡优先级，**自动识别薄弱层并主动补缺口**——这就是我现在在做的优化
2. **多模型路由**：现在是单一 Claude，应该按任务复杂度路由——简单卡走 Haiku 省成本，复杂卡走 Opus 保质量
3. **可观测性**：现在只有 git log，应该加每轮的 token 消耗、改写前后 diff 对比、retry 次数等指标

**金句**：**"loop 系统现在的状态就是 prd-tools 早期的状态——能跑、有效果、但缺指标体系。下一步要把它从'流水线'升级成'可观测的流水线'。"**
