总览
这个项目是做什么的？
一套 Claude Code 插件——把 PRD 蒸馏成可执行、可追溯的开发计划，把 AI 每条结论锁在“有证据、能审计”的边界里

是什么。一个跑在 Claude Code 里的 PRD 工作流插件，纯 Markdown Skill、零外部依赖，我独立设计迭代到 v2.20。它解决的是需求到代码这条链路上的信息损耗——PM 写完 PRD 口口相传给开发、开发理解再写，每一步都在丢信息最后返工；老员工脑子里的经验新人接不住；直接把 PRD 喂大模型，没项目上下文兜底，连该改哪个文件都会编。真正难的不是"生成"——LLM 能生成；难的是它给的结论你敢不敢信：依据在哪、影响范围估得对不对、能不能复现。把矛盾定义在这一层，方向就不是堆更大模型（那只会让"看起来更顺"的幻觉更多），是把每条结论做成可追溯、可验证、可拒绝。

怎么运转。分两段。第一段先扫一遍项目，把几万行代码和历史 PRD 蒸馏成一套领域知识库叫 reference——项目画像、编码规则、跨层契约、业务术语这些，外加一套自研的代码实体倒排索引；本质就是"项目知识库 + 倒排索引"，让 AI 拿到需求能快速定位要改哪、不用把整个仓库塞进上下文。第二段拿到新 PRD 走一条 11 步的蒸馏流水线：摄入、建证据账本、拆需求点、定位代码、对齐契约、产出 Spec，然后硬停一步等人确认，确认完才产出开发计划，最后还有个五维加权的质量评分卡交付。

怎么保证可信。三个机制贯穿全程，我管它叫 AI 产出可信三支柱：一是单一事实源（SSOT），每条知识只存一处、别处用 ID 引用，从根上压缩 AI 编造空间——同一事实散在多处正好给它挑顺口的用；二是证据链，定义了 8 种证据类型（PRD、源码、契约、git diff、负向搜索这些），每条改动都要挂证据 ID 才能进 spec，挂不上就不让过；三是多层门控，从 Spec 确认到最后 Readiness 五维加权评分（85 通过 / 60 需 owner 确认 / 小于 60 阻塞），每步都有卡点。举个具体的坑：一类需求 AI 反复编造引用说"这个改动会影响 XX 模块"但代码里根本没那模块——一开始堵不住，因为只有正向证据"找到了什么"，它会钻"没找到但你说的应该有"的空子；后来加了 negative_code_search 这类负向证据，让"我搜过源码、确实没这个东西"本身也必须留痕，才把这类编造压下去。

代码生成那段。往下我还做了个端到端原型 prd2code-gen，把 PRD 一路打通到代码：模板优先、零 token 出代码，没模板再让大模型兜底；生成完自动跑四类校验、最多重试三次，还会盯来回震荡——发现 A 改成 B 又改回 A 就整段重写；跑完带一个自动调参的反馈环。但这段是个人原型探索、非生产级闭环，没上过真实业务流量；团队在用的是停在开发计划那一版——代码生成涉及质量责任划分，我先把链路跑通验证可行性，工程化投入再看。

现状。团队里小范围在用（ADR-0005 口径约 1–5 人），前端后端各跑各的，我负责基建和 review；搭了几个真实业务 case 的回归基准。我给它的定位是工程化探索——真正的价值不在"能生成代码"，在"怎么把 AI 每条结果变成可溯源、可验证的工程产物"。提效数字说清楚：单需求理解从大概 5 小时降到 2 小时左右是前后工时对比、不是严格 AB（团队规模小没法随机分组双盲）。

往哪深挖。这张是入口，后面八个模块各有一张主卡：reference 怎么把几万行代码蒸成可检索的知识库；PRD 蒸馏 11 步怎么跑、为何硬停等人确认；整体架构为什么拆双插件、为什么不走 MCP；对比 Cursor / Copilot / BMAD 差在哪；SSOT + 证据链 + 门禁怎么串成可信链；端到端原型怎么自验证自修复；实现层踩过什么坑（幻觉 / 上下文漏 / 归因）；推广落地遇到什么阻力怎么解。

约 90 秒。被问"介绍下这个项目 / 这是干嘛的 / 说说你的 PRD 工作流"翻这张——reference 一句话带过（项目知识库 + 倒排索引），别钻 yaml 文件数；追问"reference 具体怎么做"翻 reference 模块主卡，"完整链路 / 为啥不推广代码生成"翻端到端模块。
