两种检索的本质差异是模糊语义匹配与精确结构定位之别。向量召回把文档和查询都压成 embedding，靠距离找"意思相近"的东西，擅长处理"改这个功能"这类自然语言，但字面术语（表名、字段名）精度差、缺逻辑关联；实体图把代码里的符号、调用、import 显式建边，靠拓扑回答"谁调用了谁、改这个函数影响哪里"，精确但需要预先抽取。两种范式解决的是不同问题，不是哪个更先进。我两个项目各做了一种。

向量召回做在 code_assistant，自研轻量、强项是模糊语义。text-embedding-v3 做 embedding，内存余弦算相似度，sha256 内容指纹做惰性增量（查询时按需重算，不是 git 钩子）。它的优势是无需结构化、能处理"改这个功能"这类模糊自然语言查询；缺点是短查询和字面术语（表名、字段名）精度差、缺逻辑关联，所以我混了关键词命中做兜底。

代码实体图做在 prd-tools，受 GraphRAG 启发的正则轻量版。entities.json 存代码符号、函数、组件，edges.json 存调用和 import 关系，再加一个倒排索引 term 到 entity，全部由正则扫描器产出。这块我明说受 Microsoft GraphRAG 思路启发，但做的是轻量的正则版，不是完整图数据库，更不是 AST——早期接过 GitNexus AST，v2.13 已经删了换回原生正则。它的优势是关联查询强，能给到调用链、import 关系、影响范围，结果可解释；缺点是要预先抽取、正则有遗漏、维护成本高。

实践里用实体图路由加证据 ID 锚点把两者串起来。先用实体图把业务概念定位到代码符号，再让证据链保证每条结论可回溯。

红线主动暴露：两个项目都是自研轻量实现。没用 Pinecone、没用 Neo4j、没用 Chroma。问到向量库，直接说自研 JSON 加内存余弦；问到知识图谱，直接说是正则抽取的轻量实体图、受 GraphRAG 启发但不是图数据库。如果要做更重的关联推理（比如跨服务的全局影响分析），会考虑上真正的图数据库，但目前轻量正则版够用。
