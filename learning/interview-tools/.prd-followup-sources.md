## 模块① reference(PRD 源文件知识库 / SSOT)

1. 你说的 reference 具体指哪 6 个文件?为什么是这 6 个、不是 5 个或 10 个?标准是什么?(源: S2 决策追问)
2. reference 和普通 RAG 向量检索有什么本质区别?为什么不直接把全量源码切片做 embedding?(源: S2 为什么选这方案)
3. 自研了实体倒排索引,它比现成 BM25 / 全文检索好在哪?精度和召回你怎么测的?(源: S1 trade-off articulation)
4. reference 里的源码或文档过时了怎么办?有没有更新和校验机制?谁保证 SSOT 不腐化?(源: S2 排障 + S4 可复盘)
5. code_anchor 锚到"文件 + 符号",为什么不做精确到行号?行级锚定不是精度更高吗?(源: S2 取舍)
6. 倒排索引的准确率 / 召回率有没有跑过?有没有 bad case 集来回归?(源: S1 failure mode awareness + S4 可验证)
7. 6 个文件加起来多大?塞进 context 的策略是全量还是按需?token 成本怎么控?(源: S1 cost awareness)
8. PRD 引用了一个 reference 里没有的实体,系统怎么处理?静默编造、还是报错打回?(源: S1 failure mode + S2 evaluation)
9. 这个 reference 方案是你拍的还是调研过 alternatives?对比过哪些(直接喂全量源码、外部知识库、纯 prompt)?(源: S2 decisions)
10. 谁维护 reference 质量?有没有明确"满足什么条件才算 SSOT"?(源: S4 证据链可复盘 + S2 ownership)

## 模块② PRD 蒸馏(11 步 / 加权门控 / Spec 确认)

1. 11 步蒸馏,为什么是 11 步不是 5 步?每一步都不可省吗?哪步其实是冗余的?(源: S2 取舍)

5. Spec 确认门控——人工 review 还是自动打回重跑?重跑上限几次?死循环怎么破?(源: S2 排障 + S4 复盘)
6. 蒸馏输出的"开发计划"颗粒度到哪一级?任务、模块还是文件?(源: S2 depth)

10. 这套蒸馏逻辑换业务域(比如从 BFF 换到数据)还成立吗?领域知识怎么迁移?(源: S2 scale constraints)

## 附:三个答题方法论简述(供写卡 agent 参考"怎么答")

**CLEAR(S5)** — Context(背景为何重要)/ Leadership(为何是"我"来牵头、我独特的位置)/ Execution(具体执行步骤)/ Accomplishment-Results(成果与可量化指标)。核心是 L:面试官真正想听的不是你做了什么,而是你如何主导、影响了结果。答题先讲"为什么是我"再讲"我做了什么",leadership 信号自然带出来,比 STAR 的"Task"更有分量。

**5-Why(S3 拉取失败,据通用方法论 + S4)** — 每答一句,预判面试官会再问一个"为什么",连续下钻 5 层直到根因。S4 掘金原文要求把每个项目"连续回答 5 个为什么:为什么做、为什么这样做、为什么不用另一种、失败怎么办、结果怎么衡量"。用法是在**准备阶段**自检:你的每个技术决策(为什么倒排索引、为什么 11 步、为什么双路径)能不能扛住 5 连问——把"决策→理由→理由的理由"链路写满,面试时才不会被问散。

**证据链审查(S4)** — 把每个项目声明拆成三标准:**可追问**(能被连续下钻到细节)、**可验证**(有指标 / 产物 / bad case 能证明)、**可复盘**(失败或复盘能变成下一轮训练任务)。答题按"结论 → 场景 → 取舍 → 结果 → 复盘"顺序,让面试官顺着证据链走,而不是听一堆结论堆砌。AI 面试官的追问模式正是"说技术→追指标→追角色→追边界→追失败",证据链就是提前把这五层备好。
