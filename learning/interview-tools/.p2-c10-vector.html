向量库选型我选了自研，不是 Chroma。code_assistant 的检索我是设计者，做的是自研 JSON 加内存余弦的方案，没上 Chroma 或 Milvus。code RAG 的索引规模就一个项目几百到上千个文件块，内存里全装得下，这种规模上专业向量库是杀鸡用牛刀。选型不是看哪个更先进、更主流，是看场景规模和工程成本。

本质矛盾是通用能力和工程成本、可控性之间的权衡。Chroma 这类专业库功能全、ANN 索引（HNSW、IVF）、过滤、持久化都有，但引入了进程依赖、运维负担、调试黑盒；自研 JSON 加内存余弦反过来——轻、零依赖、索引能直接打开看，但能力有限，大规模和低延迟并发扛不住。本质是按规模和场景权衡，不是"哪个更先进"——大规模该上专业库就上，小规模自研更合适，不存在普适最优。

我的方案是 JSON 加内存余弦加 sha256 增量。具体做法：文件名 token 预筛快速过滤候选集，再调 text-embedding-v3 这个现成 embedding 服务切块向量化，内存里逐块算余弦相似度打分，索引落盘成一个 JSON 文件，下次启动直接 load。增量用 sha256 内容指纹——文件没改就直接复用缓存的 embedding，不用重新调 API 烧钱。我拒绝了 Chroma，因为量小不需要、重依赖不值当；也拒绝了 git 钩子做增量，因为 sha256 内容指纹更轻、不绑工作流。

自研方案在 code 仓库规模上跑得通，这是 eval。召回准——索引是 JSON 我能直接打开看哪个块的向量是什么、为什么召回它，排错比黑盒 Chroma 容易；增量快——sha256 让重复 embed 的成本几乎为零，没改的文件直接复用缓存。eval 不需要"召回率 XX%"这种数字，看的是行为是否符合设计：相关文件能召回、增量更新不重算、索引可观察可调试，就算 work。

方法论一句话——选型按场景规模权衡，不追"更先进"。大规模、低延迟、要并发，就上 Milvus 或 pgvector；小规模、要可控可调试，自研够用。局限我主动讲清楚：自研只在量小的时候成立，规模上去必须上专业向量库——真到百万级向量，内存里逐块算余弦扛不住，那时老老实实换 Milvus 或 pgvector。我的策略是"先够用、留好替换接口"，索引层抽象出来不绑死实现，未来要换库改一层就行。
