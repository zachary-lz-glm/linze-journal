◆ 实现
实现过程踩的坑——幻觉、上下文漏文件、输出不稳定
四大坑(LLM 输出飘、幻觉、上下文漏文件、错误流到产物)用同一思路治——确定性优先 + 多层防幻觉把偏差转成可拦截信号,不指望模型自己更准

踩的坑就这四类。LLM 输出不稳定(同输入不同次跑出不同质量)、幻觉(没把握也编路径编字段)、上下文漏文件(全仓塞不下、定向搜又漏)、工程错误流到产物(蒸馏阶段的错误沿 11 步流水线向下传染到最终 plan 和代码)。源码里没有所谓"工具多了模型选错"的指标——prd-tools 是纯 Skill 形态、不注册一堆外部 function-calling 工具,这个坑在本项目根本不存在,不把外部研究数据往本项目上贴。

治法是确定性优先,把偏差转成可拦截的工程信号。输出飘靠"有模板走 Handlebars 零 token、无模板才走 LLM"的双路径规避;幻觉靠多层防幻觉——reference 建库时证据门控(关键事实必须挂 evidence)、distill 消费时源码二次确认(进 spec/plan 的结论不能只靠 reference)、bff-gen 生成时字段严格比对(生成字段必须 ⊆ 蒸馏 JSON 字段)、auto-tune 自动检测幻觉字段;上下文漏文件靠三阶段检索(Reference 路由定向 → rg/glob 补充扫描 → 汇总成 ≤800 行 context-pack),是单向流水线不是迭代收敛;产物侧四层拦截兜底。

每个坑都有根因和被拒绝的替代方案。幻觉根因是模型强完成倾向——不修模型,把"编"转成显性可拦信号(negative_code_search 搜不到也是证据),拒绝"只靠 prompt 写得好"的软约束;输出飘根因是 LLM 本身非确定——拒绝"全程靠 LLM 生成",用模板兜走大部分确定性场景;上下文漏文件根因是全仓 token 爆——拒绝"全量召回"和"纯向量检索",用 reference 路由先把范围收窄到几个文件;工错流根因是错误会沿步骤传染——拒绝"最后总评兜底",改成步步前置门禁早发现。代价是流程节点多、偶有过度拦截,换的是错误不向下传染、返工局部化。

诚实边界。这是研究型原型、没接真实业务流量,效果靠 benchmark oracle(单 case gasstation 97 分)和对照实验(班次签到 reference +14 分、油站清理数据泄露后真实 +9 分)小范围验证,不是生产级闭环;多层防幻觉也没建 labeled query set 测召回率,负向搜索拦得住"编的路径不存在"但拦不住"编的路径恰好存在但语义错";eval 债还没还。下次遇到 LLM 产物质量诊断,我会先建归因维度(知识库缺失/模型能力不足/上下文不够)再谈修复,而不是上来就调 prompt。
