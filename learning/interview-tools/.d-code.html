实现
向量检索不用 Chroma 自研，具体数据结构和算法怎么实现
两套独立的自研 JSON 向量库——代码文件索引按 chunk 存向量、经验库存成功 Q&A；都是磁盘 JSON 加进程内 Map 缓存加内存余弦 top-k，sha256 逐文件比内容指纹、没变就跳过重算 embedding。

数据结构 / 形态。两套独立的自研向量库，磁盘 JSON 加进程内 Map 缓存。代码文件索引的 schema 是 {version, root, embeddingModel, files: {路径: {sha256, chunks: [{startLine, endLine, vec, snippet}]}}}，每个 {root, embeddingModel} 组合对应 .data/vector-index/ 下一份哈希命名的 JSON；进程内 vectorIndexCache 按 key 单例加载、脏了才落盘，写盘走 tmp 文件加 rename 的原子写。经验库是另一份 .data/code-experience-vectors.json，一个数组每行 {id, ts, question_norm, question, vector, hint, task_kind, hint_files}，slice 保留最近 300 条，query embedding 还带短 TTL 的内存缓存避免重复算。两套都没有外部向量库进程，打开 JSON 就能 grep 调试。

核心算法 / 流程。余弦是教科书公式 dot/(|a|·|b|)，一遍 for 循环算点积加两个模长，零模长直接返回 0；两处实现略有差别——文件索引用 Math.min 容忍不等长向量，经验库严格等长。文件检索 pipeline：walkFiles 捞全仓文件 → query 分词去停用词后按路径子串命中数打分取前 60 候选（廉价路径预筛，不是真 BM25）→ 逐个 fileSha256 比指纹、没变跳过重算 → 变了就 readText、按字符预算切块（相邻块重叠几行）、批量 embedDocuments 写回索引 → query 自己 embedQuery 后对每个候选文件挑余弦最高的 chunk、按文件排序取 top-10；两个 API 入口的默认上限是 800 文件 / 60 候选 / 10 结果。经验库召回是线性扫全表、score ≥ 0.72 过滤、按 task_kind 匹配加权、取 top-3。

为什么自研、拒绝了什么。向量库自研 JSON 加内存余弦，拒绝 Chroma / Milvus 这类专业向量库：代码仓就几百到上千个 chunk、内存装得下，上专业库等于引入进程依赖、运维负担和调试黑盒，换来的 ANN 索引在这个规模用不上；代价是大仓库和低延迟并发扛不住（全量线性扫描、没 HNSW）。增量更新用 sha256 内容指纹而不是 git 钩子或全量重算：fileSha256 以固定大小缓冲流式读文件算哈希，索引里每个文件存的就是它内容的 sha256，查询时逐文件比、变了才重算 embedding、没变直接复用旧向量；拒绝 git 钩子是因为它耦合仓库生命周期、跟查询时机错位，内容指纹在查询时惰性算、不依赖外部事件，全量重算又每次都付 embedding 成本。

踩坑和边界。研究型原型，跑的是 demo 级仓库，没接真实业务流量，召回质量小范围验证过、没建 labeled query set 测 recall@K / nDCG 这类检索排序指标，是已知评测债。规模上限要诚实承认：文件索引和经验库都是全量线性余弦、没 ANN，进程内 Map 缓存重启就丢，真上量级得换近似检索加专业落库。embedding 走 @langchain/openai 的 OpenAIEmbeddings、默认 text-embedding-v3，向量维度是模型决定的、源码里没硬编，不临时报数字。同一套 sha256 也用在写盘的乐观锁上——writeText 带 expectedSha256，文件读后被改过就抛 "File has changed" 拒写，跟向量增量是同一个内容指纹思路的两种应用。
