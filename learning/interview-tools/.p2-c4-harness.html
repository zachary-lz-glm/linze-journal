Agent 不等于模型，Agent 等于模型加 Harness。这个框架来自 Addy Osmani 的 Harness Engineering——模型是发动机，Harness 是围绕模型构建的工具调度、状态管理、校验、恢复这一整套工程脚手架。我 Manager 和 code_assistant 两个项目本质上做的就是这件事：搭一套 Harness 让模型在受控边界内跑起来。理解这点能直接改变 Agent 工程的投入方向——大部分工程价值在 Harness 而不在调模型参数。

核心矛盾是给模型自由度与加约束护栏的权衡。Harness 太薄则模型乱跑、产出不可控；太厚则僵化、把模型本来的能力抵消掉。本质是工程化"可控的自主"——既不是放任模型自由发挥，也不是把每一步都写死，而是让模型在明确边界内做决策。

关键决策是分层 Harness 加关键状态门加失败恢复，拒绝了两种极端方案。分层指 L1 信息边界、L2 工具系统、L3 执行编排、L4 记忆状态、L5 评估观测、L6 约束恢复六层各自独立设计。具体机制上：KSG 关键状态门控让重要节点显式校验、Ratchet 只进不退让每条规则都可追溯到一次具体失败——拒绝了"裸调模型等结果"的方案，那种方案演示能跑、生产崩；失败恢复走重试加降级加转人工的多级策略，拒绝了"失败即终止"，因为生产里失败是常态，没有恢复路径的 Agent 扛不住。

验证方式是失败可恢复、状态可观测。Manager 的 39 节点状态机加 code_assistant 的自验证自修复循环都是 Harness 的落地——每个节点都有可观测的输入输出加状态字段、失败有明确的恢复路径。评估看 trace 完整度（每个决策可追溯）、失败恢复成功率、最终产出质量，而不是看模型一次跑通没。

可迁移的方法论是"Agent 等于模型加 Harness，价值在 Harness 不在模型调参"。局限也说清楚：两个项目偏研究型，部分机制比如 KSG 自进化闭环目前是 shadow-only（只记录不自动 promote），不是全量生产部署；Harness 的工业级严谨性——完整的 SLA 监控、灰度发布、容量规划——是后续补的方向，不是现在已经做完了的。
