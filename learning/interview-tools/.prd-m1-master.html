◆ reference
reference 是什么、为什么我自己造
主产物是 5 个数据 yaml + project-profile + 00-portal 导航,辅助层是正则扫描的实体加 term→entity 倒排索引(四个文件);索引自己造是因为全量塞 context 会打爆、纯 embedding 相似度做精确路由会漂移。

数据结构。reference v4.0 把旧版 10 文件按 SSOT 收敛成 5 个数据 yaml——01-codebase 静态清单、02-coding-rules 编码规则、03-contracts 字段级契约、04-routing-playbooks 路由打法、05-domain 业务领域,再加 project-profile 项目画像和 00-portal 导航入口;每个 yaml 顶部写 boundary 声明只放什么,字段级信息只落在 03、别处用 contract_ref 引用。主产物之外有一个辅助层 Evidence Index,四个文件:entities.json(代码实体 file/function/class/interface/enum/const 等)、edges.json(DEFINES/IMPORTS/RESOLVED_IMPORT/REGISTERS/REFERENCES 几种关系)、inverted-index.json(term→entity 倒排索引,只索引代码符号不含业务术语)、manifest.yaml(元数据)。辅助层不是主知识库,是给 /prd-distill 做确定性检索用的。

怎么跑。/reference 走 6 个 Phase:上下文收集 → 结构扫描 → 深度分析(五个子阶段顺序产出 01-05,后生成的必须查先生成的去重)→ 质量门控 → Evidence Index 构建 → 反馈回流。首次走 Mode F(上下文收集)→ Mode A(全量构建),日常用 Mode B 增量只跑受 git diff 影响的子阶段。关键是按需路由不全量塞——/prd-distill 蒸馏时从需求 IR 提取实体/字段/动作词,匹配 04-routing-playbooks 的路由打法,只加载相关层的 yaml 和命中的代码锚点,context-pack 压到 800 行内给模型消费。

为什么索引自己造。Evidence Index 是基于正则扫描的代码实体索引加倒排索引,不是图数据库查询、不是编译器语法树解析、也不是把代码切片做 embedding 相似度。设计时参考过 GraphRAG“先结构化再检索”的思路,但大幅简化——只提取几种实体类型和关系边,不建完整语义图。拒绝全量塞 context 是因为几万行代码直接进 window 必然打爆 token;拒绝纯 embedding 相似度做路由是因为实体名、字段名这类精确匹配 embedding 会给语义近但符号错的结果,确定性的 term→entity 倒排更可控、可复现。代价是正则扫描的语言覆盖有限,目前只支持 TypeScript/JavaScript 和 Go,新语言要补 scanner。

踩坑和边界。最大损耗在团队 Pull-based 模式——团队仓只读成员仓的 reference、禁止主动 rg/glob 跨仓搜源码(避免跨仓误读),代价是失去源码验证能力、跨层 contract-delta 大量标 pending、精度天花板受制于最弱的那个仓;Pull-based 方向对,但复杂的聚合策略后来简化成了原样收集。eval 层面有笔债:倒排索引本身的准确率和召回没建 labeled query set 测 recall@K,目前靠 benchmark oracle 单 case(gasstation 97 分)和 golden sample 回归间接看,不是针对索引本身 recall/precision 的严格度量。冷启动要人工引导——得在项目根目录放 prd-docs 装 2-3 份历史 PRD + 技术方案 + 接口文档先跑 Mode F,跳过直接 Mode A 质量显著下降;所以这套更适合中大型、文档相对稳定的项目,小仓库让编辑器原生 init 一遍增量价值有限。
