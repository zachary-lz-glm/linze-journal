要证明的是前端底子,不是 AI 熟悉度。这道题的潜台词是前面聊了很多 AI 和架构,需要确认前端基本盘没问题。我的证据是从零设计了 Schema 渲染引擎——完整的 JSON Schema 解析、渲染、联动、性能链路都是自己设计实现。

本质矛盾是引擎级架构 vs 调 API 写页面。难点不在「用现成 Formily/Ajv」——那是调第三方库包装一层,本质还是调 API。真正难的是从零设计联动 DSL(dAction)+ injectDActions 双求值:先扫描 Schema 收集所有联动声明构建依赖图,再注入运行时求值函数,让组件间联动从命令式 if-else 变成声明式模板表达式。这是渲染引擎层面的架构设计,不是业务页面层面的拼装。

三个关键决策,以及拒绝了什么。①联动 DSL dAction + 双求值——拒绝现成 Formily,它的联动描述力覆盖不了城市→产品线→价格策略这种跨接口联动,且实现是黑盒不可控;②虚拟滚动 + 按需渲染——拒绝全量重渲染,四百+ 组件配置页全量渲染会卡死;③Rollup 双模式打包(ESM+CJS)+ tree-shaking 精确到组件粒度——拒绝整包引入,业务只为自己用到的组件付体积代价。

结果和怎么验证。四百+ 组件的权益配置页保持流畅——虚拟滚动只渲染视口内的组件,联动触发时精确更新受影响的组件而非全量重渲染。验证不是「感觉不卡」,是可复现的:大配置页面在低端机上滚动帧率稳定,联动响应有明确的更新范围可追踪。

方法论与局限。前端扎实的证据是从零设计引擎而非用现成轮子;性能靠精确更新而非全量重渲染。局限:这套渲染引擎是为配置类页面设计的——表单结构稳定、联动可声明式描述;它不直接迁移到 C 端自由布局(拖拽编辑、实时协作、富文本编辑器这类高交互场景),那里仍然需要写代码而非靠配置。
