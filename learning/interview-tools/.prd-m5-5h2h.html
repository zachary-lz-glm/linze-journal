◆ 质量
5h→2h 和 reference 价值怎么测的
5h→2h 是单需求理解的前后工时、不是严格 AB;reference 价值有 R01/R02 对照实验,但要讲清泄露和盲区、别合成一个笼统的提升数字

5h→2h 的真实口径。这不是严格意义的 AB 实验——没有随机分组、没有对照组,是同一个团队、同一类需求、改变流程前后的工时对比。对照组取过去已完成需求的工时记录(读 PRD + 评审会 + 写技术方案,均值 2-3 天含跨天等待),实验组是同一复杂度新需求走 prd-tools 从 PRD 到产出 Spec+Plan 的纯工作时间(1-2 小时,加人工确认 Spec 共 1.5-2.5 小时)。团队规模小没法随机分组双盲,选前后对比是因为现实可行。

reference 价值的对照实验。R01/R02 做过对照,但两个 case 要分开讲、不能合成一个笼统的提升分。班次签到 ShiftCheckin 这个 case:reference 带来 +14 分(54→68),因为前端改动小、大部分判断对了 schema_driven 就够;油站 case 清理掉 ground truth 分支数据泄露后,reference 真实贡献只有 +9 分(38→47)——而泄露版本会虚高到 +35(38→73),其中 +26 来自数据泄露。说明 reference 对 schema_driven 类改动有效,对 CustomRewardRule 这种复杂组件的架构级扩展有盲区。

benchmark oracle 评分。用 5 维 oracle(requirement_recall 35 / code_anchor_accuracy 30 / blocker_quality 20 / plan_actionability 15 / false_positive_penalty -10)给产物打分,gasstation 单 case baseline 跑出 97 分、passed。注意这是单 case 不是所有 case 平均,通过线是 60 分且无 P0 missed。

诚实边界。5h→2h 是前后工时非严格 AB,R01/R02 是单 case 对照不是统计结论,reference 提升数字受数据泄露影响、真实值低于旧报告。这套度量偏原型探索阶段的内部验证、非生产级闭环,如果团队更大,理想设计是同一批需求随机分配给人工组和工具组、双盲评审产出质量,目前样本量只能支撑方向性结论。
