背景和我的角色。接手前存量是 if-else 上万行，月均线上问题一二十起（无严格统计口径），新人改一处影响多处，扩城市就在分支里加逻辑。我是这次重构的技术决策者——决定整体架构往哪个方向走、排除了哪些方案、付了什么代价。

决策本质是定位抽象层级，不是选工具。当时摆在桌上的三类方案分布在三个抽象层级：硬编码（if-else）层级太低——表达力够但完全失控；低代码（可视化拖拽）层级太高——开发效率高但复杂联动描述不了；JSON Schema 声明式配置介于两者之间。决策的本质是在抽象光谱上选一个点，既不能低到失控、也不能高到表达力不够。这跟"选什么工具"是两个不同的问题。

三个选项和排除理由。① 继续堆 if-else：维护已失控、不可持续，排除。② 上低代码平台：B 端设计稿少、以原型为主，可视化拖拽 ROI 低；更关键的是城市→产品线→价格策略这种跨接口联动，低代码描述能力够不上，排除。③ JSON Schema 声明式配置：选中。权衡是声明式可静态校验、联动关系在 Schema 里一眼可见、新人看配置就能理解页面结构；代价是学习曲线（理解 DSL 和渲染引擎协议大概一周上手），以及 Schema 有天花板——拖拽排序+实时协作仍写代码。

结果和已付代价。线上问题月均二十起→接近零；联动关系从隐式 if-else 变成可静态校验、可追溯的声明式配置。代价已经付了——团队学习曲线一周，以及明确知道 Schema 描述不了的场景仍走代码。

方法论和边界。架构决策的核心是取舍——记录排除了什么、为什么排除、付了什么代价，而不是只记录选了什么。没写代价的决策记录等于没做决策。局限：Schema 有明确天花板，超复杂交互（拖拽、实时协作）仍需写代码，配置化不是银弹。

决策深挖卡，面试官问"为什么选 Schema 不选 X / 评估过哪些方案"时翻这张。②抽象层级是本卡灵魂，和主陈述卡的项目层矛盾（表达力 vs 可控性）是两个不同维度。
