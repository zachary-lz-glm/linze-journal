第一步永远是定场景。To B 配置化、To C 高并发、内部工具提效、AI Agent 编排——场景不同选型逻辑完全不一样。离开场景谈框架优劣都是耍流氓，所以被问"为什么选 X"我一般先反问"什么场景什么团队"。这步答不上来，后面全是空的。

第二步过四大约束。团队能不能 hold 住（学习曲线 + 招聘难度，Vue 招人就比 Solid 容易得多）、性能够不够（现在够 + 半年后还够不够）、生态成熟度（issue 活跃、star 趋势、有没有生产案例）、可替换性（会不会被绑死，比如 Elysia 绑 Bun 这种）。权重按业务阶段变——0 到 1 验证期生态优先，规模化阶段性能和团队优先。

第三步列 2-3 个候选做对比表。关键是不只列优点要列代价——每选一个就放弃 N 个，放弃的那部分要写清楚。我 Schema 那次就是 if-else、低代码、Schema 驱动三选一，每条都列了"代价是什么"，最后选 Schema 是因为业务侧自主配置这个收益压过了"实现复杂度高"这个代价。

第四步决策 ADR 化。写一份 Architecture Decision Record——什么时间、什么场景、选了 X 不选 Y、原因是什么。半年后回来看不会后悔，换人接手能看懂为什么当初这么选。很多人跳过这步，但其实是选型闭环里最值钱的一环。

诚实边界。不是所有领域都做过——Agent 我做了完整自研有发言权，后端 infra 和推理部署我是"了解定位、能跟人对话"，没在生产扛过主。但选型方法论是通用的，我对任何新技术都走这四步，这是能横向迁移的能力。

挂回我的项目。这套四步不是纸上方法论——我 Schema 营销中台选型（if-else、低代码、Schema 驱动三选一）和 code_assistant 向量检索选型（自研 JSON+余弦，没选 Chroma）都是照这套定的：先卡场景、过约束、列代价、ADR 留痕。被问"为什么选 X"时，我能立刻翻出当时的候选表和拒绝理由，而不是临时编权衡——这是反复用过才有的肌肉记忆，也是这套方法论能横向迁移的底气。

约 60 秒。被问"如果重新选你会选什么"正好接 ADR 那段。
