实现
/reference 蒸馏出来的 6 个文件具体怎么生成、怎么保证每条事实只存一处
6 个 yaml 按知识在开发生命周期里的角色切，每条事实只在一个文件落户、其他文件用 contract_ref / ref_rule 这类 ID 引用；/reference 走 6 个 Phase，Phase 3 五个子阶段顺序产出五个领域 yaml、后生成的必须查先生成的去重。

数据结构 / 形态。6 个核心 yaml 按"知识在开发生命周期中的角色"单一维度切（ADR-0001），每个文件模板顶部都写 boundary: 一行声明只放/不放什么。project-profile.yaml 是项目画像——layer、tech_stack、entrypoints、capability_surfaces、related_repositories，回答"这是什么项目"。01-codebase.yaml 是静态清单——directory_tree、enums（每个值带从源码注释提取的 label）、modules、registries、data_flows、external_systems、structures，只列字段名不放字段类型。02-coding-rules.yaml 是编码规则——rules 用 severity: fatal|warning|info 区分软硬约束，外加 danger_zones 和 war_stories。03-contracts.yaml 是字段级契约的唯一权威源——request_fields/response_fields 的 type、required、producer、consumers、checked_by、alignment_status。04-routing-playbooks.yaml 是 PRD 路由——capability_inventory（已有能力清单）、prd_routing（信号→能力面映射）、playbooks（开发步骤只在这里）、golden_samples。05-domain.yaml 是业务领域——terms、implicit_rules、decision_log，术语如果能归到 01 的枚举就用 see_enum: 引用、不复制。每条事实带 evidence[]（kind: code|prd|tech_doc|git_diff|negative_code_search|human|api_doc）加 confidence: high|medium|low。

核心算法 / 流程。/reference 走 6 个 Phase：Phase 1 上下文收集自动扫 prd-docs/ 里的历史 PRD 和技术方案产出 context-enrichment.yaml；Phase 2 结构扫描用 rg/glob 限定本仓、按 frontend/bff/backend 选能力面适配器，产出 modules-index.yaml 同时沉淀 project-profile.yaml；Phase 3 深度分析有五个子阶段顺序生成五个领域 yaml——先 01-codebase，再 02-coding-rules（查 01 去重），再 03-contracts（字段级信息从 01 移出），再 04-routing-playbooks（查 02 去重），最后 05-domain 加 00-portal.md（导航入口），每个子阶段文件末尾有 Self-Check 清单、过了才进下一步；Phase 4 质量门控跑文件完整性、证据闭环、源码一致性、五条边界规则、幻觉检查加 golden sample 回归，产 quality-spec.yaml；Phase 5 跑 build-index.py 解析源码产出 entities.json/edges.json/inverted-index.json 给 /prd-distill 检索用。Mode F→A 走完整 Phase 1-5，Mode B 增量只跑受 git diff 影响的子阶段，Mode B2 健康检查只跑 Phase 4。跨文件引用规则是关键：字段 type/required 只在 03、其他用 contract_ref，编码规则只在 02、playbook 步骤用 ref_rule，开发步骤只在 04 的 playbook，术语只在 05，外部 endpoint 详情只在 03、01 用 contract_ref 引用——这就是"每条事实只存一处"靠的机制。

为什么自研、拒绝了什么。SSOT 拒绝多处冗余存储：ADR-0001 做的 10 文件→6 文件重构，根因是旧版同一事实平均出现 2-5 次、模板字段语义重叠、7 个模板缺 boundary 声明生成时不知道该放哪；散多处正好给 AI 编造空间，它会挑一份顺口的用。三个机制堵——boundary 声明、跨文件 ID 引用、step-02 末尾五条去重检查；代价是去重目前主要靠 prompt 约束、还没有 deterministic duplicate checker（ADR-0001 自己列的已知风险）。按角色切 6 文件拒绝一个大文件或全量塞 PRD：单文件塞所有知识打爆 context window、人也读不动，6 文件按角色切让 /prd-distill 按需加载相关层、spec 阶段不必读全 6 个。Claude Code 插件形态拒绝独立 SDD 平台：复用 CC 的 rg/glob、Read、Skill 加载、slash 命令入口，零外部依赖、一个 install.sh 装到目标项目就跑，不重新发明编辑器集成、权限模型、UI，专注 PRD 蒸馏逻辑本身。

踩坑和边界。团队模式 Pull-based 是有意取舍不是 bug——ADR-0012 选 pull-based 不选 push-based，团队仓只读成员仓 reference、禁止主动 rg/glob 跨仓搜源码（避免跨仓误读），代价是失去源码验证能力、跨层 contract-delta 大量标 pending、精度天花板受制于 min(各仓 reference 质量)；损耗数字是 readiness 评分推算、不是线上统计，Sprint 4 引入 A2A 协议向各仓 Agent 问源码来收窄。团队规模按 ADR-0005 实际口径是 1-5 人小范围试用、不是生产级铺开。5h→2h 是 5 个需求的前后工时对比、不是严格 AB（团队规模小没法随机分组双盲）。Evidence Index 当前只支持 TypeScript/JavaScript 和 Go，其他语言项目只能靠 reference 主产物、不能享受确定性代码锚点检索。reference 是 Claude Code 插件不直接生成代码——/prd-distill 产出 spec.md 和 plan.md（开发计划）就停，下游 Cursor / Claude Code 接着写代码、不重复造写代码的轮子。
