主上下文不直接看原图和全量代码。大 PRD 的 context 压力来自两块——PRD 里的图片、和代码搜索阶段扫出来的源码。两块都不进主上下文，先压成结构化中间产物再喂。

图片分批并行。Step 1 摄入时主上下文禁止直接 Read 图片。按每批 ≤8 张分组，>8 张拆多个 subagent 在同一条消息里并行启动(全部 foreground，禁 background，避免子 Agent 没跑完主流程就继续的竞态)，每个产出 media-analysis-part-N.yaml，合并成 media-analysis.yaml。主上下文只读合并后的结构化结果。批量 8 是上下文预算和并行效率的平衡点。

代码扫描三阶段。Step 4.2 分三阶段单向流水线：阶段 1 Reference 路由——从 IR 提取实体/字段/动作词，匹配 routing-playbooks，提取 01-codebase 的文件路径和符号，读 query-plan 预匹配锚点；阶段 2 补充扫描——rg/glob 搜阶段 1 没覆盖的，Read 命中文件，MODIFY/DELETE 追踪 callers/callees 评估 blast radius；阶段 3 汇总——写成函数级技术线索(symbol / kind / file:line / role_in_flow / callers / callees / risk)并标注来源。

最后压成 context-pack。三阶段汇总结果压到 context-pack.md ≤800 行(按 requirement 分组的精简文档)，这才是模型在 Spec/Plan 阶段实际消费的量。PRD 原文也只按相关 chunk 加载，不全文塞。

诚实边界。是单向三阶段流水线、不是迭代收敛；图片分批和三阶段扫描都没系统测过在大 PRD 上的 token 节省幅度(诚实债)，复杂度和 PRD 需求数、涉及代码量挂钩。
