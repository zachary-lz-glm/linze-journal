项目
Schema三层架构（前后端共享SSOT）
三层架构：后端API出数据，BFF模板引擎生成Schema，前端渲染引擎声明式渲染——Schema是协议不是配置

面试官想听：你做的"Schema驱动"到底驱动了什么，是协议层还是只是配置层。

判断者应答：「三层架构的核心是 Schema 是协议层，不是配置层——① 后端 API 出原始业务数据；② BFF 模板引擎把数据按 JSON Schema 规范化输出（活动配置的结构化描述）；③ 前端渲染引擎拿到 Schema 声明式渲染组件。Schema 是前后端共享的一套真相源（SSOT），类似 GraphQL 的 schema 但更轻。」

为什么是协议（被追问就讲）：① 配置是"业务方填什么"，协议是"前后端怎么对话"——后者约束力更强；② 协议层可以做版本兼容（schema v1/v2 并存）、可以做静态校验、可以做Mock。

⚠️ 雷区：① 讲"我们用 JSON 配置"——这是配置不是协议；② 讲"前端表单"——只讲了一层；③ 没讲"前后端共享 SSOT"这个本质。

项目
Schema联动引擎（发布订阅+依赖图）
联动机制不是if-else，是发布订阅+依赖图+精确订阅（dot path）——200+字段表单不重渲染

面试官想听：你做的联动是真声明式还是包装的 if-else。能不能讲清底层机制。

判断者应答：「联动机制本质是发布订阅+依赖图+精确订阅——不是 if-else。Schema 里声明字段 A 变化时影响字段 B/C，引擎构建依赖图（DAG），A 变化时 publish 事件，订阅了 A 的字段响应更新。关键优化是精确订阅（dot path）——只重渲染订阅了变化路径的组件，不是整表单重渲染，200+字段表单也不会卡。」

工程难点（被追问就讲）：① 循环依赖检测（A→B→A 死循环）——构建图时拓扑排序+环检测；② 异步组件（地址选择器等）——Schema 描述异步契约，引擎按 await 处理；③ 批量更新——同一 tick 内多个变更合并。

⚠️ 雷区：① 答"用 Redux"——这是全局订阅不是精确订阅；② 答"用 useState"——这是组件级不是引擎级；③ 没讲依赖图——等于没做声明式。

项目
Schema门控（CI/CD+静态校验+灰度）
防止不合规Schema上线：CI静态校验+灰度发布+回归Case三层兜底

面试官想听：你有没有想过线上稳定性，还是只管开发不管运维。

判断者应答：「Schema 是协议就一定要保证合规——做了三层兜底：① CI 静态校验——PR 阶段 ajv 校验 Schema 是否符合协议（字段类型/必填/联动合法性）；② 灰度发布——新 Schema 先 1% 流量灰度，监控错误率，没问题再放量；③ 回归 Case——核心场景的 Schema 快照，新版本 Schema 必须能渲染这些快照。这三层共同保证：不合规的 Schema 上不了线，上线了也能灰度拦住。」

事故预防（被追问就讲）：① Schema 版本兼容——v1/v2 并存，老配置走 v1 渲染；② 线上监控——Schema 渲染错误率报警；③ 快速回滚——Schema 配置版本化，一键回滚到上一版。

⚠️ 雷区：① 答"我们有测试"——测试是开发阶段，门控是上线阶段；② 没讲灰度——单步全量上线等于裸奔；③ 没讲回归——新 Schema 可能 break 老场景。

项目
Schema+BFF协同链路
BFF不该拼业务，只该模板化生成Schema；后端不该懂前端，只该出原始数据

面试官想听：你能不能讲清三层架构里每层的职责边界——后端、BFF、前端各干什么。

判断者应答：「三层隔离是核心——后端不该懂前端（只出原始业务数据），BFF 不该拼业务（只模板化生成 Schema），前端不该处理业务（只声明式渲染）。BFF 用模板引擎（Handlebars）把后端数据按 Schema 协议输出，零业务逻辑。这层判断的代价是 BFF 多一层，收益是三端解耦——后端改 API 字段名不影响前端，前端换框架不影响后端。」

为什么不直接前端调后端（被追问就讲）：① 后端 API 是数据接口不是视图接口，前端要拼业务就脏了；② BFF 这层可以做聚合（多个后端 API 合成一个 Schema）、可以做兜底（后端挂了返回缓存 Schema）、可以做权限（按用户角色过滤字段）。

⚠️ 雷区：① 答"BFF 是中转"——这是实现不是判断；② 没讲三层职责边界——等于没设计；③ 讲"BFF 里写业务"——这是反模式。

边界
方案边界（能解决什么、不能解决什么）
边界内的场景我能保证效果，边界外的主动fallback——不是替代人，是把人逼到该出的环节

面试官想听：你知不知道自己方案的边界。强行兜底=不专业，主动认边界=判断力。

判断者应答（AI工作流版）：「我的工作流边界很清楚——能解决：B端配置类、CRUD类、有历史代码参照的需求（占团队70%）；不能解决：完全新增的复杂算法、跨多端的硬实时需求、PRD严重不完整的需求。边界外的场景我会主动 fallback 到人工——不是工作流失败，是设计就这样。判断标准：AI 能不能从 Reference 找到足够的事实依据，找不到就 fallback。」

诚实工程的两个机制：① Confidence Score（每条结论标置信度，低于阈值自动标红让人工确认）；② fallback 提示（AI 主动说"这需求超出我的能力，需要人工介入"）。

⚠️ 雷区：① "什么都能做"——这是不专业；② "什么都做不了"——这是自卑；③ 没讲"怎么判断在边界内还是外"——等于没设计。

边界
弱保证 vs 强保证（先认本质局限）
知识库天然是弱保证——业务在变代码在变知识就有过期风险。诚实承认比假装解决更专业

面试官想听：你能不能先认"我的方案是弱保证"，再讲缓解方案。急着给解法=不诚实。

判断者应答：「先认本质——知识库天然是弱保证，因为业务在变、代码在变、知识就有过期风险。我做的不是消除这个弱保证，是把弱保证的可控性做出来：① 每次蒸馏标注时间戳和来源（verified_by 字段）；② drift-report 检测代码与知识的不一致（代码改了知识没更新就告警）；③ 明确告知用户哪些信息可能过期需要二次确认。诚实工程——承认局限比假装解决更专业。」

三类保证的区分（被追问就讲）：① 强保证（编译器/类型系统）——错了立即报错；② 中保证（测试覆盖）——错了 CI 拦住；③ 弱保证（知识库/RAG）——错了可能上线才发现。AI 工程化的核心挑战就是把弱保证做成可控。

⚠️ 雷区：① "我们做了实时同步"——这是假装解决；② "知识库准确率99%"——这是不诚实；③ 没主动认弱保证——被追问"知识过时怎么办"才被动答。

边界
provenance vs validation（证据链≠对错）
证据链解决"从哪来"，对错靠另一套验证机制——两者是叠加不是替代

面试官想听：你能不能区分"可追溯"和"可信"——这是 AI 工程化的根本性区分。

判断者应答：「这两件事是分层设计的，不能混。证据链（provenance）回答"这条信息从哪来"——通过 verified_by 字段溯源到 PRD 行号或代码 file:line。验证（validation）回答"这条信息对不对"——另一套机制：① 源码契约对比；② 运行时类型检查；③ 集成测试；④ 人工 Review Gate。证据链保证可追溯，验证保证可信度，两者是叠加关系。」

典型反例（被追问就讲）：① 只有证据链没验证——AI 可能引用对的 PRD 行号但理解错了；② 只有验证没证据链——AI 答对了但说不清为什么对。京东国际面试官直接说"证据链只能说明 TRD 从 PRD 来，怎么确保真正理解了"——就是在考这个区分。

⚠️ 雷区：① "有了证据链就保证对"——混淆了两个概念；② "对错靠 AI 自己判断"——这是放任；③ 没讲清"两层叠加"。

边界
工具增量价值（N=0 vs N>0）
市面工具解决新项目（N=0），我解决存量项目（N>0）——这是结构性差异不是相对差异

面试官想听：你的工具和市面有什么真差异。讲"更轻量/更快"是相对差异（容易被超越），讲"N=0 vs N>0"是结构性差异（护城河）。

判断者应答：「市面 OpenSpec（5万+stars）/SuperPowers（2026最流行）/GitHub Spec-Kit（9万+stars）解决的是 0→1 写新代码——给一个全新项目从 spec 到 plan 到 implement。但滴滴这种几万行已有代码的项目，AI 理解不了上下文——我的方案独特价值在 "已有项目的领域知识前置"：扫现有代码+历史文档，蒸馏成 Reference 六文件，让 AI 先"见过"项目再蒸馏 PRD。市面工具解决 N=0 的场景，我解决 N>0 的场景——这是核心差异。」

判断增量价值的三个维度（被追问就讲）：① 结构性差异（如 N=0 vs N>0）——别人做不到；② 机制差异（如 verified_by 审计链）——别人没做；③ 场景差异（如团队推广版+原型版双轨）——别人没覆盖。三层都讲，越结构性越深。

⚠️ 雷区：① "我们更轻量"——这是相对差异；② "覆盖多少人"——这是规模不是增量；③ 没讲 N=0 vs N>0 这种结构性差异。

边界
闭环完整性（双轨定位）
团队版只做PRD→Plan是ROI选择——人工确认后才开发，风险低提效明显

面试官想听：你做的是只有一段还是完整闭环。如果只有一段，为什么不做完整闭环。

判断者应答：「双轨定位是 ROI 选择——① 团队推广版聚焦 PRD→Plan，因为这是 ROI 最高风险最低的环节：人工确认 Spec/Plan 后才开发，错了改 Plan 不改代码，风险可控；提效明显（5h→2h）；不涉及代码质量责任。20+人团队在用，验证了这个判断。② 个人独立做的端到端原型 prd2code-gen，作为下一代探索。两层定位是工程判断——不是不会做端到端，是判断团队场景下推 PRD→Plan 风险收益比最高。」

为什么端到端没在团队推广（被追问就讲）：① 代码生成涉及质量责任——AI 写错代码上线就是事故；② CI 对齐成本高——团队现有流水线和 AI 生成代码对不齐；③ 验证体系不成熟——LLM-as-Judge 还在探索。先用原型验证设计可行性，再决定工程化投入。

⚠️ 雷区：① "我们做了完整端到端"但实际只做一段——被追问就崩；② "团队版就够了"——等于没思考过完整闭环；③ 没讲"为什么选 PRD→Plan 这段"的 ROI 判断。

边界
闭环回避怎么破（6步闭环）
端到端6步闭环：蒸馏→分析→Plan→双路径生成→自验证自修复→eval反哺，每步都有工程化难点

面试官想听：被问"既然完整为啥不推广"——你能不能讲清 6 步完整闭环而不是回避。

判断者应答：「端到端 6 步闭环我都设计跑通了——① 蒸馏（PRD→IR中间表示，每个需求点抽出来标来源）；② 分析（IR+Reference对比源码契约，找差异）；③ Plan（生成开发计划+文件映射）；④ 双路径生成（Handlebars 模板 0 token 优先+LLM 兜底）；⑤ 自验证自修复（4类校验+3次重试+震荡检测 A→B→A）；⑥ eval/auto-tune 反哺（黄金 Case 集+A/B 对照）。每一步都能讲出具体工程化难点。」

每步的工程化难点（被追问任何一个深入讲）：① 蒸馏——IR 设计+verified_by 溯源；② 分析——BM25 索引+context-pack 锚定；③ Plan——变更分类（新增/修改/删除判断）；④ 双路径——模板命中率+LLM 兜底策略；⑤ 自验证——震荡检测+断点续传 progress.yaml；⑥ eval——LLM-as-Judge+pass^k 一致性。

⚠️ 雷区：① "老板让先做前两步"——这是回避不是判断；② 只讲步骤不讲难点——等于流水账；③ 没讲"为什么原型能跑通但不推广"（答：PRD→Plan ROI 最高+代码生成涉及质量责任）。

边界
端到端 vs 单段（团队版vs原型版）
团队版=只做PRD→Plan，原型版=完整6步——判断依据是风险/收益/责任，不是技术能力

面试官想听：你做架构选择的判断依据是什么。技术驱动还是业务驱动还是责任驱动。

判断者应答：「三个维度判断：① 风险——PRD→Plan 错了改 Plan（成本低），代码生成错了改代码甚至上线事故（成本高）；② 收益——PRD→Plan 占研发总时间 30%+，提效 60% 明显；代码生成节省时间但和 Cursor/Copilot 重叠；③ 责任——PRD→Plan 是辅助工具人最终拍板，代码生成涉及质量责任谁担。三个维度都指向推 PRD→Plan。」

原型版的价值（被追问就讲）：「原型不是浪费——它是设计验证：① 验证 6 步设计能跑通；② 验证工程化难点（震荡检测/断点续传/审计链）有解；③ 给团队'什么时候该推广端到端'留出判断空间。原型是判断力的训练，不是工作量。」

⚠️ 雷区：① "团队版就够了"——没思考过原型；② "原型不成熟所以没推"——这是借口不是判断；③ 没讲"三个维度怎么权衡"。

边界
Gap诚实度（主动承认局限）
主动认Gap比强行兜底更专业——一致性指标是已知Gap，规划用LLM-as-Judge补

面试官想听：你能不能主动说"这个我没做"——而不是被追问才承认。

判断者应答：「Gap 我主动认——① 执行一致性指标：同一个 PRD 跑 10 次结果差异多少？目前 40%→85%（团队版优化后），但还没到 95%+ 的工业级。规划用 LLM-as-Judge（用一个 LLM 评另一个 LLM 的产出）+ pass^k（跑 k 次取一致性）补。② 代码级验证：原型版的代码生成 correctness 靠集成测试+人工 Review，还没做到完全自动化。主动认 Gap 比强行兜底更专业——前者显判断力，后者显不诚实。」

三类 Gap（被追问就讲）：① 能力 Gap（AI 还做不到的，如复杂算法生成）——fallback 到人工；② 工程 Gap（能做到但工程化没到的，如一致性指标）——规划补；③ 场景 Gap（覆盖不到的，如跨多端硬实时）——明确边界。

⚠️ 雷区：① "什么都能做"——星河二面被追问"一致性指标"硬撑被识别；② 被动承认——蚂蚁问"知识库过时是不是弱保证"才答；③ 没讲"怎么补 Gap"的规划。

原理
RAG 完整流程（4步）
RAG不是塞文档给LLM，是chunking→embedding→混合检索→reranking四步——每步都有工程难点

面试官想听：你是真懂 RAG 还是只听过概念。能不能讲清4步+每步的工程难点。

判断者应答：「RAG 是 Retrieval-Augmented Generation，4步流程：① Chunking（文档切分）——按语义切不按字数，代码按函数切、文档按章节切；② Embedding（向量化）——选模型（OpenAI text-embedding-3 / BGE / Cohere），512维左右；③ 混合检索（BM25+向量）——BM25 抓关键词、向量抓语义，结果合并；④ Reranking（重排）——cross-encoder 重排，提高 top-k 准确率。每步都有工程难点，不是简单'塞文档给 LLM'。」

每步的工程难点（被追问就讲）：① Chunking 边界——代码块和文本块用不同策略；② Embedding 模型选——通用 vs 领域专用（如金融场景用 BGE-Finance）；③ 检索召回率——top-k 设多大、要不要 Query Rewrite；④ Reranking 成本——cross-encoder 慢但准，要不要缓存。

⚠️ 雷区：① "RAG 就是检索增强"——这是定义不是流程；② "用 OpenAI Embedding 就行"——这是选型不是原理；③ 没讲 reranking——等于没做完整 RAG。

原理
KV Cache 原理（不是本地存储）
Prompt Caching是服务端的KV cache——attention张量缓存，不是客户端的localStorage

面试官想听：你是真懂 prompt cache 还是把概念搞错了。蚂蚁面试三连问就考这个。

判断者应答：「Prompt Caching 本质是服务端的 KV cache——LLM 推理时每层 attention 都计算 K/V 张量，相同前缀的 K/V 算一次就够，缓存起来下次复用。不是客户端的 localStorage 或 sessionStorage——那些是浏览器层面的，跟 LLM 推理无关。Anthropic / OpenAI 的 Prompt Caching 都是 5min TTL，相同前缀省 90% 成本、降 85% 延迟。」

底层机制（被追问就讲）：① Transformer 每层 attention 算 Q/K/V——Q 是 query、K/V 是 key/value；② 推理时逐 token 计算，每个新 token 都要算与前面所有 token 的 attention；③ 如果前缀稳定（如 system prompt + tools schema），前缀部分的 K/V 可以缓存复用；④ 命中 cache 的判断是前缀完全匹配（hash 比对）。

⚠️ 雷区：① "缓存在本地文件"——蚂蚁面试就这样答错了；② "缓存对话历史"——对话历史在变不稳定的；③ 没讲 attention 张量——等于不懂底层。

原理
Function Calling 机制
LLM输出结构化JSON告诉调用方该执行哪个函数——不是LLM直接调用函数，是LLM决定调什么

面试官想听：你是真懂 Function Calling 还是把它和"LLM调用函数"混淆。

判断者应答：「Function Calling 本质是 LLM 输出结构化 JSON 告诉调用方该执行哪个函数——不是 LLM 直接执行函数（LLM 没有执行能力）。流程4步：① 声明：把可用函数的 schema 喂给 LLM（参数名/类型/描述）；② 决策：LLM 判断该调哪个函数、传什么参数；③ 输出：LLM 输出 {"name":"foo","args":{...}} 的 JSON；④ 执行：调用方解析 JSON、执行函数、把结果回传给 LLM。LLM 决定调什么，调用方执行。」

和 MCP 的区别（被追问就讲）：① Function Calling 是协议（LLM 输出 JSON 的格式）；② MCP（Model Context Protocol）是运行时标准（工具怎么注册、怎么发现、怎么调用）；③ Function Calling 是 LLM 层的，MCP 是平台层的——不是替代，是叠加。

⚠️ 雷区：① "LLM 直接调函数"——LLM 没执行能力；② "Function Calling 就是 MCP"——概念混淆；③ 没讲4步流程——等于不懂机制。

原理
MCP（Model Context Protocol）
MCP是工具插座标准——把工具注册/发现/调用规范化，让LLM能即插即用

面试官想听：你是真懂 MCP 还是用过一两个 MCP Server 就觉得自己懂了。

判断者应答：「MCP 是 Anthropic 推的工具插座标准——把'工具怎么注册/发现/调用'规范化。类比 USB 标准：USB 之前每个外设都有自己的接口，USB 之后即插即用。MCP 之前每个 AI 应用都自己定义工具调用格式，MCP 之后工具可以跨应用复用。一个 MCP Server 实现了标准接口（list_tools / call_tool），任何支持 MCP 的客户端（Claude Code / Cursor）都能用。」

和 Function Calling 的关系（被追问就讲）：① Function Calling 是 LLM 输出 JSON 的协议；② MCP 是工具运行时的标准；③ Function Calling 决定调什么，MCP 决定怎么调——叠加关系。

Skills vs MCP（被追问就讲）：① Skills 是流程编排方案（什么时候用什么工具）；② MCP 是工具插座标准（工具怎么注册）；③ Skills 编排流程，MCP 提供工具。

⚠️ 雷区：① "MCP 就是工具调用"——太浅；② "MCP 和 Function Calling 一样"——概念混淆；③ 没讲"插座标准"这个本质。

原理
Skills 调用原理
Skills是SKILL.md+frontmatter描述的流程编排——LLM按需懒加载，不是全量灌入

面试官想听：你是真做过 Skills 还是只听过概念。腾讯面试问"LangGraph 懒加载"就考这个。

判断者应答：「Skills 本质是 SKILL.md + frontmatter 描述的流程编排方案——告诉 LLM 什么时候该做什么。机制3步：① 注册：每个 Skill 在 frontmatter 里声明 name / description / trigger；② 触发：LLM 根据用户输入匹配 Skill 的 trigger（关键词/意图）；③ 懒加载：匹配到才加载该 Skill 的 SKILL.md 内容，不是全量灌入所有 Skill——避免上下文爆炸。」

和工具调用关系（被追问就讲）：① Skills 是高层编排（什么时候做什么）；② MCP/Function Calling 是底层执行（怎么调用工具）；③ Skills 决定流程，MCP 提供工具。

工程化要点：① Skill 数量多时 trigger 匹配要准——多个 Skill 同时匹配时的优先级；② Skill 内容要简洁——SKILL.md 越长越费 token；③ Skill 失败时的 fallback——回退到人工还是重试。

⚠️ 雷区：① "Skills 是工具"——Skills 是编排不是工具；② "全量加载所有 Skill"——这是反模式；③ 没讲 trigger 匹配——等于不懂触发机制。

原理
Context Engineering（上下文工程）
窗口有限知识无限——管理上下文是AI工程化的核心，比prompt工程高一层

面试官想听：你是真懂上下文工程还是只会写 prompt。

判断者应答：「Context Engineering 比 Prompt Engineering 高一层——prompt 工程关注'说什么'，context 工程关注'把什么喂给模型'。核心矛盾：窗口有限（200K-1M token），知识无限。三层解法：① 压缩：长上下文做摘要、做 RAG 检索相关片段；② 分层：稳定前缀（system prompt / tools schema）走 cache，动态部分（用户输入 / 对话历史）按需组织；③ 调度：Subagent 独立 context、主会话只留摘要，长任务不爆主上下文。」

典型反例（被追问就讲）：① 把所有文档全塞进 prompt——爆窗口；② 对话历史无脑累积——后期 token 爆炸；③ 全量加载所有 Skill——上下文污染。

落到我的项目：① Reference 六文件就是 context 压缩（几万行代码→6个结构化文件）；② verified_by 溯源避免重复塞代码；③ Subagent 独立 context——loop 系统每轮 fresh 上下文。

⚠️ 雷区：① "Context Engineering 就是 prompt 工程"——概念混淆；② "上下文越长越好"——爆窗口+越拖越慢；③ 没讲"窗口有限知识无限"这个矛盾。

原理
Agent Loop 主循环
ReAct四步循环：思考→工具调用→工具结果→继续思考——外层harness驱动，直到stop_reason

面试官想听：你是真懂 Agent 还是只会调 SDK。

判断者应答：「Agent Loop 本质是 ReAct 循环——Reason（思考）+Act（行动）。4步循环：① 思考：LLM 判断下一步该做什么；② 工具调用：LLM 输出 tool_use（调哪个工具+参数）；③ 工具结果：harness 执行工具、返回 tool_result；④ 继续思考：LLM 基于结果判断是否结束。循环直到 stop_reason=end_turn。和单次 API 调用的本质区别是：单次是一问一答，Agent Loop 是外层 harness 驱动的多轮自主决策。」

工程化要点（被追问就讲）：① tool_use/tool_result 配对——每个 tool_use 必须有对应 tool_result，否则 LLM 会困惑；② stop_reason 判断——end_turn（自然结束）/ max_tokens（截断）/ stop_sequence（命中停止词）；③ 循环上限——防止死循环，max_iterations 兜底。

典型坑：① 工具结果格式错（如返回 NaN 或 undefined）——LLM 可能死循环；② 工具调用失败不返回错误——LLM 不知道发生了什么。

⚠️ 雷区：① "Agent 就是 LLM"——Agent 是 LLM+工具+harness 循环；② "调一次 LLM 就是 Agent"——这是单次问答不是 Agent；③ 没讲 ReAct——等于不懂原理。

原理
SSR + Hydration 详解
SSR服务端渲染HTML+Hydration客户端水合成可交互——hydration mismatch是核心坑

面试官想听：你是真懂 SSR 还是用过 Next.js 就觉得自己懂了。大麦面试问 SSR 就是考这个。

判断者应答：「SSR（Server-Side Rendering）——服务端把组件渲染成 HTML 字符串返回给浏览器，用户首屏看到内容（不用等 JS 加载）。Hydration（水合）——浏览器加载 JS 后，React 把静态 HTML'水合'成可交互的组件树（绑定事件、接管状态）。核心矛盾：服务端和客户端渲染结果必须一致——不一致就是 hydration mismatch。」

hydration mismatch 的典型原因（被追问就讲）：① 服务端/客户端时间不一致（如 Date.now()）；② 服务端/客户端环境差异（如 window 在服务端 undefined）；③ 异步数据服务端没等到客户端等到了。解法：① useEffect 里读时间（客户端才执行）；② typeof window !== 'undefined' 判断环境；③ useState 初始值保证一致。

SSR vs CSR vs SSG：SSR 每次请求都渲染；CSR 客户端渲染（SPA）；SSG 构建时渲染（博客/文档）。

⚠️ 雷区：① "SSR 就是服务端渲染"——这是定义不是原理；② 没讲 hydration——等于不懂 SSR 全流程；③ 没讲 mismatch——等于没踩过坑。

原理
Promise 三态 + 链式
三态pending→fulfilled/rejected不可逆，链式靠then返回新Promise实现

面试官想听：你是真懂 Promise 还是只会 .then .catch。

判断者应答：「Promise 三态：pending（进行中）→ fulfilled（成功）/ rejected（失败）。关键特性：状态不可逆——一旦从 pending 变成 fulfilled/rejected 就锁死，不能再变。链式调用原理：.then(fn) 返回一个新 Promise，fn 的返回值成为这个新 Promise 的 resolve 值——所以能 .then().then().then() 串起来。then 是微任务——同步代码执行完才执行。

关键细节（被追问就讲）：① Promise.resolve(thenable) 会自动展开 thenable——所以 Promise.resolve(anotherPromise) 会等另一个完成；② return 普通值——下一个 then 直接拿到；throw 或 return Promise.reject——跳到 catch；③ Promise.all 全成才 resolve，一个失败就 reject（fail-fast）；④ Promise.allSettled 全都 settled 才 resolve，不 fail-fast。

常见坑：① 忘记 return——链断了；② .then().catch().then()——catch 后的 then 也会执行（catch 返回 fulfilled Promise）。

⚠️ 雷区：① "Promise 是异步"——Promise 是状态机，then 才是异步；② "reject 后会停"——reject 后下一个 catch 会执行；③ 没讲微任务——等于不懂事件循环。

原理
事件循环（宏任务+微任务）
宏任务+微任务两队列——每轮宏任务清空所有微任务再渲染，Promise是微任务setTimeout是宏任务

面试官想听：你是真懂事件循环还是会背"先微后宏"。

判断者应答：「事件循环两队列：① 宏任务（Macrotask）——setTimeout/setInterval、I/O、UI 事件；② 微任务（Microtask）——Promise.then、queueMicrotask、MutationObserver。执行顺序：执行一个宏任务 → 清空所有微任务 → 渲染（如果有）→ 下一个宏任务。关键：每个宏任务后清空所有微任务，不是只清一个。」

经典题（被追问就讲）：

· console.log(1)

· setTimeout(() => console.log(2))

· Promise.resolve().then(() => console.log(3))

· console.log(4)

输出：1, 4, 3, 2 ——同步代码先跑（1,4），然后清微任务（3），然后下一个宏任务（2）。

Node 事件循环6阶段（BFF必考）：timers → pending callbacks → idle/prepare → poll → check → close callbacks。微任务在每个阶段切换间清空，不是每个宏任务后。

⚠️ 雷区：① "Promise 比 setTimeout 快"——是因为微任务优先于宏任务；② 没讲"清空所有微任务"——只清一个就错；③ Node 和浏览器没区分——6阶段 vs 单阶段。

闭环
自验证+自修复循环

闭环
震荡检测（A→B→A）
LLM在两种方案间反复横跳要立即识别——切整段重写而不是逐行patch

面试官想听：你有没有踩过 LLM 的真实坑。震荡是 LLM 修复的典型反模式。

判断者应答：「震荡是 LLM 自修复的核心反模式——LLM 第1次生成 A，校验失败让它改，第2次改成 B，再失败让它改，第3次又改回 A。不识别震荡就会无限重试。我的解法：对比 history，检测 A→B→A 模式，识别到立即切整段重写（清空当前段重新生成）而不是逐行 patch。核心判断：震荡说明 LLM 在两个方案间没决断，逐行 patch 永远解决不了，必须重置。」

检测机制（被追问就讲）：① history 保留每次修改的 hash；② 新版本生成后和 history 前3次对比；③ 如果 hash 匹配（A→B→A），触发震荡处理。震荡处理两步：① 标记震荡点；② 提示 LLM "你在 A 和 B 间反复，必须二选一并解释理由"，强制决策。

其他震荡变体：① A→B→C→A（长循环）；② A→A'（小幅波动，A' 是 A 的微调）——前者重写整段，后者接受 A'。

⚠️ 雷区：① "无限重试"——不识别震荡死循环；② "震荡也算失败"——A' 应该接受；③ 没讲切整段重写——等于没解法。

面试官想听：你考虑过长任务的失败恢复吗。从头重跑等于浪费时间。

判断者应答：「端到端6步是长任务（10-15分钟），中间失败必须能断点续传。解法：每步完成后写 progress.yaml，记录：① 已完成步骤；② 每步的产物路径（如 spec.md / plan.md）；③ 中间状态（如蒸馏出的 IR）。失败重试时读 progress.yaml，从失败步骤继续，不是从头跑。这比从头重跑省 80% 时间。」

关键设计（被追问就讲）：① 步骤幂等——同一步重跑结果一致（避免副作用）；② 产物版本化——每个 progress 对应一次跑（避免覆盖上次结果）；③ 校验失败和系统失败区分——前者重试该步，后者（如断网）整体暂停等人工。

典型坑：① 没持久化中间 IR——重试时 LLM 重新蒸馏结果可能不一致；② 步骤不幂等——重试有副作用（如多次写入数据库）。

⚠️ 雷区：① "失败了重跑"——长任务重跑成本高；② "全部存内存"——断电就丢；③ 没讲幂等——重试可能有副作用。

判断者应答：「知识库天然是弱保证——业务在变、代码在变、知识就有过期风险。drift-report 是把"可控"做出来的机制：定期（如每次跑 PRD 蒸馏前）扫描代码，对比知识库（Reference 六文件），生成漂移报告：① 哪些代码改了但知识库没更新；② 哪些知识库引用的代码已删除；③ 哪些契约（API 字段）变了。漂移超阈值就触发知识库重新蒸馏。」

实现机制（被追问就讲）：① 代码 hash 对比——知识库记录引用代码的 hash，跑时重新算 hash 对比；② 符号级对比——知识库引用的 symbol（函数/类）现在还在不在；③ 契约对比——API 字段名/类型变了就告警。

⚠️ 雷区：① "知识库准确率 99%"——这是不诚实；② "实时同步代码"——成本太高；③ 没讲阈值——非黑即白不专业。

选型

面试官想听：你是不是多 Agent 万能论者。能不能讲清什么场景该用哪个。

判断者应答：「选型是 ROI 驱动不是技术驱动——① 单 Agent：一个 LLM + 一组工具，适合线性流程（输入→处理→输出）。成本：1次 LLM 调用。覆盖：60-70% 场景；② 多 Agent：多个 LLM 分工，适合并行/专业分工场景。成本：N 次调用 + 编排开销。覆盖：30-40% 复杂场景。多 Agent 的代价是编排复杂度+成本翻倍。」

我的实践：「loop 系统用单 Agent + Subagent 模式——主 Agent 编排，Subagent 独立 context 干活。是介于单/多 Agent 的中间态。选择依据：主 Agent 要保持 context 简洁，Subagent 干完即销毁。」

⚠️ 雷区：① "多 Agent 更先进"——这是技术驱动不是 ROI 驱动；② "单 Agent 够了"——复杂场景编排不动；③ 没讲成本——多 Agent 成本爆炸不知道。

手写 Promise.all
三关键点：返回新Promise+计数器+fail-fast（一个失败整体reject）

判断者应答（边写边讲）：

Promise.myAll = function(promises) {
  return new Promise((resolve, reject) => {
    const arr = Array.from(promises);
    const result = new Array(arr.length);
    let count = 0;
    if (arr.length === 0) return resolve([]);  // 空数组立即resolve
    arr.forEach((p, i) => {
      Promise.resolve(p).then(val => {
        result[i] = val;  // 按下标存，不是push
        if (++count === arr.length) resolve(result);
      }, reject);  // fail-fast：一个失败整体reject
    });
  });
};

三个关键点（必讲）：

· ① 返回新 Promise——内部 resolve/reject 控制结果。

· ② 计数器——count === length 才 resolve，不是判断数组填满。

· ③ fail-fast——一个失败立即 reject，不等其他完成。

对比 allSettled：allSettled 不 fail-fast，每个都包装成 {status, value/reason}，全 settled 才 resolve。

⚠️ 雷区：① 用 push 不用下标——顺序错乱；② 没处理空数组——Promise.all([]) 立即 resolve；③ 没 Promise.resolve 包装——非 Promise 值出错。

手写 Promise.race
第一个settled就决定结果——后续无效

判断者应答（边写边讲）：

Promise.myRace = function(promises) {
  return new Promise((resolve, reject) => {
    const arr = Array.from(promises);
    arr.forEach(p => {
      Promise.resolve(p).then(resolve, reject);  // 第一个settled就resolve/reject
    });

};

核心点（必讲）：

· 第一个 settled 决定结果——不管是 fulfilled 还是 rejected。

手写
Promise 并发调度器
限制并发数+队列管理——核心是递归触发+空闲时取下一个

判断者应答（边写边讲）：

class Scheduler {
  constructor(max) {
    this.max = max;
    this.count = 0;
    this.queue = [];
  }
  add(task) {
    return new Promise(resolve => {
      this.queue.push({ task, resolve });
      this.run();
    });
  }
  run() {
    if (this.count >= this.max || this.queue.length === 0) return;
    const { task, resolve } = this.queue.shift();
    this.count++;
    task().then(() => {
      this.count--;
      resolve();
      this.run();  // 递归触发下一个
    });
  }
}

核心机制（必讲）：

· ① 计数器+队列——count < max 才执行，超了入队等。

· ② 递归触发——任务完成后调 run() 取下一个。

· ③ Promise 包装——add 返回 Promise，任务完成才 resolve。

使用示例：scheduler.add(() => fetch(url1))。

⚠️ 雷区：① 没递归触发——任务完成后队列不动；② count 没减——后续任务跑不了；③ 没用 Promise 包装——拿不到任务完成时机。

class EventEmitter {
  constructor() { this.events = {}; }
  on(event, fn) {
    (this.events[event] ||= []).push(fn);
    return this;
  }
  emit(event, ...args) {
    (this.events[event] || []).forEach(fn => fn(...args));
    return this;
  }
  off(event, fn) {
    this.events[event] = (this.events[event] || []).filter(f => f !== fn);
    return this;
  }
  once(event, fn) {
    const wrapper = (...args) => {
      fn(...args);
      this.off(event, wrapper);
    };
    this.on(event, wrapper);
    return this;

}

深拷贝：

function deepClone(obj, cache = new WeakMap()) {
  if (obj === null || typeof obj !== 'object') return obj;
  if (cache.has(obj)) return cache.get(obj);  // 循环引用
  if (obj instanceof Date) return new Date(obj);
  if (obj instanceof RegExp) return new RegExp(obj);
  const clone = Array.isArray(obj) ? [] : {};
  cache.set(obj, clone);
  Object.keys(obj).forEach(k => clone[k] = deepClone(obj[k], cache));
  return clone;
}

关键点：① WeakMap 处理循环引用；② Date/RegExp/Symbol 特殊处理；③ 递归到基本类型终止。

function debounce(fn, wait, immediate = false) {

    if (immediate && !timer) fn.apply(this, args);  // 首次立即执行
    timer = setTimeout(() => {
      if (!immediate) fn.apply(this, args);  // 末次执行
      timer = null;
    }, wait);
  };
}

LRU 缓存

判断者应答（Map 实现，最简洁）：

class LRUCache {
  constructor(capacity) {
    this.capacity = capacity;
    this.cache = new Map();
  }
  get(key) {
    if (!this.cache.has(key)) return -1;
    const val = this.cache.get(key);
    this.cache.delete(key);  // 删除再set，更新顺序
    this.cache.set(key, val);
    return val;
  }
  put(key, val) {
    if (this.cache.has(key)) this.cache.delete(key);
    this.cache.set(key, val);
    if (this.cache.size > this.capacity) {
      const oldest = this.cache.keys().next().value;  // 取最老的key
      this.cache.delete(oldest);
    }
  }
}

核心机制（必讲）：

· ① Map 天然保序——ES2015 规范：Map 按插入顺序迭代。

· ② get 时 delete+set 更新顺序——把刚访问的移到最新。

· ③ keys().next() 取最老的——迭代器第一个就是最久未访问。

网易三连追问（必答）：

· "get 里为什么 delete 再 set"——更新访问顺序，把刚访问的移到最新位置。

· "Map.keys().next() 为什么能取到最老的"——Map 按 ES2015 规范按插入顺序迭代，迭代器第一个就是最早插入的（如果没被 delete 重排过）。

双向链表实现（被问"不用 Map 怎么做"必答）：哈希表+双向链表，O(1) 查找+O(1) 调整顺序。Map 实现简单但占用稍高。

⚠️ 雷区：① 直接 set 不 delete——顺序没更新；② size > capacity 才淘汰——边界条件。

判断者应答（边写边讲）：

function buildTree(items, rootId = null) {
  const map = new Map();  // id -> node
  const tree = [];
  // 先建map
  items.forEach(item => {
    map.set(item.id, { ...item, children: [] });
  });
  // 再挂children
  items.forEach(item => {
    const node = map.get(item.id);
    if (item.parentId === rootId) {
      tree.push(node);
    } else {
      const parent = map.get(item.parentId);
      if (parent) parent.children.push(node);
    }
  });
  return tree;
}

核心机制（必讲）：

· ① Map 存 id 到节点——O(1) 查找。

· ② 两次遍历——第一次建 Map，第二次挂 children。

· ③ O(n) 不是 O(n²)——避免递归 find parent。

变种：① 树转扁平——DFS 遍历收集；② 树过滤——递归 filter+保留 children；③ 树查找——递归或 BFS。

典型应用：① 权限树/菜单树/组织架构；② 文件系统；③ 评论楼中楼。

⚠️ 雷区：① 递归 find parent——O(n²) 性能差；② 没处理 parentId 不存在——节点丢失；③ 没区分 root——可能多个根。

    let l = i + 1, r = nums.length - 1;
    while (l < r) {
      const sum = nums[i] + nums[l] + nums[r];
      if (sum === 0) {
        result.push([nums[i], nums[l], nums[r]]);
        while (l < r && nums[l] === nums[l + 1]) l++;  // 去重
        while (l < r && nums[r] === nums[r - 1]) r--;
        l++; r--;
      } else if (sum < 0) l++;
      else r--;
    }
  }
  return result;
}

滑动窗口典型题：

① 无重复字符的最长子串（LC3）：

function lengthOfLongestSubstring(s) {
  const set = new Set();
  let l = 0, max = 0;
  for (let r = 0; r < s.length; r++) {
    while (set.has(s[r])) {
      set.delete(s[l]);
      l++;
    }
    set.add(s[r]);
    max = Math.max(max, r - l + 1);
  }
  return max;
}

核心判断（被问就讲）：「看到'最长/最短子串/连续区间'就用滑动窗口；看到'有序数组+两数关系'就用双指针。」

⚠️ 雷区：① 三数之和无去重——重复解；② 滑动窗口没收缩——逻辑错；③ 边界条件（l < r）忘写——越界。

手写
快速排序+第K大
快排核心是partition；第K大用partition减治——不用全排序O(n)

快速排序：

function quickSort(nums, l = 0, r = nums.length - 1) {
  if (l >= r) return nums;
  const pivot = partition(nums, l, r);
  quickSort(nums, l, pivot - 1);
  quickSort(nums, pivot + 1, r);
  return nums;
}
function partition(nums, l, r) {
  const pivot = nums[r];
  let i = l;
  for (let j = l; j < r; j++) {
    if (nums[j] < pivot) {
      [nums[i], nums[j]] = [nums[j], nums[i]];
      i++;
    }
  }
  [nums[i], nums[r]] = [nums[r], nums[i]];
  return i;
}

第K大（LC215）——用partition减治：

function findKthLargest(nums, k) {
  const target = nums.length - k;  // 第K大 = 升序第n-k
  let l = 0, r = nums.length - 1;
  while (l < r) {
    const pivot = partition(nums, l, r);
    if (pivot === target) return nums[pivot];
    else if (pivot < target) l = pivot + 1;
    else r = pivot - 1;
  }
  return nums[l];
}

核心机制（必讲）：

· ① partition——选 pivot，把数组分成 < pivot / = pivot / > pivot 三部分。

· ② 减治——第 K 大只用递归包含 target 的那一半，不是两边都递归。

· ③ 复杂度——平均 O(n)，最坏 O(n²)（已排序数组+选末端 pivot）。

变种：① 三路快排（处理大量重复）；② 随机化 pivot 避免最坏情况；③ 小数组用插入排序。

⚠️ 雷区：① 第 K 大直接排序——O(nlogn) 不是最优；② 没去重——重复元素错；③ pivot 选末端——已排序数据退化。

手写
反转链表+环检测
反转用三指针prev/curr/next；环检测用快慢指针相遇判定

反转链表（迭代）：

function reverseList(head) {
  let prev = null, curr = head;
  while (curr) {
    const next = curr.next;
    curr.next = prev;
    prev = curr;
    curr = next;
  }
  return prev;
}

反转链表（递归）：

function reverseList(head) {
  if (!head || !head.next) return head;
  const newHead = reverseList(head.next);
  head.next.next = head;
  head.next = null;
  return newHead;
}

环检测（快慢指针）：

function hasCycle(head) {
  let slow = head, fast = head;
  while (fast && fast.next) {
    slow = slow.next;
    fast = fast.next.next;
    if (slow === fast) return true;
  }
  return false;
}

环检测+找入口（LC142）：

function detectCycle(head) {
  let slow = head, fast = head;
  while (fast && fast.next) {
    slow = slow.next;
    fast = fast.next.next;
    if (slow === fast) {
      // 相遇后，一个从head出发，一个从相遇点出发
      let p = head;
      while (p !== slow) {
        p = p.next;
        slow = slow.next;
      }
      return p;
    }
  }
  return null;
}

核心机制（必讲）：① 反转——三指针轮转；② 环检测——快慢指针，快 2 步慢 1 步，有环必相遇；③ 找入口——相遇后一个从头出发再以同样速度走，相遇点就是入口（数学可证）。

⚠️ 雷区：① 反转丢了 next——next 要先存；② 环检测 fast 走 3 步——可能跳过；③ 找入口的数学证明没讲——被追问答不出。
