◆ 架构
整体架构为什么拆双插件、为什么选 Skill 不走 MCP
整套跑在 Claude Code 里、是纯 Markdown Skill(零外部依赖);reference 建库 + prd-distill 蒸馏拆双插件做上下文隔离;选 Skill 不选 MCP——MCP 是后续路线不是弯路

整体长什么样。prd-tools 拆两个 Claude Code 插件:reference(DEFINE 阶段,长期知识库,8 模式覆盖构建、增量、反馈回流)+ prd-distill(PLAN 阶段,单次 PRD 蒸馏 11 步),各一个 Skill;prd2code-gen 再把 build-reference → prd-distill → bff-gen 三个 Skill 串起来,从 PRD 一路走到 BFF 代码。横在中间的是能力面适配器 layer-adapters,前端 9 个 surface、BFF 10 个、后端一组(api_surface、domain_model 等),把不同项目千差万别的目录抽象成统一的关注维度。

怎么跑。适配器先把 PRD 需求点映射到对应的代码能力面——走哪层、动哪个 surface,reference 按需只加载涉及的那一层、不全量塞。Skill 本身是纯 Markdown 指令,由 Claude Agent 用原生工具(Read、Edit、Glob、Grep、Bash)执行,不调用 src/ 下的 TS;那部分是离线开发辅助(模板引擎、评测、auto-tune),跟 Skill 运行时解耦。蒸馏每步产出 YAML 中间文件,下一步读上一步状态往下走。

为什么选 Skill、不选 MCP。选 Skill(纯 Markdown、零外部依赖)而不是 MCP Server,是因为 MCP 过早实现维护成本高、会分散 MVP 重点——ADR-0009 把 MCP Server 化列为 Phase 5 后续路线,源码从没实现过 MCP、不是试过放弃,代价是暂时没有跨客户端复用能力。拆双插件同理是为上下文隔离:项目知识理解和需求分析是两种认知任务,上下文混一起 AI 容易跳步、幻觉,拆开后各自独立上下文窗口、按需加载,代价是多一次加载、reference 得先建好 distill 才能用。

代价和边界。双插件多了协调成本,中间文件传递和状态一致性要靠门禁兜;能力面适配器换新项目得先把 surface 划分配置一遍,不能开箱即用;整条链路绑 Claude Code 运行时,跨客户端复用要等 MCP 化。目前是个人原型探索、非生产级闭环,reference 召回率还没有 labeled query set 量化,eval 是债。
