完整链路（要能背出来）。resource 资源准备 → tool_health 健康统计 → probe 实时探活 → decompose 子句拆解 → route 意图路由 → prefetch 预取 → plan 任务规划 → execute 执行（db/rag/code）→ synth 合成 → critic 质检 → finalize 收尾。旁路四条：信息不足走 clarify 主动追问、要查外网走 web_search、高风险走 admin_confirm 等确认节点、质检不过走 fix 修复。

为什么 probe 排在 plan 前面（最高频考点）。probe 在规划之前先并行探一遍下游能力——RAG 有没有文档、DB schema 能不能读、Code 服务通不通，拿到客观快照再喂给路由和规划。模型基于"下游当前是否可用"决策，而不是脑补下游状态，减少"规划完成后才发现下游不可用"的幻觉式路由。

为什么用 LangGraph + 路由错乱怎么兜底。多 Agent 编排有分支、回环、人工确认、失败恢复，本质是状态机，LangGraph 的节点+条件边、checkpoint（检查点）暂停恢复、HITL（人工介入）正好对位。路由错乱靠三层兜底：意图分类与路由规则分流、状态机限定节点跳转方向、日志和指标回看错误样本驱动规则进化。

我的取舍。probe 排在 plan 前面是我定的顺序——本可以让模型规划完再去探下游、或干脆省掉 probe 让模型脑补可用性，但那样路由决策建立在"假设下游活着"上，幻觉式路由就是从这来的。把客观快照前置，是用一次确定性的并发探活换掉路由环节的一个幻觉来源，代价是多一个节点、多几百毫秒。

编排层（总管）职责。维护会话状态、意图路由、任务规划、结果汇总、Critic 质检、高风险 HITL——调度编排归它，不碰任何专业领域的脏活：不写 SQL、不建向量、不改代码。避免"一个 Prompt 包打天下"的耦合。

专家层（Specialist）职责。db 做 NL2SQL（自然语言转 SQL）、rag 做检索、code 改代码、crawler 爬数据，各自独立服务、独立部署、能独立扩展。叫"Agent"是概念抽象，本质是总管通过 RPC（远程过程调用，HTTP/WebSocket）调用的独立服务，不是同进程函数，也不是 LangGraph subgraph（子图）——子图要求同进程，跨服务边界用不了。

内部协作者：何时用确定性代码、何时用 LLM。clean/visualize/report 在图内完成，不算独立 Agent。原则是 DB 出表后画图表走 chartOption 规则生成，而不是再调一次大模型画 JSON——确定性的事不交给概率模型。

我的取舍。编排层不碰专业细节这条边界是我守的——本可以把 RAG 检索或 NL2SQL 直接做进总管省一次跨服务调用，但那样编排和专家能力耦合，换一个 Specialist 实现就得动总管。把职责切干净，代价是多一跳 RPC，换的是十几个下游能独立部署、独立扩展、独立换实现。

区别在哪。一次普通 LLM 调用是无状态问答，问完就完。Critic 是状态机里的一个节点：有结构化输出（比如 {pass, reason}），专门审结果的证据完整性、事实对不对，审完不通过能触发 fix 节点进入重试闭环，不是孤立的一问一答。

为什么要单独做成节点。质检要可审计（留痕否决几次、为什么）、要能驱动重试，塞进 synthesize 里质检和生成搅在一起就模糊。顺带：Critic 审"事实对不对"，还有个 Evaluator（评估器）评"够不够好"，聚焦不同所以分开。

一个分层上的取舍。Critic 做事实审查本来需要懂专家产出（懂 SQL、懂截图），但这破坏"编排层不碰专家细节"的分层。折中是让 Critic 审"Agent 自己生成的 summary + evidence 列表"做间接审查，而不是盯 raw output。

我的取舍。Critic 做成图里的独立节点、而不是塞进 synth 顺带质检，是我定的结构——本可以在合成阶段让同一个 LLM 调用既出结果又自检，省一次调用，但生成和审查放一起等于让学生自己批自己的卷子，否决几次、为什么否决都留不下来。拆成节点要多一次 LLM 调用，换来的是质检可审计、能驱动 fix 重试，简单任务走快速路径跳过它就行。

它是什么。一个跨 Agent 的任务结构，总管派活时按这个格式组装：意图、允许调用哪些 Agent、查询内容、抽出来的实体、上下文、结果怎么回传。下游每个 Specialist 都按这份契约接活、回结果。

为什么重要（SSOT 单一数据源）。agentExecutors 是跨 Agent 调用的统一入口，十几个下游都按 managerTask 消费。如果每个 Agent 自定义一套任务格式，schema 就乱了——字段对不齐、错误模型不统一。统一契约让"怎么派活、怎么回结果"只有一份真相。

意义：解耦。总管换路由策略、下游某个 Specialist 换实现，只要还认这个协议就能互通。这是跨进程多 Agent 能独立扩展、独立部署的根基。

我的取舍。定一份跨 Agent 的统一任务契约是我定的——本可以让每个 Specialist 自己定义任务格式、总管按下游适配，那样上手快、不用先设计 schema，但十几个下游各一套格式，字段对不齐、错误模型不统一，改一处要动十几个适配层。把"怎么派活、怎么回结果"收敛成一份真相，代价是前期定义协议的成本，换的是总管和下游能独立演进。

为什么用 WebSocket。多轮对话、流式输出、确认/取消事件都更适合长连接，HTTP 单向请求撑不住整段长会话。WebSocket 入口在 manager-ws。

阶段事件（phase 协议）。不是一次性把结果返回，而是按阶段推事件——路由了、规划了、在执行哪个 Agent、等确认了、合成完了，前端能边收边渲染进度，用户看得到"现在到哪一步"。

为什么这样设计。多 Agent 任务耗时长、中间状态多，阶段事件让交互不卡在"转圈等待"，确认/取消也能在任意阶段插进来，配合 HITL 的暂停恢复。

我的取舍。按阶段推事件、而不是等整段结果一次性返回，是我定的交互协议——本可以用 HTTP 轮询或 SSE（服务器推送事件），但多 Agent 任务耗时长、中间还要插确认取消，HTTP 撑不住长会话、SSE 又是单向推不了 confirm/cancel。选 WebSocket 长连接，代价是连接管理和鉴权更复杂，换来的是任意阶段能暂停、能续跑、能边收边渲染。

为什么需要。多 Agent 系统一次请求要穿过 probe、route、plan、execute、critic 一串节点和多个子 Agent，出问题（路由错了、子 Agent 幻觉、Critic 漏审）很难靠日志反推，必须能追踪到每个节点、每次调用。

怎么做。LangSmith 做 trace 到 node 级，看每个节点的入参出参与耗时；指标接口和脚本看路由质量和稳定性；聊天页挂进化看板。把"哪一步慢、哪一步错"从主观判断变成可查的事实。

诚实边界。当前是 demo 级——trace（追踪）到 node（节点）不到 span（跨度），dashboard 和告警没做全。可观测性是多 Agent 生产化的硬骨头：demo 阶段看不到价值，生产阶段定位问题全靠它。

我的取舍。可观测我定的是 trace 到节点级、用 LangSmith 留痕——本可以只靠 console.log 加错误码兜底，那样轻量、不依赖外部服务，但多 Agent 一次请求穿好几个节点和子 Agent，出问题靠日志反推根本对不上链路。把每个节点的入参出参和边的选择都留痕，代价是 trace 数据量和接入成本，换来的是"哪一步错"能查到事实。

Probe + Health（规划前探活）。规划前先快照下游能力——RAG 有没有文档、DB schema 能不能读——把客观快照喂给路由和规划，减少幻觉式路由。Probe 放在 plan 之前是这个图最关键的顺序选择。

Retrieve-first（先检索再合成）。RAG 走先检索（/api/retrieve）再 synth 合成，比端到端 chat 稳——先拿到证据再回答，不是凭模型记忆编。

Evidence Gate（没证据不强行答）。没有引用、没有数据时不强行给最终答案，宁可走 clarify 追问或承认查不到，从机制上防幻觉。

Retry Budget + Platform Sync。多步失败有重试上限，避免无限重试拖死流程；端点和模型配置从平台拉取（编排与配置解耦），不在代码里写死。

我的取舍。这几个模式的共同取向是我定的——都是"用确定性约束堵 LLM 的幻觉口子"，本可以相信模型能力、把校验和探活都交给模型自己判断，那样实现简单、代码少，但 probe、evidence gate、retry budget 这类约束一旦拿掉，错就是概率性地出、还查不清是哪一步放水。宁可多写几道机制上的硬约束，也不把可信度押在模型自觉上。

解决什么问题。写库、调外部服务、大批量改这类高风险操作必须能暂停确认，而不是闷头执行；长任务中途崩溃要能从断点恢复，不能从头跑。

怎么做。高风险触达时 graph 不直接发最终结果，而是存 checkpoint 等人工确认；WebSocket 协议里有 confirm/cancel 事件，确认后续跑到 synth。持久化用自建业务快照而不是 LangGraph 原生 saver（持久化器）——业务字段比框架 channel（通道）快照更可控。

安全。WebSocket 暴露公网必须加鉴权和 TLS；高风险操作必须保留确认步骤；子 Agent 的密钥和内部 URL 不泄露到前端。

诚实边界。checkpoint 存的是 graph 状态和节点进度，副作用（已发邮件、已写库）不存，所以是 at-least-once（至少一次）不是 exactly-once（恰好一次），跨进程续跑可能重复执行副作用——Coding/Agent 系统的共同难题。

怎么验证 / 沉淀。checkpoint 续跑的正确性目前靠人工跑一遍高风险流程的断点恢复来验证，没接真实流量所以没有线上数据；若要正经度量，会按"断点恢复成功率 + 副作用重复率"两个指标看。这件事的本质是"高风险操作必须有兜底开关"，下次遇到任何 AI 能触发副作用的系统，第一步先把暂停确认和状态落盘做进去，而不是先优化效果。

向量召回 + 用户画像（经验回放）。成功 task 的经验写进向量记忆，用 embedding（向量编码）加相似度混合排序，下次相似 query 召回注入路由的长期记忆；会话用户画像按 sessionId 积累意图偏好与成功任务摘要。

Prompt 补丁 + 策略金丝雀。失败洞察生成 Prompt 补丁先写 shadow 文件，金丝雀按 sessionId 稳定分桶跑小流量，达标才 promote 注入 router/planner；Planner 硬规则同样走 shadow 进化路径。

Planner 硬规则 + 健康探测 + Curator。plan_lint（计划校验）强制执行 Planner 硬规则，违规直接拦；tool_health 做全 Agent 健康探测，down 的不进 allowedAgents；后台 Curator（治理器）定时治理记忆、进化 Prompt 与规则；Agent 注册表对外暴露能力清单、端点和健康快照。

关键边界与本质。自动晋级默认关——shadow 只写盘、要人工 promote；没接真实业务流量，所以没有实测效果数据。本质是失败洞察驱动 + 金丝雀验证的受控迭代，不是自主进化（模型权重一行没动）。

怎么沉淀。这套机制可迁移的判断是：任何"让系统自己变好"的需求，先分清是改模型权重还是改 Prompt/规则——前者是 online learning、成本和可复现都是硬骨头，后者就走失败洞察加金丝雀这套受控路径，不往"自主进化"上靠。下次遇到同类需求我会先问这一刀，再决定要不要动权重。

真实状态。立项逻辑不是"现在有人用"，是把多 Agent 编排做到能上生产的工程范式讲清楚。机制设计比较全——probe、Critic、HITL、自进化都有——但实测数据不多。

术语降级。简历里"self-evolving"看似自主进化，其实模型权重一行没动——本质是 Prompt 级在线适配，失败样本沉淀成 Prompt 补丁再注入。权重是否变化这类追问，直接回答没动，不往 online learning 上靠。

上生产要补什么。概念上三件：① 用官方 Postgres 检查点存储（checkpointer）替掉自研的；② 接 OpenTelemetry 拿真实 trace；③ golden（基准）测试样本从几个扩到几十个做统计显著。机制是 roadmap，不是 PPT。

怎么沉淀。讲研究型项目这件事的本质，是把"机制设计得全"和"有没有人用"分开讲——前者讲清工程范式、后者诚实认领没上生产，不拿机制的完整度冒充线上效果。下次遇到"没接真实流量的项目怎么讲"这类追问，我会按这个分法：先讲设计取舍和诚实边界，效果只说靠 trace、probe、金丝雀门槛这些机器信号判断的，不报百分比。

两种范式先分清。LLM Supervisor / Swarm（让 LLM 看运行时状态动态决定下一步去哪个 Agent，如 LangGraph 的 createSupervisor、OpenAI Swarm 的 handoff）对位的是硬编码状态机（add_conditional_edges 在构建期把边写死）。LangGraph 两种都支持，不是非此即彼——区别只是"路由决策放在 LLM 里还是放在条件边里"。

业界共识是 workflow 优先。Anthropic《Building effective agents》的核心观点：能用确定性 workflow 编排就别上自主 agent，尤其高风险、要审计的场景——自主循环是兜底手段不是默认选项。原因是自主路由会幻觉选错路、多一次 LLM 调用更贵更慢、决策链不可复现。

我的取舍是混合，不是纯硬编码。Manager 调度的下游能写库（NL2SQL）、能爬数据、未来能改代码——高风险、要 HITL、要可审计。所以把"必经的确定性步骤"用条件边钉死（probe 必在 plan 前、高风险必经 admin_confirm、critic 不过必走 fix），只把"需要语义判断的"交给 LLM（route 意图分类、decompose 子句拆解）。该确定的确定、该判断的判断。

诚实边界。反证我不会只用硬编码：Code Agent 就是 ReAct，LLM 自主选工具循环，因为代码任务探索式、步骤不可预知。硬编码的代价是不够灵活（改流程要改图），autonomous 的代价是 misroute 不可控。换到开放式场景（比如通用聊天），我会更倾向 autonomous——场景决定选型，不是只会一种。

怎么验证。硬编码 graph 的正确性目前靠 trace 验证——每条条件边为什么被选在 trace 里一目了然，misroute 能直接定位到是哪条规则的锅，这是它比 LLM 自主流转好验证的地方。没接真实流量所以没有线上 misroute 率，若要正经度量，会按"路由准确率 + 高风险操作拦截率"看，自动晋级那条路默认关着、要人工确认。

先认竞品强。市面多 Agent 编排不少：AutoGen（微软，对话式多 Agent）、CrewAI（角色分工）、MetaGPT（软件研发 SOP）、Manus（通用 autonomous）、LangGraph 官方的 Supervisor 模式。功能广度都比不过，它们是成熟框架，我的是个人研究项目。

为什么要做（动机）。三件事：① agent-main 单仓十几个 Agent（db / rag / code / crawler / 多模态…）各自为政，缺一个统一入口和编排层，这是真实工程缺口；② 想搞清楚"多 Agent 编排怎么做到能上生产"——光调 API 不难，难的是可观测、HITL、失败恢复、防误路由，这些框架给得不完整；③ 个人研究项目，验证一套工程范式。

差异化（我的强项）。① 安全第一——下游能写库/写文件，HITL + scope 鉴权 + 审计是地基不是补丁；② 可观测——LangSmith trace 到节点级，路由决策可回看；③ 自进化的非劣性设计（金丝雀 + A/B + 自动回滚），不是裸跑；④ Probe 排在 plan 前（规划前先探下游能不能用，减少幻觉式路由）——这个顺序多数竞品没有。

诚实边界。偏研究型、没接真实业务流量、功能完整度认输。做这个最大的收获是工程认知，不是用户量——这点我直说，不包装成产品。

怎么沉淀。造轮子之前先问市面方案缺什么、自己补的是不是真实缺口——这是这次做下来可迁移的判断。agent-main 十几个下游缺统一入口和编排层是真缺口，所以做总管；但功能广度认输、不跟成熟框架比角色扮演。效果没法用 benchmark 对比验证（偏研究型、没接流量），差异化靠 HITL、可观测、Probe 顺序这些机制设计讲清，不靠用户量。

怎么长出来的。不是一开始就设计 39 个节点。演进路径：单下游代理（3-5 节点）→ 多路由加 route / decompose → 流式加 phase 事件 → HITL 加 admin_confirm / checkpoint → 自进化加 Curator / tool_health，每加一类需求就加几个节点，自然涨到 39（说"接近 40"最稳，代码 39、文档 38 有出入）。

每个节点挡什么坑（讲必要性）。probe 规划前探活挡幻觉路由、decompose 子句拆解挡"一句两意图取平均"、admin_confirm 挡高风险误操作、critic 挡质检漏审、fix 做修复回环、Curator 治理记忆。不是堆砌，每个节点有它专门挡的坑。

是不是过度设计（诚实答）。有部分是。demo / 研究阶段堆了一些理想化机制——比如三套自适应路由（Bandit / RL / Causal），职责正交但可能 over-engineered。生产化第一件事是做减法：看哪些节点在生产流量下根本不走、可以删。

关键认知。复杂度本身不是问题，"无解释的复杂度"才是——能讲清每个节点为什么存在，就不算过度设计；但主动承认规模偏大、生产要精简，反而是加分。而且生产化真正要补的不是减节点，是 Postgres 原生 saver 替自研、接 OpenTelemetry 拿真 trace、golden 样本从几个扩到几十个做统计显著——节点数反而是次要的。

怎么验证。节点是不是该留，最终要靠生产流量验证——没接流量之前我的办法是看 trace 里每个节点的命中率，走不到的节点就是减法候选。目前 demo 阶段只能靠这套命中分析粗判，像三套自适应路由那种 over-engineered 的，留着是因为研究阶段要看对比，真上生产第一件事就是砍。
