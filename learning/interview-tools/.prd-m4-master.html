◆ 对比
和 Cursor / Copilot / Devin / BMAD 比,差异在哪
定位针对存量项目(几万行代码+大量历史逻辑)的需求理解,产物是带证据链和门控的计划(spec/plan)不是代码;形态选 Claude Code 插件(Skill 纯 Markdown、零依赖、复用 CC 能力、团队级可分发),拒绝正面拼从 0 到 1 的 codegen。

定位差异化。Cursor、Windsurf、Copilot、v0、bolt.new 主战场是从 0 到 1 拉新代码或单文件补全,这是它们强项也是红海。prd-tools 切的是另一条线——存量项目里几万行代码加大量历史决策,新 PRD 提的每个需求都要先判是新增、修改还是删除、改的话影响哪些现有契约和组件,这一层"先理解再动手"是市面工具没专门处理的空白。产物停在 plan.md 不是源码,把写代码留给 Claude Code/Cursor,我们负责想清楚改哪里、为什么、有什么风险。

补的是证据层。同一份 PRD 让 Claude Code init 走一遍也能给个看起来完整的 plan,但你看不出哪条是查过源码、哪条是模型猜的;AI 有强完成倾向会编路径。prd-tools 每条结论挂 evidence_id——PRD 第几页加代码 file/symbol 加 negative_code_search(搜过没找到也留痕),reference 蒸馏时再从 IR 检索相关实体和路由打法做二次确认。再叠 Spec Review Gate 硬停人审、Readiness 五维评分、benchmark oracle 回归三道门控。CC init 缺的是项目级历史上下文加证据链加门控,补的就是这层。

形态为什么选 Claude Code 插件。Skill 是纯 Markdown 指令,由 Claude Agent 用 Read/Edit/Glob/Grep/Bash 原生工具执行,零外部依赖,src/ 目录的 TS 只是离线开发辅助(模板引擎/评测/auto-tune)不在运行时调用。拒绝做成独立 IDE 插件,因为多 IDE 适配、UI 维护、版本分发都会分散重点,团队推广还要每台机器装一遍;CC 插件靠一份 Skill 文件分发,团队级可推广成本最低。聚焦 PRD→Plan 不直接生成代码,是质量责任划分——代码层错一个字段就是线上 bug,计划层错一个判断还能被人审兜住。

诚实边界。Cursor 的交互体验和实时代码补全强、Devin autonomous 端到端跑但过程不可控、BMAD 是多 Agent 方法论(对比项、不是同类产品),这些维度 prd-tools 不争。反过来 prd-tools 范围窄(只到计划生成)但深——证据链加门控加 reference 知识库组合,护城河不在单点在组合;个人原型探索非生产级闭环、团队约 1–5 人小范围试用(ADR-0005)。Copilot/Cursor 真要追"针对存量项目的需求理解"这条线,靠 codegen 广度优势压不过来,但短期内会不会有专门竞品、不好讲。
