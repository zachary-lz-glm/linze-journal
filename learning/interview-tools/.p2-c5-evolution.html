自进化是 Agent 从自身运行数据学习改进的机制，我在 Manager 里设计了一套 Self-Evolution 闭环。背景是传统 Agent 每个上下文窗口独立、不会从失败中积累经验，跑十次同样的坑踩十次。Self-Evolution 的目标是把运行中的失败模式提炼成可复用规则，让 Agent 在后续任务里避开。我负责采集加归因加验证这套链路的设计。

核心矛盾是学习信号与统计可靠性的权衡。单次反馈噪声很大，直接把每次失败都固化成规则，等于把噪声也学进来——好的经验没积累多少，坏的特殊 case 把行为带歪。本质是自进化必须配统计护栏，不是裸学习——怎么判断一个失败是真信号还是偶发噪声，是这套机制能不能上生产的命门。

关键决策是四步闭环加纯规则归因加 shadow 默认，拒绝了两种天真方案。四步指采集（记录每次运行的决策加产出加评分）、归因（分析失败归到哪类原因）、实验（候选规则在 golden sample 上回归对比）、回滚（评分下降就回滚）。失败归因走纯规则——拒绝了"LLM 归因"方案，因为 LLM 归因本身不可靠、会把归因错误放大；上线策略默认 shadow-only——拒绝了"学习结果直接 promote 到生产"，没经过统计验证的规则上生产等于让所有用户替你交学费。

验证方式是有 hold-out 加对照、不靠 vibes。Manager 的 Self-Evolution 目前跑的就是 shadow 模式——只采集加生成候选规则加离线评估，不自动影响线上行为。指标看候选规则在 hold-out 集上是否真的带来评分提升、提升是否达到统计显著门槛，而不是"加了规则感觉效果好就上线"。

可迁移的方法论是"自进化必须配统计护栏加 shadow，不能裸学"。诚实讲局限：目前是 shadow-only、没接真实流量做大规模验证，统计严谨性（power analysis 定样本量、序贯检验、完整的 A/B 框架）是后续要补的，不是现在已经做扎实了的；另外四步闭环里的归因规则覆盖面有限，遇到规则没覆盖的新型失败会漏掉，得持续维护。
