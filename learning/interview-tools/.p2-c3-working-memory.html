长对话和多步任务里工作记忆爆掉是通病，我的处理方式是自建 Checkpoint 快照。问题是对话越长上下文越满，token 线性涨、延迟增加、模型注意力被稀释，必须压；但又不能丢关键状态。Manager 多智能体编排里我设计了一套 16 字段的 Checkpoint 结构，记录任务当前阶段、各子 Agent 状态、关键中间产出、失败重试计数等，相当于给工作记忆建了一个可恢复的快照层。

核心矛盾是保留执行历史与窗口预算的权衡。全留则窗口爆、模型分心；全砍则丢断点、失败没法续。本质不是"砍或不砍"，而是有损压缩加 checkpoint——把哪些字段保留为精确槽位、哪些压成摘要、哪些直接丢，按对后续决策的重要性分级。

关键决策是 Checkpoint 快照加有损摘要加关键槽位保留，拒绝了两种偷懒方案。拒绝"裸塞对话历史"——成本爆且 lost-in-middle；拒绝"无差别截断 oldest messages"——会把决策点和上下文一起砍掉，模型恢复后不知道之前为什么走到这。我的做法是：对话层做有损摘要、关键状态（当前阶段、已确认产出、未解决疑问）保留为强类型字段、checkpoint 落盘支持断点续跑。

验证方式是断点续跑能续上、关键状态不丢。16 字段快照的设计目标是任意时刻可恢复——任务失败或被中断后，从最近 checkpoint 重启能回到一致状态，而不是从零重跑或带着残缺 context 继续。指标看恢复后任务能否正确推进、关键字段是否完整、长任务 token 是否稳定在预算内。

可迁移的方法论是"工作记忆等于有损压缩加 checkpoint，不是无差别截断"。局限也讲清楚：16 字段是为 Manager 任务流定制的，字段语义跟"多 Agent 协调加阶段门控"强耦合；通用对话场景的 hierarchical summary、Observation Masking 这些压缩策略我没深做，下次做对话型 Agent 时得补——checkpoint 这套思路能复用，但具体字段得重新设计。
