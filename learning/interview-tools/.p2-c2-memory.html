Agent 记忆设计的本质是分层管理不同生命周期的信息，不是塞越多越好。问题背景是 Agent 跑长任务时上下文里同时混着项目长期知识、本次任务状态、历史执行轨迹三类信息，混在一起既贵又乱。prd-tools 里我把它们拆成三层——Reference 是长期项目知识库、Spec 是本次任务的蒸馏产物、Plan 是执行轨迹。我负责这三层的载体设计和注入策略。

核心矛盾是召回相关性与上下文成本的权衡。把所有信息全塞进 context 召回相关性最高，但 token 成本线性涨、还会 lost-in-middle 让模型注意力被稀释；不塞又丢关键上下文。本质不是"要不要记忆"，而是分层存储加按需注入——把什么放在外面、什么时候拿进来、拿多少。

关键决策是三层划分加按需注入，结构化指令优先于纯向量召回。三层各管各的生命周期：Reference 跨任务复用、Spec 单任务内有效、Plan 仅本次执行。注入上拒绝了"全塞 context"——成本高、噪声大，走 query 触发按需召回。检索上拒绝"纯向量模糊召回"处理键值精确需求，因为像"取 A 模块的接口定义"这种请求结构化路由比余弦相似度准得多，向量只在语义模糊召回时才用。

验证指标是 context 长度可控、关键信息不丢。Reference 作为压缩层复用省 token——同一份项目知识不用每次重新蒸馏；Plan 可回放做断点续跑——失败后从最近 checkpoint 恢复而不是从零重来。具体测就看三个数：注入前后 token 增量、任务完成时关键事实是否还能在 context 里找到、断点续跑成功率。不靠"感觉信息都在"这种 vibes。

可迁移的方法论是"记忆分层加按需注入，不是全塞 context"。局限也说清楚：prd-tools 的三层是为 PRD 蒸馏这种结构化任务定制的，硬约束是"知识能被结构化拆解"；通用 Agent 的向量长期记忆比如用户偏好、跨会话事件积累我没做过，下次接触通用对话场景得补一套 embedding 写入策略加重要性评分，不能照搬这套。
