graph.ts 一个文件把节点和边收死——节点是只写 state 的纯函数,分叉靠 addConditionalEdges 的内联谓词决定下一跳;probe 节点在 plan 前并发探一遍下游能力,把客观快照喂给规划,而不是让模型脑补下游活着。

数据结构 / 形态。整张图用 LangGraph 的三件套搭:addNode 注册节点、addEdge 串固定边(主干)、addConditionalEdges 在分叉处挂一个内联谓词函数读 state 决定下一跳(全图 39 个 addNode、固定边和条件边各约 20 处)。节点本身是纯函数——入参 state、返回部分 state(比如 probe 节点只 `return { probe }`),不感知图结构、不知道自己被谁调用下一站去哪;拓扑收口在 graph.ts 这一个文件,加节点改流向都动这里,节点文件不用碰。一个 DRY 点:13 个执行节点(db/rag/code/admin/crawler/gui/clean/visualize/report/multimodal/music/video/multi)共用同一个 afterExecution 出口谓词,统一分流到 synth/clarify/finalize,不重复写 13 遍。

核心算法 / 流程。probe-before-plan 的实现是固定边把 probe 串在 plan 前面:tool_health → probe_node → metacog(第一个条件边分叉、可早退到 finalize/clarify)→ security_gate → decompose → intent_classify → route → prefetch →(条件边按 intent 分叉)→ planner。probe 节点内部用 Promise.allSettled 并发探 5 路下游:RAG `/api/probe`(发个 k=3 的轻查询看有没有文档命中)、DB `/api/probe`(schema 匹配 + ping 看可执行性)、crawler/gui 的 health 探活、code `/api/metrics`;超时分开配(DB/RAG 默认 12 秒、health 4 秒,都走 env 可覆盖),allSettled 保证任一路 fail 不阻塞其它。返回结构直接进 state.probe——`{ rag:{hasDocs,hits,sources,snippets}, db:{schemaMatched,pingOk,executable,tables}, crawler/gui:{healthy,ready}, code:{healthy} }`,下游 planner 读它定 allowedAgents:schemaMatched=false 就不派 NL2SQL、RAG 没命中就不硬加检索步。

为什么自研、拒绝了什么。拓扑用硬编码状态机、拒绝 LLM Supervisor/Swarm 那种让模型自己决定下一步去哪个节点——硬编码的好处是每条边为什么被选在 LangSmith trace 里一目了然、节点纯函数可单测、misroute 能定位到具体谓词,代价是改流程要动 graph.ts 而不是图自己适应。probe 内联在图里并排在 plan 前,拒绝"plan 完再探下游"或"省掉 probe 让模型脑补可用性":前者规划建立在"假设下游活着"上,幻觉式路由就从这来;代价是多一个节点几百毫秒,用一次确定性的并发探活换掉路由环节一个幻觉源。节点和边的职责也强行切开:节点不查"下一步去哪"、边不产数据,这样节点能脱离图单测、图能脱离节点重构。

踩坑和边界。39 是结果不是目标——每个节点对应一个显式决策或要拦的坑(probe 挡幻觉路由、decompose 挡一句两意图取平均、admin_confirm 挡高风险误操作、critic 挡质检漏审、fix 做修复回环),不是一上来画 39 个框;配套文档写 38、代码实数 39(fix 是后加的独立 addNode),面试说"接近 40"最稳。研究型项目没接真实业务流量,图正确性靠 LangSmith trace 看每条边的命中率,走不到的节点就是减法候选,没有线上 misroute 率。失败归因是纯规则(failureAttribution 十来类 if/else 分类,不是 LLM),边缘 case 落 unclear 桶。自动晋级默认关、要人工 promote;模型走阿里云 qwen-flash 的 OpenAI 兼容协议(DashScope),不是 Claude/GPT。
