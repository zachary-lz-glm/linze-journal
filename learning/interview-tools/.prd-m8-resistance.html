◆ 推广
推广遇到的最大阻力是什么、怎么破
阻力清单按人和责任分——信任、质量责任、工具偏好、学习成本、reference 维护成本,每条都有对应解法,核心思路是停在低风险环节先推、把质量责任始终留在人这边

代码生成质量责任划分难落地。生成的代码出了 bug 算模型的、算工具的、还是算研发的,敢不敢直接合主线——这是推广代码生成段最硬的卡点。解法是战略后退:停在 PRD→Plan 段先推,生成的 plan 必须人工 approved 才进开发,质量责任始终在人、不在工具;代码生成段目前是原型探索、非生产级,不主推。

各人开发工具偏好不一。团队里有人用 Copilot、有人用 Cursor、有人用 Windsurf,各自习惯成型、不愿换栈。解法是不强求替换——prd-tools 聚焦 PRD→Plan 的需求分析层,和 IDE 里的代码补全/生成工具不在一个环节,定位成补充而不是替代,愿意的人叠加用、不愿意的不勉强。

信任问题:敢不敢信 AI 出的 plan。研发第一次用都会问"这玩意儿出的方案靠谱吗、出了错谁背锅"。解法是 Spec Review Gate 硬停——spec 生成后必须人工确认才进 plan,加上每条结论挂 Evidence ID 可追溯到 PRD 或源码,出错能定位到哪步、哪个证据引入;先让人审、审了才用,信任是慢慢建的,不是一次 demo 就能换来。

学习成本和 reference 维护成本。建 reference 知识库要 1-2 天、学三步上手流程要半天,是真实门槛。解法:上手走 Reference 蒸馏一次跑通(冷启动需人工引导标注首批领域知识),之后 Mode B 增量更新和 Mode E 反馈回流自动维护、不重复建库;团队仓走 pull-based 聚合(已简化为原样收集模式),不主动搜成员仓源码、维护成本集中在团队仓一处。算 ROI 时把这笔一次性投入扣进去,别只报每需求省 2-3h 那一笔。
