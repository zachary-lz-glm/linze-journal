业务问题与 ownership。权益 SDK 最早只服务国际化一个业务,其他七八个团队各搞各的权益组件实现——同样的优惠券、红包、积分逻辑被重复造了七八遍,接入标准不一致,维护成本高。我是这个 SDK 的 owner,跨团队推广由我推动。

本质矛盾不是技术是信任。推广阻力不在方案好坏,而在别的团队愿不愿用——业务团队最在意的是接入风险和迁移成本,不是 SDK 设计本身。技术好只是入场券,真正要解决的是降低他人的采纳风险:让人敢接、接得起、接了不背锅。

推广三步,以及拒绝了什么。①先找痛点最大的业务做试点——拉美团队多国家配置最复杂,主动找过来,我陪他们一起接入并收集真实问题;②把接入过程沉淀成接入文档和最佳实践;③拿试点数据做内部分享,之后是业务主动找上门。拒绝「写个 README 等别人来用」,没人会主动读文档。设计上用 Register 注册模式让通用组件和业务自定义组件共存——通用组件注册一次全业务复用,业务定制组件只注册到自己的命名空间;拒绝「通用不够就 fork 一份改」,fork 会污染主线、后续升级对不上。

结果和怎么验证。接入工作量从两周降到三四天(试点团队接入前后自报工时对比),从 1 个业务扩到七八个业务团队主动接入。验证靠试点团队反馈的实际接入工时和真实业务跑起来的数据,不是我单方面宣称"好用"。

方法论与局限。推广 = 试点数据说服 > 文档说服;设计上让自定义与通用共存(Register 注册模式)防 fork 是可迁移的方法。局限:推广依赖试点团队的主动性,有几个团队直到自己造的组件出问题才切换,这轮没能提前覆盖到他们——下次会在试点前先做一轮痛点摸底,主动上门而不是等业务找过来。
