## 模块 1:reference(/reference 知识库构建)

### 1.1 产出文件数 — "6 yaml"口径不严谨

- ⚠️ **"6 个文件"口径要讲清楚**:
  - ADR-0001(prd-tools/docs/adr/0001-reference-SSOT优化.md:28-39)明确"旧 10 文件 → 新 6 文件(v4.0)",图示 6 文件 = `00-portal.md` + `01-codebase.yaml` + `02-coding-rules.yaml` + `03-contracts.yaml` + `04-routing-playbooks.yaml` + `05-domain.yaml`(project-profile.yaml 不在此图内)。
  - 但 reference SKILL.md(prd-tools/plugins/reference/skills/reference/SKILL.md:56-77)写"v4.0,6 文件"却列了 7 个(00-portal + project-profile + 01-05),口径自相矛盾。
  - templates 目录实际 7 个文件:`00-portal.md / 01-05.yaml(5个) / project-profile.yaml`。
  - **安全说法**:"reference v4.0 把旧版 10 文件按 SSOT 收敛;核心是 5 个数据 yaml(01-codebase 静态清单 / 02-coding-rules 编码规则 / 03-contracts 契约 / 04-routing-playbooks 路由打法 / 05-domain 业务领域),加 00-portal 导航和 project-profile 项目画像"。别说死"6 个 yaml"——严格数 yaml 是 6 个(01-05 + project-profile),00-portal 是 md。
- ✅ **旧→新去重效果(ADR-0001 自报)**:同一知识重复 2-5 次 → 1 次;信息量 ~88KB/2200 行 → ~55KB/1500 行。注意这是 ADR 自报数据,非第三方 benchmark。

### 1.2 自研实体图/倒排索引

- ✅ **Evidence Index 是辅助层,4 个文件**:`entities.json`(代码实体:file/function/class/interface/enum/const/import/switch_case/template/registry)+ `edges.json`(关系:DEFINES/IMPORTS/RESOLVED_IMPORT/REGISTERS/REFERENCES)+ `inverted-index.json`(term→entity 倒排索引,仅代码符号不含业务术语)+ `manifest.yaml`(元数据)。出处:SKILL.md:65-69, workflow.md Phase 5(prd-tools/plugins/reference/skills/reference/workflow.md:198-232), output-contracts.md。
- ✅ **构建脚本**:`python3 .prd-tools/scripts/build-index.py --repo <项目路径> --out _prd-tools/reference/index`(workflow.md:207)。支持增量(默认,基于文件 hash)或全量(`--full`)。
- ✅ **查询**:`--query <关键词> --index ...` 确定性评分检索。
- ✅ **支持语言**:TypeScript/JavaScript、Go(workflow.md:226)。
- ⚠️ **"自研"定位**:基于正则扫描的代码实体索引,不是 AST 也不是真图谱。output-contracts.md 原话"基于正则扫描的代码实体索引"。面试别说"知识图谱""AST 解析",说"正则扫描的实体+倒排索引辅助层"准确。
- ❌ **别说"向量检索/embedding"**:reference index 不是向量检索,是确定性 term→entity 倒排索引。源码未硬编 embedding 维度。

### 1.3 /reference 走几个 Phase

- ✅ **6 个 Phase**(workflow.md:23-31):
  1. 上下文收集 → `context-enrichment.yaml`
  2. 结构扫描 → `modules-index.yaml`
  3. 深度分析 → `reference/` v4.0(5 个子阶段生成 01-05 + portal)
  4. 质量门控 → `quality-spec.yaml`
  5. Evidence Index → `reference/index/`
  6. 反馈回流 → `feedback-spec.yaml`
- ✅ **8 种模式(F/A/B/B2/C/E/T/T2)**:SKILL.md:29-37, README.md:42-51。首次走 F(上下文收集)→ A(全量构建);日常用 B(增量)/B2(健康检查)/C(质量门控)/E(反馈回流);T/T2 是团队聚合/继承。

### 1.4 团队 Pull-based 模式

- ✅ **Pull-based 聚合**:ADR-0012 决策 1(prd-tools/docs/adr/0012-团队公共知识库聚合与继承.md:48-62)明确选 pull-based 不选 push-based。理由:更新需审核(契约冲突/术语歧义),全自动 push 易污染;认证集中一处;无 CI 依赖。
- ✅ **团队仓不主动搜成员仓源码**:ADR-0012 "团队仓没有源码副本,只有 reference 知识";团队 distill 跳过 rg/glob 源码扫描,从 team/01-codebase 的 cross_repo_entities 查。
- ⚠️ **ADR-0012 顶部已标废弃**:"团队 reference 聚合/继承已改为原样收集模式(references/{repo}/)"。所以 Pull-based 方向对,但具体聚合策略已简化。别说"复杂的 union_by_id 聚合算法",已改原样收集。

### 1.5 冷启动新项目

- ✅ **F→A 流程**:SKILL.md:39-43, README.md:11-23。首次在项目根目录创建 `prd-docs/` 放 2-3 个历史 PRD(.md/.docx/.txt)+ 技术方案 + 接口文档 → Mode F 上下文收集 → Mode A 全量构建。跳过 F 直接 A 质量显著下降。
- ✅ **无历史样例也能建**:SKILL.md:52 "无历史样例时也可构建,但标注业务语义低置信度"。
