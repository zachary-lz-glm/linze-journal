reference 模块高频追问
本模块预判追问 + 一句话答案(对照源码核对)
  • reference 具体哪几个文件,为什么是这几个不是 5 个或 10 个?核心 5 个数据 yaml(01-codebase 静态清单/02-coding-rules 编码规则/03-contracts 契约/04-routing-playbooks 路由打法/05-domain 业务领域)+ project-profile 项目画像 + 00-portal 导航,按“知识在开发生命周期里的角色”切;旧版 10 文件按 SSOT 收敛(ADR-0001)。别死记“6 个 yaml”——严格数 yaml 是 6 含 project-profile,00-portal 是 md。
  • 和普通 RAG 检索本质区别?为什么不直接全量源码切片做 embedding 相似度?reference 辅助层是确定性 term→entity 倒排索引(正则扫描的代码实体),不是 embedding 相似度;倒排索引做精确路由(实体名→文件:符号),embedding 擅长模糊语义相似但精确路由会漂移,且全量切片塞 context 会打爆 token。
  • 自研倒排索引比现成 BM25 / 全文检索好在哪?精度召回怎么测?比 BM25 多了实体关系边(edges.json 的 DEFINES/IMPORTS/REFERENCES)能追踪调用链做 blast radius;诚实讲精度召回没建 labeled query set,是 eval 债,目前靠 benchmark oracle 单 case(gasstation 97)和 golden sample 回归间接看。
  • reference 里的源码或文档过时了怎么办?三层——代码 diff 触发 stale(Mode B 增量只跑受影响子阶段)+ Phase 4 质量门控跑源码一致性检查 + Mode B2 健康检查;但纯动态业务规则(如会员等级阈值)的自动检测还是 gap。
  • code_anchor 锚到文件+符号,为什么不做精确到行号?源码是模块文件级、符号会随重构漂移,行号最容易失效;文件+符号(函数/类名)稳定且够 /prd-distill 定位,行号留在 evidence 的 file:line 溯源里不作为路由键。
  • 倒排索引的准确率/召回率跑过吗?有没有 bad case 集回归?诚实没建 labeled recall@K,是 eval 债;有 benchmark oracle(gasstation 单 case 97)和 golden sample 回归,但都不是针对索引本身 recall/precision 的严格度量。
  • 几个文件加起来多大?塞进 context 是全量还是按需?按需路由非全量——/prd-distill 从需求 IR 提取实体/字段/动作词匹配 04-routing-playbooks,只加载相关层 yaml 和命中代码锚点,context-pack 压到 800 行内给模型消费。
  • PRD 引用了 reference 里没有的实体,系统怎么处理?Step 4.2 阶段 2 补充扫描(rg/glob + Read)兜底 + negative_code_search 标记“搜不到也是证据”;不会静默编造,缺证据的标 confidence: low 进 spec 让人确认。
  • 方案是你拍的还是调研过 alternatives?调研过——全量塞 context(token 打爆)、纯 embedding 相似度(精确路由漂移)、图数据库(过重)都否了;选正则实体+倒排索引是轻量+确定性+精确路由的折中。
