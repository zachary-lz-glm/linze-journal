◆ 蒸馏
蒸馏过程信息损耗在哪、怎么防漏
损耗在 PRD 长尾细节(数值/枚举/边界)向 IR 转译那一环；靠 source_blocks + evidence 追溯 + coverage gate 防漏，不报百分比

损耗发生在长尾细节向 IR 转译那一环。PRD 主干需求(做什么功能)不容易丢，丢的是长尾细节——"满 100 减 20 但 VIP 打 8 折"这种数值约束、"状态只能是 A/B/C"这种枚举、"凌晨时段不生效"这种边界条件。这些在自然语言里散落，从 PRD 原文转成结构化 Requirement IR 时容易被概括掉，是信息逐层压缩里损失最大的一步。

防漏靠三道。① source_blocks——IR 每条需求挂 PRD 原文片段定位，能回溯到原文哪段，不是 AI 凭记忆概括。② evidence 追溯——每条结论进 spec 必须挂 Evidence ID，关键事实没证据、没负向搜索(negative_code_search)就标 confidence: low，不让它悄悄混过去。③ coverage gate——Spec 阶段检查 PRD 的需求点是否都被 IR 覆盖，没覆盖的暴露成 Open Question 让人确认，不靠 AI 凭空推测。

不报准确率百分比。源码里没有"蒸馏准确率"这个指标——Readiness 评分(85/60 阈值)是就绪度、benchmark oracle 是离线质量分、bff-gen scorer 是代码生成准确度，三套都不是"蒸馏过程准确率"。报百分比会穿帮，所以只讲损耗点和防漏机制。

诚实边界。长尾细节的转译损失没法归零，现在的防漏主要靠结构化约束 + Spec Review Gate 硬停人审兜底；没建 labeled query set 系统测过 recall@K / nDCG(诚实债)，防漏效果目前靠"加约束后某类错误是否消失"粗验。
