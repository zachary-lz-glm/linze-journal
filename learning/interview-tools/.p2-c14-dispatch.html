通用化的本质矛盾是特化优化与通用抽象之间的取舍。Manager_Agent 是个具体实现（12 Agent + 39 节点），但它踩的坑是所有 Multi-Agent 调度系统的共性——路由会错、工具会挂、长尾会爆、跨进程状态会丢、黑盒不可观测。把 Manager 的特化优化升华成通用模式，难点在于：太特化不通用、换一个项目就要重写；太通用又失去 Manager 在路由/重试/长尾上的具体优势，变成正确的废话。我的做法是提炼五大支柱——保留 Manager 验证过的机制（重试预算、Probe、checkpoint），但抽象到任何 Multi-Agent 系统都能套。 一、路由可靠性是 Manager 最大教训：路由器必须输出置信度，低置信度降级到 HITL 而不是硬猜。现有 14 路意图分类（M7）+ 12 Agent 路由，但路由准确率没测（M14 只是改进方向）。通用设计：(1) 路由器必须输出置信度，低于阈值降级到反问用户（HITL）而不是硬猜——这是 clarify_needed 失败类型（failureAttribution.ts）的正解；(2) 路由准确率作为一等公民 metric 线上监控，错误路由自动采样进 golden set；(3) 双路由投票——关键路径用两个不同 prompt 的路由器投票，不一致触发 HITL。坑：路由器自己也是 LLM 也会错，不能让路由决策不可逆。 二、工具失败重试预算（recursionLimit 48 + 分层预算）是 Manager 已有可复用的通用模式。recursionLimit 48（retryBudget.ts）+ 单步 1 / 多步 2 / 全局 1 的重试预算是通用模式——区分瞬时失败（超时重试）和逻辑失败（工具语义错不重试）。升华：(1) 失败归因必须先于重试——不归因就重试是烧 token，现有 11 类失败归因（policy_boundary 不该重试 / timeout 该重试）是正确的分层；(2) 熔断——同一工具连续失败 N 次触发熔断降级，不是无限重试；(3) 递归预算防 fix → synth 死循环，这个模式通用。 三、p99 长尾治理：每步设 soft timeout，超时降级返回部分结果，不 kill。Multi-Agent 最怕长尾——一个慢 Agent 拖垮整个 plan。Manager 的 Probe-before-plan（M4）是长尾预防（探活发现下游慢提前换路），升华成通用：(1) 每 Agent 步骤设 soft timeout，超时不 kill 而是降级返回部分结果让上层决策；(2) 并行优于串行——能并行的 probe / 工具调用用 Promise.allSettled（Manager 已经这么干）；(3) p99 分位监控，长尾请求采样分析根因；(4) 慢 Agent 自动降权——路由器记录每 Agent 历史延迟，慢的少派活。 四、跨进程状态：checkpoint 三后端（file / redis / dual）从单机升华到跨多机。（M6）是为单机设计的。通用调度系统跨多机：(1) 状态外置到 Redis / Postgres，进程无状态可水平扩；(2) checkpoint 必须支持 resume——进程崩了另一个进程接管续跑（Manager 的 checkpointStore.ts + checkpointRedis.ts 已有此意）；(3) 长任务用持久化工作流（temporal-style），短任务用内存 checkpoint。 五、可观测：OpenTelemetry 分布式 trace 把黑盒打开。现有只有 LangSmith（M14 改进方向接 OTel）。通用设计：(1) OpenTelemetry 分布式 trace——一个用户请求 span 贯穿多个 Agent 调用，能定位是哪个 Agent 哪一步慢 / 错；(2) 结构化日志带 trace_id / correlation_id；(3) 核心 metric 看板——路由准确率 / 工具失败率 / Critic 否决率 / HITL 通过率 / p99 / 成本，这些是 SD6 评估体系的输入。 六、tradeoff：可靠性与成本分档——高价值任务全开，低价值任务走快路径省成本。双路由投票 + Probe 探活 + 熔断都增加开销。判断：高价值任务（改生产代码 / 执行 SQL）全开，低价值任务（查文档 / 格式化）走快路径省成本。 方法论：Multi-Agent 调度系统的核心矛盾是路由不确定 + 工具不可靠 + 延迟长尾，用重试预算、置信度降级、分布式可观测三个模式系统性治理。不是靠单个模型变聪明。 诚实边界：Manager 没真实用户（MAU / DAU 为零），这些模式的效用是基于代码逻辑推演 + 小规模自测，没经过大规模流量验证。这是诚实承认——生产级验证是下一步。 对应通用 SD。这五大支柱对应通用系统设计里的负载均衡 / 重试熔断 / 长尾治理 / 状态持久化 / 可观测——Agent 域的矛盾是通用 SD 矛盾在非确定性场景下的强化版，方案可迁移。讲完通用方案后可以锚一句"这块我在 Agent 系统里做过更难的版本，因为路由非确定性，重试 / 熔断 / 降级我都做得比传统服务重"。
