◆ 推广
怎么在团队推、遇到什么阻力、怎么解
推广路径分三步——先停 PRD→Plan 这个低风险环节试点,用前后工时对照的 ROI 量化说服,定位"赋能产品/研发不替代";阻力主要在信任和质量责任,不在技术

推广路径长这样。分三步、不是一上来全量铺开:先找 1-2 个意愿高的团队做试点,切入点停在 PRD→Plan 这个低风险环节(生成的 plan 必须人工 approved 才进开发,出错最多重跑 plan 不影响线上);再用试点的前后工时对照数据做内部分享做量化说服;最后把工具定位成"赋能产品/研发不替代"——给产品质量卡口、帮研发省评审,不抢产品的产品决策、不替研发写最终代码。这个定位有出处:鸿湖二面面试官明确建议,自建 PM 工作流替产品写 PRD 看起来酷但产品会觉得在抢活、落地阻力大,做反推缺项检测的卡口工具产品反而欢迎。

推广流程里的阻力主要在人和责任,不在技术。四类:代码生成质量责任划分难落地(生成的代码出了 bug 算模型的、算工具的、还是算研发的,敢不敢直接合主线);各人开发工具偏好不一(有人用 Copilot、有人用 Cursor、有人用 Windsurf,习惯成型不愿换栈);信任问题(敢不敢信 AI 出的 plan、出了错谁背锅);学习成本(建 reference 知识库、学三步上手都要时间)。源码定位是个人/极小团队项目(git log 代码提交 1-2 人),ADR-0005 自报团队 1-5 人,所以推广是真问题不是凡尔赛。

为什么这么推、拒绝了什么。关键判断是停 PRD→Plan 先推,而不是一上来推代码生成段——代码生成段质量责任难划分、风险高,PRD→Plan 段人工确认才开发、风险可控得多。我是这套工具的设计开发者,把它从"能跑"推到"有人用"的 ownership 在我,所以更在意推广路径要选风险最低的切入点、不能把工具推翻团队现有流程。拒绝"写个 README 等别人来用"——业务面不买这个,改用王牌 case 示范(班次签到、油站做过对照实验有真实数据)+ reference 知识库共建拉参与(别的团队来共建 reference、用得越多收益越大)。拒绝"只报技术指标"——业务面考的是经营思维,要算清单环节省多少工时、再扣掉协同联调这些 AI 替代不了的成本,5h→2h 是前后工时不是严格 AB,只报单一百分比是夸大。

诚实边界。这是渐进推广不是全员铺开:ADR-0005 自报团队 1-5 人小范围试用,代码提交实际 1-2 人,目前 PRD→Plan 段小范围验证、代码生成段还是原型探索、非生产级闭环、没接真实业务流量。推广栽过的地方也得讲——代码生成段推广最吃力,质量责任划分难落地是硬卡点,所以战略上往后退、停在计划层先站稳。下次遇到 AI 工具落地推不动,我会先问它动了谁的活、把责任划分讲清楚,而不是先调工具本身。
