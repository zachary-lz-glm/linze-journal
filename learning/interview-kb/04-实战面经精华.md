## 1. AI 工程工作流（PRD→Spec→Plan 蒸馏）— 最高频主题

几乎每场必问，且追问链路极深。这是差异化护城河，也是失分重灾区。

### 1.1 整体链路与双轨定位

**核心答案**：填「AI 怎么理解已有项目」的空白——Cursor/Copilot 解决 0→1 写新代码，但几万行历史项目 AI 理解不了上下文。工作流产出 **Spec（做什么）+ Plan（怎么做）** 两份 AI 友好 MD。

- **双轨定位**（🛡️必背，应对"既然完整为啥不推广"）：
  - **团队推广版**（生产在用）：PRD→Spec→Plan 蒸馏，11 步流水线，20+ 人在用。**ROI 最高、风险最低**（人工确认后才开发）。
  - **端到端原型 prd2code-gen**（个人探索）：PRD→Code 完整 6 步闭环。**下一代探索**，代码生成涉及质量责任和 CI 对齐是更重话题，先用原型验证可行性再决定工程化投入。

🎯 **失分警示**（京东国际/快手/字节一面 ③闭环回避）：端到端原型常被藏在最后才说，被问"代码怎么验证"时绕弯。**第一句话就主动亮双轨**——别等面试官追问。

📍考察：几乎每场

### 1.2 SSOT 六文件 + 能力面适配器

**核心答案**：

- **Reference 六文件**（唯一事实源，对应新人接手项目的理解顺序）：①代码结构 ②编码规则 ③跨层契约 ④PRD 路由映射（什么场景改哪些文件）⑤业务领域 Map（PRD 关键词→代码字段）⑥导航入口 + 代码实体倒排索引。
- **能力面适配器**：约束 AI 的思考维度。前端 9 个 surface（路由/组件/表单/状态/API client/i18n/校验/预览/埋点）、BFF 10 个、后端 10 个。AI 拿到 PRD 先判端类型，只加载对应能力面。能力面只是**搜索候选**（checklist），最终结论必须源码确认。
- **为什么要 BFF 做 Schema 动态生成**：三层隔离——前端专注组件、Node 专注 Schema 转化、后端专注业务。

🛡️ **追问"5 条边界规则怎么来的"**（星河一面）：跑一遍流程看 AI 在哪类问题犯错 → 追溯是什么信息缺失 → 补一条规则。前后跑了 20+ 轮才稳定。规则**会膨胀**（目前前端+BFF 维度，后端客户端各自补）。

🛡️ **追问"知识库是全量知识吗"**（阿里一面 面试官纠正）：**不是全量知识库，是约束层+导航层**——帮 AI 减少 context、做 context 管理，不是写死的点对点映射。

### 1.3 证据链 + 负向搜索

**核心答案**：

- **8 种证据类型**：prd / tech_doc / code / git_diff / negative_code_search / human / api_doc / reference。每条结论必须挂 Evidence ID，可溯源到 PRD 第几节/代码 file:line。
- **负向搜索**（🛡️核心差异化）：AI 搜不到的东西必须显式标注"不知道"（negative_search），不能装作知道。LLM 有强完成性，给任务就编——负向搜索强制约束"没找到就说没找到"。

🎯 **失分警示**（快手 Q8/Q9 "不会编造"被直接挑战）：**绝不说"AI 不会编造/不会幻觉"**——这是危险断言。AI CR 准确率低、第三方库理解不到位是 2025 行业共识（研究显示最好的工具也只能把幻觉降到 96% 不能消除）。改口："幻觉是行业共识，靠 negative search + 静态扫描 + 4 层自校验**降低**，不能**消除**。"

🛡️ **追问"不依赖 AI 怎么验证接口在项目里没定义过"**（快手）：ripgrep 搜接口路径前缀 + 比对 OpenAPI/swagger spec，秒级返回。**负向搜索本质是确定性脚本，不需要 LLM**。

### 1.4 多层门控 + Readiness Score

**核心答案**（贯穿 11 步流水线）：

- **步骤级**：每步用 YAML 中间文件传状态，下一步前校验上一步产物完整性（HARD GATE），不通过重跑。
- **Spec Review Gate**：人工硬终止——Spec 必须用户确认（approved）才允许生成 Plan。风险点表格让人 check，不让 AI 猜。
- **Readiness Score**（五维加权）：PRD 摄入率(20) + 证据覆盖率(25) + 代码搜索命中率(15) + 契约对齐度(25) + 计划完整性(15)。对照 85/60 阈值分三档：85-100 pass / 60-84 warning 需 owner 确认 / 0-59 blocked。硬性降级：P0 契约 blocked 直接 fail、多层需求缺 contract-delta 直接 fail。
- **Final Quality Gate**：5 项确定性脚本检查（必传文件/上下文包消费/锚点覆盖率/计划完整性/阻塞项质量）。**关键：不是 LLM 判 LLM，是确定性脚本**。

🛡️ **追问"证据链只说明 TRD 从 PRD 来，怎么确保真正理解了 PRD"**（京东国际 面试官纠正）：**溯源（provenance）≠ 验证（validation）是两套独立机制**。证据链只解决"从哪来"；"对不对"靠三层独立机制：①人审 ②字段双向对齐（Spec 字段 ⊆ PRD 字段）③黄金样本回归。这是 AI 工程设计的根本区分。

### 1.5 数据口径硬伤（失分重灾区①，🔴极高发）

**核心答案**：见 §0.1。被问任何数字，**第一句就把口径讲死**。

🛡️ **60% 标准话术**："这是 PRD→Plan **单环节**口径——对照组是同经验同事人工读 PRD+需求评审+技术评审+写计划的平均耗时（2-4h），实验组是工具蒸馏耗时（30min）。**只覆盖 PRD 到技术方案，不含后续 AI coding**；全链路含联调/环境/协同约 20-30%。样本还小，主要在前端营销 B 端几个项目验证。"

🛡️ **80% 标准话术**（🛡️三层度量，快手面试官建议）：①文件匹配率（plan 涉及文件 vs 实际分支 diff 重合度，~80%）②plan 稳定性（同一需求跑多次，文件清单一致率）③**git 真实回测**（拿线上已完成需求，跑 plan 看人工有没有改、改了多少行——git 都有记录，比构造数据准）。三个维度互补。

🎯 **失分警示**（蚂蚁/快手/腾讯/花小猪）：60% 被追问"全流程还是只 Plan"才被动承认口径；80% 被追问"怎么算的"答不出。**第一句讲死，别等追问**。

### 1.6 知识库保鲜（弱保证难题，极高难度）

**核心答案**（🛡️蚂蚁/字节二面 满分模板——**先认弱保证再给解法**）：

> "您说得对，没做到 CCSD 级强管控，reference 过时本质是**弱保证**，偏差会累积。我能做的是三层缓解：
> ① **时效性分层**——永久事实（枚举/路径/类型）跟代码 diff 走；慢变事实（业务规则/契约）~30 天 review；快变事实（配置/文案/活动模板）短周期 refresh。每条带时间戳。
> ② **触发式更新**——代码 diff 涉及某条 reference 时，下次整流**强制走 source 验证而非 reference 验证**，主动感知变化。
> ③ **Golden Sample 回归**——易变更项做成 benchmark case，定期回归守护。"

🎯 **失分警示**（蚂蚁/阿里）：急着给解法、没先接住"弱保证"的定性。**诚实承认局限性反而加分**。这是 RAG/知识库最难的问题（staleness），三层防线是生产级成熟设计，远超"加个 TTL 过期"的入门答案。

### 1.7 和 OpenSpec / Cursor / Formily / 扣子的区别

**核心答案**：

| 对比项 | 区别 |
|--------|------|
| **OpenSpec / GitHub Spec Kit** | 它们是**执行层**（给详细 spec 跑 vibe coding）；**缺 PRD→spec 的蒸馏**。我填这个缺口。我的 spec 产出可喂给 OpenSpec 跑下游——**上下游关系，不是替代**。 |
| **Cursor / Copilot** | 它们解决 0→1 写新代码，**不管已有几万行项目**的需求理解。Claude Code init 只分析代码结构，**不理解业务语义**（PRD"会员等级"对应代码哪个字段它不知道）。 |
| **扣子 / Dify** | 它们是可视化编排平台；我用 Claude Code **Skill 工作流**（prompt 链 + 文件系统传状态），因为要深度定制每环节 prompt 和门控，低代码平台太受限。 |
| **Formily** | 纯前端表单方案；我的**双端联动**（BFF 根据上下文剪裁 Schema）+ DSL 表达力它覆盖不了。 |

🛡️ **追问"工作流是不是噱头/会不会随模型升级消失"**（快手/Booster/花小猪二面 面试官判断）：**概念不噱头，但落地被过度营销**。Boris Cherny（Claude Code 创始人）原话 "my job is to write loops, not prompts"。更重要的认知（🛡️行业共识）：**工作流编排会随模型增强而变轻/贬值，但业务知识库（Reference）、领域映射、踩坑经验这些"模型从外部获取不了的东西"才是长期护城河**。被下"噱头"判断时礼貌坚持，别盲从。

### 1.8 端到端 6 步闭环（prd2code-gen）

**核心答案**（🛡️开口能背，90 秒讲完）：①蒸馏（PRD→Spec）②分析（拆需求点 + 文件定位）③Plan（技术方案）④双路径生成（Handlebars 模板优先 + LLM 兜底）⑤自验证自修复（四层校验 + 震荡检测）⑥eval/auto-tune 反哺。

- **四层自校验**：①语法层（TS emit / AST 完整性）②字段双向对齐（Spec 字段 ⊇ 实际改动字段）③规则层（i18n key/命名反模式）④集成层（导入导出/破坏性变更，diff 警察）。
- **自修复循环**：差异性修正（不重写）→ 全量重演（A 改可能影响 B，跑整个依赖链）→ **震荡检测**（A→B→A 横跳找局部最优）。
- **代码生成的 harness 设置**（字节二面）：先拆 exec plan（be-made：spec→story→task）→ 逐 task 自验证 + 一保机制 → 多层校验（语义 TS/ESLint → 代码风格契约 → 沙箱端到端功能验证）。

🎯 **失分警示**（失分重灾区③）：见 §9.3。

### 1.9 边界场景：完全新增需求 / PRD 质量差 / 超长 PRD

- **完全新增、零历史上下文**（京东国际 Q13）：**主动 acknowledge 边界**——"这是我工作流的边界外场景，reference 失效，工具边际价值低。这种应该走 OpenSpec 探讨架构或直接对话，强行兜底是反模式。" **工具选型按场景来**。
- **PRD 质量差/写不清楚**（花小猪二面/千问）：Spec 自动把不确定点标成风险项（表格标红），研发拿风险清单找 PM 确认。中期反推产品补齐（PRD 质量卡口），长期沉淀 PM 工作流。**不能替产品做产品决策**。
- **PRD 超大（2 万字）爆 context**（阿里 Q12）：①分段摘要提取需求骨架（控制 2000 字内）②按需从原文检索相关段落 ③超长表格多子 Agent 分批并行。核心"先摘要定位，再按需深入"。
- **PRD 和代码分叉/过时**（Booster）：**以源码为 SSOT**，历史 PRD 仅作业务场景理解参考（错了不会有后果，因为最终落到源码确认）。

## 2. Agent / LLM 原理（AI 岗必问，原理卡壳重灾区②）

### 2.1 Skill / MCP / Function Calling（三层演进，高频）

**核心答案**（🛡️三层关系，Booster 面试官追问"Skills 怎么被调用"）：

- **Function Calling（底层机制）**：模型经微调，推理时识别意图并生成结构化函数调用（JSON：函数名+参数）。绑定具体厂商，单次调用。
- **MCP（中间协议层）**：Anthropic 主导的开放协议，基于 JSON-RPC，stdio/SSE 传输。**类比 USB-C 标准化**——一次开发的 server 任何支持 MCP 的 agent 都能用。三类能力：tools（函数）/ resources（数据源）/ prompts（模板）。运行在客户端侧，不绑模型。
- **Skills（上层工作流抽象）**：markdown/代码定义多步骤工作流（SKILL.md 声明触发条件），注入 agent system prompt。有状态、多步骤、含前置条件+中间产物+质量门控。
- **加载机制**（腾讯纠正）：**不是全量灌入**——先传 Skill description 做意图匹配，AI 判断需要哪些才按需读取内容（progressive disclosure 渐进式披露），省 token。

🛡️ **Skill vs Tool**（小红书）：Tool 给 agent **能力**（原子操作，无状态，查天气/调接口）；Skill 给 agent **判断力**（多步骤编排，有状态，教"怎么做更好"）。类比：Tool 是锤子扳手，Skill 是施工手册。

🛡️ **MCP vs Function Calling**（小红书二面）：Function Calling 是"模型决定调什么"（绑模型，单次）；MCP 是"标准化的工具发现和连接机制"（跨模型复用）。实际配合：MCP 负责工具标准化暴露，Function Calling 负责模型工具选择决策。只用一家模型且工具少 → Function Calling 够；跨模型复用/共建工具生态 → MCP。

🎯 **失分警示**：腾讯面被纠正"LangGraph 全量加载 skill"——正确是"先传描述做意图匹配，按需懒加载"。说原理前先确认机制。

### 2.2 Context 管理 + KV cache / prompt caching（🔴原理错重灾区）

**核心答案**（🛡️蚂蚁三连问满分模板）：

- **context 超窗怎么办**：别全塞短期记忆，善用磁盘长期记忆。输入侧渐进式披露 + RAG；输出侧文件分流（生成内容写磁盘不塞聊天）；中间推理过程超窗把长计划写外部 MD/memory。
- **Claude Code 的 context 管理**：①**auto-compaction**——接近容量极限（~90%）自动把旧对话总结成摘要，一次释放 60-70% 窗口；②**CLAUDE.md**——项目级持久记忆，每次会话启动读入，compaction 后重新加载。
- **memory 不会每轮越来越长**：区分三层——持久记忆（CLAUDE.md，固定）vs 短期记忆（对话，滑动窗口+compaction）vs 按需检索（@文件）。不是每轮全量塞。
- **单次请求带什么 + 缓存在哪**（🔴蚂蚁 Q13 原理错，必纠）：带 system prompt + tools schema + messages（对话历史）。**公共前缀被 prompt caching 复用**。⚠️**cache 在 Anthropic 服务端，存的是 Transformer 前向传播算出的 KV cache（attention 的 key/value 张量），不是 prompt 文本也不是 hash**。相同前缀复用张量跳过重复计算，成本降 ~90%、延迟降 ~85%。本地 `.claude` 存的是会话历史 transcript，和 prompt cache 是两回事。

🎯 **失分警示**（蚂蚁 Q13 最致命）：把"会话历史本地存储"和"prompt caching（服务端 KV cache）"混为一谈，被"缓存在哪"追穿。**必须纠正：cache 在服务端、存 KV 张量**。

- **工作记忆 Summary**（小红书）：①Progressive Summarization（递归压缩旧消息）②MemGPT/Letta（超阈值 flush 摘要到外部存储，类虚拟内存分页）③Observation Masking（占位符替换保留结构）。

### 2.3 多 Agent vs 单 Agent + 成本 ROI

**核心答案**（小红书二面，🛡️类比"个人 vs 团队"）：

- **单 Agent 优势**：上下文共享、调试简单、成本低。
- **多 Agent 优势**：各司职、独立上下文窗口、中间文件传状态。
- **何时用多 Agent**：①任务有明显异构知识（一个分析代码一个分析文档）②单 Agent prompt 已膨胀到互相干扰 ③需并行加速 ④长任务需状态持久化/可中断恢复。
- **成本**：多 Agent 约为单 Agent 的 **10-15 倍**（每个实例独立 token + 通信编排开销）。**ROI 是关键评估维度**——不是所有场景都值得 10 倍成本。

🎯 **失分警示**：别当"多 Agent 万能论者"，主动提成本 ROI 显工程判断力。

### 2.4 Harness Engineering / Agentic Engineering（前沿，小红书二面考察）

**核心答案**（🛡️Martin Fowler / Addy Osmani 2025 范式）：**Agent = Model + Harness**。传统三段式 agent（接收→规划→执行）无状态，每个上下文窗口独立。Harness 工程用控制器+沙盒做闭环驱动，核心是**短期记忆→长期记忆→知识库→技能手册**的抽象提取链路（**KSG：Knowledge-Skill-Growth**），是 agent 自我迭代的核心。

- **三类引擎**：合并引擎（整合多 agent 输出）、通信引擎（agent 间状态传递）、反思引擎（从失败提炼规则）。
- **自进化链路怎么 run**（🛡️四步）：①日志收集（决策日志/工具调用/质量评分）②反思引擎（识别高频失败模式，生成候选规则到待审核队列）③人工审批（确认后写入 AGENTS.md/技能库）④验证（golden sample 重跑，确认提升才上线）。核心是"**错误即规则**"的 ratchet 机制。
- **映射到自己项目**：SSOT=知识库层，证据链+门控=约束层，golden sample=反馈层。已有 Harness 骨架，缺自动技能抽取和自进化闭环。

### 2.5 Agent 六层架构 / ReAct / 选型

- **Agent 核心能力**（小红书）：推理循环（骨架，没有它就是工具）+ 工具调用（手脚，没有它就是聊天机器人）+ 记忆管理（短期/长期/工作记忆，没有它每次从零开始）。
- **六层架构**（千问）：信息边界 / 工具系统（MCP/FC）/ 执行编排（状态机）/ 记忆（短期+长期）/ 评估观测 / 约束恢复与自动修复。
- **框架选型**（腾讯/花小猪二面）：轻量任务用 Claude Code Skill+Tool；中等复杂度用成熟工作流；重量级用 **LangGraph 图状态机**（支持状态持久化+人工检查点+循环分支）。**LangChain 适合线性 RAG pipeline，LangGraph 适合需硬终止/回溯重做的多步骤工作流**。

### 2.6 几十个 Tools 识别率降低（小红书）

**核心答案**（🛡️研究数据）：4 个→51 个工具时 GPT-4o 准确率从 43% 降到 2%。三层优化：①分层路由（Router 先分类意图到领域，子 agent 从少量工具选）②动态激活（每次只注入相关工具，减噪音）③优化描述（`{操作}_{实体}_{数据}` 命名，参数 ≤5，枚举约束）。三策略叠加 33%→74%。

## 3. Schema 驱动营销中台（项目核心，几乎每场必问）

技术题基本扛得住，主要补"表达结构"。

### 3.1 三层架构 + BFF 模板引擎

**核心答案**（🛡️顺数据流讲）：①后端 API 给原始平铺数据 → ②BFF 用模板引擎（Handlebars）编译成 JSON Schema（含字段定义/校验/联动声明）→ ③前端渲染引擎声明式渲染，不写 if-else。Schema 是前后端共享的唯一真相源。

- **模板引擎演进**（京东 Q13）：早期按字段写方法（20 字段=20 方法，逻辑耦合）；重构为**按奖励维度拆模板**（20 奖励=20 模板文件，每个是该奖励完整 Schema 节点树），通用方法抽工具包。
- **BFF 不是必选**（京东面试官）：按业务复杂度决策。我们 20+ 活动类型/8 业务线/多国家差异化，BFF 做 数据→Schema 转换必要；简单业务直调 API 更高效。

### 3.2 联动机制（核心难点，高频深挖）

**核心答案**（🛡️本质是有向图依赖，不是 if-else）：

- **联动本质**：A 影响 B、B 影响 C、甚至循环——是**字段级有向图依赖**。引擎底层解决三个图论问题：依赖图建模（邻接表）、拓扑排序求值、循环检测（编译期静态扫描 + 运行期 visited Set）。
- **dot path 拍平**（星河 Q9）：嵌套结构 `page1.form2.field3` 拍平做 key-value 存储，依赖映射表 O(1) 查找联动目标，不用遍历整棵树（类似 Redux normalize）。
- **精确订阅**（星河 Q11）：每个组件只订阅自己的 dot path，某字段变了只有依赖它的组件更新，不重渲染整表单（类似 Redux selector）。
- **双重求值（injectDActions）**（🛡️核心设计）：联动规则声明一次，①BFF 端 safeEval 算初始状态（保证 SSR 首屏正确，不等前端 JS 加载）②前端运行时再求值（响应用户操作）。两次用同一套规则，触发时机不同。
- **循环依赖**（星河 Q10）：设计约束联动**只做清值不做赋值**，从架构断开循环链路；引擎层 max depth 限制（>3 层中断报警）双重保险。

### 3.3 两种联动（state 型 vs fetch/API 型）

**核心答案**（🛡️必答，多家问）：

- **state 型**（纯前端自闭环）：visible/disable/value，DSL 声明 dependence 数组 + 三元表达式模板字符串，发布订阅监听依赖变化，本地 eval。
- **fetch/API 型**（跨接口）：城市/产品线变化走 BFF Controller 调后端拿新配置，**重新拼接 Schema 整体下发前端**。

🛡️ **组件内部交互（点按钮更新数据）写死吗**（蚂蚁 Q15）：**主动声明设计边界**——"我们定位是配置驱动渲染不是完全无代码。Schema 管结构/配置/组件间联动；组件内部复杂交互写在组件代码里。把高频变化的页面结构做成可配置，把稳定的复杂交互沉淀成组件，避免 DSL 爆炸。"（Formily 同样边界）

### 3.4 为什么不用 Formily（自研 vs 开源）

**核心答案**（🛡️千问/大麦/花小猪/阿里）：Formily 是成熟纯前端表单方案，对纯前端够用。但两个差异：①**双端联动**——BFF 根据上下文剪裁 Schema，Formily 覆盖不了；②**DSL 表达力**——跨接口依赖 Formily 要写大量 effect，本质还是自研联动层。如果只前端渲染层，Formily 完全可以。

### 3.5 Schema 安全性 + 沙箱预渲染 + 版本兼容

- **XSS**（起点 Q10）：组件白名单（只能引用注册组件）+ DOMPurify 富文本过滤 + 后端校验，前端只渲染不执行逻辑。
- **表达式安全**（Booster Q13）：研发写 Schema 非运营直编，源头控注入；标准模板约束可用表达式。
- **防配错崩溃**（拼多多/Booster 面试官给方案）：🛡️**沙箱预渲染**——CI/CD 中用完整渲染引擎在隔离环境跑一遍 Schema 配置，捕获类型错误/渲染异常，正确才允许上线。+ Schema 静态校验（联动引用缺失/循环依赖报错）+ 版本回滚。
- **版本兼容**（星河 Q14）：Schema 根节点 version 字段，BFF 按 version 路由到对应模板版本，新老并行；升级时迁移函数转换 + 双版本并行 + 逐步下线。
- **类型错误前置发现**（Booster Q15）：沙箱预渲染 + BFF Schema 校验层（字段类型/必填/枚举）+ TS 类型约束 + Sentry 兜底。

### 3.6 性能（含系统设计题）

- **大表单（200+ 字段）重渲染**：dot path 精确订阅（§3.2）+ 动态表格可编辑单元格（hover 前字符串、hover 后输入框，避免上千控件挂载）。
- **联动接口慢（2-3s）**（拼多多/阿里）：🛡️**Service Worker 缓存**联动接口返回值——同入参直接读缓存不发请求，和业务逻辑解耦不改 BFF。
- **依赖包大小**（大麦 Q5）：构建时 tree shaking + babel-plugin-import 按需 → 加载时 SplitChunks + 路由 lazy → 运行时 moment→dayjs、CDN externals React → 监控 bundle-analyzer + size-limit 卡红线。
- **系统设计：一秒内渲染营销页**（千问 Q32）：网络（CDN+HTTP/3+preconnect <200ms）→ 资源（SSR/代码分割/critical CSS/WebP <300ms）→ 渲染（骨架屏+React 并发 startTransition+长任务拆分 <400ms）→ 感知（loading 反馈+SW 二次缓存 <100ms）→ 兜底监控 LCP/FCP。
- **首屏性能**（拼多多 Q9）：路由级 React.lazy 代码分割 + SW 缓存联动接口 + 虚拟滚动 + 组件级 dynamic import（BFF 按页面 Schema 只下发所需组件）。

### 4.1 Register 注册模式 + Rollup 双模 + external

**核心答案**：

- **Register 模式**：SDK 内置 20+ 通用组件，业务方运行时 register 定制组件（**不用 fork 改源码**避免分叉）。靠**配置组合**覆盖多国家×业务线，不是堆组件数量。
- **⚠️组件数口径**（见 §0.1）：**30-40 个基础组件**，"400+/800+"是配置 SKU 组合数（国家×业务线×产品线×权益类型）。永远别说"400+ 组件"。
- **Rollup 双模**（起点/星河）：ESM（静态可 tree-shake，给构建工具）+ CJS（兼容 Node/Jest/旧 webpack）+ d.ts。**external + peerDependencies** 声明 React——业务组件库和宿主都引相同内部包，不 external 会打进产物导致重复/版本冲突。
- **样式隔离**（起点 Q14）：CSS Modules（构建时 hash）→ CSS-in-JS（运行时 hash，Ant Design 5 cssinjs）→ CSS 原生 @layer/Shadow DOM。Ant Design 6 起换原生 CSS 变量。
- **CLS 布局偏移**（起点 Q11）：预留空间（默认高度+骨架屏）+ aspect-ratio（图片加载前算高度）+ content-visibility: auto（屏幕外延迟渲染）。

### 4.2 Monorepo + 依赖图拓扑排序（高频追问）

**核心答案**：

- **Lerna + Nx**（网易/大麦 A1）：历史项目（2022）选 Lerna（独立发版，benefit 组件库要独立发包给 8 业务）+ Nx（增量构建/任务缓存，全量构建 5min→30s）。**今天新项目选 pnpm workspace + Turborepo**（pnpm hard link 装包快磁盘省，Turborepo 配置简单）。诚实交代"项目是 2022 搭的"反而显批判性思考。
- **🛡️依赖图拓扑排序**（Booster/拼多多面试官追问）：①读每个包 package.json 的 dependencies 建 DAG ②拓扑排序定构建顺序（被依赖的公共包先构建）③无依赖关系包并行构建 ④NX 计算输入哈希做缓存（源码+依赖不变则跳过）。

## 5. 前端基础八股（实战变体，详见 01 题库）

题库已有详细版，这里只给**实战易错点 + 高频追问**。

### 5.1 React

- **Fiber**（千问 Q31）：大任务拆链表式子任务 + 优先级 + 时间分片，解决 React 15 同步渲染卡顿。可中断/恢复/优先级控制。
- **setState→视图流程**（Booster Q18）：setState→调度器（lane 优先级）→协调（Fiber 双缓冲 diff，标记 effectTag）→提交（应用 DOM）→useLayoutEffect（paint 前同步）→paint→useEffect（paint 后异步）。React 18 automatic batching 默认开。
- **useEffect vs useLayoutEffect**（Booster Q17）：LayoutEffect 在 DOM 更新后 paint 前**同步**执行（读布局/防闪烁）；Effect 在 paint 后**异步**执行。
- **HOC**（千问 Q36）：接收组件返回新组件，复用公共逻辑（withAuth 注入 props）。
- **React 18 并发 + Suspense**：startTransition；lazy+Suspense 代码分割；React 19 Concurrent 下 Suspense 包数据请求"挂起"避免瀑布。

### 5.2 网络/安全

- **HTTPS vs HTTP**（千问 Q37）：HTTP 三次握手后传明文；HTTPS 多一次 TLS 握手。TLS 1.2 五步：ClientHello→ServerHello→Certificate→Key Exchange→Finished。之后对称加密。TLS 1.3 优化 1-RTT。
- **XSS/CSRF/CSP**、**HTTP/2 vs 1.1**、**TCP 三次握手**：见 01 题库。

### 5.3 TS

- **interface vs type**（网易 Q7）：90% 互换。interface 支持声明合并（第三方类型增强必须用）；type 能表达联合/元组/映射/条件/工具类型。
- **泛型作用**（网易 Q8 🔴硬伤）：类型层面的参数化。三场景：容器类型（Array/Map/Promise）、工具函数（useState）、类型工具（Partial/Pick）。**场上别只挤得出 Partial**（那是映射类型不是泛型本身）——刻进肌肉记忆 Array/Map/Promise/useState。
- **Pick 实现**（拼多多 Q16）：`type MyPick<T, K extends keyof T> = { [P in K]: T[P] }`。

## 7. HR / 动机 / 行为面（失分重灾区④，🔴负责人面一票否决）

### 7.1 离职动机（🔴极高发，被打脸×2）

🎯 **失分警示**（字节一/二面 结构性问题）："更大平台/学前沿/大厂已落地"这套**外部参照框架被纠正两次**。字节面试官："你所谓的前沿误区是不存在的。真正的问题是**你有没有在业务场景下做到最好**，而不是拿别人最佳实践来套。"

🛡️ **改进版动机框架**（锚定目标岗位，不锚定逃离现状）：

> "我想从'在团队做 AI 工程化探索'切换到'AI 是业务生命线'的场景，把同一套能力（存量项目 AI 理解）往一个更高的完成度目标推——**不是要更大平台，是要一个 AI 能被做到 90%/bug-free 的业务密度**。我的差异化是已有几万行代码项目的上下文构建，市面上 OpenSpec/SuperPowers 解决 0→1，我解决存量项目的 AI 理解。"

🎯 **教训**：①动机第一反应别是套话（"工作固定/承诺做完/增删改查"）；②别贬低现团队（慢/不报销/堆需求）——负责人会想"你来这也这样"；③"token 自费"放"我对 AI 投入是真金白银"作能力佐证，不是离职理由。

### 7.2 反问策略（高质量反问框架）

🎯 **失分警示**（千问 Q43 被批/字节一面 Q9 反问太弱）：反问问宏大问题（公司战略/团队规划）= 跟个人无关，面试官无法答。操作性问题（"入职后最重要的事"）= 凑数。

🛡️ **三连反问框架**（按"个人强相关 → 团队真实 → 技术深度/商业化"递进，至少留 2 个硬的）：

1. **强相关**："这个岗位具体是哪个 team？近半年团队最核心的一个前端项目是什么？进来后预期重点参与哪个？"
2. **技术深度**（挂钩自己能力）："团队现在的 AI 工程化到哪一步？Prompt 工程还是 RAG/Agent 工作流？我做的 PRD 工作流思路能不能复用？"
3. **业务挑战/商业化**："你们觉得这个业务最大的瓶颈在哪？（召回率/解决率/付费意愿）"

### 7.3 landing（30-60-90，花小猪三面高频核心题）

🛡️ **30-60-90 框架**：

- **30 天"听和学"**：吃透业务架构 + 过核心源码摸清技术栈背景 + 聊遍上下游 + **找到团队当前最大痛点作为切入点**。
- **60 天"贡献验证"**：用已有经验快速解决一个实际问题让团队看到价值（如 Schema 配置化方案迁移到多城市多运力场景），做小但可见的改进。
- **90 天"建立影响力"**：参与技术规划讨论 + 提有数据支撑的改进建议 + 建沟通节奏（技术分享/CR 习惯）。
- **本质**：landing 不是"学业务"是"建立信任"——靠一件件小事做出来。

### 7.4 子方向负责人技术规划（花小猪三面）

🛡️ **四步**：①**对齐**（子方向必须在大团队年度目标框架内）②**盘点**（从业务痛点出发做普适性思考，按事前/事中/事后三维度对比行业标准找差距最大处）③**拆解**（投入成本×业务价值矩阵，右上角优先）④**度量**（前后对比指标，如故障率/MTTR——没有度量的规划是愿望清单）。

### 7.5 薪资谈判（花小猪三面 Q13）

🛡️ **两个角度**：①**市场锚点**（4 年大厂架构背景+独立主导中台项目，30K 月薪是合理区间，没要溢价）②**差异化价值**（同行还在"用 Cursor 写代码"，我做的是整套 AI 工程化体系：PRD 蒸馏+SSOT+证据链+三级门控，不是 PPT 是日常在用）+"薪资只是维度，更看重能不能做有挑战的事，双方合适可谈"。

### 7.6 业务 Sense（失分重灾区，🔴字节一面最严重）

🎯 **失分警示**（字节一面 Q7 卡壳）：天天讲"提效 80%/20 人团队在用"，但说不清自己做的东西为业务创造什么价值。字节面试官关注"你能不能用技术驱动业务"。

🛡️ **司机营销中台业务指标**（被问必答）：拉新侧（注册→首单转化率 ~30%、CAC 单司机获取成本、活动期注册增量）/ 留存侧（7-30 日留存率、完单率、调度时长——留存是核心痛点）/ 效率侧（拒单率、接单率、日均在线时长）/ 营销侧（活动 ROI、补贴率、参与率、GMV 增量）。**工作流挂钩**：上线 2 周→3 天 = 运营更快试错拉新策略，间接影响转化留存。

🛡️ **价值衡量标准**（字节面试官）：不是人头数，是**是否驱动了具体业务指标增长**。讲项目别光讲"20+ 人在用"，要讲"为业务带来什么收益"。

## 8. 系统设计 / 开放题

- **工程师核心价值**（小红书 Q21）：从"实现页面的执行者"变成"设计人机协作系统的架构师"。向上——设计 AI 和人的协作边界（哪些环节 agent 自动/哪些人审核/怎么设质量门控）；向下——AI 前端交互形态变化（对话式 UI/智能表达），把"模型想说什么"翻译成用户能理解的界面。核心：模型越强，"做什么"比"怎么做"越重要。
- **C 端 AI 渗透**（涂虎 Q19）：生产层——设计稿还原三层约束（Design Token 管样式 + 组件 Schema 管 AI 优先组合 + 截图对比视觉回归）；交互层——对话式导购/个性化推荐。**国内 APP 生态限制**（拿不到权限），H5/小程序更实际。阅文判断：C 端视觉还原 AI 只到 70%，**中后台是 AI 编码最佳落地场景**。
- **设计稿还原**（涂虎 Q20）：三层——Design Token（基础样式）→ 组件级 Schema（AI 优先组合现有组件不从零写 div）→ 视觉回归测试（Chromatic/Playwright 截图断言测像素偏差）→ 复杂布局用 Layout Token（栅格/布局模板）。
- **不同技术方法论共性**（小红书 Q24 🛡️高级工程师能力）：传统架构模式在 AI 系统里换名重出——Prompt 模板引擎=MVC View 模板；Function Calling=适配器模式；MCP=USB-C 标准化；Progressive Summarization=日志压缩轮转；语义缓存=Memoization；Flow Engineering=状态机。"AI 工程没推翻传统工程，只是同样设计思维用在自然语言接口层。"

## 9. 失分重灾区总览（跨 23 场共性，🔴临场前必扫）

这是 23 场提炼的**最高价值部分**——技术题能扛，但这四类失分是结构性的、反复出现的。

### 9.1 ① 数据被追穿

**症状**：讲"60%""80%""20 人团队"，被追问口径/怎么算/怎么测时卡壳或被动承认。
**根因**：天天讲提效，但没准备口径话术。
**解法**：见 §1.5。**每个数字第一句讲死口径**，备三套话术（怎么测/怎么算/怎么稳）。讲准确率不只讲构造数据，还讲 git 真实回测。**带 1-2 个业务侧硬指标**（拉新/留存/GMV），堵住"只懂提效不懂业务"。

### 9.2 ② 原理卡壳（AI 底层原理短板）

**症状**：蚂蚁 context/cache 三连答错（把本地存储当 prompt cache）；腾讯被纠正 LangGraph 全量加载。
**根因**：AI 底层原理（KV cache/MCP 机制/agent 架构）是高频考点但准备不足。
**解法**：见 §2。**标注概念必背**：prompt caching / KV cache / auto-compaction / Function Calling→MCP→Skills 三层 / progressive disclosure。说原理前先确认机制，不懂先承认"了解不深"再用类比展示学习能力，**别硬编**。

### 9.3 ③ 闭环回避

**症状**：被问"代码怎么验证/端到端"时绕弯，端到端原型直到反问才说；用"老板让先做 spec 段"回避。
**根因**：怕被追代码生成质量责任，藏起来。
**解法**：见 §1.8。**第一句话就上"端到端原型已跑通 6 步闭环 + 团队推广版聚焦 PRD→Plan 是 ROI 选择"**。别用"老板让先做前两步"回避——主动讲双轨定位。

### 9.4 ④ 动机/行为面塌方（🔴负责人面一票否决）

**症状**（字节二面 最致命）：动机套话被当面戳穿；情绪化+答非所问+贬低同事；自相矛盾（前面说团队推不动，后面承认推广顺利）。
**根因**：动机锚定"逃离现状"而非"目标场景"；防御性太强听不进问题。
**解法**：见 §7。负责人二面考的不是技术（技术在一面已筛），考的是：**动机真不真、能不能被管理（情绪/抗压）、会不会合作（贬不贬低同事）、有没有自我高目标**。
- 🛡️ **被追问"卡点/困难"时，答"结构性挑战+我的解法"，绝不答"别人的问题"**。区分"阻力"（别人不配合）和"挑战"（目标更高需更多输入）——后者显格局，前者显抱怨。
- 🛡️ **永远不在面试里贬低前同事/合作方，哪怕是真的。**
- 🛡️ 被戳痛处先**复述问题再答**，别防御。
- 🛡️ 主动设高目标："我给自己设的目标是 X%，现在到 Y%，差 Z"（替代"我在学/在探索"）。

### 9.5 ⑤ 表达无框架 + 口头禅（全场共性，每场诊断都点名）

**症状**（23 场几乎全部诊断）："然后"30-50 次/场、"对/对吧/就是"高频；回答偏长（3-5min/题）；多数题无结论先行；阿里系被指"信息量大但缺框架"。
**解法**：
- **结论先行**：每题第一句亮核心句，再展开 why/how/数据。
- **项目复述用四段式**（千问/阿里面试官要求）：**角色 → 做事 → 难点 → 效果**。
- **压口头禅**："然后"→停顿；宁可慢一点整理思路，也别滔滔不绝讲细节。
- **正面回答优先**：被问"X 是什么"第一句必须定义；被问"怎么保证"第一句必须机制名，**绝不绕背景**。

### 9.6 ⑥ 业务 Sense / 产品分层思维缺失

**症状**（字节一面 Q6/Q7）：被问"小需求是不是太复杂"答技术实现（渐进式披露）而非产品决策（lite 版）；被问业务指标卡壳。
**解法**：作为工具设计者**主动思考分层**（重度→完整版/中度→标准版/轻度→lite 版/临时→不走流程），不等用户反馈。业务指标见 §7.6。

## 10. 面试官洞察精华（行业趋势 + 团队画像）

> 这些是 23 场面试官主动透露的高价值信息，是公开资料查不到的"行业水位"。

### 10.1 行业趋势（可降维用于多场面试）

1. **工作流会贬值，业务知识库是护城河**（快手/Booster/花小猪二面共识）：工作流编排随模型增强变轻，但 Reference/领域映射/踩坑经验这些"模型从外部获取不了的"才是长期价值。
2. **AI Native 抹平前后端边界**（京东国际/星河）：行业终局不是"AI 帮前端写前端"，是"AI 让前后端边界消失"。前端护城河不再是"会前端"，是"会某一类问题的端到端解决"。
3. **代码生成已成熟，是基线不是前沿**（字节二面）：差异化不在"能做 PRD→方案"（人人都能），在"能做到多高准确率/自动化闭环"。
4. **AI CR / Code Review 分层**（千问/快手/腾讯/网易共识）：规范类（lint/类型）完全交 AI；小改动 Agent 编码人 CR；核心改动端到端测试+模块负责人审查。AI 先粗筛（机械性），人工精审（业务正确性）。
5. **AI 产品最难的是商化，不是做出来**（字节二面 最高维洞察）：AI 服务都要钱（以前免费），核心不是"用户心智"是"**用户愿不愿意付费**"——可感知的效益提升差是破局点。
6. **价值不必靠"大范围推广"衡量**（字节一面）：判断工作价值的标准是"是否驱动了具体业务指标增长"，不是覆盖多少人。
7. **不存在"前沿误区"**（字节二面 一句话打穿）：比的是业务场景内的完成度，不是别人最佳实践。
8. **AI 三层技术架构**（星河二面/拼多多）：工程层（模型调用+监控）→ 通用能力层（记忆压缩/推荐/工具调用）→ 调度层（网关/调度策略/配置平台）。记忆压缩和工具调用是核心基建非可选。
