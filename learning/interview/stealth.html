#1
React 18 并发 + Fiber
Fiber 把递归变链表，Scheduler 按优先级调度

之前虚拟 DOM diff 是递归，停不下来。Fiber 把节点组织成 child/sibling/return 链表，每个节点是可拆分的工作单元。Scheduler 用 MessageChannel（不是 setTimeout，最少 4ms）做优先级调度，高优先级可打断低优先级。

React 18 加了 useTransition（低优先级）、useDeferredValue（延迟渲染）、Automatic Batching（任何上下文都合并 setState）。

DOM 更新完 → useLayoutEffect → 浏览器绘制 → useEffect。

八股
闭包和作用域链
函数记住定义时的词法环境，即使在外部执行也能访问那个环境里的变量

闭包的本质就是：一个函数可以访问它定义时所在作用域的变量，即使这个函数在外部被执行。比如函数工厂 function makeCounter() 里定义了一个 count 变量，返回的函数每次调用都能访问到同一个 count——这就是闭包。

常见场景：for 循环里用 var 会导致所有回调共享同一个 i（用 let 解决）、柯里化其实就是闭包的应用——先传部分参数，返回的函数记住这些参数再接收剩余参数。

注意事项：闭包引用的变量不会被垃圾回收，所以事件监听器或定时器如果引用了外部大对象，不用的时候要手动清理，否则内存泄漏。

八股
原型链与继承
对象通过 __proto__ 逐级向上查找属性，直到 null 为止

当我访问 obj.name 时，JS 引擎先在 obj 自身上找，找不到就顺着 obj.__proto__ 去它的原型上找，还找不到就继续往上，一直到 Object.prototype.__proto__ 也就是 null，这条链就是原型链。

ES6 的 class 本质还是原型链，class Dog extends Animal 编译后就是 Dog.prototype.__proto__ = Animal.prototype。继承方案演进了很多代：原型链继承（引用类型共享问题）→ 构造函数继承（拿不到原型方法）→ 组合继承（调了两次父构造函数）→ 寄生组合继承（最优方案，ES6 class 底层就是这个）。

八股
HTTPS/SSL 握手
先用非对称加密安全协商出对称密钥，之后用对称密钥加密传输

握手过程：客户端发 Client Hello 带上随机数和支持的加密套件 → 服务端回 Server Hello 带上随机数和自己的证书 → 客户端验证证书链是否可信 → 客户端生成一个预主密钥，用服务端公钥加密发过去 → 双方用三个随机数算出会话密钥，之后的通信用这个对称密钥加密。

为什么这么设计：非对称加密安全但慢，对称加密快但不能安全交换密钥。所以用非对称交换密钥，用对称传数据。ECDHE 支持前向保密——即使私钥泄露，历史通信也解不开。TLS 1.3 把握手简化到 1-RTT。

#11
前端性能优化全链路
从网络、渲染、资源、代码四个维度逐一优化

网络层：CDN 缩短物理距离、HTTP/2 多路复用减少连接、Preload 预加载关键资源。渲染层：首屏用 SSR 或骨架屏、图片懒加载、长列表用虚拟滚动只渲染可视区域。资源层：代码分割 dynamic import 按需加载、Tree Shaking 删掉没用的代码、图片用 WebP。代码层：React.memo 和 useMemo 防止不必要的重渲染、耗时计算放 Web Worker 不阻塞主线程。

核心指标口诀：LCP 小于 2.5 秒、FID 小于 100 毫秒、CLS 小于 0.1。

#8
XSS / CSRF / CSP 安全
XSS 是往页面注入恶意 JS，CSRF 是冒充已登录用户发请求，CSP 是限制页面能加载什么资源

XSS 防御三板斧：输出的时候做 HTML 转义、CSP 头限制只能加载指定域名的脚本、敏感 Cookie 设 HttpOnly 让 JS 读不到。CSRF 防御：后端校验 CSRF Token，Cookie 设 SameSite=Strict，再检查 Referer 来源。Token 存 Cookie 加 HttpOnly 比 LocalStorage 安全——LocalStorage 里的 Token 能被 XSS 直接偷走。

#10
Webpack vs Vite
Webpack 开发时也要全量打包，Vite 开发时不打包用浏览器原生 ESM 按需加载

Webpack 不管开发还是生产都从入口递归打包所有模块，项目大了启动就很慢。Vite 开发模式完全不做打包，浏览器请求哪个模块才编译哪个，利用原生 ESM，所以启动几乎秒开。生产构建 Vite 用 Rollup 不用 Webpack。HMR 也不一样：Webpack 是模块级替换，Vite 精确到修改的那个模块。Tree Shaking 两者都依赖 ESM 的静态分析。

浏览器事件机制
事件传播分三个阶段：捕获从外到内、到达目标、冒泡从内到外

事件委托就是利用冒泡——不给每个子元素绑事件，而是给父元素绑一个，触发时通过 e.target 判断是哪个子元素。好处是减少内存占用，而且动态添加的子元素自动就有事件。注意 e.target 是实际触发的元素，e.currentTarget 是绑定监听器的那个元素。

虚拟 DOM Diff 算法
三个策略：只比同层、不同类型直接替换、用 key 标识可复用节点

React diff 的核心思路是：只比较同一层级的节点，跨层级不比；如果类型变了（比如 div 变成 span），直接整个替换，不往下比；列表用 key 标识哪些节点可以复用。key 用 index 的坑：插入或删除一个元素，后面所有的 key 都对不上了，导致不必要的更新。Fiber 架构下 diff 分两阶段：render 阶段可以中断让出主线程，commit 阶段不可中断直接操作 DOM。

八股
HTTP/2 和 HTTP/3
HTTP/2 在一个 TCP 连接上并行多个请求，HTTP/3 用 QUIC 解决 TCP 队头阻塞

HTTP/2 三个核心改进：多路复用（一个 TCP 连接并行发多个请求，不用排队）、头部压缩（HPACK 算法压缩重复 header）、服务端推送。但 HTTP/2 还有队头阻塞——在 TCP 层面，一个包丢了会阻塞这个连接上所有流。HTTP/3 把 TCP 换成 QUIC（基于 UDP），每个流独立，一个流丢包不影响其他流，还支持连接迁移（WiFi 切 4G 不断连）和 0-RTT 建连。

CSS 布局：BFC + Flexbox + Grid
BFC 是独立渲染区域管浮动和 margin，Flex 管一维，Grid 管二维

BFC 触发条件：overflow 不是 visible、float 不是 none、position 是 absolute 或 fixed、display 是 flex 或 grid。常见用途：overflow:hidden 清除浮动、防止 margin 塌陷（两个相邻块级元素的上下 margin 会合并，给其中一个加 BFC 就不合并了）。

#14
var / let / const 区别
var 有变量提升和函数作用域，let/const 有块级作用域和暂时性死区

var：函数作用域，声明提升（声明提到顶部，赋值留在原地），可重复声明。

let：块级作用域，暂时性死区（TDZ——从块开始到声明语句之间不能访问），不可重复声明。

const：同 let，但声明时必须赋值且不能重新赋值。注意：对象/数组内部属性可以改，只是引用不变。

经典坑：for 循环用 var 导致所有回调共享同一个 i，用 let 每次循环有独立 i。

#15
this 指向规则
默认绑定 → 隐式绑定 → 显式绑定 → new 绑定 → 箭头函数，优先级从低到高

默认绑定：独立函数调用，严格模式 this 是 undefined，非严格是 window。

隐式绑定：obj.fn()，this 指向 obj。但赋值后单独调用会丢失：var fn = obj.fn; fn() → this 丢失。

显式绑定：call(thisArg, ...args) / apply(thisArg, [...args]) / bind(thisArg)，bind 返回新函数不立即执行。

new 绑定：new Fn() 时 this 指向新创建的对象。

箭头函数：没有自己的 this，继承外层的 this，不可被 call/apply/bind 改变。React 事件处理用箭头函数就是为了绑定组件实例。

① 创建一个空对象 obj。

② 将 obj.__proto__ 指向构造函数的 prototype。

新生代（存活时间短）：半空间复制算法（Scavenge）。From 空间存活对象复制到 To 空间，然后交换。一次 GC 存活的对象晋升到老生代。

面试口诀：新生代靠复制，老生代靠标记，WeakMap 不挡 GC。

install：首次注册触发，这里预缓存核心资源。event.waitUntil() 保证安装完成前 SW 不被终止。

LLM 基础原理
Transformer → 自注意力 → 预测下一个 Token

Transformer：并行处理整个序列，不像 RNN 逐个处理。核心是自注意力——每个 token 看到所有其他 token，算相关性权重。

训练两阶段：① 预训练——海量文本预测下一个 token（自回归）；② RLHF——人类反馈强化学习，让输出对齐人类偏好。

推理：给定上下文，逐 token 生成概率分布，采样选一个，拼入上下文，重复直到 <eos>。SSE 流式就是逐 token 推送到前端。

E14
Token 与 Tokenizer
Token 是 LLM 最小处理单位，不等于字符也不等于词

BPE（Byte Pair Encoding）：把常见字节对合并成 token。英文约 1 词 = 1 token，中文约 1 字 = 1-2 token。

换算：100 token ≈ 75 英文单词 ≈ 50-60 中文字。GPT-4 系列用 tiktoken，Claude 用自己的 tokenizer。

实际影响：API 按 token 计费，上下文窗口按 token 算（Claude 200K、GPT-4o 128K），前端做 token 预算控制需要估算。

Few-shot：给几个输入→输出示例，再让模型做。适合格式要求严格的场景。

Chain-of-Thought (CoT)："一步步思考"，让模型展示推理过程。数学、逻辑题效果显著。

Tree-of-Thought：探索多个推理路径，选最优。适合开放式问题。

实战优先级：简单 Zero-shot → 格式用 Few-shot → 复杂推理加 CoT。

E16
AI Agent 三大模式
Agent = 感知 → 思考 → 行动 循环

ReAct：Reasoning + Acting 交替，每步先想再干。最常用，GPT/Claude 默认模式。

Plan-and-Execute：先做完整计划再逐步执行，适合多步骤复杂任务。可中途重规划。

Reflection：执行后自我评估，发现错误就修正再执行。提升输出质量但成本翻倍。

项目桥接：prd-tools 11 步工作流 = Plan-and-Execute，Evidence Index = Reflection。

E17
RAG 深度解析
先查再答——解决知识过时和幻觉

流程：用户问题 → 向量检索相关文档 → 拼入 Prompt → LLM 生成回答。

Chunking 策略：固定大小（简单但粗糙）/ 按语义分段（效果好）/ 按文档结构（段落/章节）。块大小一般 256-1024 token。

传统 RAG vs Agentic RAG：传统是检索→生成两步；Agentic 加入多轮检索、工具调用、自我评估，能判断"检索结果够不够"。

项目桥接：prd-tools evidence = RAG 工程化变体，BM25 替代向量检索，代码锚定替代 chunking。

E18
Embedding 与向量检索
把文本变高维数字向量，语义相近的向量距离近

Embedding 模型：OpenAI text-embedding-3、BGE（中文强）、Cohere。输出通常是 768-3072 维向量。

向量数据库：Milvus（开源大规模）、Pinecone（托管服务）、Chroma（轻量本地）。用 ANN 近似最近邻做高速检索。

vs 传统搜索：BM25 精确关键词匹配，向量检索语义匹配。实战中混合检索（BM25 + 向量）效果最好。

E19
MCP 深度解析
Anthropic 主导的开放协议，让 AI 应用标准化接入工具

架构：Host（Claude Desktop/Cursor）→ Client（协议客户端）→ Server（工具提供方）。一对多：一个 Host 可连多个 MCP Server。

通信：JSON-RPC 2.0，支持 stdio（本地）和 SSE（远程）两种传输。

三大能力：Tools（可调用函数）、Resources（可读数据源）、Prompts（预设模板）。

为什么重要：一次开发到处用——写一个 MCP Server，Claude/Cursor/Windsurf/Cline 都能用。解决了 AI 工具生态碎片化问题。

E20
AI 框架生态与选型
LangGraph 编排 Agent，LlamaIndex 专精 RAG，各有所长

LangGraph：基于图的状态机，节点是函数，边是条件跳转，支持循环（Agent Loop）。复杂 Agent 编排首选。

LangChain：LangGraph 上层，链式调用 LLM + 工具，适合线性流程。生态最大但抽象层多。

LlamaIndex：RAG 专精——数据摄取 → 索引 → 查询，开箱即用。数据问答场景首选。

CrewAI：多 Agent 协作，每个 Agent 有角色和工具，适合模拟团队分工。

选型：简单链式 LangChain，复杂 Agent LangGraph，数据问答 LlamaIndex，多角色协作 CrewAI。

E21
A2A 协议
Google 主导的 Agent 间通信协议，解决 Agent 协作

定位：MCP 解决 Agent ↔ 工具，A2A 解决 Agent ↔ Agent。互补关系。

架构：每个 Agent 暴露 AgentCard（能力描述卡片），通过 JSON-RPC 互相发现和通信。

场景：前端 Agent 调用后端 Agent，后端 Agent 调用数据 Agent。多个专业 Agent 组合成复杂工作流。

与 MCP 关系：MCP 是纵向（Agent → 工具），A2A 是横向（Agent → Agent）。一个 Agent 可以同时是 MCP Client 和 A2A 参与者。

E22
Fine-tuning vs RAG vs Prompt Eng
从便宜到贵，从灵活到定制

Prompt Eng：零额外成本，改提示词就行。适合大部分场景，响应最快。

RAG：低成本，外挂知识库。适合需要最新信息、企业内部文档的场景。不改模型本身。

Fine-tuning：高成本，修改模型权重。适合特定风格、领域深度定制（医疗/法律）。需要大量标注数据。

决策树：先试 Prompt → 不够加 RAG → 还不够才 Fine-tuning。大多数场景 Prompt + RAG 就够了。

E23
AGI 概念与现状
通用人工智能——一个模型做所有认知任务

ANI vs AGI：当前所有 LLM 都是 ANI（弱 AI），只在特定领域强。AGI 是在大多数经济有价值的任务上超越人类。

当前共识：还没到，但进步很快。推理模型（o3、Claude）在数学/代码上已接近或超过人类专家。

核心争议：Scaling Law（更多数据+算力）能不能持续？还是需要新架构？业界分歧大。

面试态度：不过度炒作也不贬低，承认不确定性。关注 AGI 对前端工程师职业规划的影响。

AI 评估基准
怎么衡量模型好不好

通用：MMLU（多学科知识）、HumanEval（代码生成）、MATH（数学推理）、GPQA（博士级问题）。

前端相关：WebArena（网页操作）、SWE-bench（代码修复）、Aider leaderboard（实际编程）。

D18
能力面适配器怎么设计？
目录结构千差万别，但研发协作的能力面是稳定的——按能力面描述，不按目录

不同项目目录完全不一样，但不管怎么组织代码，总得有路由、组件、表单、状态管理、API 调用这些关注点。我们定义了前端 9 个、BFF 10 个、后端 10 个能力面，每个能力面描述一种关注维度。

关键设计：路径只是搜索候选，最终结论必须读源码确认。路径命中但没读源码只能是 candidate，读到实现才能进入 reference。这样 reference 对不同项目都能用，不需要改模板。

D19
知识过期怎么办？
三层：被动衰减监控 + 主动反馈回流 + 质量门控守门

第一层被动监控：reference 文件记录 last_verified，14 天没验证就标过期，健康检查对比源码发现矛盾。第二层主动回流：prd-distill 蒸馏时如果发现代码和 reference 矛盾，会生成回流建议文件，人工确认后更新 reference，越用越准。第三层守门：质量门控检查 enum、field、contract 是否和源码冲突，冲突就是致命项直接不通过。

D20
图谱和知识库的关系？
图谱是原始发现层，reference 是精选后的企业知识库，Raw Graph 不等于 Reference

GitNexus 告诉你代码怎么连接，Graphify 告诉你为什么这样设计——但图谱结论不能直接用，需要精选、审计、确认后才能进 reference。所以每个条目有两套独立证据：可审计证据（源码/文档确认）和图谱溯源（从哪发现的）。图谱不可用时完全回退到源码扫描，核心流程不受影响。Provider 可替换，不绑定具体工具。

B4
微前端三种方案对比
iframe 隔离完美但性能差，qiankun 用 JS 沙箱适合大型拆分，Module Federation 性能最好但需 Webpack 5

iframe：浏览器级完美隔离，但独立进程性能差，SEO 也不好，适合完全独立的第三方系统。

qiankun：基于 single-spa，用 Proxy 做 JS 沙箱隔离 + Shadow DOM 或 CSS 前缀做样式隔离 + HTML Entry 直接配 URL 就能用。适合大型应用拆分。

Module Federation：Webpack 5 运行时模块共享，性能最好因为共享依赖，但接入成本高且无隔离。适合细粒度模块共享。

我们项目的选择：没用标准微前端，因为 15 个模块在同一个仓库不需要独立部署，编译时共享更高效。实际是宿主平台通过 iframe 嵌入我们的页面，URL 参数传公参。

追问
qiankun 原理深挖（阿里二面真题）
核心做三件事：JS 沙箱（Proxy 代理 window）+ 样式隔离（Shadow DOM 或 CSS 前缀）+ HTML Entry（fetch 解析执行）

JS 沙箱：创建一个 fakeWindow，用 Proxy 代理，子应用操作的全局变量都在 fakeWindow 里，卸载后自动恢复，子应用之间互不污染。不支持 Proxy 的浏览器降级为快照沙箱——激活时存 window 快照，卸载时 diff 恢复。

样式隔离两种：strictStyleIsolation 用 Shadow DOM（弹窗挂在 body 外会丢样式）；experimentalStyleIsolation 给 CSS 选择器动态加前缀，类似 scoped CSS。

HTML Entry：不像 single-spa 需要子应用暴露 bootstrap/mount/unmount，qiankun 直接配 HTML 地址，它会 fetch 下来解析出所有 script 和 link 执行，子应用不用改代码。通信用 initGlobalState 做全局状态共享，底层是 CustomEvent 发布订阅。

追问
渐进式升级系统设计（阿里二面真题）
三层：Schema 加 version + 运行时按版本动态 import + 配置中心灰度控制

面试官命题："两个 button 一个 0.0.1 一个 0.0.2，怎么支持？"三层设计：

第一层 Schema 层加 d_version 字段，BFF 生成 Schema 时通过配置中心（Apollo）控制哪些字段用新版组件。

第二层运行时组件加载器做版本感知——注册表从"一个名字对应一个组件"变成"一个名字对应多个版本"，渲染时根据 d_version 动态 import 对应版本的组件。

第三层配置中心做灰度——先给 1% 用户下发新版 Schema，观察错误率和性能，没问题再逐步放量到 100%，最后去掉灰度配置让新版成为默认。

接入层三种形态（阿里二面真题）
按消费方需要的控制力度区分：组件模式要完整 CRUD、函数调用只要单点能力、微前端直接用标准能力

第一种组件模式（App 内嵌）：消费方需要完整 CRUD 能力，import 组件+传参，整个权益生命周期都交给你。

第二种函数调用模式（Modal.info 命令式）：只需要单点能力比如查一个权益详情，暴露 Modal 方法不需要引入整个组件，BenefitAdmin 统一分发解耦内部实现。

追问
Changesets 发版流程
人工选变更级别（比 semantic-release 更适合多包 Monorepo），alpha 走预发布通道

和 semantic-release 的区别：semantic-release 基于 commit message 全自动，适合单包；Changesets 人工选 patch/minor/major，适合多包 Monorepo——一次改动可能影响多个包，Changesets 能自动追踪依赖关系同时 bump。

alpha 流程：changeset pre enter alpha 进入预发布 → changeset 记录变更 → changeset version 生成 1.0.0-alpha.0 → publish --tag alpha → 消费方 npm install pkg@alpha 联调 → 联调完 changeset pre exit 退回正式通道。

第二线：个人知识库。用 Docsify 搭了一个学习知识库（linze-journal），把前端基础、架构模式、AI 工程分模块整理。不是泛泛抄文章，是结合项目经验写的——比如闭包那篇，直接拿联动引擎的柯里化举例。

第三线：开源工具。prd-tools 本身就是一个可复用的 AI 工程工作流——6 文件 SSOT + 证据链 + 质量门控的方法论不绑定具体业务，其他团队也能用。

软技能
期望薪资和到岗时间？
薪资给范围，到岗说一个月，留谈判空间

薪资：给一个合理的范围而不是固定数字，比如"我目前的综合 package 是 XX，期望有一定涨幅，具体可以聊"。让对方先出价。如果对方追问具体数字，说一个范围的中间值偏高。

到岗：正常是一个月交接期。"目前手里还有几个重要项目在收尾，预计一个月可以到岗。如果紧急可以协商提前"。

薪资谈判核心：1. 知道自己的市场价（4年+大厂+A绩效+架构经验，应该有溢价）2. 不要先出底牌 3. 综合考虑 base+bonus+期权+通勤+技术栈。

反问
有什么想问我的？（每次面试必问）
准备 2-3 个问题，按面试官角色选

技术面（一线/资深）：

· 团队目前的技术栈和架构是什么？有没有在做的技术升级？

· 我这个岗位进来之后，前半年最重要的 1-2 件事是什么？

· 团队怎么做 Code Review 和技术决策？是集中式还是民主式？

架构师/技术负责人面：

· 团队在 AI 工程方向有什么规划？比如 AI 辅助开发、Agent 工作流？

· 目前前端架构最大的技术债或痛点是什么？

· 怎么看前端工程师在 AI 时代的角色变化？

HR 面：

· 团队规模和组成？前端多少人？

· 绩效考核标准是什么样的？

· 团队的入职 onboarding 流程？

选 2-3 个问，别全问。优先问和工作内容直接相关的——展现你对岗位的认真思考，不是来凑数的。

L1
有点印象但说不准确
说确定部分 → 给合理推测 → 展示验证方向

"这个知识点我之前遇到过，核心思路是——[能想起来的]，但具体细节我可能说得不够精确。不过从使用场景来看，我觉得它主要解决的是——[合理推测]的问题。"

L2
完全没概念（算法不会）
先说暴力思路，再问优化方向

"这道题我一时想不出最优解，但我先说一下初步思路——[暴力解法]。如果时间允许，请问期望的时间复杂度是多少？这样我可以往那个方向优化。"

L3
被追问到死角
承认盲区 + 合理推测 + 学习方向

"这个问题确实触及了我的知识盲区，我回去会重点补一下。不过基于目前的项目经验，我的理解是——[合理推测]，不知道这个方向对不对？"

L4
系统设计没思路
类比已知架构 + 列关键决策点

"这个场景我没直接设计过，但可以从 Schema 驱动架构类比——核心都是分层解耦加可扩展。我会先确定几个关键问题：[列 2-3 个关键决策点]。"
