我在滴滴国际化做了四年营销中台前端，核心做的是一套 Schema 驱动的全栈架构。

简单说就是：BFF 层用模板引擎生成 Schema，前端通过渲染引擎动态消费。中间最核心的设计叫 injectDActions，是一次声明、双重求值——BFF 端先算一次保证初始状态正确，序列化给前端运行时再算一次做联动。联动分两种，需要后端数据的走 fetchApi 回调 BFF 重生成，纯前端的走本地 eval。这样新增活动类型，前端基本不用写新代码。

这套 Schema 架构后来也复用到了权益组件库 benefit SDK，支持 400 多种权益的组件化接入。

在这个基础上我又做了 prd-tools，用 AI 把 PRD 转成有证据、可执行、可回流的结构化开发计划。

A1
Monorepo 怎么管理 15+ 模块？
Lerna 管版本 + Nx 管加速 + Yarn 管依赖

app 目录放 15 个业务模块，独立版本、独立部署；packages 放共享组件。Lerna independent version 独立发版，Nx 做构建缓存只构建变更影响部分。

踩过坑：yarn 解析器从 rc.31 升到 rc.48 不兼容 node16，整个构建全挂。所以 yarn.lock 必须锁死。

A2
Schema 驱动架构怎么设计？
BFF 生成 Schema，前端消费 Schema，新增活动零代码

三层：后端 API 给原始数据，BFF 用模板引擎生成 Schema，前端按 Schema 渲染。

最核心是 injectDActions，一次声明、双重求值：BFF 先 safeEval 算初始状态，序列化给前端运行时再算。联动分 fetchApi 型和 state 型，16 个联动模板。

前端通过 formatName 拍平嵌套路径，Listener 发布订阅执行联动。组件双轨制：基础走 switch-case，业务走 RegisteredComponents，35 个动态渲染。

新增活动类型前端基本零代码。

A3
400+ 权益组件库怎么设计？
Register 扩展 + Schema 同构，两个消费者共享架构

Register 模式——内置 24 个组件，业务运行时注册自己的权益组件。和活动表单共享同一套 Schema 渲染架构，两个消费者共享，维护成本低。

Rollup 输出 ESM + CJS + d.ts。peerDependencies 避免重复打包 React。Changesets 发版，alpha 联调，yalc 本地验证。

A7
联动引擎怎么设计？
一次声明，两端求值

BFF 端 injectDActions 声明式注入，双重求值：BFF 先 safeEval 算一次保证 SSR 初始状态，前端运行时再算一次。

两种联动：fetchApi 型（需后端数据）回调 BFF 重生成；state 型（纯前端）本地 eval。16 个联动模板，listener.on 支持链式联动。

追问
Schema 架构有什么边界和风险？
表达力有上限，复杂联动和特殊 UI 仍需定制开发

三个边界。表达力上限：联动嵌套超过三层模板难维护，拆成多个模板各管一层。调试困难：动态生成不能断点，加了 Schema diff 工具。类型安全弱化：字符串键值编译时无法检查，BFF 模板层做字段枚举校验。

追问
大表单 200+ 字段性能？
分步渲染 + 字段级更新 + 虚拟滚动

三层：渲染层——按步骤分组，当前步全渲染，前后步只渲染摘要。更新层——Listener 按依赖图触发，只重渲染受影响组件。组件层——长列表用虚拟滚动。实测首屏从 1.2s 降到 300ms。

追问
BFF 挂了怎么办？
缓存兜底 + 降级 UI + 告警联动

缓存兜底：成功生成的 Schema 缓存到 CDN，BFF 超时用缓存渲染。降级 UI：连缓存都没有展示骨架页。告警：错误率超阈值自动告警。

追问
injectDActions 竞态问题？
有，请求序列号 + 废弃标记解决

快速切换字段触发两次 fetchApi，旧数据覆盖新数据。解法：递增序列号，响应回来检查是否最新请求，不是就丢弃。state 型同步不存在这个问题。

追问
线上出过事故吗？
yarn 解析器升级导致构建全挂，15 分钟回滚

resolver 从 rc.31 自动升到 rc.48 不兼容 Node 16，整个构建全挂。15 分钟回滚 yarn.lock 恢复。复盘三件事：yarn.lock 加入 git 保护；CI 加 lockfile diff 检测；依赖升级必须 staging 验证。

事故处理 = 快速止血 + 根因分析 + 流程改进。

追问
代码审查标准？
类型安全 + 边界条件 + 性能影响 + 架构一致性

四点：类型安全（TypeScript strict 有没有 any 逃逸）、边界条件（空值/越界/并发）、性能影响（是否引起大范围重渲染）、架构一致性（BFF 逻辑不泄漏到前端）。

扩展
为什么用 Rollup 不用 Webpack？
Rollup 基于 ESM 静态分析，天然 tree-shaking，产物干净

Rollup 天生为库打包。ESM 静态 import/export 分析，未引用导出直接删除。Webpack 的 tree-shaking 靠 Terser 分析副作用，不如 Rollup 精确，且有运行时代码。

一份配置出三份产物：ESM、CJS、d.ts。peerDependencies 声明 react 避免重复打包。

扩展
Monorepo 工具链怎么选？
Lerna 管版本 + Nx 管缓存，当时 Turborepo 还不成熟

Turborepo 构建编排强但版本管理要额外配，远程缓存在内网用不了。pnpm workspace 管依赖好但缺构建编排。Lerna+Nx 一步到位。现在可能会选 pnpm+Turborepo，但迁移成本高，当前方案稳定。

扩展
为什么用 yalc 不用 npm link？
npm link 的 symlink 导致 peerDependencies 解析错误

npm link 的 symlink 让 Rollup 解析不到宿主项目的 react，类型检查全挂。yalc 真实文件拷贝，模拟真实场景。

扩展
为什么不直接用 JSON Schema？
JSON Schema 做验证，不做 UI 渲染和联动

JSON Schema 只能说字段必须是数字，不能说用 Slider 还是 InputNumber 渲染，更不能描述联动。我们扩展了 d_component（组件类型）、d_actions（联动配置）、visible/disabled（UI 状态）。

#1
React 18 并发 + Fiber
Fiber 把递归变链表，Scheduler 按优先级调度

之前虚拟 DOM diff 是递归，停不下来。Fiber 把节点组织成 child/sibling/return 链表，每个节点是可拆分的工作单元。Scheduler 用 MessageChannel（不是 setTimeout，最少 4ms）做优先级调度，高优先级可打断低优先级。

React 18 加了 useTransition（低优先级）、useDeferredValue（延迟渲染）、Automatic Batching（任何上下文都合并 setState）。

#2
useEffect vs useLayoutEffect
useLayoutEffect 绘制前同步阻塞，useEffect 绘制后异步

DOM 更新完 → useLayoutEffect → 浏览器绘制 → useEffect。

需要读 DOM 布局或防闪烁用 useLayoutEffect，其他都用 useEffect。

#3
React 19 新特性
Server Components + React Compiler + Form Actions

RSC 只在服务端运行不发 JS 到客户端（和 SSR 不同）。React Compiler 自动优化重渲染，减少手动 useMemo/useCallback。Form Actions 绑定 server action 简化数据变更。use() Hook 支持 Promise 和 Context 条件调用。

#4
浏览器渲染管线
DOM+CSSOM → Render Tree → Layout → Paint → Composite

改 width/height 触发 Reflow 最贵。改 color 只 Repaint。改 transform/opacity 只 Composite 性能最好。

#5
HTTP 缓存
强缓存不请求，协商缓存可能 304 不下载

强缓存：Cache-Control max-age 直接用本地缓存不发请求。协商缓存：发请求可能返回 304，用 ETag + If-None-Match 比对。

口诀：HTML 用 no-cache，带 hash 的 JS/CSS 设 max-age 一年，API 用 no-store。

#12
TypeScript 高级类型
infer 推断 + 条件三元 + 映射遍历

infer 在条件类型里推断子类型。条件类型是类型层面三元：T extends X ? A : B。映射类型遍历联合类型生成新类型——Pick、Omit、Partial 本质都是映射类型。

#7
事件循环
微任务先于宏任务，同一轮微任务全清完才取下一个宏任务

同步代码 → 清空微任务（Promise.then） → 取一个宏任务（setTimeout）。async/await 本质 Promise 语法糖。

陷阱：Promise 构造函数里同步执行，.then 才是微任务。输出：1 5 3 4 2。

八股
闭包和作用域链
函数记住定义时的词法环境，即使在外部执行

经典场景：for 循环 var 的坑（用 let 或 IIFE）、函数工厂、柯里化、私有变量。闭包引用的变量不被回收，注意内存泄漏——事件监听器引用外部变量没清理，或定时器引用大对象。

八股
原型链与继承
对象通过 __proto__ 链向上查找属性

ES6 class 是语法糖，底层还是原型链。继承演进：原型链 → 构造函数 → 组合 → 寄生组合（最优，ES6 class 底层实现）。

八股
HTTPS/SSL 握手
非对称加密协商密钥，对称加密传输数据

四步：Client Hello（随机数+加密套件）→ Server Hello（随机数+证书）→ 客户端验证证书链 → 生成预主密钥用公钥加密发过去 → 双方用三个随机数生成会话密钥，之后对称加密。

ECDHE 支持前向保密。TLS 1.3 简化到 1-RTT。

#11
前端性能优化全链路
网络 → 渲染 → 资源 → 代码，四个维度

网络：CDN、HTTP/2、Preload。渲染：SSR/SSG、骨架屏、懒加载、虚拟列表。资源：代码分割、Tree Shaking、WebP。代码：React.memo、useMemo、Web Worker。

核心指标：LCP < 2.5s、FID < 100ms、CLS < 0.1。

#8
XSS / CSRF / CSP 安全
XSS 注入脚本，CSRF 冒充请求，CSP 限制加载

XSS 防护：输出转义 + CSP + HttpOnly Cookie。CSRF 防护：CSRF Token + SameSite Cookie + Referer 检查。Token 放 Cookie+HttpOnly 比 LocalStorage 更安全。

#10
Webpack vs Vite
Webpack 打包构建，Vite 开发时原生 ESM 按需加载

Webpack 从入口递归打包所有模块。Vite 开发模式不打包，用浏览器原生 ESM 按需编译，启动极快。生产用 Rollup。HMR：Webpack 模块级，Vite 精确到修改的模块。

八股
浏览器事件机制
捕获 → 目标 → 冒泡，委托利用冒泡

事件委托：给父元素绑定，通过 e.target 判断子元素。好处：减少内存、动态元素自动有事件。e.target 是触发元素，e.currentTarget 是绑定元素。

八股
虚拟 DOM Diff 算法
同级比较 + 三策略 + key 优化列表

三策略：只比较同层级、不同类型直接替换、key 标识可复用节点。key 用 index 的问题：插入删除导致后面全部更新。Fiber 下 diff 分两阶段：render 可中断，commit 不可中断。

追问
React 18 并发模式踩过什么坑？
Suspense 边界设计 + useTransition 饥饿 + Batching 适配

Suspense 边界太大 loading 闪烁严重，太小太多 loading 区——按用户感知的"内容块"划分。useTransition 低优先级可能饥饿——加 3 秒超时兜底。Automatic Batching 统一了，但依赖旧行为的代码需适配。

八股
HTTP/2 和 HTTP/3
HTTP/2 多路复用，HTTP/3 用 QUIC 解决队头阻塞

HTTP/2：多路复用、头部压缩 HPACK、服务端推送。但 TCP 层还有队头阻塞。HTTP/3 用 QUIC（基于 UDP），每个流独立，支持连接迁移和 0-RTT。

前端负责向量检索的用户体验和反馈闭环

E4

MCP 是标准化工具协议，Function Calling 是模型能力，互补

MCP：Anthropic 开放协议，Client-Server 架构，JSON-RPC，不绑定模型厂商，支持长连接和工具发现。Function Calling：OpenAI 定义，绑定模型，单次调用。A2A：Google 的 Agent 间通信协议，和 MCP 互补。
