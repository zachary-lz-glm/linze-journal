追问
异步组件（地址选择器）Schema怎么描述？
不是不支持，是远程数据组件通过自定义组件+异步属性声明来处理

地址选择器这种需要按级联请求远程数据的组件，确实不是Schema声明能闭环的，因为数据不在本地。处理方式是注册为自定义组件，组件内部自己管理异步数据请求和级联逻辑。

Schema层只声明组件类型和必要的配置（比如接口地址、级联层级），具体的异步请求逻辑封装在组件代码里。这和Formily处理远程数据组件的方式本质一样——复杂异步组件都需要代码层介入，纯配置搞不定。

我们的原则是：Schema管结构和联动，代码管复杂交互。98%的表单场景Schema直接覆盖，剩下2%的自定义组件兜底。

追问
Schema 架构有什么边界和风险？
表达力有上限，复杂联动和特殊 UI 仍需定制开发

三个边界。表达力上限：联动嵌套超过三层模板难维护，拆成多个模板各管一层。调试困难：动态生成不能断点，加了 Schema diff 工具。类型安全弱化：字符串键值编译时无法检查，BFF 模板层做字段枚举校验。

追问
大表单 200+ 字段性能？
分步渲染 + 字段级更新 + 虚拟滚动 + 联动接口SW缓存

渲染层：按步骤分组，当前步全渲染，前后步只渲染摘要。实测首屏从 1.2s 降到 300ms。

更新层：Listener 按依赖图触发，只重渲染受影响组件，不触发全表单重渲染。

组件层：长列表用虚拟滚动。

联动接口层：性能瓶颈不在渲染在接口——用户改一个配置项可能触发1-2个联动接口调用。Service Worker拦截联动请求做缓存匹配，同一联动接口入参相同直接返回缓存结果不发请求，和业务逻辑解耦不需改BFF层代码。

追问
一个字段改动会全表单重渲染吗？
不会，按 dot path 精确订阅，只更新受影响组件

和 Redux selector 原理类似——每个组件只订阅自己那个 dot path 的值（如 page1.form2.field3），不是监听整棵 Schema 树。某字段变了，引擎通过依赖映射表找到受影响的联动目标，只通知这些组件更新，其余组件不触发渲染。

面试要点：先回答"不会 + 精确订阅"，再补充 API 型联动才是真正的性能瓶颈（接口耗时 2-3s），计划用 Service Worker 缓存优化。

追问
联动会不会循环依赖导致死循环？
设计上约束联动只做清值不做赋值，架构层面断开循环链路

两层防御。第一层：联动规则只能做清值操作——A 改了导致 B 的选项更新，B 之前的值清掉。不会出现"A 赋值给 B，B 又赋值给 A"的循环。需要默认值时用独立字段存储，不走联动路径。

第二层：引擎层做 max depth 限制，联动链超过 3 层就中断并报警。如果追问"更系统的方案"，可以说拓扑排序检测依赖图中的环，有环就编译报错不让上线。

追问
BFF 挂了怎么办？
缓存兜底 + 降级 UI + 告警联动

缓存兜底：成功生成的 Schema 缓存到 CDN，BFF 超时用缓存渲染。降级 UI：连缓存都没有展示骨架页。告警：错误率超阈值自动告警。

追问
injectDActions 竞态问题？
有，请求序列号 + 废弃标记解决

快速切换字段触发两次 fetchApi，旧数据覆盖新数据。解法：递增序列号，响应回来检查是否最新请求，不是就丢弃。state 型同步不存在这个问题。

追问

resolver 从 rc.31 自动升到 rc.48 不兼容 Node 16，整个构建全挂。15 分钟回滚 yarn.lock 恢复。复盘三件事：yarn.lock 加入 git 保护；CI 加 lockfile diff 检测；依赖升级必须 staging 验证。

追问
Schema 版本变了怎么兼容？
Schema 带 version 字段，BFF 层版本路由 + 迁移函数双版本并行

Schema 根节点有 version 字段。BFF 层根据 version 路由到对应模板引擎版本处理，老活动和新活动可以并行运行。引擎升级时写迁移函数把老版本 Schema 转成新版本，线上双版本并行跑一段时间确认没问题后逐步下线老版本。

通用方法论：任何配置化系统的版本化三要素——版本号标注（知道是哪个版本）、兼容路由（新老版本都能跑）、迁移函数（老版本能转新版本）。

扩展
为什么用 Rollup 不用 Webpack？
Rollup 基于 ESM 静态分析，天然 tree-shaking，产物干净

Rollup 天生为库打包。ESM 静态 import/export 分析，未引用导出直接删除。Webpack 的 tree-shaking 靠 Terser 分析副作用，不如 Rollup 精确，且有运行时代码。

一份配置出三份产物：ESM、CJS、d.ts。peerDependencies 声明 react 避免重复打包。

扩展
为什么不用 Formily？
Formily是纯前端表单方案，我们需要从BFF到前端的端到端联动DSL

Formily确实能做联动和自定义扩展，JSON Schema描述表单结构，x-decorator/x-component控制渲染， reactionary做响应式联动。但这些都在客户端闭环。

我们的特殊性在于联动不是前端自闭环的。很多联动需要调BFF接口查业务数据，BFF根据结果重新拼接Schema下发给前端做整体替换。这意味着联动配置必须嵌在BFF模板里，和后端逻辑紧耦合，Formily的客户端方案覆盖不了这一层。

所以我们需要一套从BFF模板引擎到前端渲染引擎的端到端联动DSL，而不是纯前端表单方案。如果只是前端渲染层，Formily完全可以。我们的选择不是因为Formily不行，而是场景不同。

#1
React 18 并发 + Fiber
Fiber 把递归变链表，Scheduler 按优先级调度

之前虚拟 DOM diff 是递归，停不下来。Fiber 把节点组织成 child/sibling/return 链表，每个节点是可拆分的工作单元。Scheduler 用 MessageChannel（不是 setTimeout，最少 4ms）做优先级调度，高优先级可打断低优先级。

React 18 加了 useTransition（低优先级）、useDeferredValue（延迟渲染）、Automatic Batching（任何上下文都合并 setState）。

#2
useEffect vs useLayoutEffect
useLayoutEffect 绘制前同步阻塞，useEffect 绘制后异步

DOM 更新完 → useLayoutEffect → 浏览器绘制 → useEffect。

需要读 DOM 布局或防闪烁用 useLayoutEffect，其他都用 useEffect。

#2
React 从 setState 到视图更新的完整流程
setState触发后经历调度、协调、提交、绘制四个阶段，Effect在不同阶段介入

1.调度：setState触发更新进入Scheduler，根据lane优先级安排执行时机。React 18 automatic batching合并多次setState为一次更新。

2.协调(Reconciler)：从触发组件开始DFS遍历Fiber树，双缓冲(current/workInProgress)做diff，标记effectTag。

3.提交(Committer)：将workInProgress树的变更同步应用到真实DOM。

4.useLayoutEffect：DOM更新后、paint前同步执行（读布局/防闪烁）。

5.浏览器paint。

6.useEffect：paint后异步执行（副作用/订阅/请求）。

微任务先于宏任务，同一轮微任务全清完才取下一个宏任务

同步代码 → 清空微任务（Promise.then） → 取一个宏任务（setTimeout）。async/await 本质 Promise 语法糖。

八股
闭包和作用域链
函数记住定义时的词法环境，即使在外部执行也能访问那个环境里的变量

闭包的本质就是：一个函数可以访问它定义时所在作用域的变量，即使这个函数在外部被执行。比如函数工厂 function makeCounter() 里定义了一个 count 变量，返回的函数每次调用都能访问到同一个 count——这就是闭包。

常见场景：for 循环里用 var 会导致所有回调共享同一个 i（用 let 解决）、柯里化其实就是闭包的应用——先传部分参数，返回的函数记住这些参数再接收剩余参数。

注意事项：闭包引用的变量不会被垃圾回收，所以事件监听器或定时器如果引用了外部大对象，不用的时候要手动清理，否则内存泄漏。

八股
原型链与继承
对象通过 __proto__ 逐级向上查找属性，直到 null 为止

当我访问 obj.name 时，JS 引擎先在 obj 自身上找，找不到就顺着 obj.__proto__ 去它的原型上找，还找不到就继续往上，一直到 Object.prototype.__proto__ 也就是 null，这条链就是原型链。

ES6 的 class 本质还是原型链，class Dog extends Animal 编译后就是 Dog.prototype.__proto__ = Animal.prototype。继承方案演进了很多代：原型链继承（引用类型共享问题）→ 构造函数继承（拿不到原型方法）→ 组合继承（调了两次父构造函数）→ 寄生组合继承（最优方案，ES6 class 底层就是这个）。

#11
前端性能优化全链路
从网络、渲染、资源、代码四个维度逐一优化

网络层：CDN 缩短物理距离、HTTP/2 多路复用减少连接、Preload 预加载关键资源。渲染层：首屏用 SSR 或骨架屏、图片懒加载、长列表用虚拟滚动只渲染可视区域。资源层：代码分割 dynamic import 按需加载、Tree Shaking 删掉没用的代码、图片用 WebP。代码层：React.memo 和 useMemo 防止不必要的重渲染、耗时计算放 Web Worker 不阻塞主线程。

核心指标口诀：LCP 小于 2.5 秒、FID 小于 100 毫秒、CLS 小于 0.1。

#8
XSS / CSRF / CSP 安全
XSS 是往页面注入恶意 JS，CSRF 是冒充已登录用户发请求，CSP 是限制页面能加载什么资源

XSS 防御三板斧：输出的时候做 HTML 转义、CSP 头限制只能加载指定域名的脚本、敏感 Cookie 设 HttpOnly 让 JS 读不到。CSRF 防御：后端校验 CSRF Token，Cookie 设 SameSite=Strict，再检查 Referer 来源。Token 存 Cookie 加 HttpOnly 比 LocalStorage 安全——LocalStorage 里的 Token 能被 XSS 直接偷走。

#9
CSS 布局：BFC + Flexbox + Grid
BFC 是独立渲染区域管浮动和 margin，Flex 管一维，Grid 管二维

BFC 触发条件：overflow 不是 visible、float 不是 none、position 是 absolute 或 fixed、display 是 flex 或 grid。常见用途：overflow:hidden 清除浮动、防止 margin 塌陷（两个相邻块级元素的上下 margin 会合并，给其中一个加 BFC 就不合并了）。

Flexbox 是一维布局：主轴用 justify-content 控制对齐，交叉轴用 align-items。flex:1 等价于 flex-grow:1 + flex-shrink:1 + flex-basis:0%，就是自动填满剩余空间。Grid 是二维布局：grid-template-columns 定义列，fr 单位做比例分配。选型口诀：一行或一列用 Flex，行列交叉用 Grid。

#22
十万条数据怎么渲染？
虚拟列表只渲染可视区域，或分批渲染用 requestAnimationFrame

function search(nums, target) { let lo = 0, hi = nums.length - 1; while (lo <= hi) { const mid = (lo + hi) >> 1; if (nums[mid] === target) return mid; // 左半边有序 if (nums[lo] <= nums[mid]) { if (nums[lo] <= target && target < nums[mid]) hi = mid - 1; else lo = mid + 1; } else { // 右半边有序 if (nums[mid] < target && target <= nums[hi]) lo = mid + 1; else hi = mid - 1; } } return -1; }

function lowestCommonAncestor(root, p, q) { if (!root || root === p || root === q) return root; const left = lowestCommonAncestor(root.left, p, q); const right = lowestCommonAncestor(root.right, p, q); if (left && right) return root; // 左右都有，当前就是LCA return left || right; // 透传非空的那一侧 }

经典二维 DP：dp[i][j] = 字符相同取左上+1，否则取 max(上,左)

思路：dp 表格，行是 s1、列是 s2。字符匹配 → dp[i-1][j-1]+1；不匹配 → max(dp[i-1][j], dp[i][j-1])

function longestCommonSubsequence(s1, s2) { const m = s1.length, n = s2.length; const dp = Array.from({length: m + 1}, () => Array(n + 1).fill(0)); for (let i = 1; i <= m; i++) { for (let j = 1; j <= n; j++) { if (s1[i-1] === s2[j-1]) dp[i][j] = dp[i-1][j-1] + 1; else dp[i][j] = Math.max(dp[i-1][j], dp[i][j-1]); } } return dp[m][n]; }

E16
Agent 架构全解（面试必考）
Agent = LLM + Tool Use + Memory 的循环决策系统

定义：让 LLM 自主决策"下一步做什么"。普通 Chat 是单轮问答，Agent 是多轮循环——感知（接收输入+工具结果）→ 思考（LLM 推理）→ 行动（调用工具），不断迭代到任务完成。

三大模式：

· ReAct：Reasoning + Acting 交替，每步先想再干。GPT/Claude 默认模式，适合工具调用场景。

· Plan-and-Execute：先做完整计划再逐步执行，适合多步骤复杂任务，可中途重规划。项目桥接：prd-tools 11步工作流。

· Reflection：执行后自我评估，发现错误修正再执行。提升质量但成本翻倍。项目桥接：Evidence Index。

Tool Use 底层链路：① 开发者定义 function schema（名称+参数+描述）→ ② LLM 根据语义判断是否调用工具 → ③ 输出 tool_call JSON → ④ 运行时执行工具获得结果 → ⑤ 结果注入上下文继续推理。循环直到不再调用工具。

Memory 三层：短期记忆（当前对话窗口）、长期记忆（摘要压缩/向量存储）、工作记忆（当前任务中间状态）。prd-tools reference = 长期记忆工程化。

面试答题句式："Agent 核心是 ReAct 循环——感知→思考→行动不断迭代。扩展靠 Tool Use，通过 function schema 定义工具接口，LLM 语义选择调用。框架层面 LangGraph 适合复杂编排。我的 prd-to-code 用双 Skill + 证据链约束 Agent 行为。"

E20
LangChain + LangGraph 详解
LangChain 链式编排，LangGraph 图状态机精确控制 Agent

LangChain 六大核心：

· Chain：链式调用，把多个 LLM 调用和工具串起来。LCEL（LangChain Expression Language）声明式编排。

· Agent：自动选择工具和决策的执行器，核心是 ReAct 循环。

· Tool：工具定义和注册。支持自定义函数、API 调用、代码执行。

· Memory：对话历史管理。Buffer（全保留）/ Summary（摘要压缩）/ Window（滑动窗口）。

· Retriever：RAG 检索器。对接向量数据库、BM25、混合检索。

· Callback：全链路观测。日志、trace、token 计数、流式输出。

LangGraph 核心概念：LangChain 团队出的独立框架，解决复杂 Agent 编排。

· StateGraph：有向图状态机。节点是处理步骤（LLM/工具/条件判断），边是跳转规则。

· Node + Edge：节点是函数，边是跳转规则。Conditional Edge 支持分支。

· Checkpoint：状态持久化。每步自动保存，支持暂停恢复、崩溃恢复。

· Human-in-the-loop：任意节点插入人工审核点，Agent 暂停等人确认后继续。项目桥接：prd-tools Review Gate。

LangChain vs LangGraph：LangChain 适合线性链式（A→B→C），上手快但复杂场景难控。LangGraph 适合循环、分支、人工介入的工作流，控制力更强。项目桥接：prd-tools 双 Skill 11步 = LangGraph 模式——明确步骤、条件分支、Review Gate。

其他：CrewAI（多 Agent 协作）适合团队分工模拟；LlamaIndex（RAG 专精）适合数据问答。

RAG 深度解析
先查再答——解决知识过时和幻觉

流程：用户问题 → 向量检索相关文档 → 拼入 Prompt → LLM 生成回答。

Chunking 策略：固定大小（简单但粗糙）/ 按语义分段（效果好）/ 按文档结构（段落/章节）。块大小一般 256-1024 token。

MCP 协议全解
LLM 的 USB 接口——标准化连接外部工具和数据

架构：Host（Claude Desktop/Cursor/VS Code）→ Client（协议客户端）→ Server（工具提供方）。一对多：一个 Host 可连多个 MCP Server。

通信：JSON-RPC 2.0。两种传输——stdio（本地进程通信，Server 作为子进程）和 SSE（远程 HTTP 长连接）。

三大能力：

· Tools：可调用函数。定义 name + inputSchema(JSON Schema) + description，LLM 根据语义选择调用。类似 Function Calling 但跨平台。

· Resources：可读数据源。文件、数据库记录、API 响应，用 URI 标识，Agent 按需读取。

· Prompts：预置提示词模板。参数化 prompt，Agent 调用时传参。

开发实战：TypeScript SDK（@modelcontextprotocol/sdk），定义 Server → 注册 Tools/Resources → 启动 stdio 监听，Host 自动发现。我开发过内部 MCP Server——从飞书文档提取 PRD 转 Markdown，Claude Code 和 Cursor 都能用。

MCP vs Function Calling vs A2A：MCP 解决 Agent ↔ 工具（纵向），不绑定模型厂商，一次开发到处用。Function Calling 是 OpenAI 模型内置能力，绑定模型，单次调用。A2A 是 Google Agent 间通信协议（横向），解决 Agent ↔ Agent 协作。三者互补。

E25
结构化索引 vs 向量检索怎么选？
结构化高→正则索引；语义模糊→向量检索

结构化索引（正则/倒排）：用规则从代码中提取实体关系（函数名、字段名、文件路径），构建倒排表。精准、轻量、无需模型。适合：代码库、配置文件等结构化程度高的场景。

向量检索（Embedding）：把文本切成 chunk → 用 embedding 模型向量化 → 存向量数据库 → 余弦相似度检索。能理解语义相似但字面不同的内容。适合：自然语言文档、FAQ、非结构化知识库。

混合检索：实际生产中通常两者结合——BM25 关键词匹配保精准，向量检索补语义。再接 Reranker 重排序。

面试口诀：它们不是替代关系，是互补关系。正则索引 = 精确查找，向量检索 = 语义查找。场景结构化程度高就用正则就够了，场景有自然语言模糊性就需要向量。

证据优先级：接口文档 > PRD推断 > 负向搜索。置信度三档：high/medium/low，不确定就标低不能编造。

架构
BMAD架构：多Skill拆分+中间文件传递
每个Skill独立上下文窗口，Skill间通过中间文件传递状态，不共享上下文

BMAD核心思想：把复杂任务拆成多个独立Skill（架构思考→讨论→出文档→执行→复盘），每个Skill有独立上下文窗口专注一个维度，避免窗口膨胀导致幻觉和跳步。

中间文件传递：Skill间不直接传上下文，而是每步产出JSON/YAML中间文件，下一步读取上一步的状态。人为确认中间产物后才继续下一步（渐进式披露）。

借鉴到prd-tools：Reference和PRD蒸馏拆成两个独立插件（各含2个Skill）。简化了颗粒度（BMAD拆更细，我们把蒸馏相关步骤揉在一个Skill里便于团队推广）。

适用场景：任务可拆成独立阶段、每个阶段有不同的上下文需求、需要人为介入确认中间产物的场景。

架构
双插件体系：Reference+PRD蒸馏
第一个插件建领域知识库，第二个插件做PRD蒸馏，拆开是为了上下文隔离

为什么拆两个插件：项目知识理解跟需求分析是不同认知任务，上下文混在一起AI会跳步和幻觉。核心是Context Engineering——信息筛选+排序+预算管理。

reference插件：含reference和team-reference两个Skill。输入Git Log+历史PRD+技术文档，多轮分析后产出6个领域知识文件——代码结构、编码规则、跨层契约、PRD路由映射、业务领域、导航入口——加上代码实体索引和倒排索引。

prd-distill插件：含prd-distill和team-distill两个Skill。基于reference产出的知识做11步结构化蒸馏，从PRD产出Spec和开发Plan。能力面适配器在蒸馏前约束AI思考方向。

按需加载：不是一次性把所有知识塞给AI，而是先判断涉及哪个层（前端/BFF/后端），只加载对应层的Reference文件和能力面。

架构
质量门控：prd-tools的门禁体系
三层门控层层收紧：步骤间前置检查 → 蒸馏门禁12项 → 最终五维评分

第一层：步骤间前置检查：11步蒸馏中，每步执行前检查上一步产物存在且非空。不通过就重跑上一步。

第二层：蒸馏门禁：12项检查——8项质量（文件存在、需求有证据、代码锚点、Spec确认等）+ 4项覆盖率（PRD内容全覆盖、图片全覆盖、需求全追溯，这3项是致命项，不通过直接停）。

第三层：最终评分：两套五维系统——蒸馏过程内置评分（PRD摄入/证据覆盖/代码搜索/契约对齐/计划完整）+ 独立脚本终检（必传文件/上下文消费/锚点覆盖/计划完整/阻塞项质量）。加上Spec Review Gate：Spec生成后硬停止等人工确认才允许生成Plan。

设计原则：声明约束不够加门禁，门禁不够加评分，评分不够加人工硬停止。

五个维度加权评分，85分直接通过，60分需确认，低于60必须修

五个维度：PRD摄入质量（权重20%）、证据覆盖率（25%）、代码搜索命中率（15%）、契约对齐度（25%）、计划完整性（15%）。加权汇总后对照两个阈值。

架构
PRD摄入：多格式输入与图片并行策略
支持四种文档格式输入，图片由AI子进程分批并行分析，不影响主流程

四种输入格式：.md/.txt直接读取写入_ingest/document.md，远程图片URL自动下载到media/；.docx用ingest-docx.py提取文字+图片；粘贴文本手工建来源。不用unzip -d（macOS权限坑）。

图片并行策略：主上下文禁止直接Read图片。分批≤8张开1个foreground子Agent，>8张按数量拆多个子Agent在同一条消息中并行启动。每个子Agent产出media-analysis-part-N.yaml，全部完成后合并为media-analysis.yaml。

质量约束：没有人工确认的图片内容不能生成高置信度需求。AI看图提取的信息默认medium置信度，关键结论仍需文本证据或人工确认才能升为high。extraction-quality.yaml三档：pass可继续，warn继续但spec暴露风险，block暂停。

设计思想：用foreground并行而非background——保证子Agent完成后主流程才继续，避免异步竞态。批量大小8是上下文预算与并行效率的平衡点。

追问
PRD超大（2万字）怎么不爆context？
PRD先做分段摘要提取需求骨架，再按需检索相关段落，不一次性灌入

实际场景确实会遇到20-30页的重构级PRD，直接灌入context会打爆。三层处理策略。

第一层，分段摘要：输入时先对PRD做分段，提取需求骨架——涉及哪些模块、哪些业务场景、哪些改动范围。摘要控制在2000字以内，作为主context的输入。

第二层，按需检索：根据摘要分析结果，从原文中只检索和当前需求点相关的段落做精确分析。不是把整个PRD都喂进去，而是按需求点逐个定位。

第三层，多Agent并行：对超长的表格数据或图片，通过多个子Agent分批并行处理，每个Agent只处理一个片段。

核心思路是"先摘要定位，再按需深入"，而不是把整个PRD一次性灌入context。目前这个策略还在优化中，是对工作流的重要补充。

Tools vs Skill 有什么区别？
Tool给Agent能力，Skill给Agent判断力——一个是锤子扳手，一个是施工手册

相同点：底层都走Function Calling机制，都是模型识别意图后注入上下文，结果都反馈给模型继续推理。

三个关键区别：

· 粒度不同：Tool是原子操作，比如查天气、调接口，一次调用一个结果。Skill是多步骤编排，比如分析PRD到产出Spec再到生成Plan，是一套完整的流程。

· 本质不同：Tool是外部集成，本质是API的包装器，给模型"能做什么"。Skill是可复用的过程知识，本质是一组Markdown指令集，教模型"怎么做更好"。Skill不增加新能力，而是指导已有能力的使用方式。

· 状态不同：Tool是无状态的一次调用，调完就结束。Skill是有状态的流程编排，包含前置条件、中间产物传递、质量门控，整个流程有生命周期。

面试一句话："Tool是工人手里的工具，Skill是施工手册——工具决定能做什么，手册决定怎么做最靠谱。"

Tool多了识别率下降怎么优化？
三层策略——分层路由先分类再选、动态激活按需加载、优化描述降低认知负担

问题有多严重：研究数据明确——从4个Tool增加到51个Tool时，GPT-4o准确率从43%降到2%。Tool数量和跨领域混叠是核心杀手。

策略一：分层路由。不做一次性从几十个Tool里选，先用Router模型（可以是小模型）分类用户意图到具体领域（天气、金融、客服），再让对应领域的子Agent从少量Tool里选择。本质是把一个N选1的问题变成两次小范围选择。

策略二：动态激活。每次请求不传所有Tool定义，先做意图匹配只注入相关Tool，减少上下文噪音和Token消耗。LangChain的ReAct实验证明，不相关Tool的混入会显著拉低准确率。

策略三：优化Tool描述质量。函数名用"操作-实体-数据"命名规范，比如get_stock_data而不是stock_info。参数控制在5个以内，复杂参数展平避免深层嵌套。有固定取值的参数用枚举约束，不让模型自由发挥。描述越精确，模型选错的概率越低。

传统工程思维在AI系统中的映射
大量传统架构模式在AI系统里换了新名字重新出现，本质完全一样

核心观点：AI工程没有推翻传统软件工程，而是把同样的设计思维用在了新的接口层——以前接口是JSON，现在接口是自然语言。

六个典型映射：

· Prompt模板引擎 = MVC的View模板。用户输入是变量，Prompt是模板，跟Jinja2或Mustache完全一样的思路。变量注入前要做输入校验防Prompt注入，就像防SQL注入。

· Function Calling = 适配器模式。把模型的自然语言输出翻译成结构化的API调用，就是经典的Adapter——把一个接口转换成另一个接口。

· MCP协议 = USB-C标准化。统一接口让任意AI客户端连接任意工具，跟USB-C让任意设备连接任意充电器一个道理。Write once, run anywhere。

· Progressive Summarization = 日志轮转压缩。递归压缩旧消息保持固定大小，跟Log Rotation一模一样——老的压缩归档，新的完整保留。

· 语义缓存 = Memoization模式。相似请求直接返回缓存结果，本质就是函数式编程里的记忆化——相同的输入不重复计算。

· Flow Engineering = 状态机。用LangGraph把Agent行为建模成确定性状态转换，跟XState或Redux管理前端状态是同一个模式——节点是处理步骤，边是跳转规则。

面试用法：被问"AI和传统工程有什么关系"时，用这个映射表快速展示跨领域理解力。能说出三组以上映射的候选人很少。

Agent工作记忆怎么管理？
Progressive Summarization递归压缩旧消息保持上下文固定大小，类似操作系统的虚拟内存分页

核心问题：对话越长上下文越满，Token消耗线性增长、延迟增加、模型注意力被稀释。必须压缩，但不能丢关键信息。

方案一：Progressive Summarization。把最老的消息递归压缩成摘要块放在Prompt开头，上下文保持固定大小。优点是成本和延迟恒定，缺点是有损压缩——多次摘要后会丢失细节，像传话游戏。

方案二：MemGPT/Letta分页机制。设置上下文阈值，超过就把旧内容flush成摘要存到外部存储，需要时再检索回来。跟操作系统的虚拟内存分页一个思路——主存放不下就换到磁盘，用的时候再换回来。

方案三：Observation Masking（JetBrains研究）。不真删旧消息，用简短占位符替换，保留对话结构信息但大幅减少Token。比如把一段50行的讨论压缩成"[讨论了API设计，决定用REST]"。

实际工程中的做法：Claude Code的Memory文件就是长期记忆和短期记忆的桥梁——自动提取关键决策和约束存成Markdown，下次对话自动注入。这解决了"每次对话从零开始"的问题。

面试一句话："工作记忆管理本质是信息压缩问题——像操作系统管内存一样，主存（上下文窗口）放不下就换页（flush到外部存储），但压缩策略决定了你会丢多少细节。"

当AI产物质量不达标时，门禁脚本先判断哪步出了问题，然后按三个维度归因并采取不同修复策略。

知识库缺失：AI基于不完整的信息做了错误推断。修复方式是触发Reference回流，把缺失的知识补进去，下次不会再犯同样的错。

模型能力不足：AI跳步或逻辑错误，信息给了但没用好。修复方式是加强那一步的prompt约束和硬校验，把软约束变成硬规则。

上下文不够：AI理解偏差，检索到的相关信息不足。修复方式是扩大源码扫描的召回率，增加上下文包的信息量。

目标：从"人工发现AI错了"变成"系统自动定位为什么错了"，最终形成数据飞轮。

缺点：在架构设计上有时过度追求通用性。比如 Schema 联动引擎初期设计了很灵活的嵌套联动，但实际 80% 的场景是简单的显隐和值联动，复杂的用例反而增加了理解成本。后来做了简化——16 个标准联动模板覆盖绝大部分场景，超出范围的才允许自定义。这个经历让我学会了架构要服务于实际场景，不是为了架构而架构。

缺点不能说"我太追求完美"这种假缺点。要说一个真实的、有反思的、有改进行为的缺点。

软技能
期望薪资和到岗时间？
薪资给范围，到岗说一个月，留谈判空间

薪资：给一个合理的范围而不是固定数字，比如"我目前的综合 package 是 XX，期望有一定涨幅，具体可以聊"。让对方先出价。如果对方追问具体数字，说一个范围的中间值偏高。

到岗：正常是一个月交接期。"目前手里还有几个重要项目在收尾，预计一个月可以到岗。如果紧急可以协商提前"。

薪资谈判核心：1. 知道自己的市场价（4年+大厂+A绩效+架构经验，应该有溢价）2. 不要先出底牌 3. 综合考虑 base+bonus+期权+通勤+技术栈。

软技能
"为什么选我们公司？"
技术方向匹配 + 个人兴趣/背景契合 + 想做的事正好你们在做

三句话公式：[我的方向] + [你们做的事] + [交集在哪]。

示例（奇迹）："我个人方向是想深入AI应用层，你们做的是AI+体育视频理解，这两个正好重叠。而且我本身是体育爱好者，篮球校队队长，对体育内容有真实感知，不是纯技术视角。我觉得这种结合点很难得。"

反例："觉得AI有前景，想来试试。"——太空，没有连接任何具体的东西。

面试前必须想好这句话。如果面试中没法自然说出"为什么选你们"，说明对公司了解不够。

反问
有什么想问我的？（每次面试必问）
准备 2-3 个问题，按面试官角色选

技术面（一线/资深）：

· 团队目前的技术栈和架构是什么？有没有在做的技术升级？

· 我这个岗位进来之后，前半年最重要的 1-2 件事是什么？

· 团队怎么做 Code Review 和技术决策？是集中式还是民主式？

架构师/技术负责人面：

· 团队在 AI 工程方向有什么规划？比如 AI 辅助开发、Agent 工作流？

· 目前前端架构最大的技术债或痛点是什么？

· 怎么看前端工程师在 AI 时代的角色变化？

HR 面：

· 团队规模和组成？前端多少人？

· 绩效考核标准是什么样的？

· 团队的入职 onboarding 流程？

选 2-3 个问，别全问。优先问和工作内容直接相关的——展现你对岗位的认真思考，不是来凑数的。

表达
面试回答结构化框架
先说结论再展开，每个回答都有头有尾

总分总模板：

· 总（10秒）：一句话核心结论。"这是一个 Schema 驱动的营销中台。"

· 分（30-60秒）：2-3 个关键点，用"第一...第二...第三..."引导。

· 总（10秒）：收尾呼应。"所以核心价值是 XX。"

STAR 法则（项目类问题）：S 背景 → T 任务 → A 行动（重点）→ R 结果（要有数字）。

口头禅克制：

· "然后" → 停顿或用"第一...第二..."替代。

· "就是" → 删除，直接说。

· "对对对" → 点一次头就够了。

· 练习方法：录音自己说一段话，数"然后"次数，目标降到 5 次以内。
