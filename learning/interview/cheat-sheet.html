开场白
30 秒项目开场白
Schema 驱动架构 → BFF 生成 → 前端渲染 → AI 工程化

我在滴滴国际化做了四年营销中台前端，核心做的是一套 Schema 驱动的全栈架构。

简单说就是：BFF 层用模板引擎生成 Schema，前端通过渲染引擎动态消费。中间最核心的设计叫 injectDActions，是一次声明、双重求值——BFF 端先算一次保证初始状态正确，序列化给前端运行时再算一次做联动。联动分两种，需要后端数据的走 fetchApi 回调 BFF 重生成，纯前端的走本地 eval。这样新增活动类型，前端基本不用写新代码。

这套 Schema 架构后来也复用到了权益组件库 benefit SDK，支持 400 多种权益的组件化接入，内置 24 个组件加运行时注册扩展。

在这个基础上我又做了 prd-tools，用 AI 把 PRD 转成有证据、可执行、可回流的结构化开发计划。不是让 AI 写代码，而是工程化地管理从需求到代码的全链路。

语速 250 字/分钟，这段约 45 秒。面试官追问时顺势展开。

  • 偏传统前端 → 展开 Schema 渲染、组件库、Monorepo
  • 偏 AI → 展开 prd-tools 证据链、质量门控、reference 知识库
  • 不要一上来讲太多细节，先立住主线
项目A1
Monorepo 怎么管理 15+ 模块？
Lerna 管版本 + Nx 管加速 + Yarn 管依赖

我们用 Lerna + Nx + Yarn workspace 来管理。app 目录放 15 个业务模块，每个独立版本、独立部署；packages 目录放共享组件和工具。Lerna 用 independent version 让各业务独立发版，Nx 做构建缓存只构建变更影响的部分。

踩过一个印象很深的坑：有个 yarn 解析器依赖从 rc.31 升到 rc.48，直接不兼容 node16，整个项目构建全挂。所以后来我们锁死了 yarn.lock，不允许随便删。

最后一句"踩过的坑"是加分项，体现真实工程经验。

  • Lerna vs Turborepo：历史选型 + 独立版本 + Nx 缓存
  • 共享组件放 packages，通过 workspace 协议引用
  • 风险：依赖漂移、类型版本冲突、构建缓存失效
项目A2
Schema 驱动架构怎么设计？
BFF 生成 Schema，前端消费 Schema，新增活动类型前端零代码

这套架构分三层：后端 API 给原始数据，BFF 层负责生成 Schema，前端负责按 Schema 渲染。

BFF 不是手写 JSON，是用模板引擎驱动的。每个步骤有独立模板——basic、group、rules、message、preview——Builder 编译出 content_schema。最核心的设计是 injectDActions，一次声明、双重求值：BFF 端先 safeEval 算一次设初始状态，然后序列化 d_actions 给前端，前端运行时字段变化再算一次。联动分两种，fetchApi 型回调 BFF 重生成，state 型前端本地 eval。一共 16 个联动模板。

前端拿到 Schema 后通过 formatName 把嵌套路径拍平成 dot path，然后用 Listener 发布订阅执行联动。组件渲染是双轨制，基础走 switch-case，业务走 RegisteredComponents 注册表，35 个组件动态渲染。

最大的好处是新增活动类型，前端基本不用写新代码，主要改 BFF 模板和联动就行。

如果面试官追问性能：局部字段重渲染，按依赖触发，不全表单刷新。

  • 为什么不用传统表单：23 种活动类型、字段组合和联动变化太快
  • 两种联动：fetchApi 型（需后端数据）+ state 型（纯前端条件）
  • 风险：Schema 过度动态会牺牲类型安全和可调试性
项目A3
400+ 权益组件库怎么设计？
Register 扩展 + Schema 同构，两个消费者共享架构

benefit SDK 支撑 400 多种权益的创建、编辑、查看和复制。核心设计是 Register 模式——内置 24 个组件，同时允许业务运行时注册自己的权益组件。

它和活动表单共享同一套 Schema 渲染架构，后端返回 form_data 和 content_schema，前端按 d_component 映射组件。这点很关键，意味着两个消费者共享同一套架构，维护成本很低。

打包用 Rollup 输出 ESM、CJS 和类型声明。peerDependencies 避免重复打包 React 和设计系统。发版用 Changesets，alpha 环境联调，yalc 本地验证，正式 release 发布。

项目A4
AI 工程怎么保证可靠性？
不押宝模型聪明不聪明，靠流程约束它

我没有把可靠性押在模型"聪不聪明"上，而是用流程来约束它。第一层是证据链——每个非显然事实都要能追到 PRD、源码、技术文档或人工确认，搜不到也可以作为负向证据，不确定的标 low。

第二层是代码锚定。build-index.py 做静态分析提取代码实体和调用边，context-pack.py 把每个需求匹配到具体的代码位置，按 must、should、optional 分层。

第三层是质量门控。quality-gate.py 有三个子命令，final 做五项加权评分——代码锚点覆盖率占 25%，计划可执行性占 25%。

第四层是反馈回流。蒸馏发现的新知识和矛盾会变成 reference 更新建议。还有 benchmark 系统做回归，用 oracle 用例评分防止质量退步。

  • 审计不是评分：重点是可追溯、可复查
  • 图谱是发现层，reference 是精选知识库，两者互补
  • 不要说已有明确量化提效数据，除非能拿出证据
项目A5
最有技术挑战的项目？
根据面试官兴趣选：AI 工程差异化 或 Schema 架构深度

方案 A——选 prd-tools，适合面试官对 AI 方向感兴趣的时候：

"最有挑战的是变更分类这一步。从 PRD 文本判断代码需要改哪里，核心难点是业务术语和代码字段不一致，比如 PRD 说'增加奖励配置'，可能是新增字段也可能是改选项。我的解法是 reference 路由匹配加代码锚定，每个需求都精准匹配到代码位置。"

方案 B——选 Schema 联动引擎，适合面试官偏传统前端的时候：

"联动引擎跨 BFF 和前端两层，既要保证 SSR 初始状态正确，又要前端运行时流畅。我的解法是 injectDActions 双重求值。"

优先选 A，AI 方向是你最大的差异化。

项目A6
Monorepo 包依赖管理？
Yarn workspace + yarn.lock 锁版本，踩过版本事故

Yarn workspace 做子包间 symlink 引用。yarn.lock 锁定第三方版本——这个必须锁死，因为我们踩过一次解析器升级直接崩掉 node16 的事故。resolutions 用来强制统一 @types/react 这种公共类型依赖。CI 环境加 --no-lockfile 避免锁文件冲突。

项目A7
联动引擎怎么设计？
一次声明，两端求值

联动引擎是横跨 BFF 和前端两层的设计。BFF 端 injectDActions 声明式注入联动配置，然后双重求值：BFF 先 safeEval 算一次保证 SSR 初始状态正确，序列化 d_actions 给前端，前端运行时字段变化再算一次。

联动分两种类型。fetchApi 型——字段依赖后端数据，比如选项需要从服务端取，就走回调 BFF 重新生成。state 型——纯前端条件判断，比如根据某个字段值决定显隐禁用，就本地 eval。

一共 16 个联动模板，12 个重渲染型，4 个 Placeholder 重置型。联动模板从 render 模板提取 handler 复用，保证结构一致。前端 listener.on 支持链式联动，a 变了触发 b，b 变了触发 c。

项目A8
组件库版本管理和发布？
Changesets 管理 + alpha/release 双环境

用 Changesets 管版本流程：先 changeset 记录变更，再 version 更新版本号，最后 publish 发布。双环境策略：alpha 给联调用，release 是正式版。本地联调不用 npm link——它有 symlink 问题——而是用 yalc push。打包用 Rollup 输出 ESM + CJS + 类型声明三份产物。

项目A9
AI 生成代码怎么验证质量？
不是打分，是审计 + 锚定 + 门控 + 回归

这套体系不是给代码打分，而是做四件事。第一，证据链审计——每个需求有 PRD 证据，每个影响有源码证据。第二，代码锚定——build-index 做静态分析，context-pack 做分层锚定。第三，多层质量门控——quality-gate 三个子命令，变更分类必须锚定源码，11 步工作流里有 Report Review Gate 硬停止。第四，benchmark 回归——用 oracle 用例加 snapshot 评分，总分低了说明质量退步。

项目A10
从 PRD 到上线的完整链路？
PRD → reference → 蒸馏 → 锚定 → 门控 → 实施 → 回流

完整链路是这样的：产品经理在飞书写 PRD，然后 reference 先构建知识库——六文件 SSOT 加 Evidence Index 静态分析。接着 prd-distill 做 11 步蒸馏，把 PRD 转成结构化报告。然后 context-pack 做精准锚定，每个需求匹配到具体代码位置。quality-gate 做五项评分，Review Gate 硬停止——人确认了才生成计划。

开发按 plan.md 实施，BFF 配模板和联动，前端渲染。做完之后回流建议更新 reference。核心是：prd-tools 把 PRD 从自然语言变成有证据的结构化开发计划，不是让 AI 直接写代码。

八股C16
React 18 并发 + Fiber
Fiber 把递归变成链表，Scheduler 按优先级调度

之前 React 的虚拟 DOM diff 是递归调用，一旦开始就停不下来。Fiber 把节点组织成 child、sibling、return 这种链表结构，每个 Fiber 节点就是一个可拆分的工作单元，React 可以处理一部分后把控制权交还给浏览器。

Scheduler 用 MessageChannel 做优先级调度——注意不是 setTimeout，因为 setTimeout 最少 4ms 延迟。高优先级任务比如用户输入，可以打断低优先级渲染。

React 18 在这基础上加了三个能力：useTransition 标记低优先级更新、useDeferredValue 延迟渲染、Automatic Batching 自动合并 setState，不管在什么上下文里都只触发一次渲染。

八股C17
useEffect vs useLayoutEffect
useLayoutEffect 绘制前同步阻塞，useEffect 绘制后异步

执行顺序是：DOM 更新完了，先执行 useLayoutEffect，然后浏览器绘制，最后执行 useEffect。所以 useLayoutEffect 会阻塞绘制，useEffect 不会。

实际使用就是：需要读 DOM 布局或者做同步修改防止闪烁的，用 useLayoutEffect。其他所有情况都用 useEffect。

八股C18
浏览器渲染管线
DOM+CSSOM → Render Tree → Layout → Paint → Composite

HTML 解析成 DOM，CSS 解析成 CSSOM，两棵树合并成 Render Tree。然后 Layout 算几何位置，Paint 画像素，Composite 合成图层。

性能影响最大的区分是：改 width、height 这种几何属性会触发 Reflow，最贵，整个 Layout 重新算。改 color 这种视觉属性只触发 Repaint。如果只改 transform 或 opacity，只触发 Composite，性能最好。

八股C19
HTTP 缓存
强缓存不请求，协商缓存可能 304 不下载

强缓存通过 Cache-Control 的 max-age 直接用本地缓存，不发请求。协商缓存发请求但可能返回 304 不下载内容，用 ETag 加 If-None-Match 做精确比对。

实战配置口诀：HTML 用 no-cache 每次都协商，带 hash 的 JS/CSS 设 max-age 一年强缓存，API 用 no-store 不缓存。

八股C20
TypeScript 高级类型
infer 推断 + 条件三元 + 映射遍历

infer 是在条件类型里"推断"一个子类型，比如从函数类型里推断返回值类型。条件类型就是类型层面的三元表达式，T extends X ? A : B。映射类型可以遍历联合类型生成新类型——我们常用的 Pick、Omit、Partial 本质都是映射类型。

八股
Hooks 为什么不能条件调用？
React 靠调用顺序匹配 state，顺序变了状态就错位

React 不是按变量名保存 Hook 状态的，而是按每次 render 的调用顺序。第一次 render 第一个 Hook 对应第一个状态，第二个对应第二个。你把 Hook 放在条件里，条件变了调用顺序就变了，状态全乱了。所以 Hook 必须在组件顶层调用，条件逻辑放到 Hook 内部。

八股
eval 安全隐患怎么处理？
承认历史债，收敛表达能力

eval 确实是历史技术债，我不会把它包装成最佳实践。当时它解决的是快速表达显隐和禁用条件的问题，但安全和可维护性都有边界。实际处理上要限制表达式来源、限制可访问变量。更理想的方向是用受限 DSL 或 JSON 规则描述联动，再由解释器执行。新版本已经用 Zustand 加 new Function 替代 Redux 加 eval。

八股F1
事件循环代码输出分析
微任务先于宏任务，同一轮微任务全清完才执行下一个宏任务

执行顺序：同步代码先跑完，然后清空所有微任务（Promise.then、MutationObserver），最后取一个宏任务执行（setTimeout、setInterval）。每轮循环都是先清微任务再取宏任务。async/await 本质是 Promise 语法糖，await 后面的相当于 .then。

常考陷阱：Promise 构造函数里的代码是同步执行的，.then 才是微任务。

console.log('1'); setTimeout(() => console.log('2'), 0); Promise.resolve().then(() => { console.log('3'); Promise.resolve().then(() => console.log('4')); }); console.log('5'); // 输出：1 5 3 4 2

八股F3
React 19 新特性
Server Components + React Compiler + Form Actions

三个核心变化。一是 Server Components，组件可以在服务端渲染，减少客户端 JS 体积。它和 SSR 不同——RSC 只在服务端运行，不发 JS 到客户端。二是 React Compiler，自动优化重渲染，减少手动 useMemo、useCallback。三是 Form Actions，表单提交直接绑定 server action，简化数据变更。还有 use() Hook 支持 Promise 和 Context 的条件调用。

八股F4
XSS / CSRF / CSP 安全
XSS 注入脚本，CSRF 冒充请求，CSP 限制加载

XSS 是攻击者往页面注入恶意 JS，防护靠输出转义加 CSP 加 HttpOnly Cookie。CSRF 是利用用户已登录状态，诱导浏览器发请求，防护靠 CSRF Token 加 SameSite Cookie 加 Referer 检查。CSP 通过 HTTP 头限制页面能加载哪些资源，从根本上阻止 XSS。Token 放 Cookie 加 HttpOnly 比 LocalStorage 更安全。

八股F5
Webpack vs Vite
Webpack 打包构建，Vite 开发时原生 ESM 按需加载

Webpack 的核心是打包——从入口开始递归解析依赖，把所有模块打成一个或多个 bundle。Vite 开发模式下不做打包，利用浏览器原生 ESM，请求哪个模块才编译哪个，所以启动极快。生产构建 Vite 用 Rollup。HMR 机制也不同：Webpack 是模块级热替换，Vite 精确到修改的模块。Tree Shaking 都依赖 ESM 的静态分析。

八股F6
前端性能优化全链路
网络 → 渲染 → 资源 → 代码，四个维度

四个维度。网络层：CDN、DNS 预解析、HTTP/2 多路复用、Preload 关键资源。渲染层：SSR 或 SSG 首屏、骨架屏、图片懒加载、虚拟列表。资源层：代码分割 dynamic import、Tree Shaking、图片压缩用 WebP、字体子集化。代码层：React.memo、useMemo 防不必要重渲染、Web Worker 处理耗时计算。

核心指标：LCP 小于 2.5 秒、FID 小于 100 毫秒、CLS 小于 0.1。Preload 是当前页面必须的资源，Prefetch 是下个页面可能需要的。

八股F8
HTTP/2 和 HTTP/3
HTTP/2 多路复用+头部压缩，HTTP/3 用 QUIC 解决队头阻塞

HTTP/2 核心改进：多路复用一个 TCP 连接并行多个请求、头部压缩用 HPACK、服务端推送、二进制分帧。但 HTTP/2 还有 TCP 层的队头阻塞——一个包丢了阻塞所有流。

HTTP/3 把 TCP 换成 QUIC 基于 UDP，每个流独立，一个流丢了不影响其他流。还支持连接迁移——网络切换不断连，和 0-RTT 建连。HTTP/2 下雪碧图反而有害，小文件分开更好。

八股
HTTPS/SSL 握手过程
非对称加密协商密钥，对称加密传输数据

四步握手。Client Hello 发随机数加支持的加密套件。Server Hello 发随机数加选定的加密套件加数字证书。客户端验证证书链——Root CA 到 Intermediate 到 Server，用 CA 公钥验证签名。然后生成预主密钥，用服务器公钥加密发过去。双方用三个随机数生成会话密钥，之后用对称加密通信。

为什么这么设计？非对称加密慢但安全，适合协商阶段；对称加密快，适合数据传输。ECDHE 比 RSA 握手多一步密钥交换但支持前向保密——即使私钥泄露，历史通信也不会被解密。TLS 1.3 简化到 1-RTT，合并了步骤。

百度必考：能画出完整握手时序图。

  • TLS 1.2 四步 → TLS 1.3 简化为 1-RTT
  • 证书链验证：Root CA → Intermediate → Server
  • 对称加密用 AES-GCM，非对称用 RSA 或 ECDHE
八股
CSS BFC + Flexbox + Grid 布局
BFC 是独立渲染区域，Flex 一维布局，Grid 二维布局

BFC 就是一个独立的渲染区域，里面的元素不影响外面。触发条件四种：overflow 不为 visible、float 不为 none、position 是 absolute 或 fixed、display 是 inline-block 或 flex 或 grid。常见用途：清除浮动、防止 margin 塌陷、阻止元素被浮动元素覆盖。

Flex 是一维布局，主轴和交叉轴，flex-grow 控制放大、flex-shrink 控制缩小、flex-basis 控制初始大小。justify-content 对齐主轴，align-items 对齐交叉轴。

Grid 是二维布局，grid-template-rows 和 grid-template-columns 定义轨道，grid-area 放置元素。选择原则：单行或单列用 Flex，行列交叉用 Grid。

八股
浏览器事件机制
捕获 → 目标 → 冒泡，委托利用冒泡

DOM 事件流分三个阶段：捕获从 window 到目标元素的父节点，目标阶段在目标元素上，冒泡从目标元素回到 window。addEventListener 第三个参数 true 是捕获阶段监听，false 或不传是冒泡阶段。

事件委托：不给每个子元素绑定事件，而是给父元素绑定，通过 e.target 判断是哪个子元素触发的。好处：减少内存消耗、动态添加的子元素自动有事件处理。e.stopPropagation 阻止传播，e.preventDefault 阻止默认行为。

  • e.target vs e.currentTarget：target 是触发元素，currentTarget 是绑定元素
  • React 合成事件：React 17 绑定到 root 而非 document
八股
闭包和作用域链
函数记住定义时的词法环境，即使在外部执行

闭包就是函数能访问它定义时的作用域变量，即使这个函数在外部执行。作用域链是：当前作用域 → 外层函数作用域 → 一直到全局作用域，查找变量时沿着链往外找。

经典场景：for 循环里 var 的坑——用 let 或 IIFE 解决；函数工厂——返回定制化函数；柯里化——分步传参；私有变量——通过闭包模拟私有属性。

内存方面：闭包引用的变量不会被回收，所以要注意内存泄漏。常见泄漏模式是事件监听器引用了外部变量但没清理，或者定时器引用了大对象。

八股
原型链与继承
对象通过 __proto__ 链向上查找属性

每个对象都有 __proto__ 指向它的构造函数的 prototype。属性查找时先在自身找，找不到沿 __proto__ 往上找，直到 null。instanceof 就是沿原型链检查右边 prototype 是否在左边链上。

ES6 class 本质是语法糖，底层还是原型链。class A extends B 会设置 A.prototype.__proto__ 等于 B.prototype，同时 A.__proto__ 等于 B。

继承模式演进：原型链继承（共享引用类型属性有坑）→ 构造函数继承（不能继承原型方法）→ 组合继承（调用两次父构造函数）→ 寄生组合继承（最优方案，ES6 class 底层实现）。

八股
Promise 静态方法四件套
all 全成功才成功，race 取最快，allSettled 全完成，any 有一个成功

Promise.all：所有都 resolve 才 resolve，一个 reject 就 reject。适合多个请求都成功才继续的场景。

Promise.race：取最先完成的，不管成功失败。适合超时控制——race 里面同时放 fetch 和 timeout，哪个先完成用哪个。

Promise.allSettled：等所有都完成不管成功失败，返回每个的状态和值。适合并行操作但需要逐个处理结果。

Promise.any：有一个成功就成功，全部失败才失败。适合多源尝试，哪个先成功用哪个。

手写 Promise.all 核心：计数器，每个 resolve 时加一，全到齐了才 resolve 外层。

八股
虚拟 DOM Diff 算法
同级比较 + 三策略 + key 优化列表

React 的 diff 建立在三个策略上。第一，只比较同层级节点，不跨层级比较。第二，不同类型的节点直接替换，不尝试复用。第三，通过 key 标识哪些子节点可以复用。

具体比较过程：先比类型，类型不同直接替换整棵子树；类型相同比属性，属性变了更新属性；子节点列表比较时，没有 key 就按顺序逐个比，有 key 就按 key 匹配移动。

key 用 index 的问题：插入和删除会导致后面的节点全部更新而不是移动，性能浪费。Fiber 架构下 diff 分两阶段：render 阶段找差异可以中断，commit 阶段应用到 DOM 不可中断。

八股
Generator / Iterator
Generator 是可以暂停的函数，Iterator 是统一的遍历协议

Generator 函数用 function* 声明，yield 暂停执行，next() 恢复执行。每次 next() 返回 { value, done }。

Iterator 协议：对象有 next() 方法返回 { value, done }。Iterable 协议：对象有 Symbol.iterator 方法返回 Iterator。for...of 遍历的就是 Iterable 对象。

实际用途：用 Generator 实现异步流程控制（在 async/await 之前的时代）；用 Generator 惰性生成无限序列，用一次算一次；Redux Saga 用 Generator 管理 side effects，让异步逻辑可测试。

八股
Map/Set/WeakMap/WeakSet
Map 任意类型键，Set 去重，WeakMap 弱引用不阻止 GC

Map 是键值对集合，键可以是任意类型——Object 的键只能是字符串或 Symbol。Set 是值不重复的集合，常用于数组去重：Array.from(new Set(arr))。

WeakMap 的键必须是对象，且是弱引用——如果没有其他引用指向这个键对象，GC 可以回收它，对应的条目自动消失。WeakSet 同理。

用途：Map 适合存储对象到值的映射，比如缓存计算结果；Set 适合去重和判断存在性；WeakMap 适合给 DOM 元素关联数据或存私有属性，不会造成内存泄漏。性能方面：Map 和 Set 的查找是 O(1)，比 Array 的 indexOf 是 O(n) 快很多。

追问
Schema 架构有什么边界和风险？
表达力有上限，复杂联动和特殊 UI 仍需定制开发

边界主要在三个地方。第一，表达力上限——当联动逻辑涉及多字段复杂计算、条件嵌套超过三层时，模板表达式变得很难维护。我们的做法是超过三层就拆成多个联动模板，每个只管一层。

第二，调试困难——Schema 是动态生成的，出了问题不能直接断点。我们加了一份 Schema diff 工具，生成前后的差异一目了然。

第三，类型安全弱化——Schema 字段都是字符串键值，编译时无法检查。折中方案是在 BFF 模板层做字段枚举校验，保证生成出来的字段名在预设范围内。

这三点都是如果重新设计会改进的方向，我也不会回避这些不足。

面试官问"有什么不足"时，主动说这三个点，比被追问更加分。

追问
大表单 200+ 字段性能怎么优化？
分步渲染 + 字段级更新 + 虚拟滚动，三层递进

大表单性能优化分三层。第一层渲染层——200 多个字段不一次性渲染，按步骤分组，当前步骤全渲染，前后步骤只渲染摘要。

第二层更新层——不做整表 setState，而是字段级更新。我们的 Listener 发布订阅按依赖图触发，只有受影响的组件重渲染。

第三层组件层——长列表字段用虚拟滚动，只渲染可视区域的组件。

实测 200 字段表单，首屏渲染从 1.2 秒降到 300 毫秒。

追问
BFF 挂了怎么办？降级策略？
缓存兜底 + 降级 UI + 告警联动，三步走

降级分三步。第一步缓存兜底——每次成功生成的 Schema 会缓存一份到 CDN，BFF 超时或错误时前端用缓存的 Schema 渲染，字段可能不是最新的但页面不会白屏。

第二步降级 UI——如果连缓存都没有，展示基础骨架页加提示信息，不让用户看到错误页。

第三步告警联动——BFF 错误率超过阈值自动触发告警，值班同学介入。

这套机制是在一次 BFF 服务抖动后加的，那次影响了 30 分钟，之后再也没有因为 BFF 问题导致用户看到白屏。

追问
injectDActions 有没有竞态问题？
有，用请求序列号 + 废弃标记解决

竞态问题确实有。典型场景：用户快速切换字段 A 的值，触发两次 fetchApi 联动，第一次请求慢于第二次，结果旧数据覆盖了新数据。

我的解法是给每次联动请求加一个递增序列号，响应回来时检查是不是当前最新的请求，不是就丢弃。state 型联动不存在这个问题因为它是同步的。

另外，快速连续输入时用 debounce 合并，不会每个字符都触发联动。

主动承认有竞态，然后说解法，比等面试官追问更显得经验扎实。

追问
线上出过事故吗？怎么处理的？
yarn 解析器升级导致构建全挂，15 分钟回滚恢复

印象最深的是 yarn 解析器升级事故。有个子包的依赖用了 @yarnpkg/resolver，从 rc.31 自动升到了 rc.48，直接不兼容 Node 16，整个 Monorepo 构建全挂，线上发版阻断。

处理过程：先回滚 yarn.lock 恢复服务，花了 15 分钟。然后定位到是 resolver 版本问题，锁死版本加 CI 校验防止再次漂移。

复盘后做了三件事：第一，yarn.lock 加入 git 保护不允许随便删；第二，CI 加了 lockfile diff 检测；第三，依赖升级流程规范化，必须先在 staging 验证。

事故处理 = 快速止血 + 根因分析 + 流程改进，这个三段式结构面试官很认可。

追问
代码审查标准是什么？
类型安全 + 边界条件 + 性能影响 + 架构一致性

我审查代码主要看四点。第一，类型安全——TypeScript strict 模式下有没有 any 逃逸，泛型约束够不够紧。

第二，边界条件——空值、undefined、数组越界、并发请求这些有没有处理。

第三，性能影响——这个改动会不会引起大范围重渲染，有没有不必要的计算。

第四，架构一致性——改动是不是在正确的层级，BFF 的逻辑不该泄漏到前端，前端的逻辑不该泄漏到 BFF。

快的 10 分钟过，涉及 Schema 联动或跨包改动的会拉评审会。

追问
prd-tools 提效多少？怎么量化？
不说虚假数字，说可衡量的中间指标

我不会给一个虚高的百分比，因为我没有在团队推广后的完整对比数据。但我可以说的中间指标是：蒸馏步骤从人工 2-3 小时缩短到自动化 5 分钟；变更分类的代码锚点覆盖率从 0 做到了 80% 以上；质量门控的五项评分能在计划阶段就拦截低质量输出。

追问
LLM 幻觉具体遇到过吗？举实例
三种典型：编造字段名、引用不存在的文件、混淆相似概念

遇到过三种典型幻觉。第一种是编造字段名——PRD 说"增加奖励配置"，模型直接编了一个 reward_config 字段，但实际代码里叫 benefit_options。解法是 context-pack 锚定，模型必须先匹配到代码字段才能输出。

第二种是引用不存在的文件——模型会编一个看起来合理的路径，比如 src/components/RewardForm.tsx，实际不存在。解法是 Evidence Index，每个引用都要验证文件存在。

第三种是混淆相似概念——把"活动模板"和"权益模板"搞混。解法是 reference 知识库的 domain 文件做明确的领域定义。

追问
如果要换 LLM 模型要改多少？
架构改动小，主要工作在 Prompt 模板调优

架构层面改动不大，因为我们做了 Provider 解耦。核心接口就是 chat 和 embed 两个方法，加一个 token 计数。换模型只需要实现新的 Provider，改配置文件。

但实际工作量主要在 Prompt 模板调优——不同模型的指令跟随能力不同，Prompt 需要微调。蒸馏的 11 步里，大概有 4-5 个 Prompt 需要根据新模型的特性调整。

context-pack 和 quality-gate 这些工程链路不受影响。Benchmark 系统的价值就在这里——换模型后跑一遍 oracle 用例就知道质量有没有退步。

追问
Reference 构建要多久？维护成本？
初始 1-2 天，增量维护每次 15-30 分钟

初始构建分两部分。build-index.py 自动化部分跑一次大概 5 分钟，产出代码实体清单和调用关系。人工部分是六文件 SSOT 的编写和校验，中小型项目大概一天，大型项目两到三天。

增量维护方面，每次需求变更后需要更新的是 routing-playbooks 和 contracts 两个文件，大概 15 到 30 分钟。健康检查自动跑，有冲突会告警。

总体来说 Reference 就像技术文档一样需要持续维护，但它比文档精确得多——每个字段都有源码证据支撑。我把它定位成"编译时可检查的文档"。

追问
React 18 并发模式踩过什么坑？
Suspense 边界设计 + useTransition 饥饿 + Automatic Batching 适配

三个坑。第一个是 Suspense 边界设计——Suspense fallback 会替换整个子树，边界放得太大 loading 闪烁严重，放得太小会有很多个 loading 区域。经验是按用户感知的"内容块"来划分边界。

第二个是 useTransition 标记的更新在低优先级队列里可能饥饿——高优先级更新持续进来，低优先级一直被中断。我们的解法是给 transition 加超时兜底，超过 3 秒强制执行。

第三个是 Automatic Batching 在 React 18 之前的事件处理器外不生效——setTimeout 里多次 setState 会触发多次渲染。升级到 18 后统一了，但依赖旧行为的代码需要适配。

追问
TypeScript 类型体操在项目中怎么用？
Schema 类型推导 + 泛型组件 Props + 联合类型分发，不是为了炫技

三个实战场景。第一个是 Schema 类型推导——我们定义了 SchemaNode 的递归类型，根据 component 字段的值用条件类型加 infer 自动推导该节点允许的 props 类型。

第二个是泛型组件 Props——benefit SDK 里组件注册表的类型定义，泛型参数 T extends BenefitType，注册时自动推导出入参和返回类型。

第三个是联合类型分发——活动类型的 Schema 联合类型，用 discriminated union 加类型守卫，switch-case 每个分支自动窄化类型。

不是为了炫技，是为了让 IDE 在编译时帮我们检查 Schema 结构的正确性，减少运行时的字段名拼错问题。
