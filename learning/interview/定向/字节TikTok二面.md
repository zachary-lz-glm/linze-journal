## 1. 一面 3 大失分点 — 改进版必背

### 失分点 ① 业务指标卡壳(Q7)— 必复踩
**一面表现**:被问业务方关注什么指标,先说"业务方数"停顿 21 秒,最后讲的是司机侧运营指标,面试官提醒"你做的是中后台吗"。

**改进版 30 秒口播**:
> 我做的是司机端营销中台,业务方是 B 端运营,他们关心的核心指标有三个层次——
> **拉新侧**:首单转化率(注册到首单,行业基准 ~30%)、单司机获取成本 CAC、活动期间注册增量。
> **留存侧**:7 日/30 日留存率(行业 30 日 ~30%,Uber 1 年留存只有 4%,留存是核心痛点)、完单率、调度时长。
> **活动 ROI**:投入产出比、补贴率、活动参与率。
> 我做的 Schema 中台让运营一周能上 3 个活动(原先两周 1 个),**拉新策略迭代频率提升 6 倍**——直接挂钩到 CAC 和首单转化率。

### 失分点 ② "更大平台"动机被打脸(Q8)
**一面表现**:说"想去更大平台验证通用性",面试官直接说"字节别的组也在做,可能比你还好"。

**改进版**:
> 离开滴滴不是因为"更大平台"——我在滴滴的 AI 工作流已经在 20+ 人团队落地,ROI 已经验证。真正吸引我的是**你们 99%/1% 这个数字背后的 AI 业务密度**。在滴滴 AI 是"工程化探索",在你们这是"业务生命线"——这种场景下 AI 工程的迭代频率、数据反馈闭环是完全不同的。我做的"存量项目领域知识前置"差异化能力,刚好匹配你们大量历史客服代码的场景。

### 失分点 ③ lite 版产品思维缺失(Q6)
**一面表现**:被问小需求是不是太复杂,答"渐进式披露"(技术答案),面试官喂"lite 版"。

**改进版**:
> 是的,已经设计了分层——按**改动半径**判断走哪一档:
> - **完整版**(跨团队/接口变更):六步全跑,10-15 分钟。
> - **lite 版**(单模块/UI 调整):跳过 Reference 构建和 SPEC,直接 PRD→Plan + 自评门控,1-2 分钟。
> - **不走流程**(文案/配置):直接做。
>
> 判断标准是**改动半径 × 影响面**——这是我作为工具设计者一开始就该主动思考的,不是等用户反馈。这是我从工程师到产品型工程师的关键转变。

## 2. TT 客服业务全景 — 必背硬通货

> 一面完全没准备这块,二面如果问"你怎么看我们业务"必须能脱口而出。

### 2.1 客服业务的 AI 能力分层

| 层级 | 能力 | 技术栈 |
|---|---|---|
| L1 | **智能回复 / FAQ** | RAG + 知识库匹配,99% 流量在这 |
| L2 | **工单分类 + 路由** | 意图识别、优先级排序(愤怒用户优先) |
| L3 | **多轮对话 / Agent** | Slot Filling、Tool Calling(查订单、改地址) |
| L4 | **辅助人工** | 1% 转人工时给坐席推答案、推 SOP |
| L5 | **质量监控** | CSAT 预测、坐席质量评估、对话审计 |

### 2.2 客服核心 KPI(背下来)

- **效率**:
  - **FCR**(First Contact Resolution,一次性解决率)— 行业基准 70-75%
  - **AHT**(Average Handling Time,平均处理时间)
  - **TTFR**(Time To First Response,首响时间)
  - **转人工率**(目标 ≤5%,你们已做到 1%,世界级水平)
- **质量**:
  - 智能回复**准确率** / **解决率**(两个不同指标:答对了 vs 问题真解决了)
  - **重复进线率**(24h/48h 内用户又来 — 真正的质量信号)
- **体验**:
  - **CSAT**(Customer Satisfaction,满意度)
  - **NPS**(Net Promoter Score)
- **业务**:
  - AI 解决率节省的**人力成本**
  - **AI 工具渗透率**(坐席用 AI 辅助的比例)

### 2.3 字节 TikTok 海外客服的特殊挑战

- **多语言**:全球 150+ 国家,小语种 NLP 模型怎么覆盖
- **时区**:7×24 不能停,流量峰值跨时区
- **文化差异**:中东、东南亚、欧美用户行为完全不同,模型不能一刀切
- **合规**:GDPR、数据本地化(欧盟、印度要求数据不出境)
- **内容安全**:政治、宗教、暴力、儿童保护 — 多层审核
- **业务复杂度**:TT 业务横跨直播、电商、广告、创作 — 客服场景天然多

## 3. 二面高概率问题 + 答案

### Q1: 一面问业务指标你卡壳了,我再来问 — 你做的东西怎么挂钩业务?

**核心句**:Schema 中台直接提速运营迭代频率 6 倍,这是**业务敏捷性**的核心杠杆。

**展开**:
- 原:2 周 1 个活动 → 现:1 周 3 个活动,**频率 6 倍**。
- 拉新侧:运营能更快试不同补贴策略,AB 测试周期从月级到周级,**首单转化率提升的边际收益**直接挂钩。
- 留存侧:7 日留存实验更密集,运营能在一周内验证 5 个召回策略,而不是 1 个。
- **结论**:我做的不是"提效工具",是给业务方一架"高频试错机器"。

### Q2: 你这套 AI 工作流,我们客服场景能用吗?

**核心句**:能,而且客服场景比我原来的更适合——客服天然就是"大量历史代码 + 大量领域知识 + 高频需求变更"。

**展开**:
- **领域知识前置直接复用**:客服有大量历史文档(SOP、知识库、坐席培训材料),正好适合蒸馏成 Reference 六文件。
- **PRD 蒸馏换个目标**:客服的"PRD"是工单处理流程,从工单→解决方案的工作流可以做。
- **真正的差异化应用**:客服 99% AI 背后是大量 Agent 决策日志,这些日志可以反向喂回工作流,形成**自进化闭环**——这是我原来场景没有的数据飞轮。

### Q3: 为什么是 6 个 Reference 文件,不是 5 个、不是 10 个?

**核心句**:不是拍脑袋——按"AI 蒸馏 PRD 时实际需要查的维度"反推。

**展开**:
- 代码结构 → 回答"在哪改"
- 编码规则 → 回答"怎么改"
- 跨层契约 → 回答"前后端怎么对接"
- PRD 路由 → 回答"在哪个模块"
- 业务术语 → 回答"业务概念是啥"
- 模块边界 → 回答"不会冲突到哪"

少一个会有信息盲区(SPEC 出来不全),多一个会冗余(浪费 token + 噪声)。这是**从实际查询需求反推出来的最小完备集**。

### Q4: verified_by file:line 怎么防幻觉?具体讲技术实现

**核心句**:每个 IR(中间表示)产出时强制要求挂一个源码定位,没挂上就门禁拦下来重跑。

**展开**:
- 蒸馏 PRD 时,每抽出一个需求点,LLM 必须返回 `{ "claim": "...", "verified_by": "src/foo.ts:42-58", "source_text": "..." }`。
- **门禁脚本**(确定性,不是 LLM 判 LLM)检查三件事:
  1. `verified_by` 路径存不存在(防止 LLM 编造文件)
  2. `source_text` 是否真的在该文件该行(grep 校验)
  3. 每个 SPEC 条目是否都有至少一个 IR 支撑(无根之据的 SPEC 拒绝)
- 这套机制让幻觉率从早期 ~30% 降到 <5%。

### Q5: 双轨(团队版+原型 prd2code-gen)的取舍 — 为什么不直接推端到端?

**核心句**:不是技术不能,是**风险/收益匹配**的问题。

**展开**:
- 团队推广版聚焦 PRD→Plan,**ROI 最高风险最低**——只解决"PM→研发"信息损耗,生成的产物人审后再执行。
- 端到端原型 prd2code-gen 是**下一代探索**——加了双路径生成、自验证自修复、eval 反哺。技术验证可行,但生产级落地要解决两个问题:① 代码生成的安全边界(不能让 AI 直接改支付逻辑);② 评测集建立(几万行项目没有 ground truth)。
- **节奏判断**:先用团队版把领域知识、工作流文化建起来(1 年),再逐步放开端到端(2-3 年)。这是工程化节奏而不是技术能不能的问题。

### Q6: 80% 准确率怎么算的?评测集多大?

**核心句**:两个维度——**准确率**和**覆盖率**,评测集 50 个真实 PRD,双盲人评。

**展开**:
- **准确率**:SPEC/Plan 里每个条目是否准确反映 PRD 原意(人评 5 分制,4 分以上算准确)。
- **覆盖率**:PRD 里的需求点是否都被 SPEC 抽到(防漏)。
- 50 个 PRD 覆盖 4 个业务线、3 种复杂度等级,**统计上够用但不算大**——这是后续要补的:跑半年后建一个 200+ 的评测集。
- 失分案例集中在**多义需求**和**跨业务线依赖**两类——已经在 Reference v2 里加这两类的专项蒸馏。

### Q7: 给你一个客服工单处理场景,设计 AI 工作流,你会怎么做?

**核心句**:核心结构和我做的 PRD 工作流同构——**前置领域知识 + 输入蒸馏 + 多层门控**——但多两个关键差异:**实时知识库**和**Tool Calling**。

**展开(系统设计题答法,先讲框架再讲 tradeoff)**:

```
[工单输入]
   ↓
[1. 意图分类] ← Router 小模型
   ↓
[2. RAG 检索] ← 实时知识库(SOP + 历史工单)
   ↓
[3. Agent 决策] ← 大模型 + Tool Calling
   ├─ Tool: 查订单系统
   ├─ Tool: 改地址/退款(有边界)
   └─ Tool: 转人工(置信度低时)
   ↓
[4. 答案生成] ← 带 verified_by 溯源
   ↓
[5. 自评门控] ← 5 维自评,不过关转人工
   ↓
[6. 答复用户 + 写回日志]
```

**关键 tradeoff**:
- **实时 vs 离线**:知识库必须实时(政策、活动常变),但意图分类可以离线训练。
- **Tool 边界**:查不改的 Tool 全放开,改的 Tool 必须人审或限定额度。
- **置信度阈值**:宁可多转人工也不要答错——客服答错一次的损失是几十次正确答对的收益。

### Q8: 字节内部 AI 工程化已经有别的组在做,你来了能补什么?

**核心句**:不打算"取代"已有方案,我观察到一个**空白点**:存量业务代码的 AI 上下文构建。

**展开**:
- 市面 SDD 解决 0→1(新项目从 spec 到 implement)。
- 字节内部方案大多服务新项目或单一场景(如 Code Review)。
- **空白**:大量存量业务代码(客服、推荐、风控)的 AI 理解问题没系统解。我做的"Reference 六文件 + verified_by 溯源"专门解决这个。
- **具体怎么补**:不另起炉灶,作为现有体系的"前置上下文层"——你们现有的 SDD/Agent 工具直接挂我的 Reference 输出,效果提升立竿见影。

### Q9: 离开滴滴真实原因 + 为什么选字节 TT 客服

**核心句(3 句话答完)**:
1. 滴滴架构跑通进入稳定迭代期,成长边际递减。
2. 我想找"AI 是业务生命线"的场景,不是"AI 是工程化探索"的场景。
3. 你们 99%/1% 这个数字 + 大量历史代码 + 海外多语言复杂度——是我想找的密度最高的 AI 工程化训练场。

**避雷**:
- ❌ 不说"更大平台"(被打脸过)。
- ❌ 不说"做更多 AI 的事情"(别的组在做)。
- ✅ 锚定**具体业务场景**(99%/1%)+ **自己的差异化**(存量项目 AI 理解)。

### Q10: 最大挑战/最大失败/和 PM 冲突 — 准备一个故事

**讲一个真故事**:Schema 重构时和业务方冲突——业务方要"快速上线新功能",我们要"重构中台",双方节奏打架。

**结构(STAR)**:
- **S**:业务方一周要上 3 个新活动类型,中台重构阻塞了 2 个。
- **T**:既要保业务上线,又要保重构进度。
- **A**:① 把重构拆成"兼容层 + 渐进迁移"——业务先用旧通道,新功能强制走新通道;② 跟业务方做了一次 30 分钟的演示,展示新通道 1 个活动只要 3 天(旧要 2 周),拿到业务方主动推新的承诺。
- **R**:重构周期内业务 0 阻塞,迁移完成后运营主动要求全部转新通道。
- **复盘**:技术决策的最大敌人不是技术,是节奏错配——这个故事让我学到"任何重构都要兼容旧路径,直到新路径的吸引力超过迁移成本"。

## 5. 表达修正(基于一面诊断)

### 口头禅黑名单
一面高频出现:「等于」(7+ 次)、「就是」、「对吧」、「然后」、「以及」、「我想一下」。

**对策**:回答前停 1 秒再开口,用结构化语言替代——「第一/第二/第三」「核心是 X,展开是 Y」。

### 回答长度
一面每题 1-2 分钟,信息密度低。

**对策**:
- **核心句**先甩(15 秒以内)。
- **展开** 30-60 秒,2-3 个点。
- **收口** 15 秒,带一个数字或 tradeoff。

总时长目标:60-90 秒/题。

### 开头不绕
一面有「我想一下它整体的一个背景啊」「你比如说」「业务方数」这种绕。

**对策**:任何问题,先用一句话定调——「核心是 X」「这题的关键是 Y」——再展开。

## 6. 临场节奏控制预案

| 场景 | 应对 |
|---|---|
| 问到不会的 | 不硬编。**先复述问题**(展示在思考),**讲思考路径**(「这块我没直接经验,但我会这样切入...」),**讲类比**(「我做过类似的 X,原理相通」)。 |
| 被反复追问 tradeoff | 这是好事,说明对方在考深度。答 tradeoff 三段式:**有什么选什么(列举)→ 我选什么(决策)→ 代价是什么(放弃的部分)**。 |
| 被打断 | 不要坚持讲完,跟着面试官新方向走。面试官打断 = 你已经答到他的点了。 |
| 面试官挑战你的方案 | 不要软处理(一面 Q8 「OK,了解」就是软处理)。**正面接招**:「您说的对——这块我当时的判断是 X,但您提的 Y 确实是更大的考量,如果重来我会怎么调整...」 |
| 时间快到反问环节 | 必须 3 个反问,质量高。**不要问薪资**(HR 环节问)、**不要问加班**(负面信号)、**不要问"入职后做什么"**(太空)。 |

## 8. 整体作战策略

1. **开场 90 秒自我介绍** — 直接锚定两个核心标签:**Schema 中台架构师** + **AI 工作流设计者**。不绕「我是谁我在哪」。
2. **项目讲不超过 3 分钟** — 留追问空间,把深度留给对方问。
3. **业务指标脱口而出** — 一面失分点必须二面补回来。
4. **系统设计题** — 先讲框架再讲 tradeoff,留 30 秒收口。
5. **动机题** — 锚定具体场景(99%/1%)+ 自己差异化,不空谈。
6. **反问** — 3 个高质量,业务深度 + 技术栈 + 节奏文化。

**心态**:一面 35% 已经过了,二面是 leader 在判断"你能不能来做事 + 你的潜力"。**不是考你完美,是考你深度** —— 答不全没事,答得浅才是真失分。

## 9. 一面面试官信号复盘(必看)

> 一面面试官最后那段话其实是**给二面的提示**:

|---|---|---|
