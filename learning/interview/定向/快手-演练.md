## 二、5 轮演练完整记录

### 🎬 第 1 轮：SDD 主项目 + 数据口径

**戳穿点**：5h→2h、80% 准确率怎么算的

**🎯 面试官**：先讲下你简历上 SDD 工作流的整个链路。然后这两个数——5h→2h、80% 准确率——具体怎么算的？

**📱 翻卡路径**：
- 搜 `SDD prd-tools` → 「A10 从 PRD 到上线的完整链路」
- 搜 `准确率` → 「项目关键数据 + 口径」速查卡

**💬 答全文**：

> SDD 全称 Spec-Driven Development，核心是给 PRD 到代码之间补一个可追溯的中间产物 Spec。链路三段——① **build-reference**：扫代码库构建 6 文件知识底座（代码结构/编码规则/跨层契约/PRD 路由/业务领域/实体索引），按 SSOT 原则分离关注点；② **prd-distill**：11 步蒸馏工作流把 PRD 转 Spec；③ **prd2code-gen**：端到端原型，6 步闭环（蒸馏→分析→Plan→Handlebars/LLM 双路径生成→4 类校验自修复→eval 反哺）。
>
> **数据口径必须讲死**——5h→2h 是 **PRD→Plan 单环节**对照，不是全流程；对照组是**同一 PRD 用纯手写** vs **Skill 辅助**；样本是 **5 个需求**。文件级准确率 80% 的分母是**人工标注的"应该改的文件清单"**作为 ground truth，分子是 Skill 输出和 ground truth 的交集。
>
> 选 Claude Code 不选 Cursor/Copilot 三原因：**Skill 机制**能把 11 步流水线固化进 SKILL.md；**MCP 协议**能挂我自定义的 PRD 抓取工具；**CLI 形态**能进 CI/CD 跑自动化。

**🔍 面试官追问**：80% 准确率，剩下 20% 错在哪？有规律吗？

**📱 再翻卡**：搜 `证据链 质量门控` → 「A4 AI 工程怎么保证可靠性？」

**💬 再答**：有规律。错的 20% 集中在两类——① **跨层契约误判**（如 BFF 改了但漏了前端字段），靠"跨层契约" reference + 负向搜索（negative_code_search）；② **业务领域知识缺失**（如印尼时区规则），靠"业务领域" reference + PRD 入口三级门禁。我们做了 8 种证据类型 + 三级门禁就是收敛这 20%。

**⭐ 评判**：**通过 + 加分**（口径背得硬、错的规律讲得出）。**扣分点**：R01/R02 的 54→68 实验对照（端到端原型）没主动甩——明天要主动提。

### 🎬 第 2 轮：Schema 中台 + 双重求值（快手二面真题复现）

**戳穿点**：循环依赖防御 / safeEval 安全 / 性能

**🎯 面试官**：你简历说"自研 DSL injectDActions，双重求值联动引擎"——为什么要双重，BFF 端 safeEval 怎么实现安全，循环依赖怎么防？

**📱 翻卡路径**：
- 搜 `联动引擎 injectDActions` → 「A7 联动引擎怎么设计？」
- 搜 `循环依赖` → 追问卡「联动会不会循环依赖导致死循环？」

**💬 答全文**：

> 为什么要双重——一次声明两次求值：**BFF 端 safeEval 算初始状态**（保证 SSR 首屏正确，不会"先空再跳"闪烁），同时**序列化 d_actions 给前端运行时**（用户交互时联动）。一次声明避免前后端逻辑分裂。
>
> safeEval 安全三件套：① **VM2 沙箱**（Node 隔离执行环境）；② **白名单全局对象**（只暴露 lodash.get/set、dayjs 这类纯函数）；③ **超时熔断**（500ms 强制 kill）。
>
> 循环防御三层：① **声明阶段** 配置编辑器循环检测；② **BFF safeEval 阶段** 拓扑排序 + max depth=50 拦截；③ **前端运行时** visited 集合，同字段重复触发直接 break。

**🔍 面试官追问**：拓扑排序时间复杂度？safeEval 被穿透过吗？

**📱 再翻卡**：搜 `线上事故` → 追问卡「线上出过事故吗？」

**💬 再答**：

> O(V+E)，最大表单 200+ 字段实测 <5ms。沙箱穿透——VM2 历史上有过 CVE，我们锁了补丁版本 + 加资源限制（CPU/内存）+ fail-close 兜底。
>
> **真实事故**：配置人员写了 A→B→A 联动配置，前端编辑器循环检测没拦住（配置走了批量导入接口绕过 UI 校验），BFF safeEval 在第 47 次递归时栈溢出，触发 Node 内存报警。修复三步：① 立即 BFF 加 max depth 硬拦截；② 根因 批量导入接口加循环检测（和 UI 同规则）；③ 预防 抽 `cycle-detector.ts` 共享模块在 UI/BFF/导入三处都调。

**⭐ 评判**：**强通过**（架构硬度够、防御纵深清楚、事故代表性好）。**扣分点**：VM2 vs isolated-vm 选型权衡没主动给（VM2 维护已停、isolated-vm 更安全但接入复杂）——明天要主动甩。

### 🎬 第 3 轮：AI 边界感 + 实操前置验证

**戳穿点**：Vibe vs 标准 / 模型选型

**🎯 面试官**：简历说"清晰区分 Vibe Coding 与标准化 AI 研发的边界"——你说说这条边界。明天实操环节你打算怎么用 AI？

**📱 翻卡路径**：
- 搜 `Agent 边界 工作流` → 「Agent vs Workflow 区别」
- 搜 `AI Coding 实操` → 「实操 7 步剧本」+「4 类题型 prompt 模板」

**💬 答全文**：

> 这条边界在**决策控制权归谁**。引用 Anthropic "Building effective agents" 定义——**Workflow 是预定义代码路径协调 LLM**，决策权在代码；**Agent 是 LLM 自主控制工具和路径**，决策权在模型。Vibe Coding 是 Agent 端的极端形态。
>
> 我的 SDD 选 Workflow 不选 Agent，因为 **PRD→Spec 是确定性环节**——同一 PRD 跑 10 次必须给同一 Spec。这就是为什么我把一致性从 40% 拉到 85%——控制权收回来了。
>
> 但 prd2code-gen 端到端是混合：**蒸馏/Plan 是 Workflow**（受 11 步约束），**自修复循环是半 Agent**（让模型决策怎么修，但有震荡检测 A→B→A 切整段重写）。
>
> **明天实操 7 步**：复述需求列验收 → 喂上下文（不是"帮我写 X"）→ 分步 prompt（骨架→填肉→测试）→ 边收边审（关键逻辑大声改）→ 跑验证 → 加边界 → 收口讲取舍。**实操考的不是会不会用 AI，是能不能驾驭 AI 当工程用**。

**🔍 面试官追问**：JD 提到"模型选型、效果评测"——你做过吗？怎么选 Claude 不选 GPT？

**📱 再翻卡**：搜 `Benchmark 评测` → 「Benchmark 怎么看」+ 搜 `量化 提效` → 「Agent 评测」+「量化提效」

**💬 再答**：选型三维度——① **任务匹配度**：编码任务用 SWE-bench 跑；② **成本**：每千 token 价格 + cache hit 率（Claude prompt caching 命中后省 90%）；③ **稳定性**：API SLA + 国内可用性。评测我做过 prd2code-gen 的 R01/R02 消融实验——加 reference 文件从 54 拉到 68 分，量化了 reference 对模型表现的影响。

**⭐ 评判**：**强通过 + 加分**（边界感清楚、Anthropic 定义引用准、选型有数据）。**实操环节前置验证已过**。

### 🎬 第 4 轮：落地规模 + 反馈闭环诚实度

**戳穿点**：真在用吗 / 防污染

**🎯 面试官**：你说"20+ 人团队推广"——真在用还是强行说的？怎么衡量？反馈回流闭环怎么设计？

**📱 翻卡路径**：
- 搜 `回流 自进化` → 「自进化闭环」
- 搜 `知识库 防污染` → 「知识库防污染」
- 搜 `20+ 推广` → 「项目关键数据」速查卡

**💬 答全文**：

> 三个指标——① **DAU 15+/20+**（去标识统计 Skill 调用日志）；② **PRD→Spec 链路覆盖率 70%+**（团队产出的 Spec 里 70% 走过 Skill）；③ **feedback 池每周新增 5-10 条**。光看注册数没用，看活跃 + 覆盖率 + 反馈才有意义。
>
> 反馈回流闭环：用户标记 → 收集到 feedback 池 → 周期 review 分类（数据/规则/边界）→ 转 Spec 调整 + 加入 golden sample 回归。
>
> **关键防污染**——不是所有反馈都进 Spec，要经过"反向验证"：先把反馈当 query 跑现有 Spec 看输出对不对，对的话反馈是误报，错的话才考虑入 Spec。否则用户误报会污染知识库。

**🔍 面试官追问**：用户标记的成本高吗？怎么降低？

**📱 再翻卡**：搜 `Evals 评测` → 「Evals 评测」+ 搜 `证据链` → A4

**💬 再答**：成本是关键挑战。三层降本——① **被动收集**：错误日志自动归类（参考 Sentry fingerprint），用户不用主动标记；② **智能采样**：高频错误优先入池，单次错误过滤；③ **LLM 辅助分类**：用模型预分类反馈，人工只做最终决策。用户侧成本接近零，团队侧每周 30 分钟 review 就够。

**⭐ 评判**：**通过 + 加分**（指标设计合理、闭环有防污染思考、降本有具体设计）。**亮点**：主动说"不是所有反馈都入 Spec"显工程判断。

### 🎬 第 5 轮：快手电商场景定制 — 直播间 AI 智能导购 ChatUI

**戳穿点**：架构设计 + 简历空白补救

**🎯 面试官**：快手电商直播间要做一个 AI 智能导购 ChatUI——用户提问"这个口红适合我吗"，AI 要结合商品信息、用户画像、直播实时内容回答。你怎么设计前端架构？

**📱 翻卡路径**：
- 搜 `React Suspense 流式` → 「React Suspense 核心用法」
- 搜 `Event Loop 微任务` → 「事件循环」
- 搜 `应急 兜底` → 「应急话术」（部分映射 + 临场发挥）

**💬 答全文**：

> 这是典型的**流式输出 + 多模态上下文 + 状态机**前端架构题。三层设计：
>
> **① 流式渲染层**：SSE 接 LLM streaming API，前端用 **React Suspense + use hook** 渐进式渲染（参考 React 18 并发特性）。每个 token 到达后增量更新虚拟 DOM，配合 `useTransition` 把高优先级更新（用户提问）和低优先级（流式输出）分开。
>
> **② 状态机层**：用户提问 → 并行加载商品/画像/直播上下文 → LLM 流式生成 → Markdown/LaTeX 渲染。状态用 Redux + 异步切片管理，竞态靠**序列号机制**（旧请求废弃）。
>
> **③ 性能层**：长输出**虚拟滚动**（参考我做大表单 200+ 字段的虚拟列表经验）+ 增量 diff；直播上下文超长时**降级**（只取最近 N 条弹幕 + 商品标签）。
>
> 兜底：直播流断开 / LLM 超时 / 上下文超长，分别走不同的 fallback UI。

**🔍 面试官追问**：你简历没做过 ChatUI 和多模态，这块你怎么补？

**📱 再翻卡**：搜 `应急 兜底` → 「应急话术」卡（演示 5 条避坑话术之"被问没做过的怎么补"）

**💬 再答**：

> 诚实说——"ChatUI 和多模态我确实没做过深，但**底层能力是相通的**：流式渲染（SSE + Suspense）、状态管理（Redux + 异步切片 + 竞态防护）、长列表性能（虚拟滚动）我都做过。我会用 **1-2 周时间做一个 demo**：基于现有 Suspense 经验搭流式渲染骨架，复用虚拟滚动做长输出，多模态部分先用纯文本兜底再迭代。**关键是底层工程能力可迁移，UI 形态是新组合不是新原理**。"

**⭐ 评判**：**通过 + 加分**（架构思路对、诚实承认空白、给具体补救路径）。**真实弱点但答得不崩**——演示了 5 条避坑话术怎么用。

## 四、追穿点专项防御

### 4.1 数据口径防御（必背硬）

| 数据 | 标准口径 | 防御话术 |
|------|---------|---------|
| 5h→2h | PRD→Plan **单环节**对照、5 需求样本、对照组=同 PRD 纯手写 | "这是单环节不是全流程，全链路提效约 20-30%" |
| 80% 准确率 | 分母=人工标注的"应改文件清单"ground truth | "错的 20% 集中在跨层契约和业务领域两类，有规律" |
| 40%→85% 一致性 | 同 PRD 跑 10 次的文件列表重跑匹配率 | "一致性来自控制权收回，从 Agent 退到 Workflow" |
| R01/R02 54→68 | 消融实验，加 reference 文件前后对照 | 主动甩，证明 reference 对模型表现有量化影响 |
| 20+ 人推广 | DAU 15+/20+ + 覆盖率 70%+ + 反馈 5-10/周 | 三指标综合看，不光看注册数 |
| 400+ SKU | 30-40 底层组件 + 配置组合 | 强调"配置组合"是核心，不是硬编码 |
| 400+ 组件库 18 个月零 breaking | Changesets 语义化发版 + peerDependencies 隔离 | 强调"零 breaking"是工程能力 |

### 4.2 AI 原理链路防御

| 概念 | 一句话标准链路 |
|------|-------------|
| Function Calling | 模型按 JSON Schema 调工具 → tool_use/tool_result 配对 → harness 执行 |
| Prompt Caching | 服务端 KV cache 张量（不是本地）、cache write 1.25× / read 0.1×、4 个 cache_control 断点 |
| Agent Loop | ReAct 范式（Thought/Action/Observation）+ stop_reason 决定停（不是 if） |
| Skills 触发 | description 字段语义匹配 + 用户输入 → 自动加载 SKILL.md |
| RAG（我们的） | **倒排 + BM25 + 语义混合 + RRF**（不是纯向量库）—— 这是关键区分点 |
| Subagent | 独立 context window + 工具白名单 + 结果摘要回流主上下文 |

### 4.3 端到端闭环防御

被问"你的工作流和直接调 Claude Code 区别"时标准答法：
> "直接调 Claude Code 是 Vibe Coding（无约束自主）；我的 SDD 是**给 Claude Code 加了三层约束**——① Skill 把 11 步流水线固化；② reference 6 文件做知识底座；③ 8 种证据类型 + 三级门控做质量管控。本质是把 Claude Code 从'工具'升级成'可信的工程系统'。"

## 五、明天实操环节特别提示

### 5.1 实操 7 步剧本速记

```
① 复述需求 + 列验收（30 秒，买时间立判断）
② 喂上下文（不是"帮我写个 X"，是技术栈+参考文件+约定）
③ 分步 prompt（骨架 → 填肉 → 测试，不要一锅炖）
④ 边接受边改（类型/幻觉 API/重复造轮子，关键逻辑大声改）
⑤ 跑起来验证（npm run dev / npm test，跑不通展示调试）
⑥ 补边界/异常（至少 2 个：空输入/超长/并发/错误兜底）
⑦ 收口讲取舍（AI 边界感，高级工程师 vs CRUD 工具人的分野）
```

### 5.2 实操 4 题型 → Prompt 模板速查

| 题型 | Prompt 模板核心约束 |
|------|------------------|
| **A 实现功能** | 技术栈 + 参考文件 + TS 类型完整 + 不引入新依赖 |
| **B 找/修 Bug** | 完整 stderr + 复现步骤 + 期望行为 + 让 AI 先讲根因再讲修复 |
| **C 重构** | 保持外部接口不变 + 提取公共逻辑 + **给 diff 不整体重写** |
| **D 写测试** | **先列用例清单让我确认**再写代码 + 覆盖正常/边界/异常 |

### 5.3 5 条避坑话术索引

| 死法 | 话术 |
|------|------|
| 1 AI 胡编 API | "AI 假设了 X，我们项目是 Y，我手改成 Y 因为..." |
| 2 AI 反复改不对 | "AI 改不动说明 prompt 没说清，我重组需求，关键约束是 X" |
| 3 被问"为什么这么写" | "AI 给了 3 个选项（构造对比），我选 X 因为..." |
| 4 被问"不用 AI 你怎么写" | "不用 AI 我会先 X → Y → Z；AI 加速 X 步，Y/Z 手写更稳" |
| 5 屏幕共享卡顿 | "我慢一点，每步讲一下：现在我打开 X 文件，目的是 Y..." |
