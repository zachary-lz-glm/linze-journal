## 二·补、小程序突击专题（要求项 · 最大风险 · 必突击到能聊）

> JD 要求"精通至少一种小程序（微信/支付宝/抖音）"。你没实战，目标不是装懂，是**突击到能讲清原理 + 体现迁移能力**。

### 微信小程序核心（突击重点，微信最主流）

| 问题 | 核心答案（背到脱口而出） |
|------|---------|
| **双线程模型（最核心考点）** | 渲染层（WebView）+ 逻辑层（JsCore）分离，JS 跑独立线程不阻塞渲染，通信靠 Native 桥（setData 数据序列化传输）。为什么：安全（隔离 DOM）+ 性能（JS 不卡渲染）+ 多端一致。 |
| **setData 性能（高频追问）** | setData 是跨线程通信，数据要序列化。坑：频繁 setData 卡顿 / 传大量数据慢 / 全量传无关字段。优化：只传变化字段、合并调用、列表用 key、长列表用 recycle-view。 |
| **生命周期** | App（onLaunch/onShow/onHide）、Page（onLoad/onShow/onReady/onHide/onUnload）、Component（created/attached/ready/detached）。onLoad 一次、onShow 每次。 |
| **页面路由/栈** | 最多 10 层。navigateTo 入栈、navigateBack 出栈、redirectTo 替换、switchTab 切 tab。注意栈溢出。 |
| **分包加载** | 主包 ≤2MB、总包 ≤20MB。主包放首页，按业务分包懒加载减首屏；支持独立分包 + 分包预下载。 |
| **原生组件层级** | input/video/map/canvas 层级最高，盖住普通组件（同层渲染部分解决）；z-index 对原生无效。 |
| **和 H5 区别** | 无 DOM/BOM（不能 document.xxx）、有自己的组件和 API、有审核、能力受限需权限申请。 |
| **跨端框架** | Taro/uni-app（React/Vue 语法编译到微信/支付宝/抖音多端）——JD 要三端，这是主流方案。 |

### 支付宝/抖音小程序差异（被问到能答）

三端架构同源（都是双线程 + 类微信 API），差别在 API 命名（`wx.` vs `my.` vs `tt.`）和宿主能力。支付宝有 mPaaS 背景、抖音宿主是抖音 App。

### 你的迁移话术（必背，把弱项变成"可评估的 gap"）

> "小程序我没完整做过，但底层能力高度可迁移：
> ① **组件化**——400+ 权益组件库是 Register 注册 + Schema 渲染，小程序自定义组件 Component 也是注册模式；
> ② **跨端**——做过 qiankun 微前端 + 组件库三种接入形态，多端适配是同一类问题；
> ③ **配置驱动**——Schema 中台能配置化生成页面，小程序页面同理；
> ④ **性能优化**——setData 优化和我做 React 重渲染优化（精确订阅/虚拟滚动）是同一套思路。
> 真正的 gap 是小程序 API 生态和双线程细节，我突击了原理，上手成本我能评估。"

### 万一被深追小程序实战（兜底）

- 别编项目。诚实说"没有上线项目"，立刻转到"但我拆解过 XX 小程序的实现"（如大麦购票流程：选座→确认→支付，本质是表单+状态机，和我 Schema 联动引擎同构）。
- 反问回去："团队小程序主要用原生还是 Taro/uni-app？"——把话题转到你能聊的工程化选型。

### 第一梯队（大麦真题，必会）

#### 题 1：千分位逗号

```javascript
// 方法 1：正则（面试官期望）
function format(num) {
  return String(num).replace(/\B(?=(\d{3})+(?!\d))/g, ',');
}
// \B 非单词边界；(?=(\d{3})+(?!\d)) 正向预查：后面是 3 的倍数连续数字且后面不再是数字

// 方法 2：循环（容易讲清楚）
function format(num) {
  let str = String(num);
  let result = '';
  while (str.length > 3) {
    result = ',' + str.slice(-3) + result;
    str = str.slice(0, -3);
  }
  return str + result;
}

console.log(format(1234567));    // "1,234,567"
console.log(format(123));        // "123"
console.log(format(1234));       // "1,234"
```

#### 题 2：提取括号里的内容

```javascript
// 方法 1：正则（面试官期望，秒杀）
function extract(str) {
  const matches = str.match(/\(([^)]*)\)/g);  // 匹配 (内容)
  return matches ? matches.map(s => s.slice(1, -1)) : [];
}
// \( 左括号；([^)]*) 捕获组：任意非右括号字符；\) 右括号

// 方法 2：栈遍历（候选人思路）
function extract(str) {
  const stack = [];
  const result = [];
  for (const ch of str) {
    if (ch === '(') stack.push(ch);
    else if (ch === ')') {
      if (stack.length) {
        const content = stack.slice(1).join('');
        result.push(content);
        stack.length = 0;
      }
    } else if (stack.length) stack.push(ch);
  }
  return result;
}

console.log(extract('a(b)c(d(e)f)g'));  // ['b', 'e']  注意：不匹配嵌套
```

### 7.4 编译原理 + 构建工具深度（JD 明确"深入理解编译原理"）

| 题 | 回答要点 |
|----|---------|
| **编译三阶段** | 词法分析（源码→Token）→ 语法分析（Token→AST）→ 代码生成（AST→目标代码），中间有语义分析 + 优化 |
| **AST 是什么** | 抽象语法树，源码的树状表示。babel / ESLint / Webpack / Prettier / TypeScript 都基于 AST |
| **你的 AST 实践（必讲）** | ① 手写过 markdown 渲染器（token→HTML）② babel 插件改 AST ③ JSON Schema 求值（safeEval）④ Rollup/Webpack 构建配置——都是编译原理应用 |
| **Webpack 原理** | 入口 → 模块依赖图（AST 分析 import/require）→ Loader 链（从右往左）→ Plugin 钩子（Tapable 事件）→ 输出 chunk |
| **Webpack vs Rollup** | Webpack 适合应用（代码分割/CJS），Rollup 适合库（Tree Shaking 干净/ESM 优先）——你组件库选 Rollup 正因如此 |
| **Vite 原理** | 开发用 ESM 原生 import（浏览器按需请求，esbuild 预构建依赖），生产用 Rollup。快在不做全量打包、按需编译 |
| **Tree Shaking** | 基于 ESM 静态分析 + sideEffects 标记，打包时删未引用导出 |
| **Loader 执行顺序** | 从右往左、从下往上——管道式，上一个输出是下一个输入 |
| **babel 流程** | parse（@babel/parser→AST）→ transform（@babel/traverse 遍历+插件改 AST）→ generate（@babel/generator→代码），和编译三阶段一一对应 |

**话术切入**："编译原理我不是科班系统学过，但工程实践一直在用——手写过 markdown parser、做过 babel AST 转换、组件库用 Rollup 打包、配过 Webpack Loader/Plugin。编译三阶段（词法→语法 AST→代码生成）我理解，工程上 AST 是核心，babel/webpack/esbuild 都围绕它。"
