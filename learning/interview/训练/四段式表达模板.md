# 四段式表达模板 · 项目复述训练

> 这是你 14+ 场面试反复暴露的最大问题：**知识够，表达没框架**。
> 千问一面被面试官当面戳穿："聊了这么久，我其实接不到你做了什么"。
> 这个文档是解药——每个项目用"角色-做事-难点-效果"四段式重写。

---

## 零、核心公式

```
[角色]   我在这个项目里是 X 角色，主导/独立负责 Y 模块     ← 10 秒
[做事]   具体做了三件事：①...②...③...（动词开头）          ← 30 秒
[难点]   最大的技术挑战是 X，我通过 A 方案解决（讲权衡）    ← 30 秒
[效果]   业务结果：A 指标从 X 到 Y；技术沉淀：B 复用 C       ← 20 秒
```

**铁律**：
1. **结论先行**——每段第一句就是核心句，不要铺垫背景
2. **三件事上限**——人脑记不住第四件，超过三件就合并
3. **数字必须有口径**——"提升 60%"后面跟"5 个需求对照"
4. **难点讲权衡**——不是吹方案多牛，是讲"为什么这样选"

## 一、项目 1：Schema 驱动营销中台

### 30 秒版（自我介绍用）

> 我在滴滴国际化主导了 Schema 驱动的营销中台架构。**BFF 用模板引擎生成 Schema，前端用渲染引擎解析，联动用 injectDActions 双重求值**。覆盖 20+ 活动类型，新增活动前端零代码，上线周期从 2 周缩到 2-3 天。

### 90 秒版（项目主答，**首选**）

**[角色]** 我是 Schema 中台前端架构的核心开发者，和导师 5-5 开主导，负责前端渲染引擎 + BFF 模板引擎 + 联动 DSL 三层。

**[做事]** 三件事：
1. **接手老代码做问题治理**——刚入职接手的是 if-else 硬编码几万行的版本，月均 10+ 线上事故，我做根因分析归纳 5 类问题
2. **从 0 设计 Schema 三层架构**——BFF 用模板引擎按活动类型分文件（25 个活动 × 独立模板），前端用 formatName 拍平路径 + Listener 发布订阅
3. **自研联动 DSL injectDActions**——一次声明双重求值，BFF 端 safeEval 设初始状态（SSR 正确）+ 序列化给前端运行时联动

**[难点]** 最大挑战是**双端联动**——不仅前端组件间联动，BFF 也要根据上下文（如城市）动态剪裁 Schema。Formily 这种纯前端方案覆盖不了，所以自研了 dependence 数组 + 三元表达式模板的 DSL。

**[效果]** 上线 2 年只有 2-3 起小 bug（老代码月均 10+），新活动前端开发成本归零，上线周期 2 周缩到 2-3 天。这套架构后来被 benefit SDK 复用，两个消费者共享同一套渲染引擎。

### 3 分钟版（深挖追问）

在 90 秒版基础上展开：

**[难点展开]** 双端联动的具体实现——
- DSL 对象两个属性：`dependence` 数组（依赖项路径）+ 模板字符串（三元表达式）
- 前端拿到 JSON 后用 Listener 发布订阅监听依赖项，变化时 eval 模板字符串覆盖组件属性
- BFF 端用 safeEval 在 SSR 时算出初始状态，保证首屏渲染就是对的
- 联动分两种：**state 型**（前端自闭环，如 visible/disable）和 **fetchApi 型**（前端调 BFF 重生成 Schema 整体替换）

**[性能优化]** 大表单 200+ 字段场景：
- 用 dot path 精确订阅，避免全表单重渲染
- 联动接口用 Service Worker 缓存，相同入参直接读缓存
- 动态表格用可编辑单元格形式，避免一次性渲染上千组件

**[踩坑]** injectDActions 竞态问题——用户快速切换时多个 fetchApi 并发，用版本号 + AbortController 解决

## 二、项目 2：benefit SDK 权益组件库

### 30 秒版

> 我主导设计了 400+ 权益的 React 组件库。**Register 注册模式 + Schema 同构渲染 + Rollup 双模打包**，覆盖 8 个业务项目，新权益接入从 2 天缩到 2 小时，零 breaking change 跑了 18 个月。

### 90 秒版

**[角色]** benefit SDK 的架构设计者 + 核心开发，独立完成。

**[做事]** 三件事：
1. **Register 注册模式**——构造时注册 24 个内置组件（16 通用 + 8 业务），暴露 add() / addMultipleCmps() 支持消费方运行时扩展
2. **Schema 同构渲染**——SDK 本身是 Schema 渲染引擎的消费者，后端返回 form_data（含 d_component + content_schema），FieldWrapper 迭代渲染，和 Schema 中台共享同一套架构
3. **Rollup 双模打包**——ESM + CJS + .d.ts 三输出，peerDependencies 隔离 react，Changesets 语义化发版

**[难点]** 最大挑战是**双消费者共享架构的解耦**——Schema 中台和 benefit SDK 都用同一套渲染引擎，但业务组件完全不同。我把引擎层（Schema 解析 + 联动 + 注册机制）抽成独立模块，业务组件各自实现。

**[效果]** 覆盖 8 个业务项目累计 400+ 权益 SKU，新权益接入 2 天→2 小时，零 breaking change 运行 18 个月+。误报率降 83%（错误码四层分层后只报前两层）。

### 3 分钟版

**[接入层三种形态]** 按消费方控制力度区分：
1. 组件模式（`<App>` 内嵌）→ 需要完整 CRUD
2. 函数调用模式（Modal.info 命令式）→ 只需要单点能力，BenefitAdmin 统一分发
3. 微前端模式（qiankun 挂载）→ 不限版本，每次跟最新

**[版本管理]** Changesets 追踪跨包影响，alpha 环境联调后才 release，deprecated 标记不删除保兼容

**[与 Schema 中台的关系]** benefit SDK 是 Schema 渲染引擎的第二个消费者——证明这套架构是通用的，不是某个业务的特化方案

## 三、项目 3：prd-tools AI 工程工作流 ⭐（蚂蚁岗王牌）

### 30 秒版（自我介绍末尾）

> 最近一年我独立设计了一套 AI 工程工作流，**把 PRD 自动蒸馏成有证据、可追溯的结构化开发计划**。靠 SSOT 知识库 + 证据链 + 多层质量门控保证可靠性，PRD 转化效率提升 60%+，已在 20+ 人团队推广。

### 90 秒版（**蚂蚁岗主答首选**）

**[角色]** prd-tools 的独立设计者 + 落地者，从 0 到 1 完成全流程，目前在 B 端 20+ 人团队推广。

**[做事]** 三件事：
1. **Reference 知识底座**——双 Skill 体系，6 文件 SSOT（代码结构/编码规则/跨层契约/PRD 路由/业务领域/代码实体索引）+ 5 条边界规则防知识污染
2. **PRD 蒸馏 11 步工作流**——从 PRD 摄入到 Plan 产出，每步 YAML 中间文件传递状态，三层门控（步骤级前置 + 文件级硬门 + Readiness 评分）
3. **8 种证据类型 + 反馈回流**——每条 AI 结论必须挂 Evidence ID，支持 prd/code/tech_doc/git_diff/negative_code_search 等 8 种来源，跑完回流 Reference 让知识越用越准

**[难点]** 最大挑战是**AI 幻觉和跳步**——直接丢 PRD 给 AI 会编需求、跳步骤、覆盖不全。我用三层防御：证据链让每条结论可追溯 + 步骤级 YAML 校验防跳步 + Readiness Score 五维评分不达标自动返工。同一个 PRD 跑 10 次结果一致性从 40% 提到 85%。

**[效果]** PRD 到开发计划耗时从 5h/需求降到 2h/需求（5 个需求对照），文件级准确率 80%，已在 B 端 20+ 人团队推广，后端也在我的基础上迭代。

### 3 分钟版（**深挖追问用**）

**[为什么 11 步？]** 每步对应一个认知任务：
1. PRD 摄入（图片并行分析）→ 2. Evidence Ledger 建立 → 3. Requirement IR（需求点拆分）→ 4. Code Search & Layer Impact（代码定位）→ 5. Contract Delta（契约差异）→ 6-7. Spec 生成 + 人工 Review Gate → 8. Plan 生成 → 9. Readiness Score 评分 → 10. Reference 回流建议 → 11. Final Quality Gate 确定性检查

**[能力面适配器]** 给 AI 结构化 checklist 而不是大海捞针——前端 9 个能力面（ui_route/view_component/form_or_schema 等）、BFF 10 个、后端 10 个。AI 拿到 PRD 先判断涉及哪个层，只加载对应能力面去分析，路径只是搜索候选，结论必须来自源码确认。

**[Context Engineering]** 三层上下文管理：
1. 信息选择——build-index.py 做倒排索引
2. 预算管理——按 must/should/optional 分层注入
3. 渐进披露——每步只给最关键的信息

**[和阿里淘天的差距]** 阿里已做到端到端（PRD → CI/CD 自动部署 + 云真机验证 + A loop 自修复）。我的还在蒸馏阶段，但护城河是业务领域知识——这是任何模型从外部获取不了的。

**[金融场景适配]**（蚂蚁专属）如果是金融场景，我会在 Spec Review Gate 加一层资损风险评估，关键决策点强制人工审核，AI 不能自主执行有资损风险的操作。

## 五、口头禅替换表（千问血泪）

| 当前口头禅 | 替换为 | 触发场景 |
|------|------|------|
| "然后..." ×50+ | **"第一/第二/第三"** 或 **停顿 1 秒** | 任何展开陈述 |
| "对..." ×30+ | **"是的"+ 直接展开** 或 **删掉** | 面试官说话时回应 |
| "就是..." ×20+ | **"具体来说"** 或 **删掉** | 举例时 |
| "我想想..." ×10+ | **停顿 2 秒不说话** | 思考时 |
| "相当于" | **"本质上是"** 或 **删掉** | 类比时 |
| "等于" | **"也就是"** 或 **删掉** | 总结时 |
| "说白了" | **删掉** | 任何场景 |

**练习方法**：
1. 录音回答 5 个高频问题
2. 回听标出每个口头禅
3. 第二天重录，刻意停顿替换
4. 一周后口头禅能减 70%

## 六、训练计划（一周）

### Day 1-2：项目复述肌肉记忆

- 三个项目 90 秒版各口述 5 遍录音
- 回听标口头禅
- 第三遍刻意停顿替换

### Day 3-4：追问深挖训练

- 每个项目准备 5 个追问点（写在小卡片上）
- 每个追问用 30 秒回答（不超过 1 分钟）
- 录音回听，标"答得散"的题重练

### Day 5：行为面训练

- 5 个行为面问题（最大挑战 / 失误 / 冲突 / 离职 / 规划）四段式录音
- 反问环节模拟（用反问策略库）

### Day 6：完整模拟

- 找朋友/ChatGPT Voice 做一场 45min 完整模拟
- 录音 → 回听 → 标失分点

### Day 7：面试当天

- 只看 `../题库/急救包.md` 的"面试前 5 分钟速览"
- 不要再背新东西

## 七、自我检查清单（每场面试后填）

```
[ ] 结论先行做到了吗？（每段第一句是核心句）
[ ] 三件事上限守住了吗？（没讲第四件）
[ ] 数字带口径了吗？（60% 后面跟"5 个需求对照"）
[ ] 难点讲权衡了吗？（不是吹方案，是讲为什么这样选）
[ ] 口头禅控制住了吗？（"然后"少于 10 次）
[ ] 反问环节问了强相关 + 技术深度吗？（没问宏大问题）
[ ] 算法题先说思路再写代码了吗？
```

每条 1 分，**低于 5 分就要重练**。
