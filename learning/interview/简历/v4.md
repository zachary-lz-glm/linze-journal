## 二、v4 新增 / 强化的叙事

### 1. 三 plan 完整闭环(来源:快手电商一面反馈)

**面试场景**:快手电商一面面试官明确说「单 PRD→Plan 价值不大」,要求展示完整闭环。

**简历落地**:SDD 项目核心技术第二条 + 个人优势第一条。

**真实基础**(基于 prd-to-code README):
- `plan.md` §2.5 需求→文件映射 + 实现计划(精确到行号) → **开发 plan**
- `plan.md` QA 矩阵 → **测试 plan**
- `plan.md` 回滚方案 → **上线 plan**
- `final-quality-gate.yaml` 5 维准出报告 → **准出门禁**
- `team-plan.md` + `plans/*` 多 plan 输出

**追问防御**:
- Q: 三个 plan 是三个独立文件吗?
- A: 不是。是同一份 plan.md 的三段式结构(开发计划 / QA 矩阵 / 回滚方案),加上独立的 final-quality-gate.yaml 准出报告。这样设计是因为三者强耦合——开发计划改了,Q 和回滚都要同步更新,放一份文件保证一致性。

### 2. 消融实验 82→54(来源:团队 SDD 工程分享)

**面试场景**:证明证据链层不是"锦上添花",而是"必需层"。

**简历落地**:SDD 项目末尾 + 个人优势第一条 bullet 末尾。

**真实基础**:去掉证据链层(virtual 评分)后,BFF 单仓 Readiness 从 82 掉到 54,证明证据链是必需层而非可选层。

**与 R01/R02 54→68 别混淆**:
- 82→54 = 去掉证据链层 Readiness 掉档(证据链层价值)
- 54→68 = 有没有 reference 知识库的对照(reference 价值)
- **两个实验,两个层**

### 3. verified_by 证据链(替代原"防幻觉证据链 + 显式负向检索")

**v3 表述**:「file:line 溯源 + 显式负向搜索」
**v4 表述**:「verified_by 证据链 — 每条事实附带 file:line 溯源,源码 Read 后才写入,禁止推断」

**改动原因**:
- prd-to-code README 明确写「每条事实附带 `verified_by: ["file.ts:line"]`,禁止推断」
- 「显式负向检索 anti-retrieval」是 agent 讨论引入的术语,**项目里不存在**,被追问 anti-retrieval 实现会答不上来
- v3 的「显式负向搜索 negative_code_search」是 v3.md 历史描述,实际项目里是 8 种证据类型之一(包括正向 + 负向代码搜索),表述要更准确

### 4. Multi-Agent 副标题修正

**v3 表述**:「Multi-Agent Architecture」
**v4 表述**:「Claude Code cron + subagent」

**改动原因**:
- v3 副标题是抽象架构词,被追问「用了什么 Multi-Agent 框架」会答不上来
- v4 副标题是真实技术栈(Claude Code cron 调度 + subagent 编排),可追问到具体实现

## 三、v4 剔除的"编造术语"清单(重要!)

> v3 → v4 优化过程中,3 个 agent 讨论引入了一批"AI 工程师圈共识词",但项目里根本没做。下列术语已从 v4 全部剔除,**未来简历迭代绝对不能再加回来**:

| 术语 | 来源 | 为什么剔除 |
|------|------|------|
| 显式负向检索(anti-retrieval) | agent 讨论 | prd-to-code 没做 anti-retrieval,只有 negative_code_search 证据类型。表述要严格区分 |
| Prompt 版本化 | agent 建议 | 用户做的是工具版本(v233→v251),不是 LLM Prompt 版本化。被追问「你的 Prompt 版本化怎么做的」会答不上来 |
| Tool Use 含失败重试 | agent 建议 | 项目里没显式做 Tool Use 失败重试机制 |
| 对标 Cursor / Claude Code | agent 建议 | 用户实际就是基于 Claude Code,没有"对标 Cursor"。被追问 Cursor 的 merkle index / aider 的 repo map 会翻车 |
| AI 编码准确率只有 30-40% | agent 建议 | 项目里没这个 baseline 数据。被追问「30-40% 怎么测的」答不上来 |
| 「独立架构师」自称 | v3 原文 | 4 年经验自称架构师会被资深面试官皱眉,改为「独立设计并主导落地」 |

**核心原则**:简历优化时,multi-agent 讨论的输出**只能用作叙事/结构/排序建议**,绝不能直接采纳 agent 建议的新术语,必须先回真实项目(prd-to-code / prd-to-code-gen)的 README/CLAUDE.md 验证。

## 四、真实项目术语白名单(可放心写)

基于 prd-to-code 和 prd-to-code-gen 的 README + CLAUDE.md 核对,**下列术语是项目真实做的,简历可放心写,被追问能答上来**:

### prd-to-code(团队推广版,SDD 工作流)

- `/reference` Skill — 8 种模式(F/A/B/B2/C/E/T/T2)
- `/prd-distill` Skill — Ingestion → Evidence → Requirement IR → Code Search → Contract Delta → Spec → Plan
- 6 文件 SSOT 知识库(project-profile + 01-codebase + 02-coding-rules + 03-contracts + 04-routing-playbooks + 05-domain)
- Evidence Index(entities.json + edges.json + inverted-index.json + manifest.yaml)
- verified_by 审计链(file:line 溯源,禁止推断)
- Spec Review Gate(用户 approved 才能 plan)
- final-quality-gate 5 维评分(required_files / context_pack_consumed / code_anchor_coverage / plan_actionability / blocker_quality)
- plan.md 三段式(实现计划精确到行号 + QA 矩阵 + 回滚方案)
- Sub-agent 子代理(reference README 提到)
- Conventional Commits + post-commit hook 自动发版
- 团队聚合 / 团队继承(T/T2 模式)

### prd-to-code-gen(个人端到端原型)

- 3 Skill(`/build-reference` + `/prd-distill` + `/bff-gen`)
- bff-gen 6 步工作流(蒸馏 → 分析 → 计划 → 生成 → 验证 → 报告)
- 双路径代码生成(Handlebars 模板 0 token + LLM 兜底)
- 4 类验证(语法 / 字段双向不遗漏不幻觉 / 规则 / 集成)
- 自修复循环(最多 3 次)
- 6 种 BFF 架构自动检测
- 5 级匹配链(精确 → 同义词 → 模式 → golden sample → 兜底)
- Skill 是纯 Markdown 指令,不依赖 src/ TS 代码

> **注**:v4 简历正文(投递版.md / resume.typ)只用了 prd-to-code 团队推广版的术语,**prd-to-code-gen 端到端原型刻意没放进简历**(空间紧张 + HR 担心 4 年做 3 项目 + 原型水分)。原型作为面试时的"加分材料",被问到「你这 3 个项目之外还有什么」时拿出来。

## 六、面试风险提示

### 高风险追问

1. **「三 plan 完整闭环」被追问实现细节**
   - 标准答:「plan.md 三段式结构(实现计划 / QA 矩阵 / 回滚方案)+ final-quality-gate.yaml 5 维准出。三者强耦合,放同一份文件保证一致性。」
   - 风险:不要说成「三个独立 plan 文件」。

2. **「20+ 人团队推广」被追问团队反馈**
   - 标准答:「B 端 20+ 人在生产使用,反馈主要通过 /reference Mode E(反馈回流)写回知识库。后端基于此工作流迭代后端版本。」
   - 风险:不要说成「全员都说好用」,会被追问具体反馈。

3. **「82→54 消融实验」被追问实验设计**
   - 标准答:「BFF 单仓基线 82,把证据链层(verified_by + 负向代码搜索)整个关掉,Readiness 掉到 54。证明证据链是必需层而非可选层。」
   - 风险:不要和「R01/R02 54→68」混淆——后者是 reference 知识库对照,不是证据链层。

4. **「Multi-Agent 自动化流水线」被追问是什么项目**
   - 标准答:「个人项目但真实生产——面试速查页(stealth.html)是真实交付物,每天用。基于 Claude Code cron + subagent 自研,v233→v251 共 18 版本真实在跑。」
   - 风险:不要说成「团队项目」,但强调「真实生产、真实用户(我自己)」。

### 中风险追问

5. **「verified_by 证据链」被追问和别的项目比有什么不同**
   - 标准答:「每条事实附带 `verified_by: ["file.ts:line"]`,源码 Read 后才写入,禁止推断。和 Cursor 的 repo map / aider 的 read-only mode 思路类似,但我们做的是 PRD 阶段的证据链,不是 IDE 阶段的。」

6. **「Spec Review Gate」被追问怎么实现**
   - 标准答:「/prd-distill 跑完 Spec 阶段后强制暂停,等用户 approved 才能继续 Plan 阶段。这是工作流硬门禁,不是软提示。」

### 不要主动提的(被问到再说)

- prd-to-code-gen 端到端原型(避免"4 年做这么多"被怀疑水分)
- prd-to-code 8 种模式细节(避免被追问每种模式)
- reference Evidence Index 的具体数据结构(避免被追问图谱构建)
