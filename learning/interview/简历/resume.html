Spec-Driven Development Platform — Spec-Driven Development / Agentic Coding 工作流

公司团队项目 · 主导负责 · Claude Code Skill · Context Engineering · Citation Grounding · Sub-agent

项目定位:不是又一个 PRD → 代码生成器,而是把 Spec 作为可追溯工程中间产物的企业级平台 — 用 Agentic RAG + Citation Grounding 把 LLM 编码约束在可审计的工程边界内,区别于单点 PRD → Plan 工具。

业务场景:B 端 PRD 到代码缺少可追溯中间产物 Spec,AI 直接分析 PRD 缺历史代码上下文与隐性业务逻辑,产出计划不可执行、返工频发。

核心技术:

  • 蒸馏式知识库 + Hybrid Retrieval — 历史代码 / PRD / 跨层契约蒸馏为 6 类知识（代码结构 / 编码规则 / 契约 / PRD 路由 / 业务领域 / 实体索引）,倒排索引 + 实体图按需检索（Context Engineering 范式）
  • Plan-and-Execute 闭环 — 一次蒸馏产出三份产物（开发计划精确到行号 + QA Matrix + Rollback Plan）+ Final Quality Gate（5 维准出报告:完整性 / 一致性 / 可追溯 / 可行性 / 风险）,区别于单点 PRD → Plan 工具
  • Citation Grounding — 每条事实附带 file:line citation,源码 Read 后才写入,禁止推断,降低 LLM 编码 Hallucination
  • Spec Review Gate — Spec 必须用户 approved 才能进入 Plan（HITL gate before plan execution）,避免错误计划被开发执行

量化产出:需求理解阶段耗时 5h → 2h / 需求（n=5 PRDs）;文件级覆盖率 80%+;同一 PRD 蒸馏 determinism 40% → 85%+（n=10）;20+ 人团队日常使用,后端基于 Spec 产物扩展。

Grounding 价值对比实验:去掉 grounding 层,rubric 评分（完整性 / 一致性 / 可追溯 / 可行性 / 风险,满分 100）82 → 54。

Manager_Agent — Multi-Agent Supervisor Orchestrator

个人项目 · 独立设计实现 · LangGraph · Multi-Agent · HITL · Self-Evolution · Plan-and-Execute

业务场景:企业级 AI 助手需统一调度多个专业 Agent（知识库 / NL2SQL / 代码 / 爬虫 / GUI / 办公 / 多模态）,"一个 Prompt 包打天下"无法支撑。本系统基于 LangGraph 设计 Supervisor 编排模式,调度 12 个能力 Agent（8 外部 + 4 内部）。

核心技术:

  • Multi-Agent Supervisor 编排 — 基于 LangGraph 设计 39 节点 状态机,编排层与专家层分离（总管只做意图路由 / 任务规划 / 结果汇总,不写 SQL / 不建向量 / 不改代码）;支持 14 路意图分发 与子句拆解（多意图 → 多步 plan）
  • Plan-and-Execute + Critic 闭环 — Planner 拆任务 → Executor 并行调度 → Critic 事实性审查 → Synth 综合产出;Critic 与 Evaluator 形成"重试否决"仲裁,避免无谓重试;recursionLimit 48 内长跑稳定
  • Probe + Health 双层探活 + HITL 人工确认 — 规划前先探针下游能力（RAG 文档 / DB schema）,失败则降级跳过 down 的 Agent;高风险写操作 HITL 人工确认,file + Redis 双后端 checkpoint 支持跨进程续跑
  • Self-Evolution Layer — 无标注自进化闭环 — ① 失败归因 12 分类 → ② LLM 假设生成（每 30min 产出 ≤3 条） → ③ 假设金丝雀 A/B 验证命中后晋级 Prompt 补丁 → ④ 经验回放（向量 + Jaccard 混合召回）注入 router few-shot,整套机制无需人工标注

量化产出:39 节点 / ~42 条边（含 14 条条件边）· 12 Agent（8 外部 + 4 内部）· 8 类 WS 消息 · 213 个 TS 文件（约 4.6 万行）· 7 个 eval golden + 34 个 smoke 脚本。

code_assistent_Agent — Agentic Coding / Coding Agent Safety

个人项目 · 独立设计实现 · LangGraph · ReAct · Tool Use · 代码 RAG · Agent Safety

业务场景:LLM 改仓库的高风险场景下"模型 + 写权限 = 事故"。本 Agent 把 LLM 改仓库约束在可审计的工程边界内,完整覆盖 Read-before-Write → 语义检索 → 静态分析 → Diff 预览 → 受控写盘 工程闭环。

核心技术:

  • ReAct Tool Use 工具循环 — 基于 LangGraph 编排 18 个 仓库工具（读 / 检索 / AST 分析 / Patch / 校验 / 执行 / Git）的 agent↔tools 循环;Custom CheckpointSaver 实现 BaseCheckpointSaver 接口,支持跨进程会话状态恢复（LangGraph 早期版本仅内置 MemorySaver）
  • 代码 RAG + Unified Diff 工程 — 仓库语义检索（向量召回 + 文件名粗筛 + 增量索引）,区别于端到端 chat 把全量 repo 塞 context;统一 Diff 预览再写入,read-before-write,杜绝 LLM 错误 diff 静默覆盖
  • Agent Safety — 三层写盘守卫 + Sandbox Isolation — 路径黑名单 + 受保护目录 + sha256 乐观锁,任一层拦截即拒绝;subprocess 白名单 env + docker 网络隔离双沙箱,优先强隔离不可用时自动降级
  • Auto-Validation Hook + Prompt A/B（Shadow Patch） — autoValidateAfterWrite 自动跑 typecheck / lint / test,失败针对性修复,信号回流 Prompt 进化库;shadow patch 多次命中晋级正式库,hash 分桶 A/B 对照

量化产出:18 个工具 覆盖 read / search / vector / AST / patch / validate / exec / git 全链路 · 默认服务 800 文件仓库 / 512MB 内存 / 90s 超时 · 跨进程会话恢复（回归测试 0 丢失）。
