## 项目经历

### SDD 工作流落地 — PRD 蒸馏到可执行开发计划

*独立设计 + 落地 · Claude Code Skill · MCP 协议 · Prompt Engineering · Context Engineering · Harness 约束体系*

**背景**：B 端 PRD 到代码缺少可追溯的中间产物（Spec），导致 AI Coding 产出不可控，需求理解偏差返工频发。基于 **Claude Code + MCP** 实现 Spec 自动蒸馏，用 **Cursor Agent Mode** 按 Spec 驱动编码。

**行动**：
- **Reference 知识底座**：双 Skill 体系（知识库构建 **6 模式** + Spec 蒸馏 **11 步工作流**），反馈回流闭环实现 Spec 越用越准
- **SSOT 规范体系**：**6 文件**按关注点分离（代码结构 / 编码规则 / 跨层契约 / PRD 路由 / 业务领域 / 代码实体索引），每个事实只存在一处，**5 条边界规则**避免无规范 AI 编码导致的知识污染
- **能力面适配器**：前端 / BFF / 后端通过 **9+10+10 个能力面**统一描述，消除跨端认知分歧，跨项目可复用
- **AI Coding 质量管控**：**8 种证据类型**（含负向搜索 negative_code_search），PRD 入口三级门禁 + 致命/警告/边界检查，降低 AI 编码返工率

**结果**：
- PRD 到开发计划耗时从 **5h/需求 降到 2h/需求**（5 个需求对照，含 PRD 摄入到 Plan 产出全流程）
- 文件级准确率 **80%**（AI 产出 Plan 文件列表 vs 实际开发分支 diff 对比）
- 同一 PRD 跑 10 次结果一致性从 **40% 提升到 85%+**（文件列表重跑一致性，是产物稳定性指标；单次产出质量另用 Readiness 五维评分，二者互补不互替）
- 三个验证需求蒸馏时整体 Readiness：**BFF 单仓 82 / 前端单仓 83 / 团队三仓 62**（五维 A-E 加权；团队 62 因跨仓契约 E 维 45 拖累，口诀"单仓看 D、多仓看 E"）
- 已在 B 端 **20+ 人团队推广**，后端基于此工作流迭代后端版本

**端到端完整链路原型（个人独立项目 prd2code-gen，下一代探索）**：

团队推广版聚焦 ROI 最高的 PRD→Plan 环节，同时**个人独立设计并跑通端到端 PRD→Code 完整 6 步原型**，验证从需求到可部署代码的全链路可行性，作为下一代工作流探索：

- **step-1 PRD 蒸馏**：5 级匹配链（精确 → 同义词 → 模式 → golden sample → 兜底），变更分类 ADD / MODIFY / DELETE / NO_CHANGE
- **step-4 双路径代码生成**：Handlebars 模板优先（0 token 确定性通道）+ 无模板走 LLM 兜底，LLM 结果可沉淀为新模板形成学习闭环
- **step-5 自验证 + 自修复循环**：4 类校验（语法 / 字段双向不遗漏不幻觉 / 规则 / 集成兼容）+ 最多 3 次重试 + **震荡检测（A→B→A 模式自动切整段重写）**
- **eval / auto-tune 反哺闭环**：case-loader / llm-judge / scorer / gap-analyzer / pattern-detector 五模块，自动评测产物质量并反哺 prompt 优化
- **verified_by 审计链**：每条事实带 `file.ts:line` 溯源，禁止推断，强制源码 Read；漂移检测 `drift-report.json` 自动发现 reference 与代码偏离
- **断点续传**：`progress.yaml` 记录步骤状态，步骤间只通过文件通信，可从任意失败点恢复（步骤本身幂等）
- **实验对照**：R01 无 reference 基线 54 分 → R02 加 reference 知识库 68 分，量化 reference 价值 +14 分

**口径（被追问怎么算的）**：
- 5h→2h 提效：5 个中等复杂度需求对照，人工平均 5h（读 PRD + 评审 + 三方对齐 + 出方案），工具平均 2h；注意是**单环节提效，全链路约 20-30%**
- 80% 文件覆盖：实际开发分支改动的文件集合 vs Plan 产出的文件集合做匹配
- 85% 一致性：同一 PRD 跑 10 次，文件列表完全一致的占比（产物稳定性指标）
- **量化评测体系（被追问防御）**：L1 产物质量（Code Anchor 准确率 / OQ 命中率 / IMP 执行率，已有 Readiness 五维 + Final QG 内部评分）+ L2 研发效率（端到端耗时 / QA 缺陷 / 返工率）+ L3 业务价值；L2/L3 排期补齐

### Schema 驱动营销中台 — BFF 动态生成 + 前端动态渲染

*前端架构 + BFF 架构设计（和导师 5-5 开主导） · React · TypeScript · Node.js · Serverless · Lerna + Nx · Cursor / Claude Code*

**背景**：国际化营销中台覆盖 10+ 国家、20+ 种活动类型，每种活动有不同字段组合和联动规则。传统方式每新增一种活动需要前后端各写一套页面和逻辑，开发周期长、容易出错（老代码月均 10+ 线上事故）。

**行动**：
- **BFF 模板引擎**：模板驱动 Schema 生成，按活动类型分文件管理（25 个活动 × 独立模板），联动模板从渲染模板提取 handler 复用
- **双重求值联动引擎（自研 DSL injectDActions）**：一次声明联动配置，BFF 端 safeEval 求初始状态（SSR 正确）+ 序列化 d_actions 给前端运行时联动，支持 **fetchApi 型**（BFF 重生成 Schema）和 **state 型**（前端本地 eval）
- **前端渲染引擎**：嵌套路径拍平为 dot 路径（formatName），注册表支持 **35+** 组件动态渲染，发布订阅模式（Listener）执行联动
- **Monorepo 治理**：Lerna + Nx + Yarn workspace 管理 **15+** 业务模块，独立版本 + 构建缓存 + 依赖图拓扑排序

**结果**：新增活动类型前端 **零代码**，上线周期从 **2 周缩短到 3 天**，累计支撑 **20+ 种活动类型**落地，联动配置复用率 **80%+**，上线 2 年仅 2-3 起小 bug（老代码月均 10+）。

### 营销权益组件库 — 400+ 权益 React 组件库

*架构设计 + 核心开发（独立完成） · React · TypeScript · Rollup · Changesets · AI Coding*

**背景**：营销平台管理 400+ 种营销权益（优惠券、折扣、积分、立减等）的全生命周期，权益类型持续增长，原有方案每新增一种权益都需要硬编码，维护成本高。

**行动**：
- **Register 注册模式**：内置 **24** 组件 + 运行时注册自定义权益组件（add() / addMultipleCmps()），新增权益无需修改 SDK
- **Schema 同构渲染**：组件库与活动表单共享 Schema 渲染架构，两个消费者复用同一套引擎（Schema 中台 + benefit SDK）
- **Rollup 双模打包**（ESM + CJS + 类型声明），peerDependencies 隔离 react 避免依赖爆炸，Changesets 语义化发版
- **接入层三种形态**：组件模式（`<App>` 内嵌）+ 函数调用模式（Modal.info 命令式）+ 微前端模式（qiankun 挂载）

**结果**：覆盖 **8 个业务项目**，累计接入 **400+ 种权益 SKU**，新权益接入从 **2 天缩短到 2 小时**，组件库零 breaking change 运行 **18 个月+**。

### 风险提示

- **"5h→2h" 数据**：被追问样本量时诚实说"5 个中等复杂度需求对照，样本量小但趋势稳定"，并主动补"这是单环节提效，全链路约 20-30%"

- **"400+ 权益" 不是 400 个组件**：要主动澄清是"400 个权益 SKU，底层组件 30-40 个"
- **端到端原型 prd2code-gen 的定位**：诚实说"个人独立设计的端到端原型，跑通了 6 步完整流程 + R01/R02 实验对照（54→68 分），**没在团队大规模推广**——团队推广版聚焦 PRD→Plan 这个 ROI 最高的环节"——不要说成"团队在用"，否则被追问团队反馈会答不上来
- **被问"既然设计完整为什么不推广"**：标准回答——"PRD→Plan 是 ROI 最高、风险最低的环节（人工确认后才开发），代码生成涉及质量责任和 CI 对齐，需要更成熟的验证体系，所以我先用原型验证设计可行性，再决定要不要投入工程化"
