## 一、RAG 范式演进：从 Naive 到 Agentic

### 六代 RAG 架构演进

| 代际 | 架构 | 特点 | 局限 |
|---|---|---|---|
| 1.0 | **Naive RAG** | 查询→向量检索→拼接→生成 | 召回噪声大，幻觉率高 |
| 2.0 | **Modular RAG** | 可替换的索引/检索/重排/生成模块 | 管道固定，无自适应能力 |
| 3.0 | **Graph RAG** | 知识图谱 + 向量混合检索 | 图构建成本高 |
| 4.0 | **Agentic RAG** | Agent 自主规划检索策略、多步推理 | 系统复杂度高 |
| 5.0 | **Multimodal RAG** | 跨文本/图像/音频/视频检索 | 多模态对齐仍是难点 |
| 6.0 | **Hybrid RAG** | 向量 + 图 + Agent 融合 | 架构复杂度最高 |

### Agentic RAG：2026 年的默认架构

Google Cloud 官方发布的 [GraphRAG 架构指南](https://docs.google.com/architecture/agentic-ai-multimodal-graph-rag-resource-orchestration) 标志着 Agentic RAG 进入主流。

核心模式：
- **规划**：Agent 拆解复杂查询为多步检索计划
- **反思**：评估检索结果质量，决定是否重新检索
- **工具使用**：调用搜索引擎、数据库、API 等外部工具
- **多 Agent 协作**：检索 Agent、推理 Agent、验证 Agent 分工合作

arXiv 综述论文（[2501.09136v4](https://arxiv.org/html/2501.09136v4)）将 Agentic RAG 分为六类：单 Agent、多 Agent、层级式、纠正式、自适应式和基于图的 RAG。

### "RAG is DEAD" 论战

2026 年 RAG 领域最大的争论：百万 Token 上下文窗口是否会让 RAG 过时？

**正方（RAG 将死）：**
- Gemini 2.5 Pro 支持 100 万+ Token
- 长上下文模型可直接灌入大量文档
- 简单场景不再需要检索管道

**反方（RAG 进化）：**
- 企业知识库动辄 TB 级，再大的上下文也装不下
- RAG 的价值不在"塞进去"，而在"找到对的"
- Agentic RAG 从被动检索进化为主动推理
- 成本：检索 + 小模型 远比 超长上下文 + 大模型 便宜

**结论：** RAG 没有死，而是从"简单的向量检索"进化为"Agent 驱动的知识推理系统"。

## 二、RAG 框架生态（2026 年版）

### Top 5 生产级 RAG 框架

| 框架 | Stars | 定位 | 优势 | 最佳场景 |
|---|---|---|---|---|
| **LlamaIndex** | 43K+ | 数据索引 + RAG + Agent | 检索质量最优，LlamaCloud 托管服务 | 数据密集型 RAG、知识图谱 |
| **LangChain/LangGraph** | 111K+ | Agent 编排 + 工具集成 | 生态最大，工具链最全 | 复杂 Agent 工作流 |
| **Haystack** | 22K+ | 生产级 RAG 管道 | 架构清晰，可扩展存储 | 企业级生产 RAG |
| **DSPy** | 28K+ | 编程式 Prompt 优化 | 自动调优检索和生成管道 | Pipeline 精调 |
| **Mastra** | 新兴 | TypeScript 原生 Agent | TS 生态无缝集成 | 前端/Node.js 团队 |

**社区共识：** 生产级 RAG 首选 Haystack 或 LlamaIndex（检索速度更快、管道更可扩展），LangChain 适合需要复杂 Agent 编排的场景。

### LlamaIndex vs LangChain 的 2026 定位

LlamaIndex 从 RAG 框架进化为 **Agent + RAG 双引擎**：
- LlamaCloud 提供托管索引和检索服务
- 可作为 LangChain Agent 的 Retriever 工具
- 两者互补而非竞争

## 三、向量数据库选型

### 2026 年向量数据库 Top 6

| 数据库 | Stars | 语言 | 优势 | 最佳场景 |
|---|---|---|---|---|
| **Milvus** | 35K+ | Go/C++ | 大规模分布式，水平扩展 | 企业海量数据 |
| **Qdrant** | 25K+ | Rust | 延迟最低，性能最优 | 延迟敏感应用 |
| **Weaviate** | 15K+ | Go | 混合检索（向量 + 关键词），GraphQL | 多模态检索 |
| **Pinecone** | 托管 | — | 全托管，零运维 | 快速上线团队 |
| **Chroma** | 18K+ | Python | 轻量级，开发体验好 | 原型/小规模 |
| **pgvector** | PostgreSQL 扩展 | SQL | 复用 PG 生态，ACID 事务 | 已有 PG 基础设施 |

**实测基准（Reddit 社区）：**
- Qdrant（Rust 实现）延迟最低
- Weaviate 混合检索 + GraphQL API 易用性最佳
- Milvus 在大规模分布式场景碾压
- 同机房测试比跨机房延迟低 ~50ms

### 新兴趋势：专用 AI Agent 向量库

向量数据库开始针对 AI Agent 场景优化：
- 支持 MCP 协议直连（如 zilliztech/claude-context）
- 流式索引更新（Agent 实时写入/检索）
- 多粒度索引（文档级、段落级、实体级）

## 四、RAG 评估体系

### 四大核心指标

| 指标 | 测量目标 | 工具支持 |
|---|---|---|
| **Faithfulness（忠实度）** | 生成内容是否忠于检索到的上下文 | RAGAS / DeepEval |
| **Answer Relevance（答案相关性）** | 答案是否切中用户问题 | RAGAS / DeepEval |
| **Context Precision（上下文精确率）** | 检索到的 Chunk 是否相关 | RAGAS |
| **Context Recall（上下文召回率）** | 所有必要信息是否被检索到 | RAGAS |

### RAGAS vs DeepEval 实战对比

| 维度 | RAGAS | DeepEval |
|---|---|---|
| 设计理念 | RAG 专项评估 | 通用 LLM 测试（pytest 风格） |
| 忠实度实现 | 基于自然语言推理（NLI） | 基于 LLM-as-Judge |
| 指标覆盖 | 4 核心 + 扩展 | 50+ 通用指标 |
| 生产集成 | 需要额外编排 | 原生 CI/CD 集成 |
| 社区推荐 | RAG 评估首选 | 需要与其他 LLM 评估统一时选 |

**最佳实践：** RAGAS 评估检索质量 + DeepEval 评估端到端质量，两者互补。

### 2026 年 RAG 评估工具 Top 5

1. **RAGAS** — RAG 评估的事实标准
2. **DeepEval** — pytest 风格的 LLM 测试
3. **LangSmith** — LangChain 生态的可观测性 + 评估
4. **Arize Phoenix** — 实时监控 + 漂移检测
5. **Maxim AI** — 端到端评估 + 可观测性平台

## 五、企业知识库实战方案

### 10 步生产就绪 RAG

来自 100+ 企业部署经验的实战路径：

| 步骤 | 阶段 | 关键决策 |
|---|---|---|
| 1. 文档解析 | Ingestion | PDF/HTML/Markdown 解析器选型 |
| 2. Chunking 策略 | Ingestion | 固定长度 vs 语义分段 vs 递归切分 |
| 3. Embedding 模型 | Indexing | 通用 vs 领域微调 |
| 4. 向量数据库 | Indexing | 托管 vs 自建，单机 vs 分布式 |
| 5. 混合检索 | Retrieval | BM25 + 向量 + 重排 |
| 6. Re-ranking | Retrieval | Cross-encoder 或 LLM 重排 |
| 7. 上下文压缩 | Retrieval | LLMLingua 等上下文压缩技术 |
| 8. 生成策略 | Generation | 温度、Top-P、系统提示词优化 |
| 9. 评估闭环 | Evaluation | RAGAS 四指标持续监控 |
| 10. 生产运维 | Production | 缓存、限流、A/B 测试、漂移检测 |

### 关键架构决策

**Chunking 策略选择：**
- 通用文档：递归字符切分（512-1024 Token，重叠 10-20%）
- 代码/结构化文档：AST 感知切分
- 知识图谱：实体级切分 + 关系抽取

**检索策略选择：**
- 简单问答：纯向量检索
- 精确匹配场景：BM25 + 向量混合
- 复杂推理：Agentic RAG（多步检索 + 反思）

**Embedding 模型选择：**
- 通用场景：OpenAI text-embedding-3-large 或 BGE-M3
- 中文场景：BGE-M3 或 Acme-embedding
- 领域场景：微调 Embedding 模型

## 七、关键趋势总结

### RAG 的三大转变

1. **从被动检索到主动推理** — Agentic RAG 让检索系统具备规划和反思能力
2. **从单一模态到多模态** — 文本、图像、音频、视频联合检索
3. **从技术指标到业务价值** — 评估焦点从 Faithfulness 转向用户满意度和业务 ROI

### 对前端/全栈工程师的启示

- **RAG 是最高 ROI 的 AI 落地路径** — 相比训练模型，搭建 RAG 管道门槛更低、见效更快
- **向量数据库选型看场景** — 原型用 Chroma，生产用 Qdrant/Milvus，图检索用 Weaviate
- **评估先行** — 先建评估管道（RAGAS），再迭代优化检索和生成
- **关注 Agentic RAG** — 这是 2026 年的默认架构，LlamaIndex 和 LangGraph 是主要工具
