## 问题一：JSON Schema引擎做了什么事情，以及为什么自研

1、UI与逻辑分离：前端专注于基础组件和自定义组件的渲染，JSON Schema负责下发组件相关的属性
2、统一协议规范：统一不同业务之间的组件结构形态

3、解决动态表单之间的联动问题

4、提高可扩展性：支持新增字段x\-等新功能

5、为什么自研：之前调研过飞冰、formily等方案，但过于集成导致难以支持动态性的校验等配置

> \&\#34;我们的Schema引擎主要解决两个问题：
> 
> 1. **UI与逻辑解耦**：前端只需开发通用组件（如输入框、选择器），Schema下发组件属性、校验规则和联动逻辑。
> 
> 2. **协议标准化**：统一不同业务的表单结构，比如国内需要身份证校验，东南亚需要护照\+手机号双校验。
> 自研是因为现有方案（如Formily）封装过深，难以支持动态扩展。例如我们需要在运行时动态添加`x\-currency\-format`这种自定义校验规则，自研引擎更灵活。\&\#34;
> 
>

```TypeScript
// Axios拦截器

  const { config, response } = error;

    type: 'api',
    url: config.url,
    status: response?.status,
    traceId: config.headers['x-trace-id']
  });
  return Promise.reject(error);
});
```

```TypeScript

class ErrorBoundary extends Component {
```

## 问题五：React的Diff机制

**源码级解析**：  

1. **Diff策略**：  

    - **Tree Diff**：仅对比同层级节点，跨层级移动会导致组件销毁重建

    - **Component Diff**：相同类组件更新props，不同类组件替换整个子树

    - **Element Diff**：通过key识别元素稳定性



2. **Key的作用原理**：  

    ```TypeScript
    // 没有key时的更新（低效）：
    // Old: [A, B, C]
    // New: [D, A, B, C]
    // React会认为A/B/C都变化了，导致不必要的更新
    
    // 有key时的更新（高效）：
    <div key="A" /> // 被移动到新位置，复用DOM节点
    ```



3. **性能陷阱**：  

    ```TypeScript
    // 避免在渲染函数中生成动态key
    {items.map((item, index) => (
      <Item key={Math.random()} /> // 导致重复卸载/挂载
    ))}
    ```



**项目结合**：  

\&\#34;在动态表单渲染中，对每个字段使用`key=fieldId \+ schemaVersion`，确保字段更新时正确复用组件\&\#34;

## 问题六：JS原型链机制

**原型链穿透详解**：  

```JavaScript
function Car() {}
Car.prototype.drive = function() {};

const myCar = new Car();

// 原型链查找路径：
myCar.__proto__ === Car.prototype
Car.prototype.__proto__ === Object.prototype
Object.prototype.__proto__ === null

// 手动修改原型链
function SuperCar() {}
```

```TypeScript
class RequestManager {
  private controllers: Map<string, AbortController>;

  constructor() {
```

## 问题八：最大数组和算法

**算法优化详解**：  

```JavaScript
function maxSumAfterNegations(arr, k) {
  // 步骤1：排序并优先取反负数（贪心）
  arr.sort((a, b) => a - b);
  for (let i = 0; i < arr.length && k > 0 && arr[i] < 0; i++, k--) {
    arr[i] = -arr[i];
  }

  // 步骤2：处理剩余奇数次取反
  if (k % 2 === 1) {
    // 找到当前最小值（可能已被取反）
    let minVal = Infinity, minIndex = -1;
    for (let i = 0; i < arr.length; i++) {
      if (arr[i] < minVal) {
        minVal = arr[i];
        minIndex = i;
      }
    }
    arr[minIndex] = -arr[minIndex];
  }

  return arr.reduce((a, b) => a + b, 0);
}
```



**复杂度分析**：  

- 时间复杂度：O\(n log n\) （排序主导）  

- 空间复杂度：O\(1\)  



**测试用例验证**：  


## 问题十一：版本号比较的代码题

问题：

```JavaScript
假设你有两个版本标识符 version1 和 version2，需要对它们进行比较。 版本标识符由一系列修订数字构成，这些数字通过一个 '.' 连接。每一位修订数字可能包含前导零，并且至少包含一个数字字符。版本标诈符至少含有一个修订位。修订数字按照从左至右的顺序排列，序号从 0 开始编号，最左边的数字标识为修订号 0，其后的依次增加。例如，1.0.1 和 0.2 均为合法的版本标识符。 

在比较版本标识符时，应该从左至右依次比较它们的修订数字。在比较修订位时，只应比较去除任何前导零的整数值。也就是说，修订号 1 和修订号 001 应视为等同。如果某个版本标识符中未指明某序号的修订位，则该位数值应视为 0。例如，版本标识符 1.0 在比较时应视为小于 1.1，因为它们的修订号 0 相同，而修订号 1 分别为 0 和 1，而 0 < 1。 

比较结果遵循以下规则： 若 version1 > version2，则返回 1； 若 version1 < version2，则返回 -1； 其他情况返回 0。

备注
1 <= version1.length, version2.length <= 500

version1 和 version2 仅包含数字和 '.'

version1 和 version2 都是 有效版本号

version1 和 version2 的所有修订号都可以存储在 32 位整数 中

示例1
输入
version1 = "1.01" , version2 = "1.0001"
输出
0
解释
忽略前导零，"01" 和 "001" 都表示相同的整数 "1"
示例2
输入
version1 = "0.1" , version2 = "1.2"
输出
-1
解释
version1 中下标为 0 的修订号是 "0"，version2 中下标为 0 的修订号是 "1" 。0 < 1，所以 version1 < version2
```

```javascript
var compareVersion = function(version1, version2) {
  const v1 = version1.split('.').map(Number);
  const v2 = version2.split('.').map(Number);
  
  const maxLen = Math.max(v1.length, v2.length);
  for (let i = 0; i < maxLen; i++) {
    const num1 = v1[i] || 0;
    const num2 = v2[i] || 0;
    if (num1 > num2) return 1;
    if (num1 < num2) return -1;
  }
  return 0;
};
```

## **问题三：Schema协议规范如何解决逻辑耦合问题？**

**答**：原系统逻辑耦合体现在两方面，Schema通过以下方式解耦：  

1. **UI与逻辑分离**：  

    - Schema定义字段基础属性（如类型、校验规则），UI渲染通过`x\-component`映射独立组件库；  

    - 业务逻辑（如提交、联动）封装为Hooks，通过`x\-linkage`协议配置，避免硬编码。  

2. **多场景复用**：

## 问题一：（可迭代对象）

完全不会

可迭代对象 Symbol\.iterator

将一个对象改为可迭代对象，可以使用for\.\.\.of

```TypeScript
// 一个 range 对象，它代表了一个数字区间
let range = {
  from: 1,
  to: 5,
};

// 我们希望 for..of 这样运行：
// for(let num of range) ... num=1,2,3,4,5

/**
 可以应用 for..of 的对象被称为 可迭代的。

1. 技术上来说，可迭代对象必须实现 Symbol.iterator 方法。
  1.1 obj[Symbol.iterator]() 的结果被称为 迭代器（iterator）。由它处理进一步的迭代过程。
  1.2 一个迭代器必须有 next() 方法，它返回一个 {done: Boolean, value: any} 对象，这里 done:true 表明迭代结束，否则 value 就是下一个值。
2. Symbol.iterator 方法会被 for..of 自动调用，但我们也可以直接调用它
 */
```

**解答**

```javascript
let range = {
  from: 1,
  to: 5,
  [Symbol.iterator]() {  // 实现 Symbol.iterator 方法
    let current = this.from;
    const last = this.to;
    return {
      next() {          // 迭代器的 next 方法
        return (current <= last) 
          ? { done: false, value: current++ } 
          : { done: true };
      }
    };
  }
};

// 测试
for (let num of range) console.log(num); // 输出 1,2,3,4,5
```

## 问题三：（箭头函数）

没有思路

将箭头函数改写为普通函数

```javascript
function defer(f, ms) {
  return function() {
    setTimeout(() => {
        f.apply(this, arguments)
    }, ms);
  };
}

function sayHi(who) {
  alert('Hello, ' + who);
}

let sayHiDeferred = defer(sayHi, 2000);
sayHiDeferred("John"); // 2 秒后显示：Hello, John
```

---

### 关键原因解析

箭头函数没有自己的 `this` 和 `arguments`，会自动捕获外层上下文。普通函数则需要通过闭包手动保存这些值。

---

### 分步改写方案

```JavaScript
function defer(f, ms) {
  return function(...args) {  // 使用普通函数，通过闭包保存参数
    const context = this;    // 手动保存当前 this
    setTimeout(function() {  // 改用普通函数
      f.apply(context, args); // 显式传递保存的 this 和参数
    }, ms);
  };
}

function sayHi(who) {
  console.log('Hello, ' + who);
}

let sayHiDeferred = defer(sayHi, 2000);
sayHiDeferred("John"); // 2 秒后输出：Hello, John
```

---

### 核心知识点

1. **`this`**\*\* 的差异\*\*  

    - 箭头函数：无自己的 `this`，自动继承外层函数的 `this`。

    - 普通函数：有自己的 `this`，默认指向调用者（严格模式下为 `undefined`）。

2. **参数传递**  

    - 箭头函数：通过闭包直接访问外层 `arguments`。

    - 普通函数：需通过 `\.\.\.args` 或 `arguments` 显式保存参数（注意 `arguments` 是类数组对象）。

3. **闭包的必要性**  

    - `setTimeout` 的回调函数是异步执行的，必须通过闭包预先保存 `this` 和参数，否则会丢失上下文。

---

### 错误写法对比

```JavaScript
// 错误示例：直接使用 this 和 arguments
function defer(f, ms) {
  return function() {
    setTimeout(function() {
      f.apply(this, arguments); // 此时 this 指向 window/undefined
    }, ms);
  };
}
```

- **原因**：`setTimeout` 回调中的 `this` 指向全局对象（非严格模式），且无法获取外层函数的 `arguments`。

### 分步解析与实现



要实现 `runPromiseInSequence` 函数，核心是利用 **Promise 链式调用** 和 **Array\.reduce** 的累积特性。以下是具体步骤：



```JavaScript
function runPromiseInSequence(arr, input) {
  // 初始值包装为 Promise，并通过 reduce 逐步链接处理函数
  return arr.reduce((promiseChain, currentFunc) => {
    return promiseChain.then(result => currentFunc(result));

}
```

---



### 关键点解析



1. **链式调用原理**  

    - `arr\.reduce` 从初始的 `Promise\.resolve\(input\)` 开始，逐步将每个处理函数（`p1`, `p2`, `f3`, `p4`）链接到 Promise 链中。

    - 每次调用 `\.then\(\)` 会将前一个 Promise 的结果传递给当前处理函数。



2. **处理同步与异步函数**  

    - **异步函数**（如 `p1`, `p2`, `p4`）：返回 Promise，`\.then\(\)` 自动等待其解决。

    - **同步函数**（如 `f3`）：返回值会被自动包装为 `Promise\.resolve\(value\)`。



3. **执行流程分解**  

    - 初始输入 `10` → `Promise\.resolve\(10\)`。

    - **Step 1**: `\.then\(p1\)` → `p1\(10\)` 返回 `50`（Promise 解决值）。

    - **Step 2**: `\.then\(p2\)` → `p2\(50\)` 返回 `100`。

    - **Step 3**: `\.then\(f3\)` → `f3\(100\)` 返回 `300`（同步值，自动包装为 Promise）。

    - **Step 4**: `\.then\(p4\)` → `p4\(300\)` 返回 `1200`。

    - 最终结果通过 `\.then\(console\.log\)` 输出 `1200`。

### 分步解答



要实现 `usePrevious` 自定义 Hook，核心是利用 `useRef` 和 `useEffect` 的配合，保存上一次的 `value` 值。以下是完整实现和解析：



```JavaScript
function usePrevious(value) {
  const ref = React.useRef(); // 创建一个 ref 对象

  React.useEffect(() => {
    ref.current = value; // 在渲染完成后更新 ref 的值为最新值
  }, [value]); // 依赖 value，当 value 变化时触发更新

  return ref.current; // 返回上一次的 value（当前渲染周期尚未更新时的值）
}
```



---



### 核心原理



1. **`useRef`**\*\* 的作用\*\*  

    - `useRef` 创建一个持久化存储的引用对象 `ref`，其 `\.current` 属性的变化不会触发组件重新渲染。

    - `ref\.current` 始终保存的是上一次 `useEffect` 执行后的值。



2. **执行顺序的巧妙性**  

    - **渲染阶段**：`usePrevious` 返回的是 `ref\.current`（即上一次渲染时的值）。

    - **提交阶段**：`useEffect` 执行，更新 `ref\.current = value`（为下一次渲染做准备）。



3. **依赖项控制**  

    - `useEffect` 的依赖项为 `value`，确保 `value` 变化时更新 `ref\.current`。

|---|---|---|---|
|1|初始渲染|`undefined`|`undefined`|
|2|`useEffect` 执行|`0`|\-|
|3|点击按钮，`count=1`|`0`|`0`（渲染阶段）|
|4|`useEffect` 执行|`1`|\-|



此时控制台输出：`Count changed from 0 to 1`。
