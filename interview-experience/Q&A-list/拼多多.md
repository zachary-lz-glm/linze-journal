## Q2: 组件库和Schema营销中台是一体的吗？

**核心句**：组件库是中台的一个子模块，负责权益类型渲染
**展开**：营销中台是整体架构，组件库是里面权益相关的子模块。它的作用是把多种权益类型的配置统一收容到一个组件库，通过Register模式按需加载，其他活动类型走独立的Schema模板。

## Q3: 营销中台的应用场景？

**核心句**：运营自助配置拉新留存活动，开发不介入
**展开**：国际化针对司机有多种拉新留存奖励活动，每种活动有不同的配置规则、触达人群、奖励策略。之前每次上线新活动需要前端开发2周介入，现在运营通过中台配置页面自助完成，前端零介入。

## Q4: 国际化方案？

**核心句**：i18n由专门团队维护，前端通过key-value接入
**展开**：国际化由专门团队统一维护，前端通过i18n key接入。文案通过key-value形式管理，后端下发多语言文案包，前端根据当前locale取对应文案。组件库层面也支持通过Schema的locale字段做差异化配置。

## Q5: 怎么声明组件？Schema模板描述和统一渲染层？

**核心句**：三层架构——后端管数据，BFF管协议转换，前端管渲染
**展开**：整体分三层。后端专注业务数据，按字段平铺下发，不管页面布局。BFF中间层负责把后端数据转成前端可识别的JSON Schema。前端有一套Schema解析渲染引擎。Schema结构包含：component字段映射前端组件名，name字段对应后端字段名，attrs对象承载组件配置，以及我们自研的D2联动语法来描述组件间的联动关系。D2本质上是一个模板字符串，描述"当依赖项等于某值时，触发什么动作"。

## Q6: 联动关系维护在哪里？

**核心句**：联动写在Schema模板里，分API联动和状态联动两类
**展开**：联动关系维护在Schema模板中。每个组件的联动配置包含dependencies数组（声明被依赖的组件路径）和action描述（用三元表达式模板字符串定义触发条件）。联动分两类：fetch API型，组件变化时调后端接口拉最新配置项，BFF重新生成Schema覆盖；state型，前端自闭环，比如A组件变化时直接清空B组件的值或改默认值。

## Q7: 组件间联动通过什么技术方案感知状态变化？

**核心句**：发布订阅模式，路径拍平后用listener监听变化
**展开**：因为组件涉及多层嵌套，我们先用formatName把路径拍平，比如basic.timeRange.start作为key。这样依赖项和被依赖项可以通过路径前缀匹配来查找。然后通过listener发布订阅模式：组件初始化时注册依赖字段，字段变化时emit事件通知关联组件执行联动回调。

## Q8: AB实验怎么做？

**核心句**：AB实验组件管理对照组和实验组，运营配置比例即可
**展开**：AB实验是中台内置的独立组件，运营在人群卡片里配置对照组和实验组的比例。前端负责提供比例输入和实验组的独立配置tab。实验分流由后端根据配置的比例进行，前端不参与分流逻辑，只负责根据后端下发的实验组标识渲染对应的UI方案。

## Q9: 客户端渲染怎么保障首屏性能？

**核心句**：首屏通过代码分割+按需加载+预请求优化，非首屏组件延迟渲染
**展开**：B端系统首屏确实体感不如C端敏感，但我们做了几个关键优化：第一，路由级别用React.lazy做代码分割，首屏只加载当前页面所需代码；第二，Schema驱动的组件按需注册，未使用的组件不加载；第三，16个联动接口通过Service Worker缓存，减少重复请求耗时；第四，大型表格使用虚拟滚动，只在可视区域渲染DOM。非首屏的组件区域通过IntersectionObserver延迟渲染。

## Q10: 写前端解析Schema渲染的伪代码

**核心句**：递归解析Schema，children字段触发嵌套渲染
**展开**：

```tsx
function SchemaRenderer({ schema }) {
  return (
    <FormWrapper schema={schema}>
      {schema.fields.map(field => <FieldRenderer key={field.name} field={field} />)}
    </FormWrapper>
  );
}

function FieldRenderer({ field }) {
  if (field.children?.length) {
    return (
      <FormItem label={field.label} visible={field.visible}>
        {field.children.map(child => <FieldRenderer key={child.name} field={child} />)}
      </FormItem>
    );
  }
  const Component = componentRegistry[field.component];
  return (
    <FormItem label={field.label}>
      <Component {...field.attrs} name={field.name} />
    </FormItem>
  );
}
```

## Q11: 懒加载方案？

**核心句**：目前路由级用React.lazy，组件级按需注册但未做动态import
**展开**：路由级别用了React.lazy做代码分割。组件级别目前是全量注册到组件Map里，没有做动态import。如果要优化，可以在BFF层根据当前页面的Schema只下发所需的组件列表，前端按需import动态加载。

## Q12: 监控和异常处理？

**核心句**：主动发起报警治理，新增reason tag实现精准过滤上报
**展开**：原方案直接按后端错误码上报，误报率极高。我分析了后端错误码管理混乱的问题，整理出6个维度，让后端新增reason tag字段标识错误场景。前端拿到reason tag后做一层过滤，只上报需要关注的错误。前端JS错误方面接入了Sentry，捕获运行时异常。

## Q13: 线上白屏怎么发现和排查？

**核心句**：白屏检测用MutationObserver+骨架屏超时，配合全链路TraceID定位
**展开**：白屏发现分三层：第一，JS运行时错误通过Sentry捕获，组件级白屏用React Error Boundary兜底；第二，接口异常通过axios拦截器捕获，关联TraceID可追踪后端日志；第三，真正的白屏检测——我们在关键容器上设置MutationObserver，如果页面加载3秒后根节点仍无子元素，自动上报白屏告警，截图当前DOM状态。排查时通过TraceID串联前端→BFF→后端完整链路。

## Q14: 怎么捕获JS异常？

**核心句**：全局window.onerror + unhandledrejection + Error Boundary三层兜底
**展开**：

```
1. window.onerror：捕获同步/异步运行时错误（脚本、语法）
2. window.addEventListener('unhandledrejection')：捕获未处理的Promise异常
3. window.addEventListener('error', fn, true)：捕获阶段监听资源加载错误（img/script/css）
4. React Error Boundary：组件级错误兜底，防止白屏
5. try-catch：业务代码主动捕获已知异常点
```

## Q15: 组件包版本兼容性？

## Q16: TypeScript的Pick怎么实现？

**核心句**：Pick<T, K> = { [P in K]: T[P] }
**展开**：

```typescript
type MyPick<T, K extends keyof T> = {
  [P in K]: T[P];
};

// 使用示例
interface User {
  name: string;
  age: number;
  email: string;
}

type UserNameAndAge = MyPick<User, 'name' | 'age'>;
// 等价于 { name: string; age: number }
```

核心就三步：1）K extends keyof T 约束K必须是T的key子集；2）[P in K] 遍历K中的每个key；3）T[P] 取原类型中对应key的值类型。

## Q19: 扁平数组转树结构（代码题）

**核心句**：递归filter+map，或用Map一次遍历O(n)
**展开**：

```javascript
// 方案一：递归 O(n²)
function buildTree(list, parentId = null) {
  return list
    .filter(item => item.parentId === parentId)
    .map(item => ({
      ...item,
      children: buildTree(list, item.id)
    }));
}

// 方案二：Map优化 O(n)
function buildTreeOptimized(list) {
  const map = {};
  const roots = [];
  list.forEach(item => {
    map[item.id] = { ...item, children: [] };
  });
  list.forEach(item => {
    if (item.parentId === null) {
      roots.push(map[item.id]);
    } else {
      map[item.parentId].children.push(map[item.id]);
    }
  });
  return roots;
}
```

面试中先写递归版（快、清晰），再主动提O(n)优化版。

## 面试官洞察（人形机器人公司）

> 面试官详细描述了公司的AI技术架构，是所有面试中分享AI工程实践最深入的。

### 洞察1：AI工程三层架构

**面试官描述**：
- 第一层（工程侧）：大模型调优、监控告警体系
- 第二层（RAG通用能力层）：记忆压缩、推荐、工具调用层
- 第三层（大模型调度层）：网关、调度策略、配置平台

**价值**：这是AI工程化落地的行业通用分层模式。面试中讨论"AI工程架构"时可以引用这个三层模型，展示对行业架构的认知。

### 洞察2：效果vs成本的核心权衡

**面试官直言**："你用越好的模型给越长上下文，效果越好，但成本越高"，需要"权衡效果和成本之间做个取舍"。

**价值**：揭示了AI落地不是只追求效果，成本控制是同等重要的工程命题。面试中说"AI提效"时要同时考虑成本维度。

### 洞察3：自研对话压缩方案

**面试官提到**：公司自己实现了对话压缩层，否则"每一轮都把上下文扔给模型，不用积分就肯定会超"。

**价值**：对话压缩是AI工程化的核心能力，不是可选优化。说明大模型应用的工程挑战远不止模型能力本身。

### 洞察4：幻觉/稳定性/体验需工具手段解决

**面试官指出**：大模型本身的幻觉、调用稳定性、体验问题需要用额外工具手段弥补，不能指望模型自身解决。

**价值**：再次印证"AI工程化"的核心不是模型，是在模型之上构建可靠性保障层。
