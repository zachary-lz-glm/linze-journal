## Q2: Harness、Skill、Agent（Space）之间什么关系？

**核心句**：Skill是经验显性化、Harness是上下文管理、Agent是6层基础设施

**展开**：
- **Skill**：本质是把"人怎么用AI做好任务"的隐性经验显性化、可复用，解决效率、质量、组织记忆三个问题
- **Harness Engineering**：偏向Context Engineering，是上下文管理过程——通过信息选择排序、上下文预算管理、产物渐进式披露
- **Agent**：围绕模型的基础设施，6层架构——信息边界、工具系统（MCP/Function Calling）、执行编排（智能体编排/状态机）、记忆（短期+长期）、评估观测、约束恢复与自动修复

## Q3: AI Agent里Skill的加载过程是怎样的？

**核心句**：触发→路由决策→构建请求→执行→响应返回，6步标准化流程

**展开**：以Claude Code原生能力为例：
1. **用户触发**：关键词触发或斜杠command触发
2. **路由+下载决策**：验证skill是否存在、是否有权限
3. **构建请求**：把上下文包装成标准化JSON（含开发配置）
4. **发送请求**：交给skill执行环境
5. **执行业务逻辑**：skill内部跑工作流
6. **生成响应**：包装成符合平台规范的响应，平台返回用户设备

## Q4: PRD体系搭建项目的背景？

**核心句**：解决认知偏差和隐性知识两个痛点，把PRD蒸馏成有证据的开发计划

**展开**：想把AI工程化落地到实际开发。两个核心痛点：
1. **认知偏差**：PM写PRD→口口相传→开发理解，每一步都有信息损耗，导致返工沉没成本
2. **隐性知识**：老项目经验沉淀在老员工脑子里，新员工上手成本高

工作流不是让AI写代码，而是把PRD蒸馏成有证据支撑的开发计划。

## Q5: PRD写错（认知偏差）这一块怎么解决？

**核心句**：用证据链机制——所有AI推断必须有据可循，不确定的标红让人为check

**展开**：工具结合历史PRD、技术文档、git log、项目源码，先生成一份前置领域知识文件（思维图谱约束）。新需求时如发现PRD与事实不符、图文不符、结构不符，会在spec里作为风险点标记。核心是**证据链机制**——AI所有推断必须有据可循，不确定的让人为check。

## Q6: 开发生命周期里，PM写PRD时就知道跟代码不一致，怎么调整？

**核心句**：已沉淀知识PM写错由工具自动纠正，全新需求PM写不对需后续规范化

**展开**：分两种情况：
1. **已沉淀的项目知识PM写错**（如历史记录查询写进详情，实际应在列表），工具会**自动纠正**——这种属于项目隐性知识，AI基于代码证据可以直接修正
2. **全新需求PM写不对**，属于工具范畴外的问题，需要后续规范化PM写PRD（推动产品按AI-Friendly规范来写）

## Q10: 怎么把控整体质量？

**核心句**：多层质量门控——步骤级前置检查+文件级门禁+脚本+人为check

**展开**：
1. **步骤级前置检查**：PRD蒸馏11步，每步执行前检查上一步产物
2. **文件级门禁**：文件存在性、需求证据、代码锚点、spec确认、覆盖率（PRD内容覆盖/图片需求追溯/最终评分）
3. **脚本检验+人为check**：通过独立脚本验证，最后人工check生成最终plan
4. **可追溯验证**：对比中间文件（如需求点拆解）验证是否有遗漏/偏差

## Q26: 营销中台关注哪些技术指标？

**核心句**：列表渲染+懒加载+动态表格可编辑单元格+联动接口Service Worker缓存

**展开**：
1. **列表渲染**：常规优化
2. **懒加载**：常规优化
3. **动态表格场景**：行数跟前面联动（如填100城市就有100行×多列=几百上千单元格），用**可编辑单元格形式**让它看上去是单一输入框，避免一次性渲染上千组件
4. **联动接口优化**：点A组件动态调接口获取B组件配置，A组件转圈1-2秒体验不好，用**Service Worker缓存**联动接口的响应

## Q27: 用React吧？开发场景里有哪些策略提高组件生产效率？

## Q29: Schema方案跟Formily有什么区别？为什么不用开源？

**核心句**：Formily是纯前端方案，我们的双端联动+DSL表达能力它覆盖不了

**展开**：Formily确实是成熟的Schema驱动表单方案，支持自定义组件扩展，对纯前端表单够用。但我们的核心差异：
1. **双端联动**：不仅是前端组件间依赖联动，还涉及BFF层联动接口（如用户切换城市，前端不直接拿后端数据，先调BFF，BFF根据城市产品线重新剪裁Schema返回），Formily覆盖不了
2. **DSL联动表达能力**：跨接口依赖Formily需要写大量effect，本质上还是自研联动层

## Q30: 自研方案里，表单之间联动怎么实现？

**核心句**：DSL对象含dependence数组+模板字符串，发布订阅模式监听变化

**展开**：DSL本身是一个对象，两个属性：
1. **dependence**：数组，包含依赖项的内部路径
2. **具体依赖值**：产生的影响，如A改了B的visible，后面是三元表达式模板字符串——依赖项等于某值时true否则false

前端拿到JSON后通过**Listener发布订阅模式**监听依赖项，依赖项变化时**eval模板字符串**，结果覆盖到组件。

## Q31: React Fiber架构怎么实现？解决什么问题？

**核心句**：把大任务拆成链表式子任务+优先级+时间分片，解决React 15同步渲染卡顿

**展开**：Fiber本质是把大任务拆解成无数子任务（**链表形式**），给每个任务加**优先级和时间限制**，解决React 15同步渲染导致卡顿掉帧的问题。React 15的调和（reconciliation）是不可中断的，现在通过Fiber节点可以**中断、恢复、优先级控制、优先级更新**。

---

## Q32: ⚠️ 一秒内实现营销页面渲染，从能力角度怎么设计？

**核心句**：按"网络-资源-渲染-感知"四层拆解，每层定指标

**展开**：
1. **网络层（<200ms）**：CDN静态资源+HTTP/3+预连接preconnect
2. **资源层（<300ms）**：HTML走SSR或预渲染、JS/CSS代码分割+critical CSS内联、图片WebP/AVIF+懒加载
3. **渲染层（<400ms）**：骨架屏先上、首屏React并发渲染（startTransition）、长任务拆分（避免主线程阻塞）
4. **感知层（<100ms）**：loading过渡+用户操作反馈+Service Worker缓存二次访问
5. **兜底**：监控真实用户的LCP/FCP，定位瓶颈

## Q33: SSR的渲染过程？

**核心句**：服务端拼HTML字符串+数据注入，浏览器直接绘制首屏

**展开**：
1. 客户端发起请求 → Node服务器接收
2. 服务器匹配路由 → 执行组件的数据预取函数（如`getServerSideProps`）
3. 调用后端API/数据库拿数据
4. React把组件树`renderToString`成HTML字符串
5. 数据注入HTML（`<script>window.__INITIAL_STATE__</script>`）
6. 返回浏览器 → 浏览器直接绘制（首屏快）

## Q34: 注水（hydration）水合过程了解吗？

**核心句**：React在内存构建virtual DOM，对比SSR的DOM匹配则复用，再绑定事件激活state

**展开**：
1. 浏览器下载HTML后立即显示（用户已能看到内容）
2. JS bundle下载并执行
3. React在内存里基于同样的组件构建**virtual DOM**
4. React**对比**virtual DOM和SSR的DOM节点：
   - 匹配 → **复用现有DOM**（不重新创建，保住SSR的性能收益）
   - 不匹配 → 报**Hydration Mismatch** warning，严重时丢弃SSR HTML重渲染
5. React给现有DOM节点**绑定事件监听**（这是为什么按钮在hydration前点了没反应）
6. 组件state/effect激活，页面变成可交互

**关键坑——Hydration Mismatch**：用了`Date.now()`、`Math.random()`、`window.innerWidth`等会导致服务端和客户端渲染结果不一致。解决：用`useEffect`里设置这些值，或`typeof window !== 'undefined'`判断。

**React 18增强**：Selective Hydration（优先hydrate用户正在交互的部分）+ Concurrent Features。

来源：[Josh W. Comeau - Perils of Hydration](https://www.joshwcomeau.com/react/the-perils-of-rehydration/) / [Gatsby React Hydration](https://www.gatsbyjs.com/docs/conceptual/react-hydration/)

## Q37: HTTPS和HTTP握手过程有什么区别？

**核心句**：HTTP是TCP三次握手后直接传明文，HTTPS在TCP之上多一次TLS握手协商加密

**展开**：
**HTTP**：TCP三次握手（SYN → SYN+ACK → ACK）→ 直接开始传HTTP报文（明文）

**HTTPS**：TCP三次握手 → **TLS握手** → 加密通道传输

**TLS握手5步**（以TLS 1.2为例）：
1. **ClientHello**：客户端发送支持的TLS版本、加密套件列表、客户端随机数
2. **ServerHello**：服务端选定TLS版本、加密套件、返回服务端随机数
3. **Certificate**：服务端发送SSL证书（含公钥），客户端用CA公钥验证证书合法性
4. **Key Exchange**：客户端生成pre-master secret，用服务端公钥加密发送（或ECDHE协商）→ 双方用三个随机数算出session key
5. **Finished**：双方用session key加密发送握手摘要，互相验证握手完整性

之后所有HTTP报文用session key对称加密传输。

**TLS 1.3优化**：合并步骤，1-RTT完成握手（甚至0-RTT恢复会话）。

来源：[Cloudflare TLS Handshake](https://www.cloudflare.com/learning/ssl/what-happens-in-a-tls-handshake/) / [Palo Alto Networks TLS Handshake](https://www.paloaltonetworks.com/cyberpedia/what-is-a-tls-handshake)

## Q38: ⚠️ 节流和防抖有什么区别？（比喻说反了）

**核心句**：防抖是"等用户停下来再执行"（合并），节流是"固定频率执行"（稀释）

**展开**：
**正确比喻**：
- **防抖** = 玩英雄联盟回城——你每次按都重新读条，**只有最后那次能完成**（被打断就重来）
- **节流** = 技能冷却——技能用完后**固定CD内不能再放**，但CD到了就一定能放

**核心区别**：
- 防抖：只执行**最后一次**（在quiet period之后）
- 节流：执行**第一次**，然后每隔delay执行一次

**典型场景**：
- 防抖：搜索框输入、window resize
- 节流：scroll、mousemove、按钮防连点

**实现**：
```js
// 防抖：每次清除前一个timer
function debounce(fn, delay) {
  let timer;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), delay);
  };
}

// 节流：用flag锁
function throttle(fn, limit) {
  let inThrottle = false;
  return (...args) => {
    if (!inThrottle) {
      fn(...args);
      inThrottle = true;
      setTimeout(() => (inThrottle = false), limit);
    }
  };
}
```

来源：[GreatFrontEnd - Debouncing & Throttling](https://www.greatfrontend.com/questions/quiz/explain-the-concept-of-debouncing-and-throttling)

2. **技术深度（展示专业）**：
   - "千问在做活动运营中台，会不会遇到类似Schema驱动配置化、动态表单这类需求？目前是怎么解决的？"
   - "团队对AI代码生成的接受度怎么样？有没有在用类似Cursor/Claude Code/Copilot的工具？"

3. **避免问的**：千问整体战略、阿里集团规划（跟个人无关）、加班/福利（HR面再问）、"我有什么需要改进的"（不礼貌）
