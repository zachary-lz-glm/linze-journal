## Q1: 自我介绍
**核心句**：我在滴滴国际化做了3件事——Schema中台、权益组件库、AI工作流
**展开**：面试官好，我叫邓泽林，滴滴国际化高级前端，4年经验，2年A绩效。我在滴滴主要做了三件事。第一，设计了一套Schema驱动的营销中台，覆盖20多个活动类型，新增活动前端成本降为0，上线从两周缩短到两天。第二，搭建了400+权益组件库，Schema驱动渲染，覆盖8个业务，上线两年0事故。第三，独立设计了一套AI工程工作流，把PRD自动蒸馏成结构化开发计划，开发效率提升60%。我的核心优势是从业务中抽取共性，做全流程质量把控。

## Q3: AI工作流的项目背景和痛点
**核心句**：需求链路依赖口口相传导致返工，新人上手成本高，用知识蒸馏解决
**展开**：观察到两个痛点。第一，需求从PRD到开发计划，中间经过评审、口头传达、人工理解，每一步都有信息损耗，返工很频繁。在AI时代这种沉没成本没必要。第二，历史项目经验在老员工脑子里，新员工上手困难，很多问题跑来问前端。所以我想用AI把PRD自动蒸馏成结构化开发计划，同时沉淀项目知识库，解决信息传递和知识传承两个问题。

## Q4: 知识库分层结构 + 前后端管理
**核心句**：6个领域文件覆盖代码、契约、业务三层知识，按项目维度独立维护
**展开**：知识库按项目维度分开，每个项目6个文件。代码层有代码结构和编码规则；契约层有跨层契约，记录上下游字段和接口的关系；业务层有PRD路由映射——什么场景该改哪些文件，和业务领域——PRD关键词到代码字段的映射。另外有个导航入口做总览，还有个脚本构建的倒排索引，根据业务领域文件反查到具体代码文件，帮AI快速定位候选文件。前后端虽然业务公用，但关注点不同：前端关心组件和交互，BFF关心接口和模板，所以有"能力面适配"机制，根据项目类型自动帮AI筛选该关注的文件。

## Q6: 知识过时/失效怎么办
**核心句**：知识库本质是约束层不是全量事实，靠代码验证+回流飞轮+人工校验三重保障
**展开**：首先要澄清，知识库不是全量的业务知识存储，而是帮AI做约束和导航——告诉AI某种场景下应该关注哪些文件、走什么思路，不是写死的点对点映射。过时分两类：代码结构、编码规则这类静态知识，每次构建都基于最新代码扫描，天然不会过时。业务规则比如会员等级判定确实可能过时，三层防护：冷启动用最新PRD构建；每次需求蒸馏后自动回流更新；产物输出前必须人工review。但坦白说，纯动态业务知识的自动检测目前还没做好，这是核心难题。阿里的体系用知识图谱+大量迭代积累来解决这个问题，我目前还处于冷启动阶段，后续重点是建立知识保鲜的自动化机制。

## Q11: 怎么证明带来效率提升
**核心句**：对比"PRD到开发计划"的耗时，人工2-4小时，工作流30分钟
**展开**：度量方式是对比"从拿到PRD到产出开发计划"的耗时。实验组用我的工作流（含Reference知识库+证据链约束），对照组是同样经验水平的同事，都用AI coding但走自己的流程。结果是从平均2-4小时缩短到30分钟。核心差异不在AI能力本身——现在没人不用AI coding——而在于证据链设计。直接丢PRD给AI，它会幻觉和推测，产出不可信不可重复；有工作流约束后，每个推理都有证据来源，产出可控可标准化。但我要诚实说两点：第一，这个60%的数字主要是从PRD到开发计划这一段的工时缩减，不是全流程；第二，样本量还比较小，目前主要是前端营销B端的几个项目验证过，还需要更多场景验证。

## Q12: PRD超大（2万字）怎么不爆context
**核心句**：先分段摘要提取需求骨架，再按需检索，不一次性灌入
**展开**：现实PRD确实可能2万字，直接灌会打爆200K上下文窗口。三层处理：第一层，PRD输入时先做分段摘要，提取需求骨架——涉及哪些模块、哪些场景，控制在2000字以内。第二层，根据摘要结果，按需从原文中检索相关段落做精确分析，不是把整个PRD喂进去。第三层，对超长表格数据，通过多个子Agent分批并行处理。核心思路是"先摘要定位，再按需深入"。另外如果原始PRD是doc格式，我会先转换成Markdown——图片转链接只保留纯文本，体积可以大幅缩减。

## Q14: 和Cursor codeit的区别
**核心句**：三个核心差异——业务语义理解、证据链约束、质量门控
**展开**：第一，Claude Code init只分析代码结构，不理解业务语义。PRD里说的"会员等级"对应代码里哪个字段，它不知道。我的知识库有业务领域映射（PRD关键词→代码字段的映射）和PRD路由映射（什么场景该改哪些文件），帮AI建立业务到代码的桥梁。第二，直接让AI写开发计划会幻觉——找不到依据就推测，产出不可信不可重复。我的工作流用证据链约束：每个结论必须标注来源是PRD原文第几节、还是代码哪个文件、还是契约哪个接口。第三，质量门控——整个工作流有跳步检测和中间产物完整性检查，不允许AI跳过分析直接出结论。核心不是AI能力本身，而是让AI能力变得可控、受约束、可标准化、可重复。

## Q16: 结构化产物的格式和章节
**核心句**：Spec是需求分析报告，Plan是开发执行计划，都有固定章节模板
**展开**：Spec分三段：第一段需求理解——30秒概览、需求点拆分（每个点标注PRD来源章节）；第二段契约对齐——BFF需要关注和不需关注的需求点分别列出；第三段风险检查——AI不确定的点生成表格，需人工补充后才能继续。Plan也分三段：第一段架构概览和开发顺序依赖图；第二段需求点到文件的映射，每个改动标注来源是代码还是Reference还是PRD；第三段QA矩阵——改动涉及哪些测试场景，以及上下游契约对齐确认。

## Q18: 为什么不用Formily
**核心句**：Formily是纯前端表单方案，我们需要从BFF到前端的端到端联动DSL
**展开**：调研过Formily，它确实能做联动和自定义扩展。但我们有两个特殊性。第一，很多联动不是前端自闭环的——需要调BFF接口查业务数据，BFF根据结果重新拼接Schema下发给前端做整体替换。联动配置嵌在BFF模板里，和后端逻辑紧耦合，Formily的客户端方案覆盖不了。第二，我们的Schema是BFF模板引擎生成的，从模板到渲染是端到端的。所以需要一套从BFF到前端的完整联动DSL，而不是纯前端表单方案。如果只是前端渲染层，Formily完全可以。

---

## Q19: 表单联动怎么做
**核心句**：State型前端自闭环，API型走BFF重拼Schema
**展开**：联动分两种。State型——组件的显隐、禁用、赋值，纯前端闭环。Schema里通过DSL声明：dependence指定依赖字段的namePath，模板字符串用三元表达式描述联动条件。前端通过发布订阅模式监听依赖变化，解析模板字符串得到布尔值，直接修改组件状态。API型——不同城市对应不同配置选项这种，依赖变化时调BFF联动接口，BFF查业务数据后重新拼接Schema下发给前端做整体替换。分成两种是因为State型不需要网络请求响应快，API型需要业务数据必须走BFF。

---

## Q20: 怎么保证性能（Service Worker追问）
**核心句**：联动接口结果相同参数直接读缓存，避免重复请求
**展开**：性能瓶颈不在组件渲染，在联动接口——用户每改一个配置项可能触发1-2个接口调用，响应慢的话界面就转圈。优化手段是缓存联动接口的返回：同一个联动接口，如果入参相同，直接返回缓存结果不发请求。实现方式是Service Worker拦截请求做缓存匹配，好处是和业务逻辑解耦，不需要改BFF层代码。

---

## Q21: 异步组件（地址选择器）Schema怎么描述
**核心句**：不是不支持，是远程数据组件通过自定义组件加异步属性声明来处理
**展开**：地址选择器这种需要按级联请求远程数据的组件，确实不是纯Schema声明能闭环的，因为数据不在本地。处理方式是注册为自定义组件，组件内部自己管理异步数据请求和级联逻辑。Schema层只声明组件类型和必要配置，具体异步请求逻辑封装在组件代码里。这和Formily处理远程数据组件的方式本质一样——复杂异步组件都需要代码层介入，纯配置搞不定。我们的原则是Schema管结构和联动，代码管复杂交互。

## 面试官洞察（阿里淘天B端基建团队）

> 面试官通过一系列追问帮助厘清了知识库的真实定位和AI工程化的核心难题。

### 洞察1：知识库的本质是约束层和导航层，不是全量知识

**面试官追问路径**："你这个知识库不是全量的业务知识对吧？有信息丢失的？"→"本质上都是在帮AI做约束"→"帮AI减少上下文，做context管理"。

**价值**：面试官帮助厘清了知识库的准确产品定位——不是"给AI建一个完整知识库"，而是"帮AI筛选信息和做context管理"。下次面试回答知识库设计时，应该说"约束层+导航层"而不是"全量知识库"。

### 洞察2：知识保鲜是核心难题

**面试官拷问**："滴滴什么是L7等级的会员...6月1号做的等级升级...如果知识库里还是之前的等级判断逻辑，这种知识更新不及时在AI coding环节岂不是会带来很大干扰？"

**价值**：暴露了知识库从"构建"到"运营"的核心挑战——冷启动知识天然会过时，必须有回流更新机制。面试中被问到"知识过时怎么办"时，应该主动承认这是核心难题，而非回避。

### 洞察3：PRD上下文窗口的硬限制

**面试官揭示**：现实中PRD差不多2万字，会直接打爆200K上下文窗口。这不是"能不能优化"的问题，而是物理硬限制。

**价值**：解释了为什么"PRD分段摘要+按需检索"的设计是必要的，不是过度工程化。面试中可以用来论证设计决策。

### 洞察4：方案和Claude Code init的差异

**面试官追问**："你觉得你的方案和Claude Code的init的区别是什么？"

**价值**：帮助提炼核心差异化价值——证据链机制+质量门控，而非仅仅是上下文管理。下次面试要主动说清这个差异。

### 洞察5：阿里淘天AI研发全托管体系（最核心洞察）

> 面试官最后分享了阿里淘天完整的AI研发工程化体系，是目前所有面试中分享的最完整的工业级实践。

**面试官描述（ASR清洗后）**：

阿里淘天已经有一套**端到端全托管的需求交付产品**，整体架构如下：

**1. Coding Agent架构**
- 自研的Coding Plan + Coding Agent架构，在云端运行
- 本地开发场景基于Claude Code
- 起点是业务PRD，最终产物直接和线上CI/CD流水线对齐，完成需求托管和自动部署

**2. 前置知识能力**
- 完整的知识图谱能力承接文档和业务知识
- 大量迭代积累的迭代知识库

**3. C端快速交付**
- C端场景大量上新页面时，能快速生成生产可用的代码
- 不是demo级代码，是直接可上线的高质量产出

**4. 自验证+自修复闭环（核心亮点）**
- **Harness能力建设**：AI coding完成后，如何让AI感知自己写的对还是不对？靠大量自动化测试能力做配套
- **云真机环境**：提供真实运行环境验证
- **测试用例PRD驱动生成**：从PRD自动生成测试用例并自动执行
- **运行环境感知 + 修复Agent Loop**：感知运行时错误，自动触发修复Agent，形成"编码→验证→修复"的闭环

**整体架构一句话概括**：PRD → Coding Agent生成代码 → 知识图谱提供上下文 → Harness自动验证 → Agent Loop自修复 → CI/CD自动部署上线。

**价值**：这是大厂AI研发工程化的完整图景。面试中讨论"AI研发的未来"时，可以用这套体系作为愿景来描述——从PRD到上线的全链路自动化，核心不只是代码生成，而是**自验证+自修复的闭环**才是真正的工程化壁垒。我的工作流目前只做到了PRD→Plan这一段，阿里的体系展示了完整链路的可能性。

### 洞察6：小仓库场景下自建知识库的必要性会被挑战

**面试官追问路径**：面试官直接指出"5个页面50个组件的仓库，让AI全仓库扫描一遍或者用Claude Code做一次init，理论上也能达到一样效果，有没有知识库影响会有那么大吗？"

**价值**：这是一个很犀利的问题，说明面试官在评估"方案的增量价值"。下次面试回答这类挑战时，要明确说清两个差异：一是Claude Code init只分析代码结构不理解业务语义（PRD关键词→代码字段的映射），二是直接让AI写开发计划没有证据链约束会幻觉。不能笼统说"提升60%"，要说清楚增量价值来自哪个具体机制。同时要诚实承认——对于小仓库，增量价值确实不如大仓库明显。

### 洞察7：面试官对产物格式和章节的考察目的是验证深度

**面试官追问**：面试官详细追问Spec和Plan的格式、章节组织，候选人说"有点记不太清了"。

**价值**：面试官不是在考记忆力，而是在验证你是否真的深度参与了设计。如果连自己设计的产物格式都说不清楚，面试官会判断这是"纸上谈兵"。下次面试前，一定要把Spec的9个章节和Plan的11个章节的关键名称和内容背熟，至少能准确说出前3-4个章节的名称和作用。
