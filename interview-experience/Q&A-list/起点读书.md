## Q7-Q9: 组件库架构
**核心句**：组件是业务的，引擎是通用的——三层架构
**展开**：组件确实是业务定制化的，但背后的架构是通用的。核心分三层：Schema描述层定义UI结构，联动引擎处理组件间依赖，渲染引擎解析执行。比如动态表格，联动机制是通过dependence声明依赖路径，Form统一监听触发——这套机制是所有组件共用的。

## Q10: XSS安全性
**核心句**：三层防护：组件白名单+内容转义+配置校验
**展开**：安全性是动态下发必须考虑的。我们做了三层：第一，组件白名单机制，Schema只能引用注册过的组件，不能注入任意HTML。第二，富文本内容通过DOMPurify做XSS过滤。第三，动态配置经过后端校验，前端只负责渲染不执行逻辑。

## Q11-Q12: 布局偏移(CLS)
**核心句**：三层防御——预留空间+异步资源占位+加载顺序控制
**展开**：动态组件确实会有CLS问题，核心思路是"让浏览器在渲染前就知道元素尺寸"。三层方案：
第一，预留空间——给每个动态组件配置默认高度，渲染前用骨架屏占位，避免组件加载后把下方内容挤下去。
第二，异步资源占位——图片用`aspect-ratio`属性（如`aspect-ratio: 16/9`）配合`width`属性，浏览器在图片加载前就能算出高度；字体用`font-display: swap`避免FOIT导致的布局跳动。
第三，加载顺序控制——用`content-visibility: auto`让屏幕外组件延迟渲染，按拓扑排序加载有依赖关系的组件。对于营销活动页常见的图片密集场景，还可以用`loading="lazy"`配合显式`width`/`height`属性，既延迟加载又预留空间。

## Q14-Q15: 样式隔离/CSS Modules
**核心句**：样式隔离有三种主流方案，按场景选择
**展开**：样式隔离是动态组件的常见问题，业界有三种主流方案：
第一，CSS Modules——构建时给每个类名加hash后缀（如`.btn`变成`.btn_abc123`），从根本上避免全局冲突。Webpack配置`modules: true`，样式文件命名`.module.less`即可。
第二，CSS-in-JS（如styled-components、Ant Design 5.x的@ant-design/cssinjs）——运行时根据组件props和主题生成唯一hash类名，每个组件实例的样式互不干扰。Ant Design 5从less变量方案迁移到cssinjs，也是为了解决动态主题切换和样式隔离问题。
第三，CSS原生方案——`@layer`声明样式优先级层叠，CSS Shadow DOM做组件级隔离。这是浏览器原生支持的方向。
我们项目用Less，组件间有冲突风险时可以切CSS Modules。如果需要动态主题切换，参考Ant Design 5的cssinjs方案。

## Q18: Rollup双模打包
**核心句**：ESM静态可tree-shake，CJS兼容Node/旧工具链
**展开**：两种格式为了最大化兼容性。ESM的import/export是静态声明，打包工具编译时分析未使用导出直接删掉，减小包体积。CJS的require是动态调用，没法静态分析。但Node和一些旧工具（Jest、老webpack）还在用CJS。双模输出确保组件库任何环境都能用。

**展开**：MCP解决工具和模型的绑定问题。之前每接入一个工具要给每个AI平台单独写适配。MCP定义了标准Client-Server协议，任何模型通过标准接口调用任何MCP Server。和Skill区别：Skill是特定平台定制指令集，MCP是跨平台开放协议，工具一次开发到处接入。

## Q20: 手写get方法
**核心句**：路径分段，reduce迭代，可选链兜底
**展开**：
```js
function get(obj, path, defaultValue) {
  const keys = path.replace(/\[(\d+)\]/g, '.$1').split('.');
  return keys.reduce((acc, key) => acc?.[key], obj) ?? defaultValue;
}
```

## Q21: 手写flatten方法
**核心句**：递归遍历+路径拼接，跳过数组和null
**展开**：
```js
function flatten(obj, prefix = '') {
  return Object.entries(obj).reduce((res, [key, val]) => {
    const path = prefix ? `${prefix}.${key}` : key;
    if (val && typeof val === 'object' && !Array.isArray(val)) {
      Object.assign(res, flatten(val, path));
    } else {
      res[path] = val;
    }
    return res;
  }, {});
}
```

## Q22: CSS一行三个正方形
**核心句**：aspect-ratio主方案，padding-bottom hack做兼容
**展开**：
```css
/* 方案一：aspect-ratio */
.item { width: calc((100% - 2 * 10px) / 3); aspect-ratio: 1; }

/* 方案二：padding-bottom hack（兼容旧浏览器） */
.item { width: calc((100% - 2 * 10px) / 3); height: 0; padding-bottom: calc((100% - 2 * 10px) / 3); }
```
关键：padding-bottom百分比相对于父元素宽度计算。

## 面试官洞察

### 洞察1：Ant Design从5.x起用CSS-in-JS替代less变量——样式隔离的行业趋势

**面试官原话/描述**：讨论样式覆盖问题时，面试官主动补充：“从6开始（Ant Design）已经完全换成了原生的CSS变量”，并解释了css-in-js方案的原理——给每个类名加hash前缀，因为每个组件的样式内容都不一样，所以生成的hash也不同，从根本上杜绝了样式冲突。

**价值**：这是前端样式隔离的行业趋势——从CSS Modules → CSS-in-JS → CSS原生变量（@property）。面试时如果被问样式隔离，可以按演进路线回答：CSS Modules（构建时hash）→ CSS-in-JS如styled-components/Ant Design 5（运行时hash）→ CSS原生层@layer。起点读书面试官对这个话题非常熟悉，说明阅文内部在做组件样式架构升级。

### 洞察2：AI做C端视觉还原只能到70%，中后台已可落地

**面试官原话/描述**：反问环节聊AI实践时，面试官明确说：“偏C端的话对视觉要求比较高，AI做的话可能只能做到70%，尤其是视觉还原还有欠缺。但做中后台，视觉要求不那么高，AI已经做得还可以了。”同时提到团队在做自动化——让AI根据产品需求+设计稿自己生成代码。

**价值**：这是来自阅文/起点内部的真实AI工程化判断。关键洞察：**C端视觉还原是AI当前最大短板，但中后台是AI编码的最佳落地场景**。面试时讨论AI编码落地可以引用这个判断——“行业头部内容平台（阅文）的实践是C端70%、中后台已可用”。

### 洞察3：腾讯内部AI工具生态——自研编辑器+CodeBuddy+扣子

**面试官原话/描述**：面试官分享了腾讯内部的AI工具现状：“腾讯自己搞了个编辑器叫CodeBuddy，强制用这个编辑器”，模型方面GPT、Claude都可以用（三选），但用的是腾讯自研的模型为主。还提到Claude 4.7之后速度明显变慢。

**价值**：了解大厂内部AI工具生态对面试有帮助——腾讯系公司倾向自研AI工具（CodeBuddy、扣子），模型选择上虽然多选但实际推自研。面试腾讯系公司时可以主动问“团队用什么AI编码工具，是自研还是第三方”。

### 洞察4：起点/阅文前端技术全景——PC站老代码+M站+APP内嵌+抖音小程序

**面试官原话/描述**：反问环节面试官完整介绍了前端技术全景：起点中文网PC站（非常老的代码）、M站（移动版）、APP内嵌H5活动页（充值活动等）、抖音小程序（Taro，维护为主）。PC站详情页做过一次Vue改版。

**价值**：了解目标公司的技术全貌有助于面试准备。起点前端覆盖面广（PC/Mobile/H5/小程序），但核心挑战是**老代码维护+新技术栈迁移**。面试时可以强调自己在Schema中台迁移旧系统方面的经验。

### 洞察5：漫剧方向——起点在探索漫画改编短剧

**面试官原话/描述**：候选人问“后面会有扩展短剧方向吗”，面试官回答：“漫剧我们现在有，漫画改编的那种”，确认公司有人在做这个方向。

**价值**：漫剧是当前内容行业的热门赛道（漫画→短剧），起点/阅文正在布局。如果对这个方向有兴趣，面试时可以主动聊漫剧相关的技术挑战（视频播放器、漫画渲染、互动式阅读）。
