## Q2: Schema是自研解析器还是用现成方案？
**核心句**：基于JSON Schema Draft-07扩展，自研渲染引擎+模板引擎
**展开**：我们用的是JSON Schema Draft-07作为基础协议，但在上面扩展了联动描述（dAction DSL）和组件映射。前端有一套渲染引擎解析Schema生成表单，BFF有模板引擎生成配置接口。之所以没有用Ajv这类现成库，是因为我们的场景需要联动逻辑的声明式描述，标准JSON Schema的`$ref`和`if/then`不够用，所以做了DSL层扩展。

## Q6: 工作流除了Skills之外还有什么特点？
**核心句**：证据链+质量门禁，解决AI幻觉和跳步
**展开**：Skills只是载体，核心是两个设计思想。第一，证据链——AI生成的每个结论必须附带源文件引用和行号，让思考路径可追溯、可验证。第二，质量门禁——工作流每一步通过XML格式严格约束输入输出，最后用Python脚本校验中间产物完整性，缺文件或格式错误就触发AI回溯重做。两个机制组合，把AI的幻觉率和跳步率控制在可接受范围。

## Q7: Schema配置怎么防止错配漏配、保证稳定性？
**核心句**：声明式Schema+联动DSL+三层防控
**展开**：预防层：Schema本身是声明式的，组件间依赖关系清晰可见，开发者改之前就能判断影响面。校验层：渲染引擎对Schema做静态校验，联动引用的组件不存在或循环依赖会直接报错。监控层：错误码分类上报（我重构了6类reason tag），前端白屏JS错误也有监控。三层叠加，线上问题从月均20起降到接近零。

## Q8: 怎么预防事故？（追问）
**核心句**：静态校验+灰度发布+自动化回归
**展开**：预防有三个层面。第一，Schema静态校验——渲染引擎在开发阶段就能检测联动引用缺失、类型不匹配、循环依赖。第二，灰度发布——新活动配置先小流量验证再全量。第三，自动化回归——核心活动类型的端到端测试覆盖，配置变更自动跑通主流程。再加上监控层的reason tag分类报警，做到"预防为主、监控兜底"。

## Q9: 手写代码——数组转树
**核心句**：递归filter+map，O(n²)可优化为Map
**展开**：先写递归版本——传入list和pid，filter出所有parentId等于pid的项，然后对每个item递归找children。如果要优化，先用Map按parentId分组，一次遍历建索引，再递归组装，时间复杂度从O(n²)降到O(n)。
```js
function listToTree(list, pid = null) {
  return list
    .filter(item => item.pid === pid)
    .map(item => ({
      ...item,
      children: listToTree(list, item.id)
    }));
}
```
