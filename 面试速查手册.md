## 二、30 秒项目开场白（必背）

> 我在滴滴国际化做营销中台，核心是 **Schema 驱动架构** ——BFF 层用模板引擎驱动 Schema 生成，字段按模板分布式管理，通过 injectDActions 声明式注入联动配置，**一次声明双重求值**：BFF 端 safeEval 设初始状态，序列化 d_actions 给前端运行时联动。联动分 **fetchApi 型**回调 BFF 重生成和 **state 型**前端本地 eval。前端 Schema 渲染引擎拿到后通过 **formatName** 将嵌套表单拍平为 dot 路径，通过 **Listener 发布订阅**执行联动，**35+ 组件**动态渲染。这套 Schema 驱动模式不只是活动表单在用，**权益组件库 benefit SDK** 也是 Schema 消费者——**24 个内置组件**通过 Register 注册，两个消费者共享同一套架构。在此基础上做了 **prd-tools**，用 AI 把 PRD 转成有证据、可执行、可回流的结构化开发计划。

**背诵要点**：5 个加粗关键词 + 三句话逻辑线（Schema 生成 → Schema 渲染 → AI 工程化）

### A2. Schema 驱动架构怎么设计？

```
三层架构：
后端 API → BFF(Schema 生成) → 前端(Schema 渲染)

BFF 核心：模板引擎驱动（不是手写 JSON）
├── 每步骤独立模板（basic/group/rules/message/preview）
├── Builder.parseOnce 编译出 content_schema
├── injectDActions 双重求值（BFF safeEval + 前端 eval）
└── 16 个 linkageSchema（12 重渲染 + 4 Placeholder）

前端核心：Schema 渲染引擎
├── formatName 拍平嵌套路径（rules.reward.0.method）
├── d_actions 声明式联动 + Listener 发布订阅
├── 双轨映射（switch-case 基础 + RegisteredComponents 35+）
└── Redux + Form 实例 + Listener 三层状态
```

**必说**：injectDActions 双重求值 + fetchApi/state 两种联动 + 新增活动类型前端零代码

### A4. AI 工程怎么保证可靠性？

```
多层保证：

## 五、C 组：前端八股速查卡

### C16. React 18 并发 + Fiber + Scheduler

```
Fiber：链表结构（child/sibling/return）→ 可中断增量渲染
Scheduler：MessageChannel（不是 setTimeout 最小4ms）→ 优先级调度
React 18：useTransition + useDeferredValue + Automatic Batching
```

### C17. useEffect vs useLayoutEffect

```
useEffect：DOM 更新后异步执行，不阻塞渲染
useLayoutEffect：DOM 更新后同步执行（绘制前），阻塞渲染
顺序：DOM 更新 → useLayoutEffect → 浏览器绘制 → useEffect

Hook 不能条件用：React 靠调用顺序匹配 state，顺序变就乱了
```

### C18. 浏览器渲染管线

```
HTML → DOM  }  → Render Tree → Layout(Reflow) → Paint(Repaint) → Composite
CSS  → CSSOM }

Reflow：改几何属性（width/height/margin）
Repaint：改视觉属性（color/background）
Composite only：transform/opacity（最佳性能）
```

### C19. HTTP 缓存

```
强缓存（不发请求）：Cache-Control: max-age=xxx
协商缓存（304）：ETag/If-None-Match（精确但贵）> Last-Modified/If-Modified-Since

实战配置：
HTML → no-cache（每次协商）
JS/CSS(带hash) → max-age=31536000（强缓存一年）
API → no-store
```

### C20. TypeScript 高级类型

```
infer：类型推断    type ReturnType<T> = T extends (...args:any[])=>infer R ? R : never
条件类型：三元判断  type IsString<T> = T extends string ? true : false
模板字面量：字符串   type EventName = `on${Capitalize<string>}`
映射类型：遍历      type Pick<T,K> = { [P in K]: T[P] }
```

### D25. 图谱和知识库的关系？

```

```

## 九、高频追问防御

### 万能应答框架

```
1. "这个问题我实际遇到过..." → 锚定实战（不背书）
2. "核心思路是..." → 一句话说清
3. "具体做法是..." → 展开细节
4. "踩过一个坑..." → 真实经验
5. "如果让我重新设计..." → 反思能力
```

### 高频刁钻问题防御

| 刁难问题 | 防御策略 |
|---------|---------|
| "你这个项目有什么不足？" | "eval 安全隐患是新版本的改进点，已经用 Zustand + new Function 替代。另外 reference 构建依赖人工确认，自动化程度可以提高" |
| "如果让你重新设计会改什么？" | "三件事：1. 联动引擎用 JSON Schema 规范替代 eval 2. Reference 加自动化构建 3. BFF 加 Schema diff 可视化" |
| "你的方案和 XX 开源方案比？" | "我们的场景是 23 种活动类型的高频配置，不是通用表单方案。injectDActions 双重求值是针对 BFF-前端协同的特化设计" |
| "为什么不直接用 Formily / FormRender？" | "我们的场景有 BFF 层的联动计算需求，纯前端方案解决不了 SSR 初始状态。另外 23 种活动类型的联动关系跨前后端，需要 BFF 统一编排" |
| "AI 生成的代码你敢上线吗？" | "prd-tools 不生成代码，它产出的是结构化开发计划。AI 的角色是知识管理和需求分析，不是代码生成。最终实施是人做的" |
| "你觉得 AI 会取代前端吗？" | "会取代重复性的 DOM 操作和基础组件开发。但 Schema 架构设计、联动引擎、AI 工程工作流这些需要架构思维的不会被取代。前端工程师的价值在系统设计，不在写代码" |

## 十一、2026 新增必考：SSE 流式 + AI Agent（面试官最爱问）

> 这部分是 2026 年大厂前端 AI 岗**最高频的新考点**，你的竞争对手可能还不知道。字节 AI 方向几乎必问。

### E1. 设计一个支持 AI 流式输出的前端架构

```
面试官会问："设计一个类似 ChatGPT 的流式对话前端，怎么处理逐 Token 输出？"

核心回答框架：
┌────────────────────────────────────────────────────────┐
│ SSE 连接层                                             │
│ ├── EventSource / fetch + ReadableStream               │
│ ├── 断线重连 + Last-Event-ID 消息去重                   │
│ └── 超时处理 + 心跳保活                                 │
├────────────────────────────────────────────────────────┤
│ 流式解析层                                             │
│ ├── 逐 chunk 累积 buffer                               │
│ ├── Markdown 增量解析（不能每次全量 re-parse）            │
│ ├── 代码块检测（```open 时不渲染 markdown）              │
│ └── 公式/表格的特殊处理                                 │
├────────────────────────────────────────────────────────┤
│ 渲染优化层                                             │
│ ├── 虚拟列表（万级 Token 不能全量渲染）                   │
│ ├── requestAnimationFrame 节流渲染                      │
│ ├── useDeferredValue 降低渲染优先级                     │
│ └── Web Worker 做 Markdown 解析不阻塞主线程              │
├────────────────────────────────────────────────────────┤
│ 状态管理层                                             │
│ ├── 消息列表（streaming 状态：loading → streaming → done）│
│ ├── 中断控制（AbortController 取消请求）                 │
│ ├── 重试策略（指数退避）                                 │
│ └── IndexedDB 分段缓存（支持离线续写）                   │
└────────────────────────────────────────────────────────┘
```

**如果你被问到，怎么关联到你的项目**：

> "我的 prd-tools 的 PRD Ingestion 层也处理过类似问题——不是把原始 .docx 直接丢给 LLM，而是先做 MarkItDown 转换 + 质量门禁。如果是流式输出场景，我会在证据链机制基础上加 SSE 连接管理，保证每个流式 chunk 都有证据追溯"

### E2. AI Agent 架构前端怎么实现？

```
面试官会问："前端怎么实现一个 Agent 工作流的编排和执行？"

核心回答框架：
┌────────────────────────────────────────────────────┐
│ Agent 循环（ReAct 范式）                             │
│ Think → Act → Observe → Think → Act → Observe...  │
│                                                    │
│ 前端核心职责：                                      │
│ ├── 展示 Agent 的思考过程（Chain of Thought 可视化） │
│ ├── 工具调用状态展示（哪个工具在执行、参数、结果）    │
│ ├── 执行流程图（节点 + 边 + 条件分支）               │
│ └── 异步执行管理（超时、重试、结果合并）              │
├────────────────────────────────────────────────────┤
│ 状态机设计（推荐 XState）                            │
│ ├── idle → thinking → acting → observing → ...     │
│ ├── 每个状态有明确的 transitions                     │
│ ├── 支持暂停/恢复/回滚                               │
│ └── 错误状态回退到最近的安全点                        │
├────────────────────────────────────────────────────┤
│ MCP (Model Context Protocol) 集成                  │
│ ├── 前端作为 MCP Client 连接多个 MCP Server          │
│ ├── 工具发现 + 参数校验 + 结果展示                    │
│ └── 我的 prd-tools 就是 MCP 协议的实践               │
└────────────────────────────────────────────────────┘
```

**你的关联话术**：

> "prd-tools 的双 Skill 体系（build-reference + prd-distill）本质上就是 Agent 工作流——build-reference 是知识采集 Agent，prd-distill 是分析 Agent，它们通过 reference 这个共享上下文协作。而且 prd-tools 使用了 MCP 协议，就是 Model Context Protocol，前端作为 MCP Client 调用工具"

### E3. 怎么识别 AI 生成的代码问题？

```
面试官会问："给你一段 AI 生成的代码，3 分钟内找到问题"

必须知道的 5 类 AI 代码问题：
┌────────────────────────────────────────────────────┐
│ 1. 幻觉库/幻觉 API                                  │
│    → AI 编造了不存在的库或方法                        │
│    → 检查：npm 查包名是否存在、API 文档是否真实       │
├────────────────────────────────────────────────────┤
│ 2. 依赖注入攻击                                     │
│    → eval() 或 new Function() 使用了未过滤的用户输入 │
│    → 检查：搜索 eval/Function/exec/child_process     │
├────────────────────────────────────────────────────┤
│ 3. 敏感信息泄露                                     │
│    → API Key、Token 硬编码在前端代码中               │
│    → 检查：搜索 key/token/secret/password/credential │
├────────────────────────────────────────────────────┤
│ 4. 逻辑漏洞                                        │
│    → 边界条件未处理、竞态条件、内存泄漏               │
│    → 检查：空值处理、异步并发、事件监听清理            │
├────────────────────────────────────────────────────┤
│ 5. 权限绕过                                        │
│    → 前端做了鉴权但后端没做、IDOR 漏洞                │
│    → 检查：API 调用是否有鉴权、是否只前端校验          │
└────────────────────────────────────────────────────┘
```

**你的关联话术**：

> "prd-tools 的证据链机制就是在解决 AI 产出可靠性的问题——我们不相信 AI 的输出，要求每个结论都有可追溯的证据。这和识别 AI 生成代码问题的思路一致：不信任，要验证"

### E4. 前端会被 AI 取代吗？（必问的观点题）

```
回答框架（展现思考深度）：

"会取代的部分："
├── 重复性的 DOM 操作和基础组件开发（已经是事实）
├── 常规的 CRUD 页面（Vibe Coding 正在解决）
└── 简单的状态管理和数据绑定

"不会被取代的部分："
├── Schema 架构设计（需要理解业务本质做抽象）
├── 联动引擎设计（需要理解前后端协同的约束）
├── AI 工程工作流（需要设计证据链和质量门控）
└── 系统级架构决策（需要权衡 trade-off）

"前端工程师的转型方向："
├── 从写代码 → 设计 AI 工作流
├── 从 DOM 工程师 → AI 产品架构师
└── 核心竞争力从"会写代码"变成"能设计 AI 解决不了的问题"
```
