## P03 痛点页 · 🔴必讲 · 3min

这一页是"为什么做"。

我们在座很多人用过 AI 做 PRD 分析。先看上面的工作流条——读 PRD、搜代码、找接口、问同事、写计划、开发——其中**搜代码和找接口是最耗时的**，也是 AI 最能帮忙的环节。

但使用 AI 工具做需求分析有 5 个结构性问题，我重点讲两个：

**第 3 个"错误传染"**——AI 一开始判断错了一个文件路径，后面所有建议都基于这个错误展开。比如它说"改 CampaignService.ts 的第 142 行"，你按它改完，发现那个文件根本不存在这个需求相关的逻辑——但 AI 已经基于这个判断生成了一整套改动方案，看起来完整，底层已经错了。

**第 4 个"AI 幻觉"**——AI 会编造不存在的文件路径和函数名。你问它"这个需求要改哪些文件"，它给你一串路径，看起来很专业，但你一 grep 发现一半不存在。

下面这个恶性循环图说明这 5 个问题是互相强化的——上下文不够导致 AI 猜测，猜测导致编造，编造导致错误传染，最终人得重做，做完知识也没沉淀，下次再来一遍。

**所以问题的本质是：不是 AI 不聪明，而是我们缺少一套机制来约束 AI 的行为、追溯它的结论。**还有一个容易被忽略的问题——AI 有强烈的"完成倾向"，为了完成任务会添油加醋，明明不确定的东西也说得言之凿凿。这恰恰说明，更需要证据链来兜底。

## P04 定位页 · 🔴必讲 · 2.5min

prd-tools 的定位是三个字——**可验证**。

不是让 AI 更聪明，而是给现有 AI 工具补上缺失的一层。这页 PPT 三栏对比——**Superpowers 管纪律**，强制 AI 分阶段执行，但 plan 基于推理，幻觉逐级传染；**OpenSpec 管规范**，结构化 spec 管理、delta 追踪，但 spec 不锚定源码，不知道哪些文件受影响。两者都有价值，但都缺了一样东西——**证据**。

**prd-tools 补的就是证据层**——源码搜索到证据锚定到质量门控，每条结论追溯到源码文件和行号。三个工具互补不竞争：Superpowers 写代码之前，prd-tools 先搜源码再给 plan，让纪律基于真相而非猜测；OpenSpec 维护 spec 之前，prd-tools 先锚定受影响文件，让规范连到源码而非悬空。

具体怎么做？三样东西：**证据系统**——每条结论必须标注"PRD 哪一页 + 代码哪一行"。有个反直觉的设计：搜不到也是证据，AI 搜不到某个功能会显式标记"搜不到"而不是假装找到了。**项目记忆**——6 个 YAML 文件作为单一事实源，关掉对话再开，reference 还在。**质量门控**——分步检查，不让错误往后传，特别是 Step 7 硬停止，report 写完必须你确认了才能继续写 plan。

两个插件的关系：reference 负责构建项目记忆，prd-distill 负责蒸馏 PRD，中间的箭头是经验回流——每完成一个 PRD，新发现写回 reference，下次更准。

## P05 架构页 · 🔴必讲 · 2min

整个工具分三层，每层职责很清晰。

**Layer 1 Agent 层**——就是 Claude Code 运行时，读 Markdown 规程文件来控制 AI 行为。里面有 5 层编排——SKILL.md 管入口、workflow.md 管顺序、steps 管约束、references 管知识、templates 管格式。**核心思路是用 Markdown 规程约束 AI 行为，而不是靠 AI 自觉。**

**Layer 2 脚本层**——三个 Python 脚本，不依赖 AI，跑一万次结果一样。build-index 扫源码建知识图谱，context-pack 做查询匹配，quality-gate 做质量检查。

**Layer 3 文件层**——所有状态通过 YAML 和 Markdown 文件持久化，没有数据库，没有后端服务。

底部这句是关键：**智能靠 Agent 推理，可靠靠脚本确定性计算，状态靠文件持久化。零基础设施——只需要 Claude Code + Python3。**

## P06 Reference 五层编排 · 🔴必讲 · 2min

这页讲 reference 插件的编排机制。5 层设计是整个工具最核心的架构决策。

大家看这 5 层，从上到下是 **SKILL → workflow → steps → references → templates**。

打个类比——SKILL.md 相当于函数签名，定义什么时候该跑、读什么写什么；workflow.md 相当于调用图，定义执行顺序和门控；steps 相当于函数体，写清楚每步能干什么不能干什么；references 相当于库文档，放格式规范和领域知识；templates 相当于类型定义，定义每个 YAML 文件的结构边界。

左下是 6 个 YAML 文件，每个只负责一个维度——代码事实进 01、编码规则进 02、跨层契约进 03、路由打法进 04、领域术语进 05。

右下是消费关系——reference 建好之后，prd-distill 消费全部 5 个 YAML + index，不需要再读源码。01 提供代码地图，02 提供 fatal 规则检查，03 提供契约基线，04 提供路由和打法，05 提供术语和领域知识。注意 04 和 05 容易混淆——04 回答"该去哪改"，是路由+打法手册；05 回答"这个词什么意思"，是领域知识词典。

## P07 Mode A 六步构建 · 🔴必讲 · 2.5min

Mode A 是全量构建 reference 的 6 个步骤。

Step 1 扫目录结构，Step 2 读历史 PRD 提取术语映射，这两步是准备。从 Step 3 开始是核心。

我重点讲 **Step 3 能力面映射**——这是 reference 不硬编码路径的关键。我们定义了 29 个能力面维度（前端 9 + Node.js 服务 10 + 后端 10），AI 先识别需求命中哪些能力面，再按能力面定位候选文件。注意是"候选"——还不确定。

**Step 4 读源码验证**——对每个候选文件 Read 源码确认，从 candidate 变成 verified code_anchor。这步最耗时但不可跳过。

**Step 5 写 6 个 YAML**——关键设计是 SSOT：每个事实只存在一个地方。契约字段只进 03，编码规则只进 02，跨文件用 ID 引用。好处是更新改一处就行。

**Step 6 build-index**——确定性脚本建索引，跑一万次结果一样。

底部总结：6 步的关键不是"AI 推理出了什么"，而是"每步产出什么文件、受什么约束"。顺便提一句，为什么用 YAML 不用 JSON——实测 LLM 输出准确率 YAML 62% > Markdown 54% > JSON 50%，YAML 的缩进结构对 LLM 更友好。

> → "六步里最关键的是 Step 3——能力面映射。"

## P08 能力面 · 🟡选讲 · 1.5min

这页讲能力面（capability surfaces）的设计动机和具体内容。

**设计动机**——为什么需要能力面？因为代码文件路径会变，但业务能力的边界相对稳定。能力面是对"这个项目能做什么"的抽象层，让 reference 不依赖具体的文件路径，而是依赖业务能力维度。

**工作案例**——油站新司机需求，通过能力面定位追踪过程：需求先命中 Node.js 服务层的"活动注册"能力面，再从能力面定位到具体文件。这就是能力面的价值——路径是候选，能力面是搜索方向。

⚪可跳过部分：设计灵感来源（Clean Architecture / GoF Adapter / Pact 契约测试），口头一句"借鉴了 Clean Architecture 的稳定依赖原则"即可。

底部：**能力面是"需求语言"到"代码语言"的翻译层，让定位过程不受文件重命名和目录调整的影响。**

## P10 蒸馏流水线 + 编排 · 🔴必讲 · 3min

进入第二部分——prd-distill 蒸馏流水线。

上半部分是 4 层编排——和 reference 共用同一套机制，但蒸馏只有 4 层（没有 templates 目录，格式规范直接写在 references/output-contracts 里）。

下半部分是 11 步流水线。看 U 型流程条——Step 1 到 6 自动执行，**Step 7 是硬停止**（橙色高亮带暂停符号），Step 8 到 11 确认后执行。

流水线里嵌入了 **7 个质量检查点层层递进**——Step 1 后双检查（PRD 解析质量 + block 覆盖率），每步后 Agent 自检，Step 7 人审硬停，Step 9 Readiness 自评，Step 11 Final QG 确定性评分。**错误越早拦截成本越低。**

下面 3 张大卡片分三个阶段：
- 理解 PRD（Step 1-3）——类比编译器前端，把自然语言编译成需求 IR
- 定位影响（Step 4-5）——三层代码搜索 + 跨层契约差异
- 生成方案（Step 6-11）——report → 人确认 → plan → 打分 → 回流 → 终检

底部洞察：**11 步不是线性流水线——7 个门控层层递进，Step 7 是最关键的硬断点。理解 PRD 和写方案之间必须有人确认。**

## P11 需求 IR · 🔴必讲 · 2min

需求 IR 是整个蒸馏的起点。类比编译器——编译器不会跳过 IR 直接生成机器码，prd-tools 也不会跳过 IR 直接生成 plan。所以 IR 不是"多此一举"，而是**质量上限的保障**。

左上 IR schema 核心字段——最关键的是 **change_type: ADD | MODIFY | DELETE | NO_CHANGE**。

右上对比——传统做法是 AI 觉得不相关就静默跳过，你不知道它漏了什么。我们的做法是 **NO_CHANGE 显式标记**——AI 觉得不相关也必须标记出来，你可以在 report 里检查它是不是真的不相关。

中间证据链——4 个来源：PRD 页码 → 文档结构解析 → 经验库命中 → 源码搜索结果。每条结论都标注来源，你可以去验证。

PPT 中间还有三条保真度原则——主输入始终是原始 PRD 不是 AI 重述版；先证据后结论，不凭空推断"应该改这个文件"；宁可报 fail 也不让产物偏离原文。

底部的消融实验数据也值得提——去除 IR 层，直接 PRD 到 plan，readiness 从 82 降到 54，错误率翻倍。**跳过"编译"步骤的代价是很具体的。**

> → "IR 建好了，接下来最关键的一步——在代码里找到它。"

## P12 代码搜索 · 🔴必讲 · 3min

代码搜索是蒸馏里最关键的步骤，对应流水线的 Step 4 和 Step 5，两步紧密关联但职责不同——Step 4 找代码，Step 5 查接口。

**Step 4 代码搜索——三层递进：**

**层 1 查经验**——先看 reference 里的路由记录，"上次改 Campaign 类需求，去了这些文件"。有记忆的搜索，不是从零开始猜。消费 04-routing-playbooks 的 prd_routing 字段。

**层 2 精确匹配**——PRD 说"冲单奖"，术语对照表翻译成 CampaignType.Order，再按符号名精确搜索，不是全文 grep。比如 PRD 写"完单油站券"，domain 桥接成 CompleteOrderGas，再从 index 里找到 `enum:campaignType.ts:CompleteOrderGas:42`。消费 05-domain 术语表 + build-index 倒排索引。

**层 3 读源码确认**——打开文件确认候选是否真的相关，从 candidate 变成 verified code_anchor。不确定的标注"低置信度"。这一步最耗时但不可跳过。

PPT 左下红色区域讲一个重要设计——**搜不到也是证据**。代码里不存在某功能，证明需要新增。这防止了 AI 编造路径。比如 AI 说"改 CampaignService.ts:142"，一搜不存在，直接拦截。

**Step 5 契约分析——跨层接口差异追踪：**

代码搜索完成后，拿 03-contracts 的字段基线做对比，产出 contract-delta。每个接口变更记录三件事：谁负责、谁依赖、对齐了吗。4 个状态——✓ aligned、⚠ 待确认、✗ 阻塞、— 不涉及。举例：dlp_level 字段，后端已确认 → Node.js 已确认 → 前端未确认 ⚠。

PPT 中间还有 Domain Bridge 实例——一次术语命中自动串起 4 条锚点：PRD 提取关键词"可选择奖" → 05-domain 术语表对应 CampaignType 枚举 → inverted-index 打分得到候选文件 → Read 源码锁定 code_anchor。**没有 Domain Bridge 的话，grep "可选择奖" 零结果，AI 只能猜。**

底部对比——没有这三层搜索：grep "campaign" 出 200+ 结果，分不清哪些要改；有了三层搜索 + 经验路由：14 个精准文件 + 接口差异全覆盖。**PRD 的中文术语和代码的英文标识符之间有鸿沟，术语对照表就是这个鸿沟上的桥。**

> → "找到了还不够，还得验证找得对不对——Quality Gate。"

## P13 Quality Gate 评分 · 🟡选讲 · 2min

这页聚焦 Quality Gate 的评分公式和判定逻辑。蒸馏链路有两套自动评分系统，**互补不替代**：

**Readiness Score**（Step 9，LLM 自评）：评语义——需求理解得对不对、影响面找全了没。5 维加权出 0-100 分：证据覆盖、契约对齐、PRD 保真、代码定位、任务落地。

1. **evidence_coverage（25%）**——证据够不够。每条需求有没有足够证据支撑。P0 需求缺证据或验收标准标了 not_testable，直接拉分。权重最高，因为证据是一切结论的根基。
2. **contract_alignment（25%）**——接口对齐没。跨层契约有多少 aligned / needs_confirmation / blocked。并列最高权重，因为跨层契约是团队协作的核心风险。任一 P0 契约 blocked 则整份 readiness 直接 fail。
3. **prd_ingestion（20%）**——PRD 读懂没。文档解析质量、图片表格提取是否完整、有没有转换警告。PRD 都没读懂，后面的分析全白做。
4. **code_search（15%）**——代码找到没。MODIFY/DELETE 类型的改动有没有 code_anchor，搜索未命中的比例高不高。
5. **task_executability（15%）**——方案能落地吗。文件路径有没有精确到行号、验证命令完不完整、checklist 格式到不到位。

**Final Quality Gate**（Step 11，确定性脚本）：评结构——产物合不合格。脚本正则扫文件，跑一万次结果一样。5 维：锚点覆盖、方案可执行、文件齐全、搜索真实、阻塞项质量。

1. **code_anchor_cover（25%）**——锚点覆盖。改动有没有定位到文件路径 + 行号。权重最高，因为没有锚点等于没有证据。
2. **plan_actionability（25%）**——方案可执行。plan 有没有 checklist、filepath、验收标准、分阶段。并列最高权重，因为 plan 最终要交给开发执行，可执行性是底线。
3. **required_files（20%）**——文件齐全。report.md、plan.md、context/ 目录是否齐备。文件不全说明流程有跳步。
4. **context_pack_used（15%）**——搜索真实。code_anchor 是不是真的来自搜索结果，而不是 AI 编造的。防止 AI 跳过搜索直接写结论。
5. **blocker_quality（15%）**——阻塞项质量。"这个接口可能有问题"和"dlp_level 字段后端已确认、前端未确认，需 @前端owner 对齐"——质量差很远。

**三个阈值**（两套共用）：
- ≥85：ready_for_dev，可直接开发
- 60-84：needs_owner_confirm，需 owner 确认
- <60：blocked，必须回去补分析

还有**硬失败规则**——不管总分多高，只要命中一条就直接 blocked：PRD 解析失败、P0 契约阻塞（影响核心功能的跨层字段未对齐）、多层契约分析缺失。有些底线不能靠分数糊弄过去。

一句话：**Readiness 测"分析对不对"（语义），Final QG 测"产物齐不齐"（结构）。两者互补，不替代。**

## P14 report.md 九章 · 🟡选讲 · 1.5min

这页展示 report.md 的 9 个章节卡片。

**第一行 5 个章节**：§1 摘要（30 秒决策：做什么、影响多大、风险）、§2 PRD 过滤声明、§3 需求分解（需求 IR + change_type 标记）、§4 业务规则、§5 影响范围（代码锚点 + 改动点列表）。

## P16 案例实测 · 🟡选讲 · 2min

三个真实项目跑了完整蒸馏，拿到三个 readiness 分数——先解释下这个分数是什么意思。

**Readiness 是蒸馏产物的综合质量分**，0 到 100，三个阈值：≥85 代表 ready_for_dev，可以直接开发；60 到 84 是 needs_owner_confirm，需要 owner 确认几个阻塞项；低于 60 就 blocked，必须回去补分析。评的不是"AI 判断对不对"，而是"证据覆盖全不全、代码锚点有没有、plan 能不能直接执行"——是结构完整度，不是语义正确性。

看三列结果。**Node.js 单仓 82 分**——14 个改动点全部定位到函数级，0 个待搜索锚点，IMP-009 发现了一个关键路径问题：afterAll 只支持单字段返回，但需求要双字段 object。这个阻塞项是 AI 主动发现的，不是人审时才看到。**前端单仓 83 分**——最干净的一个，直接 ready_for_dev。两个亮点：PRD 里 5 个章节工具自动排除 3 个不归前端管（C端交互归外卖、CGC 归独立后台、开奖归独立后台），不是跳过而是显式标注 NO_CHANGE；负向搜索 5 个关键词全部 0 hit，证明功能确实需要从零新建。**团队模式 62 分**——needs_owner_confirm，主要拖累是契约对齐维度只有 45，因为团队仓只读 reference 不读源码，12 条 delta 只能标"待确认"。还有个有趣发现——工具自动检测到 PRD 内部冲突：订单量范围 §3.1 写的 1-10，§3.3 写的 1-99，矛盾自动标为阻塞项。

**效率对比**——传统人工做需求分析 2-4 小时，prd-tools 蒸馏一次 30 分钟左右；人工准确率大约 50%，prd-tools 结构评分稳定在 80 以上；人工做一次知识不沉淀，下次从零开始，prd-tools 每次蒸馏回流让 reference 越来越准。20 分钟首次投入，换回来的是每次蒸馏节省 2-3 小时。

## P18 团队案例详解 · 🔴必讲 · 2.5min

油站新司机三仓联调——同一个 PRD，三个仓库各自有 reference，团队仓聚合 18 个文件（3x6）。

底部：**团队仓看到的是二手知识（reference），不是一手代码。reference 质量决定团队模式的天花板。**

## P19 展望 · 🔴必讲 · 1.5min

三段式管道——Prompt-to-PRD（规划中）→ PRD-to-Code（已完成）→ 研发实施（已完成）。今天分享的是中间这段。

**Prompt-to-PRD（规划中）**——PM 写 PRD 三大痛点：缺少历史上下文、业务规则遗漏、缺少技术细节。输入可以是一句话需求、竞品截图、历史 PRD 片段、会议记录，输出是可评审的 PRD 草稿，而且生成的 PRD 可以直接进入 `/prd-distill`。人机协同：PM 改一版 → 再蒸馏 → 差异进 reference 回流，和现有的 Mode E 飞轮一致。

**Plan-to-ExecPlan（探索中）**——plan.md 的每个 IMP → exec-plan.md 的具体开发步骤，自动生成代码骨架 + 单测 + PR 描述。类比：从建筑蓝图到施工说明书——plan.md 告诉你"盖几层、用什么材料"，exec-plan.md 告诉你"第 1 天浇地基、第 2 天砌墙"。

底部是双轮飞轮——PRD 作为唯一业务 SSOT：Prompt / 想法 → 人审 PRD → PRD-to-Code 蒸馏 + 人审 → 研发实施 → reference 回流，循环往复。

三个"不是"定位澄清——**不是让 AI 直接写业务代码上线，不是取代 PM 写 PRD，不是一次做完。**AI 辅助规划 + 证据，人审人写代码。

> → "总结一下，三个关键词。"

## Q&A 速查卡片

> 现场按需选用。围绕 PPT 内容和 prd-tools 本身回答，不发散。

### T1 必备（15 个，高概率被问）

**T1-1: prompt 驱动靠谱吗？AI 不听话怎么办？**
三层防跳步，不是靠 AI 自觉：第一，每个 step 文件的 XML 状态头写死了当前能做什么不能做什么。第二，Step 7 硬停等人确认。第三，每步结束有机器可验证和人类判断两类检查清单。历史上试过脚本硬门控，scripts 膨胀到 19 个文件 9666 行，最终回归 prompt 驱动加最小化确定性脚本。本质是智能靠 Agent，可靠靠脚本确定性计算。

**T1-2: 为什么不用 LangGraph / CrewAI？**
三句话。第一，prd-tools 跑在 Claude Code 内部，已有 LLM 运行时，再架 LangGraph 等于 Docker 里装 Docker。第二，LangGraph 需要管 Python 运行时、State 持久化、API 调用编排，我们利用 Claude Code 自身处理这些，零部署。第三，改工作流等于改 Markdown，非工程师也能 review。

**T1-3: 和 Cursor / Copilot 有什么区别？**
Cursor 解决写代码，prd-tools 解决写代码之前。Cursor 有五个结构性缺失：没有持久记忆、没有过程产物、没有证据链、没有硬停节点、没有知识回流。prd-tools 的产出 plan.md 可以直接喂给 Cursor 执行——我们负责想清楚，Cursor 负责写代码。互补不是竞争。

**T1-4: 和 Superpowers / OpenSpec 有什么本质区别？**
三个工具解决不同层面。**Superpowers**（纪律层）强制 AI 分阶段执行，但 plan 基于推理——如果 brainstorm 阶段幻觉了文件路径，后续全错。**OpenSpec**（规范层）管理结构化 spec + delta 追踪，但不搜索代码库，spec 不锚定源码。**prd-tools**（证据层）唯一做源码搜索 + 证据锚定——plan 里每条标注"PRD 第几页 + 代码第几行"。三层互补不竞争。

**T1-5: 准确率 80% 怎么算的？还有 20% 呢？**
80% 是产物形态评分（readiness），衡量证据覆盖度、代码锚点定位、契约对齐等结构性指标，不保证每个判断都正确，需要人审。20% 缺口主要是三块：复杂业务逻辑隐性规则推断不全、跨仓契约团队模式只有 45% 对齐、历史上没出现过的需求类型 playbook 覆盖不到。两个硬停节点兜底：Step 7 人审 + quality-gate 低分不放行。

**T1-6: evidence 层就错了怎么办？**
证据链保证可追溯性，不是正确性。AI 读错代码 evidence 确实会错。但价值在于错误可低成本发现：report 每条结论标注了 PRD 哪页+代码哪行，你可以去验证。没有证据链时 review 是重新分析整个 PRD；有证据链时变成抽查几条 evidence_id。证据链不消灭错误，但把发现错误的成本从重做降到抽查。

**T1-7: 20 分钟建 reference 值得吗？**
20 分钟是一次性投入。之后 Mode B 增量更新只要 2 分钟。飞轮机制：每次蒸馏完 Mode E 回流把新发现写回 reference。用得越多 reference 越准，构建成本越低。

**T1-8: 最小启动成本是什么？**
五步约 30 分钟：install.sh 安装 → prd-docs/ 放 2-3 个历史 PRD → `/reference F` 收集上下文（5min）→ `/reference A` 全量构建（20min）→ `/prd-distill` 跑一次蒸馏。拿到 report+plan 后评估质量决定是否继续。非前端/Node.js/Go 技术栈需额外 10-15 分钟定义能力面维度。

**T1-9: Readiness 和 Final QG 有什么区别？**
两个完全不同的评分系统。Readiness Score（Step 9，LLM 自评）评语义——需求理解得对不对、影响面找全了没。Final Quality Gate（Step 11，确定性脚本）评结构——文件齐不齐、锚点用了没、plan 有没有 checklist。一句话：**Readiness 测"分析对不对"，Final QG 测"产物齐不齐"。两者互补不替代。**

**T1-10: plan 拿给开发说"不对"，还有价值吗？**
即使 plan 不完美，report 的四块价值依然在：§5 影响范围（14 个改动点定位到文件+行号）、§7 Open Questions（阻塞项提前暴露）、§8 契约风险（跨层接口差异清单）、plan 的 §5 QA 矩阵（冒烟+回归用例）。prd-tools 的价值不是生成完美方案，而是让开发不用从零开始。即使调整 30% 的 plan，也比重写一份技术方案快得多。

**T1-11: 为什么只能在 Claude Code 上跑？**
三个层面。**第一，插件机制**：prd-tools 用 `.claude-plugin/plugin.json` + `SKILL.md` 自动发现注册，输入 `/prd-distill` 自动加载 4 层编排。Cursor 有 Rules 但没有 Skill 自动加载。**第二，运行时**：prd-tools 的"代码"是 600+ 行 Markdown 指令，LLM 就是执行引擎。不是"应用跑在 Claude 上"，而是"Markdown 编排了 Claude 的行为"。**第三，工具依赖**：Read 工具原生多模态看图 + 300+ 行指令需要长上下文精确遵循。迁移到其他工具是重写不是移植，Python 脚本层可直接复用。

**T1-12: 代码安全怎么保证？**
prd-tools 只通过 Claude Code 发送代码片段（Read 工具），不调用外部服务、不上传第三方平台。Python 脚本全部本地运行。不引入新的数据暴露面——用的是 Claude Code 已有的代码读取能力。如果公司允许用 Claude Code 辅助编码，prd-tools 在同一个安全边界内运作。

**T1-13: report 为什么不叫 spec？**
老板的直觉是对的。report.md 实际角色是技术规格书——定义"做什么"，plan.md 定义"怎么做"。report 是 plan 的约束：plan 只能实现 report 里确认的范围。现在叫 report 是历史原因，后续会考虑重命名 spec.md。

**T1-14: 图片解析不可靠怎么办？**
已知限制。图片分析走 Claude 多模态（不是 OCR——OCR 只能抠文字，看不出布局和交互语义），置信度硬顶 medium。对前端确实不够，改进方向：图片分类差异化 + 图文交叉验证 + Reference 组件库锚定。v3 方向。

**T1-15: 蒸馏一次花多少钱？**
中等 PRD 约 ¥10-18/次（$1.5-2.5），简单 ¥5-8，复杂 ¥20-35。对比人工 2-4 小时的时薪。reference 构建 Mode A 约 $1-1.5，Mode B 增量约 $0.2-0.3。
