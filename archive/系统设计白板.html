为什么不用纯前端表单库？核心问题不是"怎么画表单"，而是 BFF 和前端怎么共同编排活动规则。很多字段的初始显隐、选项和联动依赖后端数据，必须由 BFF 参与生成。injectDActions 的双重求值就是为了解决跨层联动：BFF 保证初始状态，前端保证运行时体验。
injectDActions 双重求值是什么？一次声明联动配置，BFF 端 safeEval 求初始状态（SSR 就是对的），同时序列化 d_actions 给前端运行时再次求值。联动分 fetchApi 型（回调 BFF 重生成）和 state 型（前端本地 eval），避免前后端各写一套联动逻辑。
性能怎么保证？按字段依赖局部触发，不全表单刷新。state 型联动前端本地 eval 不走网络。formatName 拍平路径做 listener 事件名，发布订阅按需触发，只有依赖该字段的组件才会重新渲染。
eval 安全怎么办？承认是历史技术债，不包装成最佳实践。实际处理：限制表达式来源、限制可访问变量、避免用户输入直接进入表达式。重构方向是用受限 DSL 或 JSON 规则描述联动，再由解释器执行。
怎么保证 Schema 正确？联动模板从 render 模板提取 handler 复用，保证联动返回与初始渲染的 Schema 结构一致。16 个 linkageSchema 文件按维度拆分，每新增活动类型只需加模板文件和联动文件，前端零改动。后续加强 Schema diff 和类型生成。
怎么扩展新活动类型？加枚举值 + 在 rules/details/ 加规则模板 + 配 linkageSchema 联动文件。联动模板从 render 模板提取 handler 复用，保证结构一致。前端通过 Schema 渲染引擎自动适配，不需要为新活动类型写页面代码。
核心是 Register 扩展、Schema 同构、打包发版治理。

怎么避免依赖爆炸？peerDependencies 声明 react/react-dom，避免重复打包。Rollup 打包时 external 化设计系统库。resolutions 强制统一 @types/react 版本，避免多个版本导致类型冲突。
怎么支持业务自定义？Register 模式：构造时注册 24 个内置组件（通用 16 + 业务线 8），业务通过 add()/addMultipleCmps() 运行时注册自定义权益组件，不侵入 SDK 核心代码。
Schema 渲染怎么和活动表单复用？SDK 本身也是 Schema 驱动渲染的消费者——后端返回 form_data 含 d_component 和 content_schema，前端用 FieldWrapper 迭代渲染，和 营销平台 完全同构。两个消费者共享一个架构。
怎么发版？Changesets 管理版本：开发者选变更包 + 填描述，changeset version 自动 bump 版本号和 CHANGELOG。先 build+yalc 本地联调确认没问题，再发 alpha 联调，最后正式 release。zx 脚本自动化流程。
怎么回滚？锁版本到上一 release，业务侧灰度控制。yalc 本地联调可以在发版前发现问题，alpha 环境联调确认后才能进正式版。
业务包独立发版，共享包统一复用，构建只跑受影响部分。

为什么 Lerna + Nx？历史选型（项目启动时 Turborepo 还不够成熟）+ Lerna independent 支持业务模块独立版本避免全量发版 + Nx 做构建缓存只跑受影响部分。Yarn workspace 做依赖链接。
怎么避免包互相污染？包边界清晰：app/ 放业务模块，packages/ 放共享组件和工具。子包显式声明 workspace 协议引用。根目录统一 ESLint + Prettier + Husky + commitlint，子包继承，CI 跑 lint 和类型检查。
构建慢怎么办？Nx affected 只构建变更影响范围。构建缓存避免重复编译。共享组件抽离到 packages/ 减少重复编译。支持并行构建。踩过 @yarnpkg/parsers 升级不兼容 node16 的坑，所以锁版本是硬要求。
依赖冲突怎么办？强制用 yarn 不用 npm，yarn.lock 严格锁定版本。resolutions 强制统一冲突依赖（如 @types/react 统一 ^18.0.0）。peerDependencies 处理包间共享的大型依赖。CI 用 --no-lockfile 避免锁文件冲突。
怎么做本地联调？组件库 build+yalc 推到本地仓库，目标项目 yalc add 拉取本地版本。避免 npm link 的符号链接问题（符号链接会导致 react 多实例）。

为什么不用向量库？向量适合模糊召回，不适合做权威事实源。BM25 倒排索引做精确检索，可审计、可重现、支持增量构建。向量搜索和 BM25 可以共存，但 reference 作为 SSOT 必须是精确匹配。
知识过期怎么办？三层机制：被动监控（last_verified 14 天过期 + B2 健康检查对比源码发现矛盾）、主动回流（蒸馏发现矛盾生成 reference-update-suggestions.yaml + 人工确认后更新）、守门（质量门控检查 enum/field/contract 与源码冲突为致命项不通过）。
AI 错了怎么办？多层兜底：证据链（8 种证据类型，搜不到也是 negative_code_search 证据）+ 代码锚定（build-index 精准匹配到代码位置，context-pack 按 must/should/optional 分层）+ Report Review Gate 硬停止（用户确认报告后才能生成 plan）+ 反馈回流闭环让知识库越用越准。
怎么量化收益？目前强调工程化原型和方法论。quality-gate.py final 做 5 项加权评分（代码锚点覆盖率 25% + 计划可执行性 25% + 必需文件 20% + context pack 消耗 15% + 阻塞项质量 15%）。benchmark 用 oracle 用例跑完整蒸馏链路，需求召回 35 分 + 代码锚点准确率 30 分，总分低了说明退步。
Evidence Index 为什么用正则不用 AST？零依赖（Python 标准库即够）、跨语言（TS/JS/Go 统一处理，AST 通常绑定特定语言）、文件哈希增量构建快（只重建变化部分）。AST 精度更高但引入 tree-sitter 等重依赖，当前规模下正则够用。
团队模式怎么工作的？T 模式将成员仓库 reference 聚合到团队仓库，跨仓库公共知识只维护一份。T2 模式反向继承团队知识到成员仓库。build-index 和 context-pack 都支持多仓库索引加载，团队级 reference 可以覆盖多个子项目。
PRD 蒸馏最难的一步是什么？变更分类（ADD/MODIFY/DELETE）。需要理解项目现状 + PRD 描述可能模糊 + 跨模块影响。解法：reference 路由信号快速定位 + Evidence Index BM25 精准搜索 + 负向搜索（搜不到也是证据）+ 置信度分级（low 强制进入 questions.md）。
图谱和知识库的关系？图谱是原始发现层，reference 是精选后的企业知识库。GitNexus 提供代码结构，Graphify 提供业务语义，prd-tools 做证据治理。图谱结论不能直接用，需要精选、审计、确认后才能进入 reference。图谱不可用时回退到 rg/glob 源码扫描。

问题回答方向
SSE vs WebSocket？SSE 是单向服务器推送，协议简单（HTTP + text/event-stream），浏览器原生支持自动重连，适合 AI 流式输出。WebSocket 是双向实时通信，适合多人协作编辑场景。单向流式输出用 SSE 更轻量，不需要 WebSocket 的双向能力。
Markdown 卡怎么办？不能每个 token 都全量 re-render。用 requestAnimationFrame 节流渲染频率，虚拟列表处理长对话，增量 Markdown 解析只处理新增部分。Worker 线程做解析不阻塞主线程。代码块未闭合时要做渲染保护。
断线怎么办？EventSource 自动重连 + Last-Event-ID 告诉服务器从哪续传。解析层按 chunk 维护 buffer 处理去重。网络恢复后重建状态，不丢消息。超时和取消也要处理。
Tool call 怎么展示？把 Agent 过程显式状态化：conversation → message → run → step → tool call，每步都有状态（streaming/waiting_tool/executing/done/failed）。工具调用展示输入输出和耗时，敏感操作需要用户二次确认。和 prd-tools 类似——每个结论都要可追溯。
和 Agent 架构怎么衔接？流式输出是 Agent 过程的一部分。前端要把执行过程状态化——不能只展示最终答案，tool call 的输入输出、权限、耗时、错误都要展示。核心模型包括 conversation、message、run、step，每一步都有状态机管理。
