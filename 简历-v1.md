## 项目经历

### prd-tools — PRD 工程工作流（AI 方向）

*独立设计 + 开发 · 已落地生产使用*

**背景**：B 端业务中 PRD 到代码的核心难点不在"写几行代码"——PRD 业务词和代码字段不一致、前后端对同一字段认知分歧、同类需求反复返工但历史经验没有沉淀。需要一个工程化的方案把 PRD 变成可执行、可追溯的开发计划。

**设计**：

- **双 Skill 体系**：build-reference 构建 `_reference/` 领域知识库（6 种模式：全量构建/增量更新/健康检查/质量门控/反馈回流），prd-distill 消费 reference + PRD 蒸馏出 report + plan + questions + artifacts，两个 Skill 通过反馈回流闭环形成越用越准的知识生命周期
- **Reference v4.0 SSOT 架构**：6 文件按关注点分离（codebase / coding-rules / contracts / routing-playbooks / domain / portal），每个事实只存在一处，跨文件通过 `contract_ref` / `ref_rule` / `playbook_ref` 引用，5 条边界铁律避免知识污染
- **能力面适配器**：不绑定固定目录结构，前端/BFF/后端通过 20+ 能力面（ui_route / edge_api / api_surface 等）统一描述项目能力，路径只是搜索候选，结论必须来自源码证据
- **证据链 + 质量门控**：8 种证据类型（code / prd / tech_doc / git_diff / negative_code_search / human / api_doc / reference），PRD Ingestion 质量门禁三级（pass / warn / block），reference 质量门控覆盖致命项 + 警告项 + 跨文件边界检查
- **三层图谱融合**：GitNexus（代码结构维度）+ Graphify（业务语义维度）+ prd-tools（治理维度），图谱是原始发现层，reference 是精选后的企业知识库，图谱不可用时完全回退到源码扫描

**技术栈**：TypeScript · Claude API · Handlebars · 知识图谱（GitNexus / Graphify）· MCP 协议

### 营销中台 Schema 驱动架构（营销平台 + marketing-bff）

*核心角色：前端架构 + BFF 架构设计 · Lerna + Nx Monorepo（15+ 业务模块）*

**背景**：国际化营销中台需支持 10+ 国家、23 种活动类型（优惠券/推送/完成单奖励等），每种活动有不同的字段组合和联动规则。传统做法前后端各写一遍，新增活动类型需 2-3 人天。通过 Schema 驱动架构实现 BFF 动态生成 → 前端动态渲染，新增活动类型前端零代码。

**架构设计**：

- **BFF 模板引擎**：设计 `@company/schema-builder` 模板引擎，每个步骤独立模板函数（basic / group / rules / message / preview），字段定义按模板分布式管理（不存在集中注册中心），联动模板从 render 模板提取 handler 复用，保证联动返回与初始渲染的 Schema 结构一致
- **injectDActions 双重求值联动**：一次声明联动配置，BFF 端 safeEval 求初始 visible/disabled 状态（SSR 初始状态正确），同时序列化为 d_actions 给前端运行时联动。联动分 fetchApi 型（回调 BFF 重生成 Schema）和 state 型（前端本地 eval 表达式），16 个 linkageSchema 覆盖全场景
- **前端 Schema 渲染引擎**：formatName 将嵌套表单拍平为 dot 路径（同时是 Redux 存储 key 和 listener 事件名），RegisteredComponents 注册表支持 35+ 组件动态渲染，listener 发布订阅执行联动
- **Monorepo 治理**：Lerna（independent 版本）+ Nx（构建缓存）+ Yarn workspace 管理全部 15+ 业务模块，yarn.lock 严格锁版本 + resolutions 统一冲突依赖，根目录统一 ESLint / Prettier / Husky / commitlint

**结果**：新增活动类型前端零代码，BFF 加枚举 + 模板 + 联动配置即可上线，开发效率 [待补具体数据]

**技术栈**：React · TypeScript · Redux · Node.js · Serverless · Lerna + Nx · Yarn Workspace

### ibt-benefit-sdk — 权益组件库

*核心角色：架构设计 + 核心开发*

**背景**：营销平台需管理 400+ 权益的创建/编辑/复制/查看全生命周期，权益类型持续增长，需要可扩展的组件架构支持业务快速接入新权益。

**设计**：

- **Register 注册模式**：构造时注册 24 个内置组件（通用 16 + DLP 3 + UserCamp 1 + Wang 5），`add()` / `addMultipleCmps()` 支持消费方运行时注册自定义权益组件，无需修改 SDK 源码
- **Schema 同构渲染**：SDK 是 Schema 渲染引擎的第二个消费者——后端返回 `form_data`（含 `d_component` + `content_schema`），前端 FieldWrapper 迭代渲染，与 营销平台(dive) 活动表单共享同一套 Schema 渲染架构，体现系统级复用
- **Rollup 双模打包**：ESM（tree-shakable）+ CJS + `.d.ts` 类型声明，`exports` 字段条件导出，peerDependencies 声明 react 避免重复打包
- **发版流程**：Changesets 语义化版本 + alpha/release 双环境 + yalc 本地联调 + zx 脚本自动化

**技术栈**：React · TypeScript · Rollup · Changesets · Less
