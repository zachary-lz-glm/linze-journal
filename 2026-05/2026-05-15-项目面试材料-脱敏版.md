## 二、项目一：营销中台前端（Monorepo）

### 2.1 架构要点

**Monorepo 规模**
- Lerna + Nx + Yarn workspace 管理 **15个子应用**（不同业务线各自独立部署）
- 3个共享包：组件库、工具库、富文本编辑器
- 共享组件包包含 36+ 表单字段组件（TargetCrowd、AiTool、CouponDrawer 等）

**微前端架构**
- 采用 微前端框架（腾讯开源微前端框架）接入宿主应用
- 子应用支持独立运行 + 嵌入宿主双模式
- 通过 `window.__MICRO_APP__` 判断运行环境

**Schema 驱动渲染（核心亮点）**
- BFF 返回 `content_schema` JSON，描述表单字段结构、组件类型、校验规则
- 前端 Register 注册表：字符串组件名 → React组件映射（35+组件）
- FieldWrapper 递归渲染 Schema 树
- `d_actions` 声明字段间联动规则（显隐、联动请求、属性变更）

**动态 Reducer 注入**
- 每个表单步骤对应一个 Redux Slice
- 步骤由 BFF Schema 驱动，运行时动态创建和销毁 Reducer
- `store.injectReducer(key, reducer)` / `store.removeReducer(key)`

**多步骤向导表单**
- 创建页面：基本信息 → 目标人群 → 规则配置 → 消息/预览
- 每步"下一步"提交数据给 BFF，BFF 返回下一步 Schema
- 步骤流程完全由后端驱动，前端无需硬编码步骤顺序

### 2.2 性能优化

| 优化手段 | 说明 |
|---------|------|
| 路由级懒加载 | 所有页面用 `React.lazy()` + `Suspense`，首屏只加载当前路由 |
| Content Hash | `[name].[contenthash:8].js/css`，长期缓存 |
| Webpack 文件系统缓存 | `cache.type: 'filesystem'`，二次构建加速 |
| Babel 缓存 | `cacheDirectory: true` |
| Runtime 内联 | `InlineChunkHtmlPlugin` 避免 runtime 额外请求 |
| Moment.js Locale 精简 | `IgnorePlugin` 移除所有语言包 |
| Service Worker | Workbox `InjectManifest` 离线缓存 |
| Web Vitals 监控 | CLS、FID、FCP、LCP、TTFB 上报 |
| 请求去重 | dajax 封装自动取消重复请求 |

### 2.3 国际化

- 4种语言：中文、英文（美国/澳洲）、葡萄牙语（巴西）
- 自研 i18n 插件 + CLI key 提取
- Elvish 库处理多币种、多时区

### 2.4 面试话术（STAR）

**问：说说你做的最有技术含量的项目**

> 我负责国际化营销中台的前端架构。这个平台覆盖10多个国家、23种活动类型，每种活动的表单字段和联动规则完全不同。传统的做法是每种活动写一套表单页面，维护成本随活动类型线性增长。
>
> 我设计了一套 Schema 驱动架构：BFF 层作为 Schema 工厂，根据活动类型动态生成表单描述 JSON；前端通过 Register 注册表将字符串组件名映射到 React 组件，FieldWrapper 递归渲染整棵 Schema 树。字段间的联动通过 d_actions 声明，支持 fetchApi 型（回调 BFF 重新生成）和 state 型（前端本地计算）。
>
> 结果是新增活动类型前端**零代码**，只需 BFF 配置模板。Redux Reducer 也是动态注入的，每个步骤的 Reducer 在运行时从 Schema 生成，步骤完成后销毁，避免内存泄漏。整个系统通过 Lerna + Nx 管理15个子应用，各团队独立开发部署。

**问：微前端方案为什么选 微前端框架？**

> 我们的场景是子应用需要嵌入一个已有的宿主管理平台。微前端框架 基于 WebComponent + iframe 的隔离方案，CSS 和 JS 天然隔离，避免样式冲突，同时通信机制比纯 iframe 更轻量。子应用也支持独立运行模式，开发调试时不需要启动宿主。

**问：Schema 驱动架构的难点在哪？**

> 三个核心难点：
> 1. **联动引擎**：一个字段变化可能触发其他字段的显隐、选项更新、属性变更，且联动可能是链式的（A→B→C）。我们用 d_actions 声明依赖图，发布订阅模式执行，BFF 端预计算初始状态保证 SSR 正确
> 2. **动态 Reducer**：每个步骤的表单字段由 Schema 决定，Redux Store 结构必须在运行时构建。我设计了 injectReducer/removeReducer 机制，步骤切换时动态挂载和卸载
> 3. **组件注册表**：35+ 组件的统一接口抽象，每个组件需要处理 Schema 中 attrs 到 React Props 的映射、校验规则注入、联动事件绑定

### 2.5 可能的追问

**Q: Schema 的嵌套层级有多深？循环依赖怎么处理？**
> Schema 一般3-4层嵌套（页面 → 区块 → 字段组 → 字段）。FieldWrapper 用递归渲染，通过 `content_schema.children` 向下展开。联动依赖在 BFF 端生成时就有拓扑排序保证，不会出现循环。

**Q: Schema 版本兼容性怎么处理？**
> Schema 是无版本的——BFF 和前端同时部署，不存在版本不一致。前端组件注册表是向后兼容的：如果遇到未知组件名，FieldWrapper 降级渲染一个通用 Input 并打日志告警。

**Q: Webpack 构建速度如何优化？**
> 文件系统缓存 + Babel 缓存 + contenthash。二次构建从 2 分钟降到 30 秒左右。生产构建用 Terser 压缩，关掉了 comparisons 优化和 inline > 2 的限制以兼容 Safari 10。

## 三、项目二：权益组件库（NPM SDK）

### 3.1 架构要点

**定位**
- 内部 NPM 包，发布到公司私有 Artifactory
- 管理 400+ 权益的全生命周期（创建、查看、编辑、复制）
- 被多个业务线的 B 端页面消费

**组件分层**
- **表单字段组件（27个）**：Common（16个基础组件）+ DLP 业务（3个）+ 富文本业务（6个）+ 时间业务（2个）
- **模块级组件（6个）**：App（完整权益管理）、BenefitAdmin（统一调度）、View/Edit/Copy/Create（命令式调用）

**核心设计模式**

| 模式 | 说明 |
|------|------|
| Register 注册表 | 内置24组件 + 运行时 `register.add()` 注册自定义权益组件 |
| Schema 同构渲染 | 与营销中台共享 Schema 渲染架构，后端返回 Schema 前端迭代渲染 |
| 命令式 Modal API | `viewBenefitByUser(type, params)` 直接调用 `Modal.info()`，无需声明式渲染 |
| forwardRef + useImperativeHandle | 暴露方法给父组件 |
| EventEmitter | 手写发布订阅，组件间通信 |

**构建方案**
- Rollup 3.x 双模输出：ESM（`es/index.js`）+ CJS（`dist/index.js`）+ 类型声明（`es/index.d.ts`）
- CSS 内联到 JS（PostCSS + `postcss-url({ url: "inline" })`），消费者无需额外引入样式
- peerDependencies：react、react-dom、UI库、富文本编辑器由宿主提供
- Changesets 语义化版本管理，`alpha` 预发模式

**Schema 驱动的表单渲染**
- 组件库内的表单完全由后端 JSON Schema 驱动
- 每个 Schema 节点有 `d_component`（组件名）、`content_schema`（子节点）、`attrs`（属性）、`errs`（校验）、`value`（默认值）
- Register 类将字符串名映射到 React 组件，和营销中台的注册表是同一套架构

### 3.2 面试话术（STAR）

**问：权益组件库是怎么设计的？**

> 营销平台有 400 多种权益，类型持续增长。如果每新增一种权益就写一套表单，维护成本不可控。
>
> 我设计了 Register 注册表模式：内置 24 个通用表单组件（Input、Select、DatePicker 等），业务方通过 `register.add()` 注册自定义权益组件，无需修改 SDK 代码。表单结构由后端 Schema 驱动，Register 根据组件名查找并渲染。
>
> 对外暴露命令式 API：`BenefitAdmin('view', params)` 即可弹窗查看权益，底层用 `Modal.info()` 实现，消费者不需要管理弹窗的显隐状态。构建用 Rollup 输出 ESM + CJS 双模包，CSS 内联，peerDependencies 避免版本冲突。

**问：为什么选 Rollup 而不是 Webpack？**

> 组件库是 NPM 包，消费者是其他应用。Rollup 天然适合库的打包——Tree-shaking 更彻底、输出更干净。Webpack 适合应用打包（代码分割、运行时 chunk 等），对库来说过重。我们用 `rollup-plugin-dts` 单独生成类型声明，打包产物就是一个 JS 文件 + 一个 `.d.ts` 文件。

**问：CSS 内联有什么取舍？**

> CSS 内联到 JS 意味着消费者不需要单独引入样式文件，安装即用。代价是包体积稍大（~1.6MB），但 gzip 后可接受。另一个考虑是避免样式冲突——如果提取为独立 CSS 文件，不同版本的组件库可能产生样式覆盖问题。

### 3.3 可能的追问

**Q: 组件库和业务项目怎么联调？**
> 开发时用 yalc——本地构建后 `yalc push` 推送到本地仓库，业务项目 `yalc link` 链接。比 npm link 更可靠，不会有符号链接问题。组件库有一个完整的 CRA example 项目做本地开发测试。

**Q: 版本管理策略？**
> Changesets 管理版本，支持 alpha 预发。发布流程：进入预发模式 → 安装依赖 → 构建 → changeset → 版本号 → 发布 → 推送 tag。每次发布有明确的变更记录。

**Q: 如何保证类型安全？**
> TypeScript strict 模式 + `rollup-plugin-dts` 生成声明文件。导出了完整的 Props 接口（BenefitProps、BenefitType 等）。但业务组件中 `any` 使用较多——因为 Schema 是动态的，字段类型在运行时才确定。

## 四、项目三：营销 BFF 层

### 4.1 架构要点

**定位**
- Serverless BFF，部署在公司内部 Serverless 平台
- 核心职责：Schema 工厂——将后端原始数据转化为前端可渲染的表单 Schema

**技术选型**
- 自研 Serverless 框架（装饰器路由 `@Controll` + `@Request`）
- esbuild 构建（比 tsc 快 10x+），目标 Node 14
- axios 代理请求到后端服务
- Elvish 安全库处理加密/验签

**核心模式：Schema 工厂**

```
后端 API（原始数据） → BFF（Schema 生成） → 前端（Schema 渲染）
```

- `ComponentModel` 单例 Builder，带上下文注入（i18n、币种、工具函数）
- **模板组合**：每个页面/步骤有模板函数，返回字段定义数组
- **字段联动（d_actions）**：最复杂的部分——字段声明依赖关系，生成联动元数据
  - 显隐联动：根据其他字段值决定当前字段可见性
  - 选项联动：依赖字段变化时回调 BFF 重新获取选项
  - 属性联动：动态修改 label、disabled 等

**请求处理流程**
```
1. initElvish()         — 安全初始化
2. cors(res)            — CORS 头
3. Service.getXxxData() — 代理请求到后端
4. 检查 errno           — 错误早返回
5. Model.getXxxSchema() — 通过 Builder.parseOnce() 生成 Schema
6. 合并/转换            — 拼装后端数据 + Schema
7. 返回富化数据
```

**路由设计**
- 装饰器注册路由：`@Controll('create')` + `@Request({ method: 'get', routerName: '/render' })`
- 26+ 个路由覆盖：创建/编辑/复制渲染、字段联动、预览、批量上传、策略列表、安全门禁

**错误处理**
- Go 风格的 `to()` 工具函数：将 Promise 转为 `[result, error]` 元组
- 后端统一 `{errno, errmsg, data}` 响应格式
- 错误早返回模式：`if (data.errno !== 0) return data`

### 4.2 面试话术（STAR）

**问：BFF 层的核心职责是什么？**

> 营销中台覆盖 23 种活动类型，每种活动的表单完全不同。如果把表单逻辑放在前端，每新增一种活动就要写一套新页面。
>
> 我把表单生成逻辑抽到 BFF 层，设计成 Schema 工厂。BFF 根据活动类型，通过模板组合 + Builder 引擎，动态生成表单 Schema JSON。前端只做一件事：拿到 Schema，递归渲染。
>
> 最复杂的是字段联动——一个字段变化可能触发其他字段的显隐、选项刷新、属性变更。我在 BFF 端用 d_actions 声明联动规则，支持 fetchApi 型（回调 BFF 重新生成部分 Schema）和 state 型（序列化给前端本地计算）。BFF 端预计算初始联动状态，保证 SSR 首屏正确。

**问：为什么用 Serverless 而不是传统 Node 服务？**

> BFF 层是 I/O 密集型的——主要工作就是代理请求 + Schema 拼装，没有长连接或内存状态。Serverless 按请求计费、自动扩缩容，和这种无状态的 BFF 天然匹配。冷启动方面，esbuild 打包成单文件，启动很快。

**问：esbuild 为什么不用 tsc？**

> tsc 编译需要 15 秒左右，esbuild 只需要不到 1 秒。BFF 是 Serverless 函数，构建速度直接影响开发体验和 CI 效率。esbuild 不做类型检查（用 IDE + ForkTsChecker 做），只做转译，足够了。

### 4.3 可能的追问

**Q: Schema Builder 单例有什么问题？高并发下安全吗？**
> Builder 实例是单例，但每次请求调用 `parseOnce()` 时传入当前请求的上下文（i18n、币种等），上下文是请求级别的，不共享。所以是并发安全的。

**Q: BFF 和后端的 API 契约怎么保证？**
> 后端统一 `{errno, errmsg, data}` 响应格式，BFF 层有类型声明。但没有做运行时校验（如 Zod），依赖 TypeScript 类型 + 约定。这是一个可以改进的点。

**Q: 没有缓存层，性能怎么保证？**
> 目前每次请求都是实时代理到后端。表单 Schema 生成是 CPU 密集型的，但计算量不大（毫秒级）。如果后续有性能瓶颈，可以在 Serverless 层加 Redis 缓存热点 Schema。

## 六、高频追问预备

### 关于 Schema 驱动

**Q: 为什么不直接用 Formily / FormRender？**
> 我们评估过，但有两个原因：1）我们需要 BFF 端生成 Schema（不只是前端定义），现有方案都是纯前端；2）联动规则需要支持 fetchApi 型（回调后端），现有方案的联动是纯前端状态。所以自研了 Builder + d_actions。

**Q: Schema 的学习成本高吗？新成员多久上手？**
> Schema 的 JSON 结构很直观（组件名 + 属性 + 子节点），新人通常 1-2 天理解渲染机制。复杂的是联动规则——d_actions 的声明方式需要理解依赖图。我们写了详细的文档和 CLAUDE.md（AI 协作指南），新成员可以边问 AI 边上手。

### 关于 Monorepo

**Q: Lerna + Nx vs Turborepo？**
> 我们开始做的时候 Turborepo 还不够成熟。Lerna + Nx 的组合已经能满足需求：Lerna 管理包版本和发布，Nx 提供构建缓存和依赖图分析。如果重新选型，可能会考虑 Turborepo，但迁移成本不低。

**Q: 15个子应用怎么防止互相影响？**
> 每个子应用独立部署，共享包通过 npm 依赖管理。构建时 `lerna run build --scope=$module` 只构建目标模块。共享包升级时通过 Changesets 管理版本，消费者按需升级。

### 关于性能

**Q: 首屏加载优化做了哪些？**
> 路由级 React.lazy 懒加载，非当前路由的代码不加载。Webpack contenthash 做长期缓存，runtime chunk 内联减少请求。Service Worker 缓存静态资源。Web Vitals 监控核心指标上报。

**Q: 大表单渲染性能怎么保证？**
> 23 种活动类型的表单，字段最多的可能有上百个。优化手段：1）只渲染当前步骤的字段，未激活步骤的组件卸载；2）Redux Reducer 动态注入，步骤切换时清理；3）Schema 渲染用 React.memo 避免不必要的重渲染。
