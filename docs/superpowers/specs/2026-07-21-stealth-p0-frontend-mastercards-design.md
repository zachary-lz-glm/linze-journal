## 3. 架构决策

### 3.1 双层结构(复制三件套)

- **主陈述/故事卡**(本轮改造 5 张):开场/被让展开时用,套五步
- **机制知识点卡**(10 张保留):被追问单点机制时用,问答型

老前端项目此前**缺主陈述入口**(只有散落知识点+故事),本轮补上。

### 3.2 改造现有卡,不另立新卡

5 张都已有雏形(STAR/架构决策/推广/前端扎实/事故),保留 core 与好句,把 card-body 改写成五步、补缺的②⑤。卡1标题改(§3.4),其余标题保留。

### 3.3 给 5 张加 data-g="前端项目"(必要的小结构改动)

5 张现在无 data-g,靠正则兜底散到"Schema项目"/"产品思维"隐式组。加 `data-g="前端项目"` 让主陈述聚拢成组——这是"主陈述入口"的必要条件。getGroup 开头 `if(c.dataset.g)return c.dataset.g` 已支持,不动物理位置(5 张都在 2759-2829 区,本就相邻)。

**执行时必须**:在 `subOrder` 对象补"前端项目"的排序值(放 proj tab 里 Schema项目之后、agent 三件套之前或之后,执行时看现有 subOrder 决定),并测试 quickbar 分组显示正常。

### 3.4 卡1 标题改 + data-kw 同步

- 标题:"最有价值项目(STAR逐字稿)" → **"最有价值的项目:Schema 营销中台"**(保留高频问法入口 + 点明项目 + 去 meta 后缀)

## 6. 五张卡内容骨架

### 6.1 卡1:最有价值的项目:Schema 营销中台(原 STAR 逐字稿,line 2794)
- **core(精炼)**:从 if-else 地狱到声明式配置——新增活动前端零开发、上线两周缩三天、线上问题月均二十起归零
- ①业务+ownership:国际化营销 20+ 活动类型 × 七八国多城,老代码 if-else 上万行,扩城市就加分支,月均线上问题一二十起(无严格统计口径),新活动前端要两周。**我是前端负责人,要在不停业务下重构整个配置体系**。
- ②本质矛盾(项目层):难点不是"写个配置系统",是"配置表达力 vs 可控性"——要灵活描述复杂联动(城市→产品线→价格策略跨接口),又不能失控变成新 if-else 地狱。**本质是把命令式硬编码联动抽象成声明式、可静态校验的配置,不牺牲表达力**。
- ③决策+拒绝:Schema 驱动(**拒绝低代码**:B 端设计稿少,可视化拖拽 ROI 低,且复杂联动描述不了);双端解析架构(**拒绝单端**:SSR 初始状态不对,得等前端 JS 加载);联动 DSL dAction(**拒绝命令式 if-else**:不可静态校验、新人看不懂)
- ④结果+eval:新增活动前端零开发 / 上线两周→三天 / 月均二十起→接近零;并入 A5 拆出的「Schema 引擎月均十来个 bug→归零」。**怎么验证**:静态校验 + 灰度 + reason tag 分类报警三层防控(非 vibes)
- ⑤方法论+局限:**「声明式配置驱动——人依赖配置而非代码,把命令式业务逻辑抽象成可静态校验的配置」**。局限:Schema 有天花板(拖拽排序+实时协作仍写代码);只适用配置类页面,C 端自由布局/数据大屏不适用。

### 6.2 卡2:架构决策故事:为什么选 Schema(line 2759)
- **core(保留)**:排除了 if-else 维护和低代码平台,Schema 在灵活性和可控性间取到最优
- ①业务+ownership(补):存量 if-else 维护失控(月均二十起问题)。**我是这个重构的技术决策者**。
- ②本质矛盾(决策层):不是"选什么工具",是**"抽象层级选哪层"**——硬编码(if-else)太低失控、低代码(可视化)太高表达力不够,需要一个介于其间的声明式抽象。决策的本质是定位抽象层级。
- ③(已有最完整):三选项(①if-else ②低代码 ③JSON Schema)+ 排除①② + Schema 权衡 + 代价(学习曲线一周/天花板)
- ④(补 metrics):月均二十→接近零;静态校验可追溯。代价已付:学习曲线
- ⑤方法论+局限(补):**「架构决策核心是取舍——记录排除了什么 + 代价,不是记录选了什么」**(拔自原 hint)。局限:Schema 天花板(超复杂交互仍写代码)

### 6.3 卡3:跨团队推广故事:组件库如何从 1 到 8(line 2771)
- **core(保留)**:先试点解决痛点再文档化,不是写 README 等别人用
- ①业务+ownership(补):权益 SDK 最早只服务国际化,其他七八个团队各搞各的权益组件,重复开发。**我是 SDK owner**。
- ②本质矛盾(补):**推广阻力不是技术是信任**——方案好坏 vs 别人愿不愿用。本质是降低他人采纳风险/成本,技术好不够,要让人敢用。
- ③(补拒绝):推广三步(试点痛点最大的业务 → 沉淀接入文档 → 试点数据内部分享)。**拒绝"写 README 等别人来用"**(没人看);Register 注册模式(**拒绝"通用不够就 fork 改"**:污染)
- ④eval(补):接入工作量两周→三四天;从 1→8 团队
- ⑤方法论+局限(补):**「推广 = 试点数据说服 > 文档说服;设计上让自定义与通用共存(Register)防 fork」**。局限:推广靠试点团队意愿,非所有团队都主动(诚实)

### 6.4 卡4:怎么证明前端扎实(line 2829)
- **core(保留)**:举 Schema 渲染引擎——从零设计了完整解析→渲染→联动→性能链路
- ①业务+ownership(补):证明前端底子(面试官确认你不是只聊 AI)。**我从零设计了 Schema 渲染引擎**。
- ②本质矛盾(补):难点不是"用现成 Formily/Ajv"(那是调 API),是**"从零设计联动 DSL + injectDActions 双求值"——渲染引擎级架构 vs 调 API 写页面**。
- ③(补拒绝):联动 DSL dAction + 双求值(**拒绝现成 Formily**:联动描述力不够/黑盒);虚拟滚动 + 按需渲染(**拒绝全量重渲染**:四百+组件卡);Rollup + tree-shaking(**拒绝整包**:体积)
- ④eval(补):四百+ 组件配置页流畅(虚拟滚动只渲染视口内);联动精确更新受影响组件而非全量
- ⑤方法论+局限(补):**「前端扎实的证据是从零设计引擎而非用轮子;性能靠精确更新而非全量重渲染」**。局限:这套引擎是为配置类页面设计,不直接迁移到 C 端自由布局(诚实)

### 6.5 卡5:线上事故反思 + 基础设施架构判断(line 2807)
- **core(保留)**:事故响应三部曲 + 容灾架构判断:两地同区域不算真容灾
- ①业务+ownership(补):**我经历过 yarn.lock 事故**(解析器自动升级导致构建全挂,十几分钟回滚),主导复盘。
- ②本质矛盾(补):事故处理本质不是"快速修",是**"快速止血 + 根因(追机制不追责)+ 流程改进"——救火 vs 防火**。容灾本质是物理距离非机房数(两地同区域=假容灾)。
- ③(已有):事故三部曲 + 两地部署判断 + 补偿四项(降级/多级监控/预案演练/客户端容错)
- ④eval:yarn.lock 事故十几分钟回滚;复盘三件事(yarn.lock 入 git 保护 / CI lockfile diff 检测 / 依赖升级 staging 验证)
- ⑤方法论+局限(已有雏形 + 补局限):事故三部曲=方法论。**局限(补诚实边界)**:我的事故是构建级(构建挂)非用户级(数据丢失/资损),别夸大严重性;容灾判断是理论推演,不是我亲自架构的生产容灾
