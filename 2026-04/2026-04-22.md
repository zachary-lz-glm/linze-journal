# 2026-04-22 周二 — Reference 领域知识标准化 & PRD 蒸馏优化

## 一、调研背景

bff-gen 要推广到 B 端全栈（前端/后端/BFF），面临两个核心挑战：

1. **Reference 领域知识标准化** — 当前是单层（BFF）× 单项目（marketing-bff），无法扩展
2. **PRD 蒸馏质量优化** — 蒸馏效果最大瓶颈是 PRD 输入质量（★★★★★），而非蒸馏算法

**研究方法**：代码库深度分析 + 30+ 篇行业文献调研 + BMAD Architecture 阶段正式产出

## 二、行业趋势：Context Engineering

2025-2026 年，AI 代码生成的核心演进方向是 **Context Engineering**——系统化地构建和管理提供给 AI 的知识架构。

| 趋势 | 说明 |
|------|------|
| Context Engineering 作为学科 | 迭代式构建、优化、测试 AI 的输入 |
| Knowledge Priming | Martin Fowler："知识预注入"——AI 生成代码前先共享策划好的上下文 |
| 3-Tier Knowledge Architecture | 企业级 AI coding agent 采用 3 层知识架构（共享/领域/任务） |
| Virtual Monorepo Pattern | 跨 35 个仓库为 Claude Code 提供全系统上下文 |
| .md 作为核心工程资产 | Markdown 文件成为 AI 协作中最有力的工程制品 |

### 5 层上下文架构（行业共识）

| 层级 | 目的 | bff-gen 对应 |
|------|------|-------------|
| 指令层 | 清晰具体的指令 | `steps/*.md` |
| 领域层 | 领域特定知识 | `reference/` |

### AI 工具知识管理对比

| 工具 | 格式 | 层级 | 置信度 |
|------|------|------|--------|

| Claude Code | `CLAUDE.md` | 多层 | 无 |
| **bff-gen** | `reference/*.md` + Schema | **多层 + 校验** | **✅ confidence 3级** |

## 三、PRD 蒸馏优化

### 蒸馏效果 7 因素分解

```
工具效果 = reference/ 质量 × PRD 质量 × LLM 能力
```

| # | 因素 | 权重 | 当前 | 优化方向 |
|---|------|------|------|---------|
| 1 | **PRD 结构化程度** | 25% | 低 | PRD Schema 1.0 |
| 2 | **Reference 覆盖度** | 25% | BFF 75% | 三层标准化 |
| 3 | **字段映射明确性** | 15% | 中 | PRD 显式标注 target_file |
| 4 | **PRD-Reference 语义匹配** | 15% | 低 | PRD 用标准术语 |
| 5 | **后端契约可用性** | 10% | 低 | 强制后端文档 |
| 6 | **蒸馏 Prompt 质量** | 5% | 高 | 持续迭代 |
| 7 | **LLM 模型能力** | 5% | 中 | 模型升级 |

### 蒸馏 4 种错误类型

| 错误 | 根因 | 发生率 | 解法 |
|------|------|--------|------|
| 字段归属错误 | PRD 没标注字段属于哪个步骤 | ~30% | PRD 按步骤分区 |
| 组件类型误判 | "选择" = Select 还是 Radio？ | ~15% | PRD 标注组件白名单 |
| 遗漏字段 | PRD 信息分散 | ~20% | 表格化替代自由文本 |
| 编造信息 | LLM 倾向填满所有字段 | ~10% | 强调"未提及标 low" |

### AI 友好 PRD Schema 1.0

```markdown
## Meta | campaign_type | country | is_new_type | reference_campaign |
## 基础信息（basic）     | PRD 字段名 | dataIndex | 组件类型 | 必填 | 校验规则 |
## 奖励条件（rewardCondition）| 同上 |
## 规则详情（detail）    | PRD 字段名 | dataIndex | 组件类型 | 梯度类型 |
## 联动关系              | 字段 | 联动类型 | 依赖字段 | 联动逻辑 |
## 校验规则              | 规则 | 错误提示 | 作用范围 |
## 预览占位符            | 占位符 | 说明 |
```

核心设计：按步骤分区（归属准确率 30%→90%）+ 强制白名单选组件（准确率 95%）+ 表格替代自由文本（遗漏率 20%→5%）

### 蒸馏准确率提升路线

| 阶段 | 动作 | 准确率 |
|------|------|--------|
| 0（当前） | 自由文本 + BFF reference | ~70% |
| 1 | PRD Schema 1.0 | ~85% |
| 2 | 三层 reference 标准化 | ~90% |
| 3 | 联动显式 + 后端契约 | ~95% |

## 四、最终架构：8 文件标准 Reference 结构 ⭐

> 以下为 BMAD Architecture 阶段正式产出，否决了之前调研中的 `contract.json + domain/` 方案。

### 核心定位转变

**工具是流程化能力提供者，不是数据存储**：

```
用户维护 reference（领域知识）→ 工具读取 reference → 理解项目 + 蒸馏 PRD → 结构化开发文档
```

Reference 是用户的输入，不是工具的输出。工具的核心职责是**强制 AI 严格按用户注入的 reference 执行**，防止 AI 凭"经验"判断导致偏差。

### 为什么否决 contract.json + domain/ 方案

| 之前的方案 | 问题 |
|-----------|------|
| `contract.json` 作为共享契约 | 过于简单，只覆盖"字段注册表"和"术语表"两类，实际有效的 BFF reference 有 **8 类知识** |
| `domain/` 层无关知识 | 先不考虑跨层共享，各团队代码差异太大，纯靠自动生成不靠谱 |
| 工具自动从代码生成 reference | ❌ 各团队代码结构差异大，不可靠 |
| 工具从 git diff 推断规则 | ❌ 不可靠 |

### 8 文件标准结构

```
reference/
├── 01-registry.json       # 结构化目录：有什么（字段、枚举、组件清单）
├── 02-topology.md         # 代码拓扑：放哪里（字段归属决策树、文件映射）
├── 03-conventions.md      # 命名规范：怎么命名（文件/代码/i18n/枚举）
├── 04-patterns.md         # 代码模式：怎么写（模板结构、handler 类型、hook 用法）
├── 05-constraints.md      # 约束规则：什么允许/禁止（白名单、校验规则、边界）
├── 06-anti-patterns.md    # 反模式：什么错误要避免（已知错误 + 检测方法）
├── 07-mechanisms.md       # 框架机制：系统怎么运作（内部行为、数据流边界）
└── 08-glossary.yaml       # 术语表：业务术语 ↔ 技术术语（PRD 关键词映射）
```

### 为什么是这 8 类

| 知识类型 | 占比 | 解决什么 AI 偏差 |
|---------|------|-----------------|
| Registry（目录） | ~15% | 编造枚举值、重复提取已有字段 |
| Topology（拓扑） | ~15% | 字段归属错误（basic vs rewardCondition） |
| Conventions（规范） | ~12% | 命名不一致、风格漂移 |
| **Patterns（模式）** | **~33%** | 违反代码模式、框架约定 |
| Constraints（约束） | ~14% | 使用白名单外的组件/函数 |
| Anti-patterns（反模式） | ~16% | 犯已知错误（options: null 等） |
| Mechanisms（机制） | ~10% | 误解框架行为、重复定义自动处理的字段 |
| Glossary（术语） | ~5% | PRD 术语 → 字段名映射错误 |

### 现有 BFF reference → 新结构映射

| 现有文件（6 个） | → 新文件 |
|-----------------|---------|
| field-mapping.md (186行) | 01-registry.json + 02-topology.md |
| naming.md (143行) | 03-conventions.md |
| scaffold-patterns.md (399行) | 04-patterns.md |
| validation.md (172行) | 05-constraints.md |

| (无) | 08-glossary.yaml **新增** |

### 防偏差保障机制

工具的核心价值：**强制 AI 严格按 reference 执行，每个产出字段可追溯到具体规则**。

| 偏差类型 | 无 reference 时 | 有 reference + 工具强制时 |
|---------|----------------|----------------------|
| 字段归属错误 | AI 凭感觉判断 | 严格按 02-topology 决策树 |
| 命名不一致 | 每次生成不同风格 | 严格按 03-conventions |
| 编造枚举值 | AI 自由发挥 | 严格从 01-registry 白名单选择 |
| 违反代码模式 | AI 用自己的风格 | 严格按 04-patterns 模板 |
| 犯已知错误 | 重复 options:null 等错误 | 06-anti-patterns 逐条校验 |
| 误解框架行为 | 不知道后端自动 merge | 07-mechanisms 告知哪些不需定义 |

## 六、Reference 构建流程（3 步）

```
Step 1: ref-init（初始化）
  npx bff-gen ref-init --layer <bff|frontend|backend> [--example marketing-bff]
  → 8 文件骨架（含格式说明 + 填写引导注释）

Step 2: 填充内容
  工具提供: 每个文件的 prompt 模板 + 面试题清单 + marketing-bff 完整示例
  团队执行: 用 Claude Code / Cursor / 手工填写
  → 8 个填好的 reference 文件

Step 3: ref-validate（校验）
  npx bff-gen ref-validate --target /path
  → 格式合规 + 一致性报告
```

### ref-validate 检查项

1. **格式合规** — JSON Schema / YAML Schema
2. **必填完整** — 每个文件必要章节是否填写
3. **内部一致性** — registry 枚举与 constraints 白名单匹配，glossary field id 与 registry 对应
4. **freshness** — registry.updated_at 是否存在且合理

### 工具内部资产

```
prd2code-gen/
├── src/reference/
│   ├── schemas/       # JSON/YAML Schema 定义
│   ├── skeletons/     # 8 文件骨架模板（.tpl）
│   ├── prompts/       # 按层分类的 prompt 模板
│   └── interviews/    # 按层分类的面试题模板
├── src/installer/     # ref-init 命令
└── examples/marketing-bff/ # BFF 层完整示例
```

## 七、置信度校准（重要发现）

**Nature 2026**：48 个 LLM 置信度与实际准确率严重不匹配，LLM 常对流畅但不正确的输出赋高概率。

| 方法 | 可靠性 | 推荐度 |
|------|--------|--------|
| LLM 自评 | ❌ 低 | 当前方案，需改进 |
| Structured Logprobs | ✅ 高 | 生产推荐 |
| 多次运行一致性 | 高但贵 | 高价值决策 |
| **混合验证（推荐）** | ✅✅ 最高 | 业务规则 + Contract + 回测 |

## 十、5 个关键结论

1. **工具是流程提供者，不是数据存储** — reference 由各团队维护，工具只定义格式+提供引导+消费执行
2. **8 类知识缺一不可** — Patterns（33%）占比最大，之前只做了 Registry 和 Topology 就以为够了
3. **防偏差 > 生成能力** — 工具的核心价值是强制 AI 按 reference 执行，不是帮 AI 生成更好的代码
4. **PRD 输入质量是最大杠杆** — 投入 PRD 标准化的 ROI 远高于优化蒸馏算法
5. **LLM 自评置信度不可靠** — 需要多源校验（混合验证），不能只靠 LLM 自己打分
