# 2026-04-22 周二 — Reference 领域知识标准化 & PRD 蒸馏优化

## 一、调研背景

bff-gen 要推广到 B 端全栈（前端/后端/BFF），面临两个核心挑战：

1. **Reference 领域知识标准化** — 当前是单层（BFF）× 单项目（marketing-bff），无法扩展
2. **PRD 蒸馏质量优化** — 蒸馏效果最大瓶颈是 PRD 输入质量（★★★★★），而非蒸馏算法

**研究方法**：代码库深度分析 + 30+ 篇行业文献调研（Karpathy、Martin Fowler、Nature 2026、ChatPRD 等）

## 三、Reference 三层分离架构

### 3.1 当前知识分布问题

| 维度 | BFF (marketing-bff) | 前端 (营销平台/app/dive) |
|------|---------------|---------------------|
| 知识格式 | reference/*.md（5 文件） | .claude/docs/*.md（8+ 文件） |
| 核心注册表 | field-registry.ts | component-registry.tsx |
| 组件映射 | field-mapping.md | ACTIVITY-TYPE-FIELD-MAPPING.md |
| 反模式 | anti-patterns.md | TROUBLESHOOTING.md |
| 业务术语 | 分散 | BUSINESS-GLOSSARY.md |

**核心问题**：无共享层、格式不统一、无构建流程、后端层缺失。

### 3.2 标准化 Reference 结构

```
reference/
├── _shared/                          # 共享契约层（三层共用）
│   ├── contract.json                 # API 数据契约（字段定义 + 类型 + 枚举）
│   ├── campaign-types.json           # 活动类型枚举（单一来源）
│   ├── business-glossary.json        # 业务术语对照表
│   └── field-registry.json           # 全局字段注册表（field name → 跨层分布）
│
├── bff/                              # BFF 层特有知识
│   ├── field-mapping.md              # 字段归属决策树（引用 _shared/contract）
│   ├── naming.md                     # 命名规范
│   ├── validation.md                 # 校验规则
│   ├── anti-patterns.md              # 反模式
│   ├── hooks.md                      # 框架机制
│   └── scaffold-patterns.md          # 代码模板模式
│
├── frontend/                         # 前端层特有知识
│   ├── component-registry.md         # 组件注册表
│   ├── component-ref.md              # 组件参数参考
│   ├── anti-patterns.md              # 前端反模式
│   └── workflow.md                   # 修改代码流程
│
└── backend/                          # 后端层特有知识
    ├── api-contract.md               # API 契约说明
    ├── data-model.md                 # 数据模型
    └── anti-patterns.md              # 后端反模式
```

### 3.3 3 个核心设计原则

**原则 1: 共享契约层（Shared Contract Layer）**
- `contract.json` 作为三层之间的"共享真理"——API 契约、活动类型枚举、字段注册表
- 各层 reference 从共享契约派生，不跨层直接引用

**原则 2: 每层独立维护（Layer Autonomy）**
- 每层有自己的 reference 目录，维护该层特有知识
- 安装方式：`npx bff-gen install --example <name>`，部署 `_shared`（只读）+ 对应层 reference（可维护）
- 演进成熟后可 PR 回工具仓库惠及其他项目

**原则 3: Schema 强制校验（Schema Enforcement）**
- 每个 reference 文件有对应的 JSON Schema / YAML Schema
- CI 中校验格式完整性和跨层一致性


### 3.5 知识构建流程（4 阶段）

```
Phase 1: 自动生成（零人工）
  项目代码 + API 文档 → AST 解析 → contract.json 骨架 + naming.md 骨架

Phase 2: Diff 学习（需 2-3 个历史 diff）
  git diff → 字段归属统计 → field-mapping.md + anti-patterns.md（带置信度标注）

Phase 3: 人工审核（标准化交互）
  低置信度规则列表 → 逐条确认/修正 → 补充隐性知识

Phase 4: Schema 校验 + 回测验证
  JSON Schema 格式检查 → 历史 diff 回测准确率 → ≥90% 冻结规则
```

### 3.6 知识质量度量

| 度量 | 指标 | 目标 |
|------|------|------|
| 覆盖率 | field-registry 字段数 / 实际字段数 | > 95% |
| 模板命中率 | 模板匹配成功 / 总生成 | > 70% |
| 准确率 | reference vs 代码实际行为 | > 98% |
| 时效性 | 最后更新距今天数 | < 30 天 |
| 蒸馏成功率 | high confidence 字段占比 | > 80% |

## 四、PRD 蒸馏优化

### 4.1 蒸馏效果 7 因素分解

```
工具效果 = reference/ 质量 × PRD 质量 × LLM 能力
```

进一步量化为 7 个因素：

| # | 因素 | 权重 | 当前状态 | 优化方向 |
|---|------|------|---------|---------|
| 1 | **PRD 结构化程度** | 25% | 低（自由文本） | PRD Schema 1.0 |
| 2 | **Reference 覆盖度** | 25% | BFF 75%，前端/后端 0% | 三层标准化 |
| 3 | **字段映射明确性** | 15% | 中（决策树 + LLM） | PRD 显式标注 target_file |
| 4 | **PRD-Reference 语义匹配** | 15% | 低 | PRD 用标准术语 |
| 5 | **后端契约可用性** | 10% | 低（常缺失） | 强制后端文档 |
| 6 | **蒸馏 Prompt 质量** | 5% | 高（已优化） | 持续迭代 |
| 7 | **LLM 模型能力** | 5% | 中（Sonnet） | 模型升级 |

### 4.2 蒸馏 4 种错误类型

| 错误类型 | 根因 | 发生率 | 优化方案 |
|---------|------|--------|---------|
| 字段归属错误 | PRD 没有标注字段属于哪个步骤 | ~30% | PRD 按步骤分区 |
| 组件类型误判 | "选择" = Select 还是 Radio？ | ~15% | PRD 标注组件白名单 |
| 遗漏字段 | PRD 信息分散，LLM 无法识别 | ~20% | 表格化替代自由文本 |
| 编造信息 | LLM 倾向填满所有字段 | ~10% | 强调"未提及标 low" |

### 4.3 AI 友好 PRD Schema 1.0

```markdown
# {活动名称} PRD

## Meta（必填）
| campaign_type | country | is_new_type | reference_campaign |

## 基础信息（basic）— 按步骤分区，消除 target_file 推断歧义
| PRD 字段名 | dataIndex | 组件类型 | 必填 | 校验规则 | 默认值 | 说明 |

## 奖励条件（rewardCondition）
| PRD 字段名 | dataIndex | 组件类型 | 必填 | 校验规则 | 说明 |

## 规则详情（detail）
| PRD 字段名 | dataIndex | 组件类型 | 梯度类型 | 说明 |

## 联动关系
| 字段 | 联动类型 | 依赖字段 | 联动逻辑 |

## 校验规则（跨字段）
| 规则 | 错误提示 | 作用范围 |

## 预览占位符
| 占位符 | 说明 |
```

**核心设计决策**：

| 设计 | 原因 | 收益 |
|------|------|------|
| 强制按步骤分区 | 消除 target_file 推断歧义 | 字段归属准确率 30% → 90%+ |
| 强制标注 dataIndex | 消除命名歧义 | 命名一致性 100% |
| 强制白名单选组件 | 消除 Select vs Radio 歧义 | 组件准确率 95%+ |
| 表格替代自由文本 | 结构化提取更可靠 | 遗漏率 20% → 5% |
| 显式声明联动 | 消除跨字段推断 | 联动完整率 80%+ |

## 五、置信度校准（重要发现）

### LLM 自评置信度不可靠

**Nature 2026 研究表明**：48 个 LLM 在专业领域问题上，置信度与实际准确率严重不匹配。LLM 常对流畅但不正确的输出赋予高概率。

来源：[Nature - LLMs Poorly Calibrate Confidence](https://www.nature.com/articles/s44355-026-00053-3)

### 6 种置信度评分方法对比

| 方法 | 描述 | 可靠性 | 适用场景 |
|------|------|--------|---------|
| **LLM 自评** | LLM 对自己输出打分 | ❌ 低 — 过度自信 | 当前方案（需改进） |
| **Logprobs** | token 级对数概率 | 中高 — 客观数学信号 | 有 API 访问时 |
| **Structured Logprobs** | JSON 逐字段置信度 | ✅ 高 — 当前最佳 | 生产环境推荐 |
| **多次运行一致性** | 多次采样检查一致性 | 高但昂贵 | 高价值决策 |
| **LLM-as-Judge** | 第二个 LLM 评估 | 中 | 评估场景 |
| **混合验证** | 业务规则 + 置信度阈值 | ✅✅ 最高 | **推荐方案** |

**推荐方案**：多源校验 = LLM 自评 + Contract 一致性 + 历史回测，三源融合比单一 LLM 自评更可靠。

来源：[structured-logprobs](https://github.com/arena-ai/structured-logprobs), [DACA Calibration](https://openreview.net/pdf?id=I4PJYZvfW5)

## 七、集成实现方案

### 蒸馏引擎消费共享契约

Step 1 蒸馏时，从 contract.json 加载：
- `campaign_types` → 判断 is_new_campaign_type
- `component_types` → 校验 d_component 白名单
- `fields` → 检查字段是否已存在
- `steps` → 校验 target_file 合法值

### 共享契约版本管理

- `contract.json` 使用 semver
- 新增活动类型/字段 → minor 升级
- 删除字段（破坏性） → major 升级
- CI 检查各层 reference 与 contract.json 版本兼容

## 十、5 个关键结论

1. **PRD 输入质量是蒸馏效果的最大杠杆** — 投入 PRD 标准化的 ROI 远高于优化蒸馏算法
2. **bff-gen 已走在正确方向** — reference/ 本质是 Context Engineering，有差异化优势（Schema 校验 + 置信度机制）
3. **共享契约 contract.json 是枢纽** — 连接三层知识的单一来源，类似 Schema Registry 模式
4. **LLM 自评置信度不可靠** — 需要多源校验（Contract + Logprobs + 回测），推荐混合验证
5. **"给 AI 写 PRD"是一种新技能** — PM 需学结构化表达，但也让 PRD 对人更清晰
