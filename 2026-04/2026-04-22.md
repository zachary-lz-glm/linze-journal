# 2026-04-22 周二 — Reference 领域知识标准化 & PRD 蒸馏优化

## 一、调研背景

bff-gen 工具要推广到 B 端全栈（前端/后端/BFF），当前 reference 知识结构存在两个核心问题：

1. **单层（BFF）× 单项目（marketing-bff）** — 无法扩展到前端和后端
2. **PRD 蒸馏准确率瓶颈** — 最大影响因素是 PRD 输入质量（★★★★★），而非蒸馏算法

**研究方法**：代码库深度分析（marketing-bff, 营销平台/app/dive, mcp-bff-gen）+ 行业最佳实践调研（Karpathy 知识架构、spec-kit workflow、ChatPRD 2026 等）

## 二、Reference 领域知识标准化

### 2.1 现状分析

当前 reference 知识结构（mcp-bff-gen 内）：

| 文件 | 内容 | 问题 |
|------|------|------|
| `field-mapping.md` | PRD 字段 → BFF 组件映射 | 仅覆盖 BFF 层，无前端/后端映射 |
| `validation.md` | 验证规则白名单 | BFF-centric，前端验证规则缺失 |
| `naming.md` | 命名规范 | 仅 BFF 文件/字段命名 |
| `bff-hooks.md` | BFF 框架 hooks | 纯 BFF 实现，其他层无对应 |
| `anti-patterns.md` | 反模式检测 | 仅 BFF 层反模式 |
| `scaffold-patterns.md` | 脚手架代码模式 | 单一项目(marketing-bff)模式 |

### 2.2 三层分离的领域知识架构

基于 marketing-bff 的 Schema-Driven 架构：`Backend (数据源) → BFF (Schema 生成) → Frontend (Schema 渲染)`

每一层对同一业务概念有不同视角。例如"活动标题"：

| 维度 | Backend | BFF | Frontend |
|------|---------|-----|----------|
| 语义 | 数据库字段 `task_title` | Schema 字段定义 + 组件类型 | 渲染 `Input` 组件 |
| 关注点 | 存储类型、长度约束 | d_component、linkage、validation | onChange、状态管理 |
| 命名 | snake_case | data_index (snake_case) | props (camelCase) |

**reference 分 4 个层级组织**：

```
reference/
├── domain/                    # 层无关的领域知识（共享）
│   ├── glossary.md            # 业务术语表（统一词汇）
```

### 2.3 4 个核心设计原则

#### 原则 1：field-registry 为中心（Field Registry as Source of Truth）

参考 Karpathy 的 LLM 知识库架构，知识需要"编译"成 LLM 可查询的结构。`field-registry.md` 定义每个字段的三层映射：

```yaml
field: task_title
domain_name: 活动标题

bff:
  data_index: task_title
  d_component: Input
  target_file: basic
  validation: [required, max_length:50]

frontend:
  component: Input
  props: { maxLength: 50, showCount: true }
  state_path: dataSet.basic.task_title

backend:
  db_column: task_title
  db_type: VARCHAR(50)
  api_field: task_title
```

#### 原则 2：领域知识层无关化

`domain/` 不包含实现细节：
- `glossary.md` — "冲单奖 = 司机在指定时段内完成订单数达标后获得奖励"，不提 BFF 或 React
- `business-rules.md` — "活动开始时间不可早于当前时间"，不指定在哪一层实现

新增层时 domain 知识零修改。

#### 原则 3：模板按层×活动类型矩阵组织

| | gas-station | courier-rush | shift-checkin | 新活动 |
|---|---|---|---|---|
| **BFF** | `templates/bff/gas-station/` | `templates/bff/courier-rush/` | `templates/bff/shift-checkin/` | LLM 兜底 |
| **Frontend** | `templates/frontend/gas-station/` | ... | ... | LLM 兜底 |
| **Backend** | `templates/backend/gas-station/` | ... | ... | LLM 兜底 |

#### 原则 4：知识构建流程标准化

```
阶段 1：Domain Knowledge 采集
├── 1.1 术语表构建 — 与业务方对齐统一词汇
├── 1.2 活动类型梳理 — 枚举所有活动类型及分类
├── 1.3 业务规则提取 — 从已有代码/文档中提取
└── 1.4 字段注册表初始化 — 扫描现有代码建立字段清单

阶段 2：Layer Knowledge 采集
├── 2.1 选择目标层（BFF/Frontend/Backend）
├── 2.2 代码模式扫描 — AST 解析提取模式
├── 2.3 映射表构建 — 手动 + LLM 辅助
├── 2.4 反模式识别 — 标记常见错误
└── 2.5 命名规范固化 — 从代码库提取模式

阶段 3：模板构建
├── 3.1 高频场景模板化
├── 3.2 Handlebars 模板编写
├── 3.3 模板回测
└── 3.4 模板注册

阶段 4：持续维护
├── 4.1 新活动类型 → 更新 field-registry
├── 4.2 代码模式变更 → 更新 scaffold patterns
├── 4.3 定期验证 reference vs 代码一致性
└── 4.4 反模式库持续积累
```

### 2.4 知识质量度量

| 度量维度 | 指标 | 目标 |
|---------|------|------|
| 覆盖率 | field-registry 字段数 / 实际字段数 | > 95% |
| 模板命中率 | 模板匹配成功次数 / 总生成次数 | > 70%（高频场景） |
| 准确率 | reference 描述 vs 代码实际行为一致性 | > 98% |
| 时效性 | reference 最后更新距今天数 | < 30 天 |
| 蒸馏成功率 | high confidence 字段占比 | > 80% |

## 三、PRD 蒸馏优化 & AI 友好 PRD 格式

### 3.1 蒸馏效果依赖分析

| # | 依赖因素 | 影响程度 |
|---|---------|---------|
| 1 | **PRD 输入结构** | ★★★★★ |
| 2 | Reference 覆盖度 | ★★★★☆ |
| 3 | 蒸馏 JSON Schema | ★★★☆☆ |
| 4 | 置信度校准 | ★★★☆☆ |
| 5 | 上下文窗口 | ★★☆☆☆ |

**关键发现：PRD 输入质量是最大的影响因素。**

### 3.2 蒸馏 4 个核心瓶颈

#### 瓶颈 1：PRD 术语不统一
- "目标司机" vs "参与人群" vs "司机筛选条件" → 都指 `target_crowd`
- "活动时间" vs "活动周期" vs "有效期" → 都指 `task_valid_period`
- **解决**：`glossary.md` 术语表与 PM 对齐，PRD 模板中固化

#### 瓶颈 2：字段归属判断模糊
- field-mapping.md 决策树在 PRD 描述跨越模块时判断困难
- **解决**：PRD 模板按步骤分区描述字段，与 BFF target_file 一一对应

#### 瓶颈 3：组件类型推断不可靠
- "选择" = Select 还是 Radio？ "日期" = DatePicker 还是 DateRangePicker？
- **解决**：PRD 模板使用标准化组件标签（`[下拉单选]`、`[日期范围]`）

#### 瓶颈 4：验证规则隐含在自然语言中
- "活动标题不超过 50 字" → `max_length: 50`，LLM 容易遗漏
- **解决**：PRD 模板中验证规则用结构化标签标注

### 3.3 AI 友好的 PRD 标准格式

**设计原则**：
1. **Markdown 优先** — LLM 解析效率最高
2. **表格化字段描述** — 结构化数据 > 自然语言
3. **分区对应 target_file** — PRD 结构与 BFF 文件结构对齐
4. **标签化组件和验证** — 消除推断歧义
5. **引用术语表** — 使用统一术语

**标准 PRD 模板结构**：

```markdown
# PRD: {{活动类型名称}}

## Meta
| campaign_type | country | product_line | is_new_campaign_type | reference_campaign |

## 1. 基础配置 (basic_fields)
| # | 字段名 | data_index | 组件[标签] | 类型 | 必填 | 验证规则 | 默认值 | 说明 |
### 字段联动 (d_actions)
| 触发字段 | 条件 | 动作 |

## 2. 规则详情 (rule_detail)
| # | 字段名 | data_index | 组件 | 类型 | 验证规则 | 说明 |

## 3. 奖励条件 (reward_condition_fields)
### 阶梯奖励
| 阶梯 | 条件 | 奖励方式 | 奖励金额 |

## 4. 消息配置 (message_fields)

## 5. 预览配置 (preview_fields)

## 6. 业务验证规则 (validations)
| 规则描述 | 错误信息(i18n key) | 触发时机 |

## 7. 后端数据契约 (backend_contract)
| API | 字段 | 类型 | 说明 |
```

### 3.4 蒸馏 JSON Schema 扩展

新增 `layers` 维度支持跨层生成：

```json
{
  "fields": [{
    "prd_name": "活动标题",
    "domain_key": "task_title",
    "layers": {
      "bff": { "data_index": "task_title", "target_file": "basic", "d_component": "Input" },
      "frontend": { "component": "Input", "state_path": "dataSet.basic.task_title" },
      "backend": { "db_column": "task_title", "db_type": "VARCHAR(50)" }
    },
    "confidence": "high",
    "source_ref": "PRD §1 #1"
  }],
  "cross_layer_deps": [{
    "field": "target_crowd",
    "bff_to_frontend": { "linkage": "visible_when", "value": "custom" },
    "bff_to_backend": { "api_mapping": "target_crowd_tag" }
  }]
}
```

## 六、关键结论

1. **PRD 输入质量是蒸馏效果的最大杠杆** — 投入 PRD 标准化的 ROI 远高于优化蒸馏算法
2. **Reference 的核心是 field-registry** — 连接 domain 知识和各层实现的枢纽
3. **三层分离架构让扩展成本线性** — 新增层只需添加子目录，domain 零修改
4. **"给 AI 写 PRD"是一种新技能** — PM 需要学结构化表达，但也让 PRD 对人更清晰
5. **模板命中率决定效率上限** — 高频场景模板化后 70%+ 不需 LLM 参与
