## 二、架构落地：从单体到双 Skill

### 整体架构

**之前**：一个 `/bff-gen` Skill 承担全部 6 步，reference 仅服务 BFF 层，npm 分发。

**之后**：两个独立 Skill + Shell 安装器，reference 跨前端/BFF/后端通用。

```
prd2code-gen (工具仓库)
├── install.sh                    ← curl | sh 一键安装
├── skills/
│   ├── build-reference/          ← Skill 1：领域知识构建
│   │   ├── SKILL.md + workflow.md + steps/ (3 个)
│   ├── prd-distill/              ← Skill 2：PRD 蒸馏
│   │   ├── SKILL.md + workflow.md + steps/ (3 个)
│   └── bff-gen/                  ← 仅 BFF 内部使用
├── examples/                     ← marketing-bff / dive-frontend / dive-backend
└── src/                          ← Node.js（仅 BFF 模板引擎+验证）
```

### 关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 安装方式 | **Shell 脚本**替代 npm | 后端无 Node.js，Shell 是最小公共依赖 |
| 工具拆分 | **2 个独立 Skill** | 前端/后端只需前两步，代码生成各端自己处理 |
| bff-gen | **保留但不推广** | 模板引擎+验证仍需 Node.js，仅 BFF 团队用 |
| 蒸馏输出 | **Markdown + 嵌入 YAML** | 人 review 表格 + AI 读 YAML，不双写 |

### 三端使用方式

| 步骤 | BFF | 前端 | 后端 |
|------|-----|------|------|
| 领域知识构建 | `/build-reference` | `/build-reference` | `/build-reference` |
| PRD 蒸馏 | `/prd-distill` | `/prd-distill` | `/prd-distill` |
| 代码实现 | `/bff-gen`（模板生成） | AI/手动开发 | AI/手动开发 |

### 安装器（install.sh）

- 自动检测项目类型：BFF（serverless.yml）/ 前端（React）/ 后端（pom.xml）
- 部署 `.claude/skills/` + `_reference/` 骨架 + `_output/`
- 支持 `INSTALL_BRANCH` 指定分支

## 四、/prd-distill Skill

### 3 步工作流

| 步骤 | 步骤文件 | 产出 |
|------|---------|------|
| Step 1 解析+路由+代码锚定 | `step-01-parse.md` | 路由结果 |
| Step 2 分类+结构化 | `step-02-classify.md` | 蒸馏草稿（ADD/MODIFY/DELETE） |
| Step 3 确认+输出 | `step-03-confirm.md` | 最终蒸馏报告 |

### 关键设计

1. **代码锚定（Code Grounding）**：`implemented` 仅作初始假设，关键判断必须 Grep/Read 源码
2. **验证来源透明**：`verification_source`（`code_verified` / `reference_only` / `code_contradicts_reference`）
3. **层感知**：从 reference 读 `layer`，按前端/BFF/后端适配
4. **docx 分级回退**：mammoth → pandoc → textutil → 提示用户

## 六、PRD 蒸馏深度调研（28K 字）

### 行业对标

**小红书（arXiv 2603.01460）**：NER 式 PRD 分解
- 配置实体 → 7 类 UI 组件分类，持久化为 Requirement Understanding Artifact
- 微调 Qwen2.5-72B，F1=0.848

**MetaGPT（arXiv 2308.00352）**：结构化通信防级联幻觉
- 消融实验：去掉结构化输出 → 代码质量显著下降
- **结论：结构化中间产物 = 代码质量上限**

### 蒸馏产出格式：Markdown + 嵌入 YAML

- 上半部 Markdown 表格 → 人 review
- 下半部 YAML 代码块 → 下游解析
- **不存在双写**：先生成 YAML，再渲染表格

### 蒸馏报告结构（10 个模块）

```
📋 摘要         ← 活动类型/国家/产品线/新增枚举/影响模块/置信度
⚠️ 需确认项     ← 歧义编号 + 建议 + 置信度
📊 字段清单     ← 按 target_file 分组表格
🔗 联动关系     ← 触发→联动→条件
✅ 校验规则     ← 规则/错误信息/触发条件/严重度
📐 奖励条件     ← 门槛/范围/分支
🧩 业务规则     ← 5 类（campaign_type/visibility/reward_branch/placeholder/alert）
📦 占位符       ← 按奖励分支分组
📦 变更范围     ← 预判影响文件
📦 结构化数据   ← YAML 代码块（source of truth）
```

### 组件锚定（对标小红书 NER）

8 类 d_component 分类完备且互斥：

| 小红书 | 我们 |
|--------|------|
| Input | `Input` / `InputNumber` |
| Select | `Select` / `Radio` / `Checkbox` |
| Picker | `DatePicker` / `CityGroup` |
| List | `GradientTable` |
| Button/Tab/Dialog | 框架/组件库处理 |

## 七、PM 友好模板（长期目标）

### 核心发现

蒸馏产物中 **~70% 是 PM 能填的业务信息**，~30% 是 AI 技术映射。

PM 按模板填 → AI 查表映射 → 蒸馏 Token ≈ 0。

### 模板结构

```
§0 可见性条件 → §1 基础信息 → §2 目标人群 → §3 活动规则
  → §3.1 奖励条件 → §3.2 奖励规则（梯度+分支）
  → §4 触达 → §5 预览 → §6 校验规则 → §7 业务规则
  → §8 批量配置 → §9 查询/列表页 → §10 补刀需求
```

字段表格格式：`# | 字段名称 | 说明 | 输入方式 | 必填 | 默认值 | 可选值 | 数值范围 | 提示文案 | 联动条件`

### 歧义消除效果

| 问题 | 自由文本 PRD | 结构化模板 |
|------|-------------|-----------|
| 订单量上限矛盾 | §3.1 写 1-10，批量写 1-99 | PM 填 `数值范围: 1-10`，无歧义 |
| DLP 是条件还是字段 | AI 需推断 | §0 填可见性变更 + §1 列字段，两者明确 |

### 推广路径

| Phase | 内容 | 状态 |
|-------|------|------|
| Phase 1 | AI 蒸馏 → 模板反向验证 | **当前** |
| Phase 2 | PM 试用模板 → 反馈迭代 | 下一步 |
| Phase 3 | 模板内嵌 PM 工具 → 跳过蒸馏 | 长期 |

## 十三、思考与观察

### 今日是从"设计"到"实现"的转折点

04-22 和 04-23 两天的架构文档（v3→v4→v5）一直在迭代设计方向。04-24 开始把设计落地为实际代码：

1. **Skill 拆分从"长期目标"变成"核心实现"**：之前说 Phase 2 稳定后再拆，现在直接拆了
2. **npm → Shell 是务实选择**：后端没有 Node.js 是硬约束，Shell 是最小公共依赖
3. **代码锚定是关键防御**：reference 可能过期，但源码不会骗人。`code_contradicts_reference` 标签让偏差可见
4. **PM 友好模板是最有想象力的部分**：如果 ~70% 的蒸馏工作可以前置给 PM，那整个管线就变成"PM 填模板 → AI 查表 → 代码"
5. **反馈闭环形成知识复利**：每次蒸馏都在校准 reference，越用越准

### 风险

1. **Skill 编写量不小**：build-reference 3 阶段 + prd-distill 3 步 + install.sh，全是新代码
2. **Shell 安装器的边界情况**：Windows 兼容性、权限问题、网络代理
3. **PM 模板是长期目标**：需要 PM 实际试用验证覆盖率，可能和当前 PRD 格式有 gap
