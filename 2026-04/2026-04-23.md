## 三、格式选型：为什么统一 YAML

基于两项基准测试数据：

| 维度 | YAML | JSON | Markdown |
|------|------|------|----------|
| LLM 理解准确率 | **62.1%** ✅ | 50.3% | 54.3% |
| Token 效率 | **35 tokens**（基线） | 50 (+43%) | 70 (+100%) |
| 注释支持 | **原生 `#`** ✅ | 无 | 原生 |
| 人类可读性 | **优** ✅ | 中 | 优 |

> 来源：[Improving Agents](https://www.improvingagents.com/blog/best-nested-data-format/) 1000 题基准；[ShShell](https://shshell.com/blog/token-efficiency-module-13-lesson-2-format-comparison) Token 对比。

### YAML 统一规范

```yaml
# 文件头（必须）：Step N: 标题 + 用途一句话
# 元数据（必须）：version / layer / project / last_verified / verify_cadence / owner
# 章节分隔：--- 一级章节，-- 二级分组
# Key 命名：snake_case，简洁无冗余后缀
# 注释：字段说明写 key 上方，长说明用 | 折叠块
# 行数限制：每个文件 < 300 行（01 例外）
```

## 四、多轮扫描自动构建方案（⭐ 核心新增）

### 方法论来源

三家权威实践不约而同的设计决策：

| 来源 | 核心方法 | 关键成果 |
|------|---------|---------|
| **Meta Pre-Compute Engine** | 50+ agents 分阶段扫描，5 问框架 | 质量分 3.65→4.20，零幻觉，tool calls -40% |
| **Anthropic Context Engineering** | Sub-agent + Structured Note-taking | 每个 agent 返回 ~1000-2000 tokens 摘要 |
| **Addy Osmani Ralph Wiggum Loop** | 每次迭代任务单一 + 全新上下文 | 四层持久化（Git+Progress+Task+Knowledge） |

**四点共识**：
1. 分阶段扫描，不是一次塞入全部代码
2. 每轮聚焦一个小任务，做完写文件、重置上下文
3. 不确定的标 TODO，人工校准
4. 多轮 Critic 质量评审，防止幻觉

> **Meta 关键洞察**：私有代码库的部落知识不存在于任何模型训练数据中，因此 reference 对私有项目有显著正效果。但对知名开源项目，额外上下文反而是冗余噪声。

### 3 阶段扫描流程

```
Phase 1: Structure Scan（1 轮）
  输入：项目目录 + Git 历史
  动作：扫描目录结构、枚举模块、识别注册入口
  输出：modules-index.yaml（模块地图）

Phase 2: Deep Analysis（2~3 轮）
  输入：modules-index.yaml + 源代码
  动作：对每个模块回答 5 个关键问题
  输出：各模块 .yaml 知识文件
  方式：Sub-agent 并行，每个返回 ~1000 tokens

Phase 3: Quality Gate（3 轮 Critic）
  Round 1: 完整性（自动化）— 路径验证 + TODO 计数
  Round 2: 准确性（人工）— 抽样对照源码
  Round 3: 实用性（端到端）— 真实需求测试
```

### Meta 五问框架

每个模块回答 5 个问题：

| # | 问题 | 产出价值 |
|---|------|---------|
| 1 | 这个模块配置/管理什么？ | 功能定位 |
| 2 | 常见的修改模式是什么？ | 怎么改 |
| 3 | 有哪些非显而易见、会导致构建失败的陷阱？ | 隐式坑 |
| 4 | 跨模块依赖有哪些？ | 依赖关系 |
| 5 | 代码注释中埋藏了哪些隐式知识？ | **产出最多价值** |

> Meta 发现第 5 问产出最多价值：挖掘出 50+ 个从未被文档化的隐式知识。

### Sub-agent Prompt 模板

```markdown
你正在分析 {project} 的 {module} 模块。
阅读文件并回答 5 个问题，严格 YAML 格式输出。
不确定的一律标 TODO + confidence: low。
文件路径必须实际存在，不猜测。
总长度控制在 35 行以内（~1000 tokens）。
```

### TODO 标记与人工校准

```yaml
# 好的 TODO：具体、可操作
todo: "人群定向 6 种方式的优先级和互斥关系需要运营确认"

# 差的 TODO：模糊
todo: "不确定"
```

校准流程：统计 TODO → 确认正确的删 TODO → 错误的修正 → 无法确认的标 needs_domain_expert → 全部清零后进入 Round 3。

### 持续更新

- 周期：14 天自动刷新
- 触发：新增活动类型 → 增量更新；目录结构变更 → 重新结构扫描
- > "Context that decays is worse than no context at all." — Meta

## 五、实验验证结果

### 设计

| 组别 | Reference | 控制变量 |
|------|-----------|---------|
| 基线组 | 无 | 同一 PRD / 项目 / 模型 |
| 实验组 | 有 | 同一 PRD / 项目 / 模型 |

### 评分维度

| 维度 | 权重 | 评分标准 |
|------|------|---------|
| 变更分类 | 35% | schema_driven / modification / new_code / deletion / no_change |
| 影响范围 | 25% | 命中目标文件数 / 实际改动文件数 |
| 具体变更描述 | 25% | 描述是否准确反映实际改动 |
| 工作量判断 | 15% | 无/小/中/大 |

### dive-frontend ShiftCheckin 活动实测

| 指标 | 基线组 R01 | 实验组 R02 |
|------|-----------|-----------|
| **总分** | 54 | **68** (+26%) |

**关键发现**：reference 帮助 AI 正确识别 d_actions 联动边界。

**踩坑经验**：reference 必须反映 PRD 实施**之前**的项目状态。如果 reference 已包含新需求的能力（implemented=true），实验组会误判为「无工作」。

### 目标指标

| 指标 | 目标 |
|------|------|
| 变更分类准确率 | > 90% |
| 枚举正确率 | 100% |
| 反模式检出率 | > 80% |

## 九、风险与防御（新增）

| 风险 | 防御 |
|------|------|
| **幻觉**（编造文件/函数） | Phase 3 Round 1 自动验证所有路径 |
| **业务盲区**（AI 无法推断业务规则） | 5 问第 5 问 + TODO + 人工校准 |
| **Context Rot**（上下文过长） | Sub-agent 独立上下文 + 文件 ≤ 300 行 |
| **过度文档化**（对公开项目噪声干扰） | Compass 原则：只写项目特有的 |
| **过期知识** | 14 天自动刷新 + 变更触发 |
| **实验偏差**（reference 反映实施后状态） | 实验前检查 implemented 状态 |

## 十二、思考与观察

### 方向性变化评估

本次更新不是增量调整，而是**方向性重构**：

1. **9→7 文件 + 主题→Step**：更符合 AI 消费习惯，每个 Step 只读一个文件
2. **混合→统一 YAML**：有基准测试数据支撑，LLM 准确率 +12pp、token 省 30%
3. **人工构建→多轮扫描**：引入 Meta 50+ agents 经验证的自动化方案
4. **无数据→有实验**：基线 54 vs 实验 68 提供了 reference 有用的初步证据

### 待观察的风险

1. **YAML 300 行限制**：01-prd-distill 的 prd_routing 表可能超过 300 行，需要分区策略
2. **Sub-agent 工具化成本**：将 3 阶段扫描封装为 CLI/Skill 需要额外开发
3. **实验样本量**：目前只有 1 个 PRD（ShiftCheckin）的测试数据，需要更多样本
4. **跨层 YAML 格式统一**：后端层的 reference 格式是否能同样精简存疑
