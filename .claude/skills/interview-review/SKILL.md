# /面试复盘

面试结束后使用。自动诊断失分点，提取面试官洞察，生成改进方案，并回流新知识点到手机速查表。

## 用法

```
/面试复盘 星河觉醒 一面
```

然后粘贴录音转写文本。也可以一句话全给：

```
/面试复盘 星河觉醒 一面
[粘贴转写文本]
```

## 执行流程

### 第一步：清洗录音转写

1. **识别说话人**：区分面试官和候选人（候选人发言通常包含项目细节和技术术语）
2. **纠正ASR错误**：常见映射：
   - AR → AI, 四编码/SSC码 → Schema, PSG → PRD
   - note → Node.js, 壁府/地F → BFF, 衣服/VIVO → eval
   - PI pd/PI ID → PRD, 歪理 → Value, 模板琴 → 模板引擎
   - roll2 → Rollup, 仿修 → 返工, 跑领领008 → 跑了0-0-8
3. **提取Q&A对**：整理成结构化问答

输出清洗后的 Q&A 对列表。等用户确认后继续。

### 第二步：逐题深度分析

对每一道题输出：

**Q[N]: [问题]**
- 表面问的是：X
- 实际考察的是：Y
- 答到的要点：列出来
- 漏掉的关键点：列出来

| 维度 | 分数(1-10) | 问题 |
|------|-----------|------|
| 知识完整度 | ? | 漏了什么 |
| 表达清晰度 | ? | 哪里冗余/绕弯/口头禅多 |
| 追问防御力 | ? | 哪些追问没防住 |

**改进版答案**：
- 核心句（20字以内）
- 口语展开（3-5句）

**搜索增强**：对以下情况的题目，先用 WebSearch 搜索相关知识再写改进答案，确保答案有高质量的外部知识支撑：
- 候选人明显答不上来或答错的题（如被问"Tools vs Skill区别"时卡住）
- 面试官补充了候选人不了解的知识点的题
- 涉及行业前沿趋势但候选人的回答停留在表面的题

搜索策略：
- 用英文关键词搜索，获取更高质量的结果
- 优先参考权威来源（官方文档、学术论文、行业头部博客）
- 将搜索结果中的关键数据/结论融入改进答案，并标注来源

**保存复盘产物**：逐题分析完成后，**立即**将所有改进版答案写入 `reviews/qa/[公司名].md`，不要延迟到后续步骤。格式：

```markdown
# [公司名] [轮次] — 面试优化答案

> 日期：YYYY-MM-DD | 通过概率：XX%

---

## Q1: [面试官问题]
**核心句**：[20字以内]
**展开**：[3-5句口语化回答]

## Q2: ...
```

如果 `reviews/qa/` 目录不存在就创建。

### 第三步：面试官洞察提取

**录音转写中不仅包含求职者的回答，还包含面试官分享的高价值信息。** 这些信息来自行业前沿实践，对未来面试和职业判断都极有价值。

对每道题，额外分析面试官说了什么：

1. **公司实践**：面试官描述了什么AI工程化实践？（架构设计、技术选型、落地阶段、团队规模、工作方式）
2. **技术纠正/补充**：面试官纠正了求职者的哪些错误认知？补充了哪些求职者不知道的知识？（概念纠正、原理补充、方案建议）
3. **行业判断**：面试官对技术趋势有什么前瞻性判断？（能力边界判断、趋势预测、成本权衡）
4. **追问深层意图**：面试官的追问链路暴露了什么考察维度或价值观？
5. **反问环节**：候选人问面试官的问题，面试官的回答中包含大量有价值信息——团队规划、技术方向、工作方式、产品策略。这些信息对后续面试和职业判断都有直接参考价值。

**特别注意面试官的"喂答案"行为**：面试官有时会引导、纠正或补充候选人的回答。这些内容本身就是高质量的行业实践信息，必须记录下来。比如：
- 面试官说"传播性是不是更好"→ 候选人没想到的维度
- 面试官说"这块你没做过吧"→ 暴露短板的同时也给了方向
- 面试官介绍团队工作方式（80-90%代码AI生成）→ 行业前沿实践

**保存面试官洞察**：分析完成后，将面试官洞察作为独立板块追加到 `reviews/qa/[公司名].md` 末尾，格式：

```markdown
---

## 面试官洞察（[公司名] [团队]）

> 面试官描述的一句话概括。

### 洞察1：[标题]

**面试官原话/描述**：[摘录或概括]

**价值**：[这个洞察对求职者未来面试或技术判断的价值]
```

### 第三步半：改进版答案二次增强

面试官洞察提取完成后，回到 Q&A 文件中的改进版答案，做一次二次增强：

1. **融入面试官补充**：如果面试官对某题补充了知识或纠正了候选人的认知，将这些内容融入对应题目的改进答案中
2. **融入反问回答**：如果反问环节中面试官的回答与某道题相关，将关键信息作为"行业佐证"补充到答案中
3. **标注来源**：改进答案中涉及面试官补充的内容，用括号标注来源公司，如"（腾讯面试官确认了类似机制）"

这样每条改进答案就同时包含了：候选人的反思 + 面试官的补充 + 外部搜索的权威知识。

### 第四步：整体评估

| 项目 | 结果 |
|------|------|
| 通过概率 | XX%（理由） |
| 最大加分点 | |
| 最大失分点 | |
| 面试官风格判断 | |

**表达诊断**：
- 口头禅统计（"然后""就是""对吧"出现几次）
- 回答平均长度（太长/太短）
- 结构性（是否有开头-展开-收尾）

**AI 工程化 3 硬伤专项扫描**（13 场面经归纳的系统性失分，必检）：
- ① 数据被追穿：是否出现量化数据（提效/准确率/一致性）但说不清口径或被追问崩盘？每个数字必须备"怎么算的"话术
- ② 原理卡壳：是否被问到 AI 底层原理（图片传参/Function Calling/Skills 触发/LangGraph/RAG 机制）答不出或讲错？标注具体概念
- ③ 闭环回避：是否被问端到端闭环（执行 / Review / 自修复 / eval 反哺 / CI 对齐）而只用"老板让先做前两步"回避？**必须答出端到端原型 prd2code-gen 完整 6 步**（蒸馏 → 分析 → Plan → 双路径生成 → 自验证自修复 → eval/auto-tune 反哺），并讲清**双轨定位**——团队推广版聚焦 PRD→Plan 是 ROI 选择（人工确认后才开发、风险低、提效明显），端到端原型已设计跑通作为下一代探索。每一层都能讲出具体工程化难点（断点续传 progress.yaml / 震荡检测 A→B→A / verified_by 审计链 file:line 溯源 / drift-report 漂移检测 / Handlebars 模板 0 token 优先 + LLM 兜底）才算过。被问"既然完整为啥不推广" → "PRD→Plan 是 ROI 最高风险最低环节，代码生成涉及质量责任和 CI 对齐，先用原型验证设计可行性再决定工程化投入"。

### 第五步：知识回流

**这是最重要的一步。** 分析面试中暴露但现有知识库没覆盖的知识点，生成 learning/interview-tools/stealth.html 卡片。

**回流来源有三类（按优先级排序）**：
1. **面试官洞察**：面试官分享的行业实践、技术纠正、前瞻判断——这些是面试中获取的最珍贵信息，值得沉淀为卡片
2. **求职者短板**：卡壳、回答不准、被追问穿的知识点
3. **多源验证的行业共识**：同一知识点在多场面试中被不同面试官提到——说明是高频考点，必须沉淀

**结合源码分析**：如果面试涉及候选人自己的项目（如 prd-tools、Schema中台等），优先在 `reviews/transcripts/` 找面经原文核对候选人当时的技术描述；若需源码级核实，向用户索取项目源码路径（项目源码不一定在本机，滴滴内网项目尤甚），基于实际实现生成准确的知识卡片，不能凭记忆或面试时的模糊回答来写。

判断回流方式：
- 某题卡壳，且现有卡片没覆盖 → **新增卡片**
- 面试官补充了现有卡片缺少的关键信息 → **更新现有卡片**
- 同一知识点在多场面试出现 → **新增卡片**（多源佐证让答案更有说服力）
- 涉及自己项目但讲不清楚 → 必须读源码后再回流
- 现有卡片已覆盖且无新信息 → **跳过**

先读取 learning/interview-tools/stealth.html 确认当前卡片状况。对每个回流点判断是"新增"还是"更新现有"：

**新增卡片**——为每个回流点生成卡片HTML：

```html
<div class="card" data-c="[分类]" data-kw="[关键词1] [关键词2]">
  <div class="card-head">
    <span class="tag [分类]">[标签]</span>
    <div class="card-title">[面试题/知识点]</div>
  </div>
  <div class="core">[核心答案，20字以内]</div>
  <div class="card-body">
    <p>[展开说明]</p>
    <p><strong>[关键点]</strong>[详细解释]</p>
  </div>
</div>
```

分类规则（data-c）：
- React/浏览器/TS/CSS基础 → `bagu`，标签 `#N`
- 项目设计/架构/工程化 → `proj`，标签 `A N`
- AI/LLM/Agent/RAG → `ai`，标签 `AI`
- 手写代码/算法 → `code`，标签 `手撕`
- 面试策略/话术/反问 → `emer`，标签 `软技能` 或 `反问`

输出：
1. 先列出所有待回流的知识点，标注"新增"或"更新现有：[卡片标题]"
2. 等用户确认
3. 确认后输出完整卡片HTML（新增）或更新内容（更新现有）
4. 告诉用户应该插入/修改 learning/interview-tools/stealth.html 的哪个位置（行号范围）
5. 用户确认后执行插入/编辑，然后 bump sw.js 的 CACHE_NAME

**更新现有卡片**：在现有卡片的 card-body 末尾追加新段落，补充面试官的额外信息或纠正。格式：
```html
<p><strong>[来源：XX面试官补充]</strong>[补充内容]</p>
```

### 第六步：改进计划

**今晚必做3件事**：每件说清补什么 → 怎么补 → 补到什么程度

**下次面试注意**：2-3条针对性建议

### 第七步：自动提交推送（不询问用户）

复盘流程结束后，自动执行以下操作，**不需要等用户说"提交"**：

1. `git add` 所有变更文件（录音转写、Q&A答案、learning/interview-tools/stealth.html、sw.js）
2. `git commit`，消息格式：`feat: [公司名][轮次]复盘 — [新增/更新卡片摘要] + Q&A答案`
3. `git push` 到远程 main 分支
4. 输出 commit hash 和推送结果

### 可选第八步：历史Q&A批量Review

如果用户要求对历史Q&A文件做批量Review（如"review一下之前的Q&A"），执行以下流程：

1. **扫描文件**：列出 `reviews/qa/` 和 `reviews/transcripts/` 中所有文件，匹配录音和Q&A对
2. **并行Review**：对每个Q&A文件，启动独立的sub-agent并行处理：
   - 读取录音和Q&A，对比找出遗漏的面试官洞察
   - 识别需要改进的答案，用 WebSearch 搜索补充
   - 直接编辑Q&A文件补充洞察和改进答案
3. **每个agent独立提交推送**：避免合并冲突
4. **汇总输出**：所有agent完成后，汇总新增洞察数量和改进答案数量

## 硬性原则

- 基于事实分析，表现差直说差在哪，不安慰
- 不编造用户没说过的回答
- 面试官洞察必须标注来源（哪家公司、什么团队），不能模糊化
- 知识回流只生成真正缺失的卡片，不重复已有知识点
- 面试官分享的行业实践和技术判断是比求职者短板更珍贵的回流素材
- 每张卡片的核心答案必须能在手机上3秒内读完
- 改进动作具体到"今天花30分钟做xxx"，不要笼统说"加强练习"
- 插入 learning/interview-tools/stealth.html 后必须 bump sw.js 的 CACHE_NAME（kb-vN → kb-vN+1）

## 卡片内容质量规则（防止"写得好但面试念不出来"）

### 1. 聚合不拆散

**同一面试题的知识点必须聚合到一张卡片里，不要拆成多张。**

反面案例：面试官问"介绍你的AI工作流"，答案散在5张卡片（定位/双插件/证据链/质量门控/评分），面试时翻不过来。

正面案例：一张"AI工作流完整介绍"卡片覆盖：是什么→为什么→怎么做（三大支柱）→效果→边界。

**规则**：如果回答某个面试题需要翻2张以上的卡片，说明需要合并。细节卡片保留给追问环节。

### 2. 可朗读性检查

**每张卡片的内容必须能直接念出来。禁止以下写法：**

| 禁止 | 原因 | 替代 |
|------|------|------|
| 缩写不展开（BF、SSC、PI ID） | 念不出来 | 首次出现写全称 |
| 箭头流无解释（A→B→C） | 口头说不清 | 每步用完整句子 |
| 括号压缩10个东西 | 口头报不完 | 拆成独立行或只列3个关键的 |
| 层级编号无上下文（L1/L2/L3） | 不知道在说什么 | 写明"第一层XX、第二层YY" |
| 代码/文件名当正文 | 面试不说文件名 | 用自然语言描述 |

### 3. core 字段 = 面试第一句话

`core` 字段必须是面试时脱口而出的**完整句子**，不是一个短语。

| 错误 | 正确 |
|------|------|
| `SSOT+证据链+多层门控保质量` | `把PRD自动蒸馏成结构化开发计划，靠SSOT+证据链+多层门控保质量` |
| `目录千差万别，按能力面描述` | `目录结构千差万别，但研发协作的能力面是稳定的——按能力面描述` |
| `WASM + ONNX Runtime 跑小型模型` | `能，WASM + ONNX Runtime 跑小型模型` |

### 4. 质量自检清单

生成每张卡片后，逐项检查：
- [ ] core 是完整句子，能直接念出来？
- [ ] card-body 每段是自然语言，不是缩写/代码/文件名列表？
- [ ] 面试官问这个问题，翻这一张卡就能答完，不用再翻第二张？
- [ ] 跟 Q&A 文件中的优化答案对比，卡片内容是否更差？如果是，用 Q&A 答案替代。
