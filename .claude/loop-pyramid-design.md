# Loop 系统金字塔优化设计 v1

> **用途**：(1) loop 系统从"造卡机"升级到"金字塔能力建设机器"的设计文档；(2) 面试场上讲"我在做什么、下一步怎么演进"的素材。
> **现状**：loop v251 已稳定运行（单 cron 串行 + subagent + 状态机），自动产出 18 版本。
> **目标**：让 loop 不仅优化"卡的风格"，更按 **L1-L5 五层金字塔**系统化建设面试能力，自动识别薄弱层并主动补缺口。

---

## 一、为什么升级（钩子）

**当前 loop 的局限**：
- 选卡只按"卡类型"优先级（`ai > proj > AI 工程化 > 综合答题型`），**没有金字塔层级维度**
- 补缺口（第 8 步）只是"随机搜面经"，**不针对薄弱层**
- 没有"五层覆盖度"的可观测指标——不知道哪层强、哪层弱

**升级的核心洞察**：**面试能力 = 五层金字塔 × 每层深度**。当前 loop 在"每层深度"上做得好（口语化改写），但**没有层级的体系感**——可能 L3 改了 30 张，L4 只有 2 张，L5 一张没有，但系统不知道。

**金句**：**"流水线能跑只是起点，能往战略方向跑才是工程。"**

## 三、选卡逻辑改造（最关键的工程变更）

### 当前逻辑（loop-worker.md 第 1 步）
```
优先级：AI 卡 > Schema 项目卡 > AI 工程化卡 > 综合答题型
```

### 升级后逻辑（金字塔驱动）
```
1. 计算五层覆盖度（grep data-kw + 关键词映射 → 每层卡数 / 平均质量分）
2. 找最薄弱层（卡数 < 阈值 或 平均质量分 < 阈值）
3. 选卡优先级：
   a. 薄弱层的"待优化"卡（最高优先）
   b. 薄弱层的"补缺口"（次高，走第 8 步 WebSearch + 造卡）
   c. 非薄弱层的"待优化"卡（按原 ai > proj 排序）
4. 每个层级的"待优化"卡，改写时 fresh 读对应层样板（保证深度对齐）
```

**关键创新**：**worker 启动时先跑"金字塔健康度检查"**——输出五层覆盖度报告，再决定本轮改谁。

## 四、金字塔健康度算法（可观测性）

> 这是 loop v2 → v3 的核心升级：从"流水线"升级到"可观测流水线"。

### 五层覆盖度计算

```python
# 伪代码
def pyramid_health():
    layers = {'L1': 0, 'L2': 0, 'L3': 0, 'L4': 0, 'L5': 0}
    quality = {'L1': [], 'L2': [], 'L3': [], 'L4': [], 'L5': []}

    for card in all_cards:
        layer = classify_card(card)  # 按 data-kw 关键词分类
        layers[layer] += 1
        quality[layer].append(score_card(card))  # 长度/数字密度/念法完整度

    return {
        layer: {
            'count': layers[layer],
            'avg_quality': mean(quality[layer]) if quality[layer] else 0,
            'gap': max(0, THRESHOLD - layers[layer])
        }
        for layer in layers
    }
```

### 卡片分类规则（data-kw 关键词映射）

| 层级 | 关键词触发 |
|---|---|
| L1 前端基础 | JS / HTTP / 浏览器 / React / Vue / Webpack / Vite / 事件循环 / 缓存 |
| L2 AI 概念 | LLM / Transformer / Attention / Agent 架构 / RAG / 微调 / Embedding |
| L3 AI 工程化 | 流式 / Token / Function Calling / MCP / Agent Loop / Subagent / Hooks / 上下文 / 错误降级 / Prompt 注入 / 成本 / 质量门控 |
| L4 系统设计 | 架构 / 高并发 / 设计 / SPARE / 整体方案 / 系统设计 / 抢票 / 工作台 |
| L5 业务判断 | 为什么 / 取舍 / 边界 / 该不该 / 业务价值 / 产品判断 / 飞轮 |

### 阈值（v1 起步）

- 每层最少 **8 张** 卡（低于则触发补缺口）
- 每层平均质量分 ≥ **0.7**（低于则优先优化该层）

## 五、补缺口策略（按层级定向补）

### 当前补缺口（loop-worker.md 第 8 步）
```
WebSearch "大厂 AI 前端 面试 面经 2026 字节 蚂蚁 大麦 TikTok"
→ 抓 1 篇 → 对照现有 data-kw 找真缺口 → 补 1 张新卡
```

### 升级后补缺口（金字塔驱动）
```
1. 跑 pyramid_health() 找最薄弱层
2. 按层级定向搜索：
   - L1 薄弱 → WebSearch "前端面试 2026 高频 JS React 浏览器"
   - L2 薄弱 → WebSearch "LLM Agent RAG 面试题 大厂 2026"
   - L3 薄弱 → WebSearch "AI 工程化 流式 Function Calling MCP 面试"
   - L4 薄弱 → WebSearch "AI 前端 系统设计 面试 真题 大厂"
   - L5 薄弱 → WebSearch "AI 产品经理 业务判断 取舍 面试"
3. 抓 1 篇 → 提炼 → 造该层级的卡（data-kw 必须含该层关键词 + 项目桥接必须真实）
4. 改写时 fresh 读对应层样板
```

**关键升级**：补的卡**必须明确归到某层**，并在 hint 标注 `[loop·L层级·已优化·待复核]`。

## 六、新增度量指标（v3 可观测性）

> loop 现在的短板是"没有指标"，这套是 v3 的核心。

### 核心指标

1. **五层覆盖度**：每层卡数 + 占比 + 缺口
2. **每层质量分布**：高/中/低质量卡占比
3. **改写效率**：每轮 token 消耗 / 改写前后字数比
4. **复核通过率**：worker 产出 vs reviewer 通过比（当前 ≈ 100%，可能太宽松）
5. **风格漂移度**：每轮卡与样板的相似度（cosine similarity of TF-IDF）
6. **金字塔演进**：每周各层卡数变化趋势（向上 = 能力在补齐）

### 报告输出

每跑完 20 轮（约 1 小时）输出一份 `.claude/loop-pyramid-report.md`：
- 当前五层覆盖度（数字 + 表格）
- 本轮改了哪张卡 + 归到哪层
- 薄弱层提示（"L4 系统设计仅 3 张，建议补"）
- 历史趋势（v233 → 当前）

## 九、风险与边界（诚实交代）

1. **金字塔分类的准确性**：data-kw 关键词映射不是 100% 准，会有错分（如系统设计卡被分到 L3）—— v1 接受 80% 准确率，后续考虑加 LLM 分类
2. **补缺口的卡质量**：联网搜来的面经质量参差，可能补进低质卡—— 靠 reviewer 把关 + golden case 回归
3. **度量指标的局限性**：质量分是启发式（长度/数字密度/念法完整度），不一定反映"真能答好面试"—— 后续考虑接真实面试结果回流（哪张卡被问到了、答得好不好）
4. **过度自动化的风险**：让 AI 自己决定学什么，可能加剧信息茧房—— 每月人工 review 一次金字塔健康度，强制干预薄弱层
