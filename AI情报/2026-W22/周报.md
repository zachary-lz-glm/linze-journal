## 三、架构/方案速写

### 方案：n8n + Claude Code + RAG 的工业化内容生成管道

- **解决什么问题**  
  传统 SEO 内容团队依赖人工写手，产出慢且风格不一致。引入 AI 后，单次生成质量无法保证，且无反馈闭环。本方案通过工作流引擎标准化了整个流程，实现批量、可监控、可迭代的内容生产。

- **核心设计思路**  
  用 n8n 作为调度中心，一条典型的流水线如下：
  ```
  [关键字列表读取] → [RAG 检索相关文档/FAQ] → [组合提示词] 
  → [调用 Claude Code 生成 HTML/Markdown] → [质量评分（如清晰度、重复率）] 
  → [不通过则自动追加修改指令重新生成] → [发布或存入 CMS]
  ```
  形象类比：这就像把“一个编辑团队”拆成了“检索助理 + 写手 + 校对”，而 n8n 是那个不留情面的主编，自动推进流程并卡住不合格的稿件。

- **关键 trade-off**  
  牺牲了单篇内容的极致创意（因为受提示词和 RAG 结果约束），换来了**每周 100 页的稳定产出和品牌风格一致性**。对于工业内容，这个牺牲是值得的。

- **与前端工程师的结合点**  
  可将此管道扩展为“PRD 自动生成器”：输入产品变更描述 → RAG 查询现有 PRD/API 文档 → 生成结构化 PRD → 输出为 Notion 或 Markdown。你仅需定义 RAG 索引和输出模板，无需训练模型。

## 四、面试弹药库（1-3 个）⭐

**1. 面试官问：“AI 编码代理在长期任务中经常忘记上下文，你如何解决？”**  
- **30秒回答**：我会引入一个专用的记忆层，比如 `ai-memory` 项目，它用 Rust 实现，支持不同 CLI 代理之间的 handoff。实测中，我们可以把一次重构拆成“分析 → 生成 → 测试”三个 agent 调用，每次的结果存入记忆，最后一个 agent 直接读取完整上下文，不再需要人工复制粘贴。

- **深入追问准备**：可以讲记忆存储的具体数据结构（会话 ID、任务描述、文件 diff 摘要），以及如何在 Rust 中使用 `sled` 实现持久化，并且讨论如何避免记忆污染导致错误决策。

**2. 面试官问：“你们团队在使用 Claude Code 这类工具时，如何确保安全和成本可控？”**  
- **30秒回答**：我们用 `ccglass` 做全量流量监控。它是一个本地代理，能捕获 Claude Code 发送给 API 的所有请求和响应，提供 Web 界面搜索。上周我们发现某个同事的配置错误导致重复上传 300KB 的无用上下文，每月浪费约 $40，通过仪表盘直接定位到问题配置。

- **深入追问准备**：可以说明如何扩展 ccglass 加入 token 用量告警、基于规则的敏感信息过滤，以及如何与公司现有的日志系统（如 Loki）集成。

**3. 面试官问：“如果让你设计一个企业级 AI 网关，你会考虑哪些核心模块？”**  
- **30秒回答**：我会参考 `awesome-architecture` 中的 AI 网关架构图，核心至少包括四层：认证与鉴权、路由与负载均衡（多模型多供应商）、缓存层（语义缓存减少重复调用）、以及观测层（日志/指标/追踪）。这样前端团队只需对接网关的单一 API，后端可以随意切换模型或进行 A/B 测试。

- **深入追问准备**：可以画出简图，对比 AWS API Gateway 模式的异同，讨论缓存的 key 设计（基于 embedding 的相似度阈值），以及请求重试和降级策略。
