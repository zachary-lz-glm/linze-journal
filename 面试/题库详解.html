A1 Monorepo 怎么管理 15+ 业务模块？ ▶
核心答案

营销平台 是滴滴国际化营销中台的前端主项目，用 Lerna + Nx 管理 15+ 业务模块（营销、司机、乘客、激励等），采用 Yarn workspace 做依赖管理。

架构分两层：

  • app/ 目录：15+ 业务 App，每个是独立子包（营销、司机、激励等）
  • packages/ 目录：共享包（components、RichText、utils）

关键设计决策：

  1. 独立版本：lerna.json 配置 "version": "independent"，每个业务模块独立发版
  2. Nx 加速："useNx": true 开启构建缓存，只构建变更影响的包
  3. Yarn 锁版本：强制用 yarn，yarn.lock 锁定依赖版本，避免第三方包升级导致环境崩溃（踩过 @yarnpkg/parsers 升级不兼容 node16 的坑）
  4. 共享组件抽离：公共组件放 packages/components，业务模块通过 workspace 协议引用
  5. 统一工具链：根目录统一 ESLint + Prettier + Husky + commitlint，所有子包继承
口语化表述
营销平台 是我们营销中台的前端主项目，用 Lerna + Nx 管理。架构分两层：app 目录放 15 个业务模块，packages 放共享组件和工具。每个业务模块独立版本、独立部署，通过 workspace 协议引用公共包。选 Lerna 而不是纯 Turborepo 的原因是历史遗留——项目启动时 Turborepo 还不够成熟。我们开了 Nx 加速构建缓存，只构建变更影响的部分。

A2 Schema 驱动架构怎么设计？ ▶
核心答案

marketing-bff + 营销平台 采用三层 Schema 驱动架构：

后端 API (原始数据) → BFF 层 (Schema 生成) → 前端层 (Schema 渲染)

BFF 层 — Schema 生成（marketing-bff）：

BFF 用 自研模板引擎 模板引擎驱动 Schema 生成，不是手写 JSON。每个步骤有独立模板函数（basic/group/rules/message/preview），模板定义字段的取值路径 + 生成逻辑。

1. 模板驱动 + 步骤分发：

前端请求 Schema 渲染 API（指定步骤）
    ↓
getRenderSchemaTemplate(step, ctx)
    ↓ switch(step)
    ├── BASIC  → basic.ts   // ~1200 行，20+ 字段
    ├── GROUP  → group.ts    // 目标人群
    ├── RULES  → rules/      // 规则（最大最复杂，25 种活动类型各有子模板）
    ├── MESSAGE → message.ts // 推送/标题/描述 ~980 行
    └── PREVIEW → preview.ts

每个模板返回 TemplateStuff（hooks + template entries），Builder.parseOnce 遍历 entries 执行 handler，hooks 自动合并后端数据（mixinDefaultFields 合并 visible/disabled/options，mixinValue 合并 value）。

2. 字段管理 — 分布式定义 + handler 复用：

字段定义按模板分布式管理（不存在集中注册中心）：

  • basic.ts：city, campaign_type, trigger_type, budget 等 20+ 字段
  • group.ts：range_type, target_group, control
  • message.ts：title, description, push, os2_push
  • rules/details/*.ts：25 种活动类型各自的规则详情

联动模板（linkageSchema）从 render 模板中提取单个 handler 复用，保证联动返回与初始渲染的 Schema 结构一致。

3. injectDActions — 联动双重求值机制（BFF 最精巧的设计）：

联动配置通过 injectDActions 声明式注入，一次声明、双重求值：

  • BFF 端：safeEval 用当前值求值 → 设置初始 visible/disabled 状态（SSR 初始状态正确）
  • 序列化到 d_actions → 前端运行时再次求值（客户端联动）

联动分两种：

  • fetchApi 型（重服务端）：依赖变化 → 前端调 BFF 接口 → BFF 重新生成该字段 Schema → 前端替换
  • state 型（纯前端）：依赖变化 → 前端本地 eval 表达式 → 更新 visible/disabled

4. 联动模板 — 16 个 linkageSchema 文件：

  • 字段重渲染型 12 个（campaignType, triggerType, rewardCondition, rules 等）：从 render 模板提取 handler 用新数据重新生成
  • Placeholder 重置型 4 个（titlePlaceholder, descriptionPlaceholder, pushPlaceholder, os2pushPlaceholder）：只清空 value

前端层 — Schema 渲染引擎（营销平台/app/dive）：

1. Schema 迭代渲染（StepForm）：BFF 返回 content_schema 数组，前端遍历渲染。映射方式是双轨制：

  • switch-case：基础组件（Radio、Select、DatePicker 等）按 d_component switch
  • RegisteredComponents 注册表：35+ 业务组件按名字注册，支持动态加载

2. 字段路径拍平 — formatName（核心设计）：

页面有层级（步骤 → 卡片 → Form.List 行 → 字段），formatName 拍平为 dot 路径：

"rules.reward.0.method" → 规则 > 奖励 > 第 0 行 > 发放方式

这个路径同时是 Redux 存储 key 和 listener 事件名。

3. 联动引擎 — d_actions + listener 发布订阅：

字段A 变值 → onFormChange → listener.emit("basic.campaign_type", data)
                                         ↓
字段B 的 d_actions.dependencies 包含 "campaign_type"
→ FieldWrapper 挂载时 listener.on("basic.campaign_type", parseCallback)
→ parseCallback → fetchApi 型回调 BFF / state 型本地 eval

4. 联动表达式解析：state 型联动通过 eval 轻量解析（正则提取 {{...}} → 变量替换 → eval），四套解析函数（RuleCard 的 parseVisible / RewardCondition 的 getExpression / FuelReward 的 parseVisible / tools.ts 的 schemaExpression），eval 可访问的变量因组件而异（dataSet / rowValue / formInstance / props）。

5. 状态管理三层：

  • Redux（saveFormData）：全局表单数据，key 是拍平路径
  • Form 实例（Form.useForm()）：字段级操作
  • Listener 事件：字段变更通知 + 联动触发
口语化表述
Schema 驱动架构分 BFF 生成和前端渲染两层。BFF 层用模板引擎驱动——每个步骤有独立模板，模板定义字段取值路径和生成逻辑，Builder 编译出 content_schema。字段定义按模板分布式管理，联动模板从 render 模板提取 handler 复用，保证结构一致。BFF 最精巧的设计是 injectDActions 双重求值——一次声明联动配置，BFF 端用 safeEval 求初始状态（SSR 就是对的），同时序列化为 d_actions 给前端运行时联动。联动分 fetchApi 型回调 BFF 重生成和 state 型前端本地 eval。前端拿到 Schema 后通过 formatName 拍平嵌套路径，用 listener 发布订阅执行联动。这样新增活动类型前端零代码，BFF 配模板和联动就行。
加分点
讲清楚 injectDActions 双重求值 + fetchApi/state 两种联动类型 + 模板 handler 复用 → 体现对 BFF+前端全链路的深度理解
A3 400+ 权益的组件库怎么设计？ ▶
核心答案

权益组件库（v1.x）是营销平台的权益组件库，管理 400+ 权益的创建、编辑、复制、查看全生命周期。

架构设计：

  1. Register 注册模式 — 内置 24 组件 + 运行时扩展：构造时注册 24 个内置组件（通用 16 + 3个业务线 8），add()/addMultipleCmps() 支持消费方运行时注册自定义权益组件
  2. Schema 驱动渲染 — 和 营销平台(dive) 完全同构：SDK 也是 Schema 渲染引擎的消费者——后端返回 form_data（含 d_component + content_schema），前端通过 FieldWrapper 迭代渲染。和 营销平台(dive) 用同一套 Schema 渲染模式，两个消费者共享一个架构
  3. 双入口设计 + CRUD 独立 Schema：
    • 组件模式（<App> 内嵌）+ 函数调用模式（Modal.info 命令式），统一用 BenefitAdmin 分发
    • 每个 CRUD 操作（创建/查看/编辑/复制）从不同 API 获取专用 Schema
  4. Rollup 双模打包 + 三输出：ESM（es/index.js）+ CJS（dist/index.js）+ .d.ts（rollup-plugin-dts），exports 字段条件导出
  5. 设计系统集成：深度依赖 公司设计系统库，保持跨业务线 UI 一致
  6. Changesets + zx 发版：alpha/release 双环境，本地 yalc 联调，zx 脚本自动化
口语化表述
benefit-sdk 是我们的权益组件库，覆盖 400 多种权益的 CRUD。核心设计是 Register 注册模式——构造时注册 24 个内置组件，同时支持运行时扩展。最关键的是，SDK 本身也是 Schema 驱动渲染的消费者——后端返回的 form_data 里有 d_component 和 content_schema，前端用 FieldWrapper 迭代渲染，和 营销平台(dive) 完全同构。也就是说 Schema 驱动不是"某个前端项目的事"，而是横跨整个营销中台的统一架构。打包用 Rollup 双模输出 ESM+CJS+类型声明，发版用 Changesets + zx 脚本自动化。

A4 AI 工程工作流怎么保证可靠性？ ▶
核心答案

prd-tools（v2.19.9）是一套 PRD 工程工作流，核心不是"生成代码"，而是把 PRD 转成有证据、可执行、可回流的结构化开发计划。可靠性靠证据链 + 代码锚定 + 多层质量门控 + 反馈回流闭环：

  1. 证据链驱动：每个非显然事实必须有证据（8 种类型：code/prd/tech_doc/git_diff/negative_code_search/human/api_doc/reference），搜不到也是证据（negative_code_search），不确定标 low confidence
  2. 变更分类 + 代码锚定：PRD 蒸馏阶段不是直接生成代码，而是先分类（ADD/MODIFY/DELETE/NO_CHANGE），每个分类必须锚定到实际源码
  3. PRD Ingestion 质量门禁：不把 .docx 直接丢给 LLM，先用 Python zipfile 提取（零第三方依赖）+ 图片提取 + 质量门禁（pass/warn/block），保证可追溯
  4. Reference v4.0 SSOT 架构：6 文件（codebase/coding-rules/contracts/routing-playbooks/domain/portal），每个事实只存在一处，跨文件通过 ID 引用，避免矛盾
  5. Evidence Index 静态分析：build-index.py 对 TypeScript/JavaScript/Go 做正则扫描，提取 enums/classes/interfaces/functions/constants/imports/switch-cases，构建 entities.json + edges.json（含跨文件 RESOLVED_IMPORT）+ inverted-index.json（BM25 检索），支持文件哈希增量构建
  6. Context Pack 精准锚定：context-pack.py 合并蒸馏产物和 Evidence Index，输出 query-plan.yaml（每个需求的匹配实体+置信度）和 context-pack.md（分层代码锚点：must/should/optional），支持团队模式多仓库索引加载
  7. 多层质量门控：quality-gate.py 三个子命令——distill（检查产物完整性/IR/layer impact/索引桥接）、final（5项加权评分：required_files 20% + context_pack_consumed 15% + code_anchor_coverage 25% + plan_actionability 25% + blocker_quality 15%）、reference（6文件完整性/索引/YAML可读性/证据声明）
  8. 反馈回流闭环：prd-distill 发现新知识/矛盾 → 生成回流建议 → reference 人工确认后更新 → 下次蒸馏更准
  9. Benchmark 回归：oracle 评分（需求召回 35分 + 代码锚点准确率 30分 + 阻塞项质量 20分 + 计划可执行性 15分 + 误报扣分 -10分）+ snapshot 回归防止退步
口语化表述
prd-tools 的核心不是让 AI 直接生成代码，而是把 PRD 转成有证据的结构化开发计划。可靠性靠多层：第一层是证据链，每个非显然事实必须有证据，搜不到也是证据，不确定就标 low；第二层是代码锚定——build-index.py 做静态分析构建代码实体索引，context-pack.py 把需求匹配到具体代码锚点，按优先级分层（must/should/optional）；第三层是多层质量门控——quality-gate.py 有三个子命令，final 子命令做 5 项加权评分，不是简单的通过/不通过；第四层是反馈回流，蒸馏发现的新知识和矛盾会回流到知识库。还有 benchmark 系统做回归——拿 oracle 用例跑完整蒸馏链路，需求召回 35 分 + 代码锚点准确率 30 分 + 阻塞项质量 20 分 + 计划可执行性 15 分，总分低了就说明知识库质量退步了。
加分点
讲清楚"证据链 + 质量门控 + 反馈回流"三层保证 → 体现工程可靠性思维
A5 你做过最有技术挑战的项目？ ▶
核心答案

两种讲法，根据追问方向选：

选法 A（选 marketing-bff）：

最有挑战的是 marketing-bff 的 Schema 驱动架构设计。23 种活动类型，每种有不同的字段组合和联动规则。难点是联动引擎——字段之间的显隐、禁用、选项联动关系非常复杂，比如选了"推送"活动类型，要联动显示推送模板字段，推送模板的 placeholder 又根据操作系统不同而不同。我们设计了 17 个联动文件，每个文件管理一种联动维度。字段注册中心统一管理所有字段的复用，同个字段在不同活动类型下可以有不同的行为。

选法 B（选 prd-tools）：

最有挑战的是 prd-tools，一套 PRD 工程工作流。核心挑战是怎么让 AI 可靠地分析 PRD 并产出工程可执行的计划。我的解法是"证据链 + 代码锚定"——每个非显然事实必须有证据，8 种证据类型，搜不到也是证据。设计了 reference v4.0，6 文件 SSOT 架构，每个事实只存在一处，跨文件 ID 引用避免矛盾。还做了 build-index.py 静态分析器，正则扫描 TypeScript/JavaScript/Go，提取实体、构建跨文件调用边和倒排索引，让每个需求都能精准锚定到代码位置。context-pack.py 再把蒸馏产物和索引合并，按优先级分层输出代码锚点。质量门控有 5 项加权评分，benchmark 系统做回归防止退步。还做了三层图谱融合——GitNexus 提供代码结构，Graphify 提供业务语义，prd-tools 做证据治理。反馈回流闭环让知识库越用越准。

加分点
两种都能讲，根据追问方向灵活切换
A6 Monorepo 的包依赖管理怎么做？ ▶
核心答案

营销平台 的依赖管理策略：

  1. Yarn workspace：根目录 package.json 声明 "workspaces": ["packages/*", "app/*"]，所有子包自动链接
  2. workspace 协议引用：子包间引用用 "@company/ui-components": "workspace:*"，本地开发直接 symlink
  3. yarn.lock 锁版本：严格锁定第三方依赖版本，遇到过 @yarnpkg/parsers 从 rc.31 升到 rc.48 导致不兼容 node16 的事故
  4. resolutions 强制统一：根目录 "resolutions": {"@types/react": "^18.0.0"} 确保所有子包用同一个 react 类型
  5. Lerna bootstrap：lerna bootstrap 处理嵌套依赖（Yarn workspace 覆盖不到的场景）
  6. CI 策略：Lerna npmClientArgs: ["--no-lockfile"] 避免 CI 环境锁文件冲突
口语化表述
依赖管理主要靠 Yarn workspace + yarn.lock。workspace 做子包间的 symlink，yarn.lock 锁第三方版本。踩过一个坑：yarn.lock 里锁的 @yarnpkg/parsers 从 rc.31 被升到 rc.48，而 rc.48 要求 node>=18，我们 node16 环境直接崩了。所以项目强制不删 yarn.lock，用 yarn 不用 npm。还用 resolutions 强制统一 @types/react 版本，避免多个版本导致类型冲突。
加分点
有真实的版本事故案例 → 说服力强
A7 联动引擎怎么设计？ ▶
核心答案

联动引擎横跨 BFF + 前端两层，核心是 injectDActions 声明式注入 + listener 发布订阅 + eval 表达式解析。

BFF 端 — 联动配置生成：

联动配置不是手写 d_actions JSON，而是通过 injectDActions 函数声明式注入：

// 模板 handler 中调用
injectDActions(schema, [
  {
    dependencies: ["basic.city", "basic.product_id"],
    callback: {
      fetchApi: "/api/linkage/fieldName",  // fetchApi 型：回调 BFF 重生成
      method: "post"
    }
  },
  {
    dependencies: ["basic.campaign_type"],
    callback: {
      state: { visible: "$deps[0]===32" }    // state 型：前端本地 eval
    }
  }
])

injectDActions 双重求值：

  1. BFF 端：safeEval 用当前值求值 → 设置初始 visible/disabled（SSR 初始状态正确，不需要等前端 eval）
  2. 序列化到 d_actions → 前端运行时 listener 触发时再次求值

16 个 linkageSchema 文件（config/linkage-templates/）分两类：

字段重渲染型（12 个）— 从 render 模板提取 handler，用新数据重新生成完整 Schema：

文件触发场景复用来源
campaignType.ts城市/产品变化 → 重渲染活动类型选项basic.ts
triggerType.ts活动类型变化 → 重渲染触发方式basic.ts
rewardCondition.ts触发方式变化 → 重渲染奖励条件rules/index.ts
rules.ts多字段变化 → 重渲染规则详情rules/index.ts
budgetAllocation / cycleType / effectMode / guaranteeType / deliveryRiderType / title / description / push 等类似

Placeholder 重置型（4 个）— 不重新渲染字段，只清空 value：

文件行为
titlePlaceholder.ts清空标题内容
descriptionPlaceholder.ts清空描述内容
pushPlaceholder.ts清空推送模板
os2pushPlaceholder.ts清空 iOS 推送模板

联动请求链路：

前端字段变化 → 读取 d_actions.fetchApi → POST BFF联动API
    ↓
Controller → 调后端 API 获取新 options
    ↓
从 basic.ts 提取 campaign_type handler → 用新数据重新生成 Schema
    ↓
返回单个字段 Schema → 前端替换原字段

前端端 — 联动执行：

字段A 变值 → onFormChange → listener.emit(formatName, value)
    ↓
FieldWrapper listener.on callback → parseCallback
    ↓
├── fetchApi 型 → POST BFF → 返回新 Schema → 替换字段
└── state 型 → eval 表达式 → 更新 visible/disabled
口语化表述
联动引擎横跨 BFF 和前端两层。BFF 端核心是 injectDActions——在模板 handler 里声明式注入联动配置，一次声明双重求值：BFF 用 safeEval 求初始状态保证 SSR 就是对的，同时序列化 d_actions 给前端。联动分 fetchApi 型和 state 型：fetchApi 型是字段变化后回调 BFF 重新生成 Schema 前端替换，state 型是前端本地 eval 表达式更新显隐。16 个联动文件按维度拆分，字段重渲染型从 render 模板提取 handler 复用保证结构一致，Placeholder 重置型只清空值。前端通过 listener 发布订阅执行联动，formatName 拍平路径做事件名。
加分点
讲清楚 injectDActions 双重求值 + fetchApi/state 两种联动类型的区别 + handler 复用机制 → 体现 BFF+前端全链路理解
A8 组件库的版本管理和发布流程？ ▶
核心答案

ibt-benefit-sdk 的发版流程：

  1. Changesets 管理版本：
    • npx changeset：开发者选择变更的包 + 填写变更描述
    • npx changeset version：自动 bump 版本号 + 生成 CHANGELOG
    • npx changeset publish：发布到 npm registry
  2. 双环境发布：
    • release:alpha：发布 alpha 版本到内部 registry，用于联调

B12 设计 Schema 驱动 BFF ▶
核心答案

以 marketing-bff 为实战基础，Schema 驱动 BFF 的核心是模板引擎驱动 Schema 生成 + injectDActions 双重求值联动 + 多场景数据适配。

后端 API (原始数据) → BFF 模板引擎 (Schema 生成) → 前端 (Schema 渲染)

1. Schema 生成层 — 模板引擎驱动

请求 → Controller → 调后端 API → 拿原始数据
    ↓
getRenderSchemaTemplate(step, ctx) → 返回 TemplateStuff
    ↓
Builder.parseOnce(template, apiData)
    ├── 遍历 template entries，每个 entry.handler(ctx) → 生成 Schema 节点
    ├── afterEach: mixinDefaultFields + mixinValue → 自动合并后端数据
    └── afterAll: 提取 content_schema 数组
    ↓
返回 { content_schema: [...], steps: [...], preview: [...] }

字段定义按模板分布式管理（basic.ts 管理 20+ 基础字段，message.ts 管理推送/标题，rules/details/ 下 25 个文件管理各活动类型规则），不存在集中注册中心。

2. 联动引擎 — injectDActions 双重求值

联动配置通过 injectDActions 声明式注入，一次声明双重求值：

  • BFF 端 safeEval：用当前值求值 → 初始 visible/disabled 就是对的（SSR）
  • 序列化 d_actions → 前端 listener 触发时再次求值

两种联动类型：

  • fetchApi 型：依赖变化 → 回调 BFF → BFF 从 render 模板提取 handler 重新生成 Schema → 前端替换
  • state 型：依赖变化 → 前端本地 eval 表达式 → 更新显隐/禁用

16 个 linkageSchema 文件（12 个字段重渲染型 + 4 个 Placeholder 重置型），从 render 模板提取 handler 复用保证一致性。

3. 多场景数据适配

同一字段在不同请求场景下数据路径不同（创建/编辑/联动/批量），getAnyDataGadget.ts 按优先级尝试所有路径，injectShadowValue 用双下划线约定注入用户最新选择覆盖后端旧值。

4. 扩展性设计

  • 新增活动类型：加枚举值 + 在 rules/details/ 加规则模板 + 配联动，前端零改动
  • 新增字段：在对应模板文件加 template entry，联动模板自动可复用
  • 新增联动：加 linkageSchema 文件，从 render 模板提取 handler

5. 边界划分

  • BFF 负责：Schema 结构、字段类型、联动规则声明、i18n key、初始状态求值
  • BFF 不负责：options 内容（后端返回）、校验逻辑（后端实现）、国际化内容（只传 key）、联动表达式执行（前端 eval）
口语化表述
Schema 驱动 BFF 的核心是模板引擎驱动生成，不是手写 JSON。每个步骤有独立模板，模板定义字段的取值路径和生成逻辑，Builder 编译出 content_schema。最精巧的是 injectDActions 双重求值——一次声明联动配置，BFF 端 safeEval 求初始状态（SSR 就是对的），序列化为 d_actions 给前端运行时联动。联动分 fetchApi 型回调 BFF 重生成和 state 型前端本地 eval。扩展性很好：新增活动类型就加模板文件和联动文件，联动模板从 render 模板提取 handler 复用，前端零改动。
加分点

C16 React 18 并发模式 + Fiber 架构 + Scheduler ▶
核心答案

Fiber 架构：

  • React 16 引入，把递归的 vdom 树改为链表结构的 Fiber 树
  • 每个 Fiber 节点有 child、sibling、return 三个指针
  • 可中断的增量渲染：每次处理一个 Fiber 节点，处理完检查是否需要让出主线程

Scheduler（调度器）：

  • 基于 MessageChannel（不是 setTimeout，因为 setTimeout 最小 4ms）
  • 优先级调度：Immediate / UserBlocking / Normal / Low / Idle
  • shouldYield() 判断是否需要让出主线程

React 18 并发特性：

  • createRoot 开启并发模式
  • useTransition：标记低优先级更新，不阻塞用户输入
  • useDeferredValue：延迟更新值，让紧急更新先处理
  • Automatic Batching：所有状态更新自动批处理（不只是事件处理函数内）
口语化表述
Fiber 是 React 16 引入的链表结构替代递归 vdom。每个节点有 child/sibling/return 指针，可以中断和恢复。调度器用 MessageChannel 做优先级调度，比 setTimeout 精度高。React 18 的并发模式在此基础上加了 useTransition 和 useDeferredValue，让非紧急更新不阻塞用户输入。还有 Automatic Batching，所有状态更新自动批处理，不再限于事件处理函数内。

提到 MessageChannel 而不是 setTimeout → 体现源码级理解
C17 useEffect vs useLayoutEffect ▶
核心答案
维度useEffectuseLayoutEffect
执行时机DOM 更新后异步执行DOM 更新后同步执行
阻塞渲染不阻塞阻塞
用途数据请求、事件监听、日志读取/修改 DOM 布局
SSR正常会警告（用 useIsomorphicLayoutEffect）

执行顺序：

  1. React 更新 DOM
  2. useLayoutEffect 回调同步执行（浏览器还没绘制）
  3. 浏览器绘制
  4. useEffect 回调异步执行

为什么不能在条件里用 Hook：React 靠调用顺序匹配 Hook 和 Fiber 节点的 state。如果条件里用，顺序变了，state 对不上。

口语化表述
useEffect 在浏览器绘制后异步执行，不阻塞渲染，适合做数据请求。useLayoutEffect 在 DOM 更新后同步执行但绘制前，适合读 DOM 布局或修改样式。如果 useEffect 里改了 DOM 样式，用户会看到闪烁——先渲染旧样式再变新样式，用 useLayoutEffect 就不会有这个问题。Hook 不能在条件里用是因为 React 靠调用顺序匹配 state，顺序变了就乱了。
C18 浏览器渲染管线 ▶
核心答案
HTML → DOM树
CSS → CSSOM树    } → Render树 → Layout(布局) → Paint(绘制) → Composite(合成)

关键步骤：

  1. Parse：HTML 解析成 DOM，CSS 解析成 CSSOM
  2. Render Tree：DOM + CSSOM 合并，去掉不可见元素（display:none）
  3. Layout（Reflow）：计算每个节点的位置和大小
  4. Paint（Repaint）：填充像素（颜色、文字、图片）
  5. Composite：图层合成，GPU 加速

触发条件：

  • Reflow：改几何属性（width/height/margin/padding/position）
  • Repaint：改视觉属性（color/background/visibility）
  • Composite only：transform/opacity（最佳性能）

优化：

  • transform 替代 top/left 做动画（只触发 composite）
  • will-change 提升图层
  • 批量 DOM 修改用 requestAnimationFrame
  • 读写分离：先读布局信息再写，避免强制同步布局
口语化表述
渲染管线分五步：解析、Render 树、Layout、Paint、Composite。Reflow 是布局计算，改几何属性触发；Repaint 是绘制，改视觉属性触发。性能最好的是只触发 Composite，transform 和 opacity 就是。所以动画用 transform 别用 top/left。优化核心是减少 Reflow，批量改 DOM、读写分离。
C19 HTTP 缓存策略 ▶
核心答案

强缓存（不发请求）：

  • Cache-Control: max-age=3600（秒数）
  • Cache-Control: no-cache（跳过强缓存，走协商）
  • Cache-Control: no-store（完全不缓存）
  • Expires（HTTP/1.0，已被 Cache-Control 取代）

协商缓存（发请求，304 则用缓存）：

  • ETag / If-None-Match：内容哈希，精确但计算成本高
  • Last-Modified / If-Modified-Since：最后修改时间，精度秒级

优先级：Cache-Control > Expires > ETag > Last-Modified

实战配置：

  • HTML：Cache-Control: no-cache（每次协商）
  • JS/CSS（带 hash）：Cache-Control: max-age=31536000（强缓存一年）
  • API：Cache-Control: no-store 或短时间 max-age
口语化表述
缓存分强缓存和协商缓存。强缓存不发请求，Cache-Control 的 max-age 控制有效期。协商缓存发请求，服务端返回 304 就用缓存。ETag 比内容哈希，Last-Modified 比时间，ETag 更精确但计算成本高。实战配置一般是 HTML 不缓存（每次协商），带 hash 的静态资源强缓存一年，API 不缓存。
C20 TypeScript 高级类型 ▶
核心答案

1. infer（类型推断）

type ReturnType<T> = T extends (...args: any[]) => infer R ? R : never;
type Unpacked<T> = T extends Array<infer U> ? U : T;

2. 条件类型

type NonNullable<T> = T extends null | undefined ? never : T;

type CSSProperty = `${string}-${string}`;     // "margin-top" ...

type Partial<T> = { [P in keyof T]?: T[P] };
type Pick<T, K extends keyof T> = { [P in K]: T[P] };

5. 实战用法（marketing-bff 中）：

  • 字段类型映射：FormDComponentType 枚举 → 不同组件的 props 类型
  • 联动类型：LinkageType = 'visible' | 'disabled' | 'api'
口语化表述
infer 用于在条件类型里推断子类型，比如 ReturnType 推断函数返回值。条件类型类似三元表达式，对类型做判断。模板字面量类型是 TS 4.1 引入的，可以在类型层面做字符串拼接，比如自动生成事件处理器类型。实战中最常用的是映射类型——Pick、Omit、Partial 这些内置工具类型的原理就是映射类型。

D22 PRD 到代码最难的一步？ ▶
核心答案

最难的是变更分类——判断 PRD 里的每个功能点到底是 ADD（新增）、MODIFY（修改已有）、DELETE（删除）还是 NO_CHANGE。

为什么难：

  1. 需要理解项目现状：不知道现在有什么字段，就没法判断是新增还是修改
  2. PRD 描述模糊：运营写"增加奖励配置"，可能是新增字段，也可能是修改已有字段的选项
  3. 跨模块影响：改一个枚举值可能影响联动规则、i18n、预览配置

prd-tools 的解决方案：

  1. Reference 路由匹配：04-routing-playbooks.yaml 的 PRD 路由信号 + 字段映射，快速定位到能力面
  2. Evidence Index 精准搜索：build-index.py 构建的倒排索引支持 BM25 检索，context-pack.py 合并蒸馏产物和索引，输出 query-plan.yaml（每个需求的匹配实体+置信度）和分层代码锚点
  3. 代码锚定：每个变更分类必须验证源码，evidence.kind 标注来源
  4. 负向搜索：搜不到也是证据，用 negative_code_search 记录 query 和范围
  5. 置信度分级：high/medium/low，不确定的标 low，强制进入 questions.md
  6. Contract Delta：多层需求必须有契约差异分析，alignment_status 标注对齐状态
  7. Report Review Gate：第 7 步硬停止，用户确认报告后才能生成 plan，避免方向错误
口语化表述
最难的是变更分类——判断 PRD 里每个功能点是 ADD/MODIFY/DELETE。运营写"增加奖励配置"，你不知道是新增字段还是修改已有字段选项。prd-tools 的解法是分层：先用 reference 的路由信号快速定位到能力面，然后 build-index.py 的倒排索引做 BM25 检索找到相关代码实体，context-pack.py 合并索引和蒸馏产物，按 must/should/optional 分层输出代码锚点。每个变更分类必须验证源码，搜不到就用 negative_code_search 记录。低置信度的强制进入 questions.md 让人确认。还有 Report Review Gate 硬停止——报告确认前不生成计划，避免方向错了白费功夫。

D23 能力面适配器怎么设计？ ▶
核心答案

能力面适配器解决了"不同项目目录结构完全不同，但研发协作中的能力面相对稳定"的问题。

设计分三层：

1. 统一能力面定义：前端 9 个 / BFF 10 个 / 后端 10 个能力面，每个能力面描述一种关注维度：

D24 知识过期怎么办？ ▶
核心答案

prd-tools 用三层机制应对知识过期：

1. 衰减监控（被动）：

  • 每个文件有 last_verified 和 verify_cadence（默认 14 天）
  • 健康检查（B2 模式）对比 reference 和当前源码，发现矛盾
  • 质量门控（C 模式）检查 enum/field/contract 是否与源码冲突

    ↓

    ↓
build-reference 反馈回流模式 → 人工确认
    ↓
更新 reference → 下次蒸馏更准

3. 质量门控（守门）：

  • 致命项：enum/field/contract 与源码冲突 → 不通过
  • 警告项：last_verified 过期 → 需要更新
  • 样例回归：用 golden sample 反推 PRD → IR → Layer Impact → Contract Delta 是否走通
口语化表述

D25 图谱和知识库的关系是什么？ ▶
核心答案

prd-tools 采用了三层架构：

Graphify (业务维度)          GitNexus (代码维度)          prd-tools (治理维度)
"为什么这样设计"              "代码怎么连接"               "怎么从 PRD 到代码"
PRD/技术方案/截图/历史文档     代码仓库                     编排 + 证据治理 + 质量门控
        │                          │                           │
        └──────────────────────────┼───────────────────────────┘
                                   ▼
                         _reference/ 企业级可治理知识库

核心原则：图谱是原始发现层，reference 是精选后的企业知识库。Raw Graph ≠ Reference。

两个关键设计：

1. 两套独立证据追踪：

  • evidence: ["EV-xxx"] — 可审计证据（源码/文档/人工确认），关键条目必须至少有 EV
  • graph_evidence_refs: ["GEV-xxx"] — 图谱溯源，不能替代 EV

2. 置信度映射：

  • GitNexus AST 提取 → high（不需要源码确认）
  • Graphify EXTRACTED + source locator → high
  • Graphify INFERRED ≥ 0.8 → medium（需源码确认）
  • Graphify INFERRED < 0.8 → low（必须确认）
  • Graphify AMBIGUOUS → low（需人工确认）

Provider 可替换：不绑定 GitNexus/Graphify，未来可换 Neo4j/Sourcegraph/CodeQL。

口语化表述
图谱和知识库的关系是"原始发现层"和"精选企业知识库"的关系。图谱帮你快速发现知识——GitNexus 告诉你代码怎么连接，Graphify 告诉你为什么这样设计——但图谱结论不能直接用，需要精选、审计、确认后才能进入 reference。所以每个条目有两套独立证据：一套是可审计证据（源码/文档确认），一套是图谱溯源（从哪发现来的）。图谱不可用时完全回退到源码扫描，核心流程不受影响。另外 Provider 是可替换的，不绑定具体工具。团队模式下还有 T 模式聚合成员仓库 reference、T2 模式继承团队知识，跨仓库的公共知识只维护一份。build-index.py 和 context-pack.py 都支持团队模式的多仓库索引加载。
