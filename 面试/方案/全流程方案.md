# 全流程面试方案 — 大厂高级前端 / AI 工程向

> **这是什么**：面试备战总纲 — 策略、工具架构、冲刺节奏
> **什么时候看**：Day 5（5/24）搭 AI 训练环境时看，之后按需查阅
> **核心原则**：面试前用 AI 把准备效率拉满；面试中只用合规工具；面试后立刻复盘回流

---

## 0. 一句话战略

你不是去面"普通高级前端"，而是去面：

**"做过 Schema 驱动中台架构、懂 BFF 编排、能把 AI 工程化落到流程治理里的高级前端 / AI 工程型候选人"。**

所以全流程工具的目标不是帮你背答案，而是让你在 3 个场景里稳定发挥：

1. **项目深挖**：任何问题都能回到 `Schema 生成 -> Schema 渲染 -> 组件库同构 -> prd-tools 工程化` 这条主线。
2. **基础与算法**：不追求题海，只覆盖面试官最容易验证你硬功底的高频点。
3. **AI 工程差异化**：把 prd-tools 讲成你的职级杠杆，而不是"玩了个 AI prompt"。

## 1. 重要边界：把原来的"实时辅助"改造成"合规作战系统"

你现有的 `面试方案/AI辅助提示词.md` 很适合做"训练器"，但不适合原样作为隐蔽实时辅助。

原因：

- CoderPad、HackerRank、CodeSignal 等平台已经在 2025-2026 年开始支持"AI-assisted interviews"，但核心是**平台内启用、交互可记录、面试官可观察**。
- HackerRank 的候选人说明里明确提到：AI Assistant 是否可用、模式是什么，由招聘公司设置。
- CoderPad 的公开说明也强调：允许 AI 的面试需要透明使用；直接读脚本、把 AI 当提词器、无法解释答案都会负面影响评价。
- 一些公司仍然禁止未授权 AI / 提词器，使用隐蔽工具有被取消资格的风险。

因此建议改成两套模式：

| 模式 | 使用时机 | 工具行为 |
|---|---|---|
| 训练模式 | 面试前 / mock / 复盘 | AI 扮演面试官，连环追问，打分，生成短答案卡 |
| 透明辅助模式 | 面试官明确允许 AI / 平台内置 AI | 先说明"我会用 AI 做检索/校验"，所有 prompt 和输出可解释 |

面试中如果对方没说允许 AI：只用你自己的脑子 + 纸质速查表 + 已背熟的话术。你的文档可以在面试前看，但不要做实时隐蔽代答。

本质是把你已有文档变成 5 个"可调用能力"：

1. **项目素材检索**：从 `题库/速查手册.md`、`题库/题库详解.md`、`简历/v2.md` 中抽取真实经历。
2. **高压追问模拟**：按目标公司风格追问，不放水。
3. **短答案卡生成**：所有答案压成"核心句 + 3 句话 + 追问点"。
4. **算法最小集训练**：10-20 道题覆盖 Map / 双指针 / BFS / DFS / 滑窗 / DP。
5. **复盘回流**：每场面试后把新问题变成题库、速查卡和简历措辞修正。

## 5. 你的"强主线"讲法

### 5.1 30 秒开场，建议终版

```text
我在滴滴国际化主要做营销中台前端架构，核心主线是 Schema 驱动。
BFF 层用模板引擎生成 Schema，不是手写 JSON；联动通过 injectDActions 一次声明、两端求值，BFF 先算初始状态，前端再做运行时联动。
前端侧用 formatName 把嵌套路径拍平，通过 Listener 发布订阅和 35+ 组件注册表动态渲染。
这套模式后来也复用到 benefit SDK，400+ 权益通过 Register 模式和 Schema 同构渲染接入。
在这个基础上，我又做了 prd-tools，把 PRD 转成有证据、可执行、可回流的结构化开发计划，所以我的优势不是单点页面开发，而是前端架构、BFF 编排和 AI 工程化结合。
```

### 5.2 所有回答都往这 4 个锚点落

| 锚点 | 你要证明什么 | 典型问题 |
|---|---|---|
| Schema 生成 | 你懂 BFF 编排和模板化 | 为什么不用传统表单？为什么前端零代码？ |
| Schema 渲染 | 你懂复杂前端状态和组件架构 | 联动怎么执行？性能怎么保证？ |
| benefit SDK | 你懂组件库工程化和复用边界 | 400+权益怎么扩展？怎么发版？ |
| prd-tools | 你懂 AI 工程化和质量治理 | AI 怎么可靠？知识过期怎么办？ |

## 8. 高频考点补齐：2026 必补

你的现有文档已经覆盖 React/Fiber/TS/缓存/Monorepo，但下周建议额外补 5 个"AI 前端"题。

### 8.1 SSE 流式输出

连接层：EventSource / fetch ReadableStream
协议层：text/event-stream、event/id/data、心跳、Last-Event-ID
解析层：chunk buffer、增量 markdown、代码块未闭合保护
渲染层：requestAnimationFrame 节流、虚拟列表、useDeferredValue
可靠性：断线重连、去重、超时取消、错误态

项目桥接：这个和我做 Schema 联动类似，核心都是"事件流进入前端后，如何分层解析、状态化、再稳定渲染"。区别是 SSE 是服务端连续推送，Schema 联动是字段变化触发局部重算。

### 8.2 AI Agent 前端架构

会话层：conversation / message / run / step
工具层：tool call 展示、确认、回滚、权限控制
状态层：streaming / waiting_tool / executing / done / failed
可观测：每一步 trace、输入输出、耗时、错误
安全：敏感操作二次确认，工具白名单，不让模型直接执行高危动作

项目桥接：我会参考 prd-tools 的证据链思路：Agent 每个结论都要能追溯，每个工具调用都要有输入输出记录，不能只看最终答案。

### 8.3 React 并发渲染

关键点：

- `useTransition` 用来把某些状态更新标成非阻塞更新。
- `useDeferredValue` 让某个值的消费方滞后更新，避免慢列表/图表阻塞输入。
- 注意：它们不是让代码变快，而是调度优先级更合理。

### 8.4 AI 生成代码怎么验收

不看"像不像"，看四件事：

1. 需求证据是否完整
2. 影响面是否覆盖
3. 契约是否一致
4. 测试和回归是否能证明

### 8.5 前端会被 AI 取代吗

重复性页面和简单 CRUD 会被大幅自动化，但高级前端的价值会往系统边界、交互复杂度、质量治理和 AI 协作迁移。我的 prd-tools 其实就是这个方向：不是让 AI 替我写完，而是把需求、代码、证据和质量门控组织起来。
